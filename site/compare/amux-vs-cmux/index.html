amux vs cmux

One is a native macOS terminal app. The other is a web-accessible orchestration platform. Both run parallel AI coding agents. Here's exactly when to pick which.

TL;DR

amux and cmux both run parallel AI coding agents, but they are fundamentally different tools built on different architectures for different workflows.

  • cmux is a native macOS terminal application built with Swift and libghostty. It replaces your terminal entirely — GPU-accelerated rendering, vertical tabs, embedded browser, notification rings, and native macOS integration. Desktop-first, visually polished, macOS-only.
  • amux is an orchestration platform built as a single Python file. It runs a local HTTPS server with a web dashboard, REST API, self-healing watchdog, kanban board, and mobile PWA. Platform-agnostic, API-first, accessible from any device.

Pick cmux if you work on macOS, want a beautiful native terminal experience, and prefer managing agents visually at your desk.
Pick amux if you need cross-platform support, remote access from your phone, self-healing overnight runs, agent-to-agent orchestration, or a REST API for custom automation.

What Each Tool Is

cmux

cmux is an open-source native macOS terminal application from Manaflow (YC S24) that manages parallel AI coding sessions. It uses libghostty — the GPU-accelerated rendering library from Mitchell Hashimoto's Ghostty terminal — to draw terminal output, and embeds a WebKit browser for agent-driven web interaction.

Key features:

  • GPU-accelerated terminal rendering via libghostty with Ghostty config compatibility
  • Vertical tabs sidebar showing git branch, PR status, working directory, and listening ports per workspace
  • Notification rings on pane borders when an agent needs attention, with macOS desktop notifications
  • Embedded scriptable WebKit browser that agents can snapshot, click, fill forms, and evaluate JS against
  • Split panes (horizontal and vertical) within each workspace tab
  • Native Claude Code Teams integration via cmux claude-teams
  • Session layout restore on app restart (directories, scrollback, browser URLs — not live processes)
  • SSH support with SCP drag-and-drop and localhost routing

cmux is not a wrapper around tmux. It is the terminal — a standalone macOS app built with Swift and AppKit that replaces iTerm2, Terminal.app, or whatever terminal you currently use.

amux

amux is an open-source agent orchestration platform built around amux-server.py — a single Python file that runs a local HTTPS server, manages tmux sessions, and serves a web dashboard. It treats parallel agents as a fleet that needs to be monitored, recovered, coordinated, and accessed from anywhere.

Key features:

  • Web dashboard at https://localhost:8822 with live session status, token spend, and a send bar
  • Mobile PWA for monitoring and controlling the fleet from your phone
  • Self-healing watchdog: auto-restart crashed agents, auto-compact context overflow, detect stuck sessions
  • SQLite-backed kanban task board with atomic claiming so agents don't duplicate work
  • Full REST API for sessions, board, notes, CRM, email, browser automation, and cron
  • Inter-session messaging and agent-to-agent orchestration via the shared board
  • Per-session token and cost tracking with cumulative spend visibility
  • Browser automation via Playwright with saved auth profiles
  • Notes, CRM, email integration, and calendar primitives

amux runs alongside your existing terminal. It manages tmux sessions in the background and serves a dashboard you access from any browser. The server persists across reboots and auto-restarts when the file is modified.

Architecture Differences

amux and cmux are built on fundamentally different architectures. This is not a matter of features bolted onto similar foundations — the foundations themselves are different.

cmux architecture

┌──────────────────────────────────────────────────────────┐
│  cmux.app  (Swift + AppKit, native macOS)                │
│                                                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌─────────┐ │
│  │libghostty│  │  WebKit  │  │ Vertical │  │ Notifi- │ │
│  │ terminal │  │ browser  │  │   tabs   │  │ cations │ │
│  └──────────┘  └──────────┘  └──────────┘  └─────────┘ │
│                                                          │
│  ┌─────────────────────────────────────────────────────┐ │
│  │  Unix socket API  (/tmp/cmux.sock)                  │ │
│  └─────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────┘
                          │
                          ▼
            ┌──────────────────────┐
            │  Your AI agents run  │
            │  inside cmux panes   │
            └──────────────────────┘

cmux is the process that renders the terminal. Your agents run inside cmux the same way they'd run inside iTerm2 or Terminal.app. There is no separate server, no daemon, no tmux layer — cmux is the container.

amux architecture

┌──────────────────────────────────────────────────────────┐
│           amux-server.py (long-running)                   │
│                                                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌─────────┐ │
│  │ Web UI   │  │ REST API │  │ Watchdog │  │  Board  │ │
│  └──────────┘  └──────────┘  └──────────┘  └─────────┘ │
│                                                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌─────────┐ │
│  │  Notes   │  │   CRM    │  │  Tokens  │  │  Cron   │ │
│  └──────────┘  └──────────┘  └──────────┘  └─────────┘ │
└──────────────────────────────────────────────────────────┘
                          │
                          ▼
        ┌────────┐  ┌────────┐  ┌────────┐
        │ tmux 1 │  │ tmux 2 │  │ tmux N │
        │ Claude │  │ Claude │  │ Claude │
        └────────┘  └────────┘  └────────┘

amux is a server that manages tmux sessions. It does not render the terminal — it delegates that to tmux and whatever terminal you already use. The server is the source of truth for sessions, tasks, costs, and notes, and it exposes everything over HTTPS.

What this means in practice

cmux's native architecture gives it a visual experience that no server-based tool can match. GPU-accelerated rendering, notification rings directly on pane borders, an embedded browser you can see next to your terminal output — these require being the window manager.

amux's server architecture gives it platform independence and remote access that no desktop app can match. The dashboard works in any browser, the API is callable from anywhere on the network, and tmux sessions survive app crashes, machine reboots, and SSH disconnects. You can close your laptop, walk away, and the fleet keeps running.

Platform Support

This is the most important difference for many developers.

cmux runs on macOS only. It is a Swift + AppKit application that requires macOS. There are unofficial community forks (wmux for Windows, cmux-linux) but they are not maintained by the cmux team and lag behind the main release.

amux runs anywhere Python and tmux run. macOS, Linux (Ubuntu, Debian, Arch, etc.), cloud VMs (GCP, AWS, any VPS), WSL on Windows. The single-file architecture means scp amux-server.py your-server: is a deployment.

If you develop on Linux, work on remote servers, or want to run agents on a cloud VM, amux is your only option between these two tools. If you're on macOS and plan to stay on macOS, both work.

Workflow Comparison

Same goal: run 5 parallel agents, each working on a different module's test suite.

The cmux way

# Open cmux (it's your terminal now)
# Cmd+T creates a new workspace tab for each task
# In each tab, run Claude Code:
claude "Add auth tests, target 90% coverage"

# cmux shows:
#   - Vertical sidebar with all 5 tabs
#   - Git branch and PR status per tab
#   - Blue notification rings when agents need attention
#   - Jump to next unread: Cmd+Shift+U

# To check a running agent, click its tab
# Split pane horizontally: Cmd+D
# Open embedded browser next to terminal: Cmd+Shift+B

Strengths: Beautiful UI, instant visual feedback, notification rings catch your attention without context-switching, embedded browser lets agents interact with web UIs natively.
Limitations: You must be at your Mac to see what's happening. No way to check from your phone or another machine. If an agent crashes, you notice by looking at the tab — nothing auto-recovers it.

The amux way

# Register sessions
for module in auth api payment notification search; do
  amux register $module --dir ~/Dev/project --yolo
done

# Queue tasks on the board
amux board add "Add auth tests, target 90% coverage" --session auth
amux board add "Add API tests, target 85% coverage" --session api
amux board add "Add payment tests, target 90% coverage" --session payment
amux board add "Add notification tests, target 80% coverage" --session notification
amux board add "Add search tests, target 85% coverage" --session search

# Start all and open dashboard
amux start --all
amux serve  # → https://localhost:8822

# Monitor from any device: laptop, phone, tablet
# Watchdog auto-restarts crashes, auto-compacts context overflows
# Check progress from bed at midnight via PWA on your phone

Strengths: Tasks are persisted on the board, accessible from any device, self-healing handles failures overnight, agents can delegate to each other via the shared board.
Limitations: More setup ceremony. No GPU-rendered terminal UI — the dashboard shows session output, but it's a web view, not a native terminal.

The takeaway

cmux optimizes for the at-your-desk interactive experience. It's the best way to visually manage parallel agents if you're sitting in front of your Mac. amux optimizes for the platform experience. It's the best way to run agents as a fleet you can monitor and control from anywhere, whether you're at your desk or not.

Dashboards & Remote Access

cmux has no dashboard and no remote access. It is a desktop application. To see your agents, you look at the app on your Mac. To check agents remotely, you SSH into your machine and... you can't use cmux over SSH, because it's a GUI app. You could use the Unix socket API to query status programmatically, but there's no web interface.

amux has a full web dashboard accessible from any browser on any device. Open https://localhost:8822 (or your machine's IP/Tailscale address) and see:

  • All sessions with live status (running, idle, crashed, stuck)
  • Per-session token spend and cumulative cost
  • Last N lines of output from every agent, scrollable
  • A send bar to type messages directly into any agent
  • The kanban board with drag-and-drop task management

The dashboard works as a PWA on your phone. Install it to your home screen and it behaves like a native app with safe-area insets and touch-optimized controls. With Tailscale, you can access it from anywhere on the internet without exposing ports.

If you ever want to start agents before bed and check on them from your phone in the morning, amux is the only option.

Browser Automation

Both tools have browser capabilities, but they work very differently.

cmux: embedded visible browser

cmux embeds a WebKit browser as a split pane next to your terminal. Agents can see the rendered page, snapshot the accessibility tree, click elements, fill forms, and evaluate JavaScript — all while you watch. The browser uses a "Snapshot + Refs" strategy for token efficiency, giving agents a concise reference list instead of the full DOM.

This is ideal for interactive use where you want to see what the agent is doing in the browser in real time. The downside: the browser is tied to the cmux desktop app and cannot be accessed remotely or programmatically from external tools.

amux: headless Playwright via REST API

amux drives a Playwright browser via its REST API. Start a browser with a saved auth profile, navigate, screenshot, click, type, evaluate JS — all via HTTP calls. The browser runs headless (or headed if you want), and screenshots are viewable from the dashboard.

This is ideal for automated workflows, CI/CD, and unattended operation. Any agent can drive the browser via the API, and multiple sessions can share the same browser instance. The trade-off: you don't see the browser visually alongside your terminal the way cmux shows it.

Unattended Operation & Self-Healing

Running agents overnight or while you're away from your desk is where these two tools diverge most sharply.

cmux: no recovery

cmux does not monitor agent health or recover from failures. If an agent crashes, the pane shows the error and the agent is dead until you restart it. If the agent hits its context limit, cmux doesn't notice. If cmux itself is quit and reopened, it restores the layout (directories, scrollback, browser URLs) but does not restore live terminal processes — your agents are gone.

For interactive use at your desk, this is fine — you see the crash immediately and restart it. For overnight runs, a crash at 2am means 6 hours of dead time.

amux: self-healing watchdog

amux's watchdog runs continuously and handles common failure modes:

  • Crash: Process dies → auto-restart, replay last user message
  • Context overflow: Detected via output pattern → trigger /compact and continue
  • Stuck: No new output for N minutes → send a nudge or restart
  • Permission prompts: Auto-respond in YOLO mode
  • Rate limits: Back off and retry transparently

Because amux uses tmux underneath, sessions persist across everything: the amux server restarting, your terminal closing, SSH disconnects, even machine reboots (if tmux is configured to persist). The watchdog picks sessions back up automatically.

If you run agents for hours at a time, the self-healing watchdog is the single most important feature differentiator between these two tools.

API & Extensibility

Both tools offer programmatic control, but the scope and accessibility differ significantly.

cmux: Unix socket API

cmux exposes a JSON-over-Unix-socket API at /tmp/cmux.sock. You can create workspaces, split panes, send keystrokes, open URLs, set sidebar metadata (status pills, progress bars), and manage notifications. The cmux CLI wraps this socket API.

The socket API is local only. External tools, remote scripts, cron jobs, and mobile apps cannot reach it over the network. It is designed for same-machine integration.

amux: full HTTPS REST API

amux exposes a comprehensive REST API over HTTPS covering:

  • Session management (list, create, start, stop, send messages, peek at output)
  • Kanban board (CRUD, atomic task claiming, status transitions)
  • Notes (documents, write-ups, reference material)
  • CRM (contacts, interactions, follow-ups)
  • Browser automation (start, navigate, screenshot, click, type, eval)
  • Email (send and read via Mail.app)
  • Calendar (iCal export with S3 sync)
  • Cron and scheduled triggers

The API is network-accessible, so cron jobs, CI/CD pipelines, mobile apps, other agents, and remote scripts can all interact with amux. This is what enables the agent-to-agent orchestration pattern where one agent creates board tasks for another.

Agent-to-Agent Orchestration

cmux agents are independent. Each pane runs its own agent in isolation. There is no mechanism for one agent to discover, communicate with, or delegate work to another agent. Coordination happens through you, the developer, looking at the tabs and deciding what to do.

amux agents can orchestrate each other. Every agent has access to the REST API and can:

  • Discover other sessions via GET /api/sessions
  • Send messages to other agents via POST /api/sessions/{name}/send
  • Create tasks on the shared board for other agents to claim
  • Atomically claim tasks so no two agents work on the same thing
  • Read notes and CRM data for shared context

This enables workflows where a lead agent decomposes a task, creates board items for sub-agents, and the sub-agents claim and complete them independently. The shared board is the coordination mechanism — no custom integration code required.

Which Should You Choose?

Choose cmux if...

  • You work exclusively on macOS and don't need Linux or cloud VM support
  • You want the best visual terminal experience — GPU rendering, notification rings, embedded browser
  • You work interactively at your desk and don't need remote monitoring
  • You prefer a standalone desktop app over a server + dashboard
  • You want native Claude Code Teams integration with cmux claude-teams
  • You value visual polish and native macOS integration (drag-and-drop SSH, Ghostty themes)
  • Your sessions are interactive and you'll be there to handle failures yourself

Choose amux if...

  • You need cross-platform support — Linux servers, cloud VMs, or WSL
  • You want to run agents unattended — overnight, while in meetings, while away from your desk
  • You need a web dashboard you can check from any device, including your phone
  • You want self-healing recovery from crashes, context overflows, and stuck sessions
  • You need agent-to-agent orchestration — shared board, delegation, task claiming
  • You need a REST API for custom automation, cron jobs, or CI/CD integration
  • You want token spend visibility per session
  • You need a kanban board, notes, CRM, or email integration alongside your agents
  • You're running 10+ agents and need fleet-level observability
  • You want a single-file deployment you can scp to any server

The honest summary

cmux and amux are not really competing for the same user. They solve the same high-level problem — "I want to run multiple AI coding agents" — but their answers reflect very different philosophies.

cmux says: "The terminal is the interface. Make it beautiful, fast, and integrated. If you're running agents, you should be looking at them."

amux says: "The platform is the interface. Make it accessible from anywhere, resilient to failures, and programmable. If you're running agents, you should be able to forget about them and check in when it's convenient."

If your workflow is "sit at my Mac and manage agents visually," cmux is genuinely excellent at that. If your workflow is "start agents, walk away, check from my phone, and trust that crashes are handled," amux is built for exactly that.

Frequently Asked Questions

What is the main difference between amux and cmux?

cmux is a native macOS terminal application that replaces your existing terminal — GPU-accelerated rendering, embedded browser, visual notifications. amux is a Python server with a web dashboard that manages tmux sessions and is accessible from any device. cmux is a desktop app you look at. amux is a platform you access from anywhere.

Does cmux work on Linux?

No. cmux is macOS-only (Swift + AppKit). Unofficial community forks exist for Linux and Windows but are not maintained by the cmux team. amux runs anywhere Python and tmux run — macOS, Linux, cloud VMs, WSL.

Can I access cmux from my phone?

Not currently. cmux is a desktop application with no web interface. An iOS companion app is on the Founder's Edition roadmap. amux has a full web dashboard that works as a mobile PWA — install it on your phone home screen and monitor agents from anywhere.

Does cmux have a REST API?

cmux has a Unix socket API at /tmp/cmux.sock for local programmatic control, but no REST API. External tools, cron jobs, and remote scripts cannot talk to cmux over the network. amux exposes a full HTTPS REST API for everything.

Which supports more AI coding agents?

Both support any terminal-based AI agent: Claude Code, Codex CLI, Gemini CLI, Aider, Goose, and more. cmux provides first-class support via its embedded browser's agent-aware features. amux's self-healing watchdog and token tracking are Claude Code-specific, though other agents run fine as regular sessions.

Is cmux free?

cmux's core is GPL-3.0 open source and free. A paid Founder's Edition adds priority support, early access to the cmux AI context engine, iOS app sync, cloud VMs, and voice mode. amux is MIT-licensed and fully free with no paid tier.

Which is better for overnight agent runs?

amux, by a wide margin. Its self-healing watchdog auto-restarts crashed agents, auto-compacts context overflows, detects stuck sessions, and handles rate limits. cmux has no recovery system — if an agent crashes at 2am, it stays dead until you manually restart it.

Can I use amux and cmux together?

Not in a meaningful way. cmux is a standalone terminal app that manages its own panes. amux manages tmux sessions. They operate independently. Most developers pick one based on whether they prioritize the desktop experience (cmux) or the platform experience (amux).

Which has more GitHub stars?

As of April 2026, cmux has approximately 15,800 stars. cmux launched on Hacker News in February 2026 and reached #2, which accelerated its growth. amux has a smaller but growing community. Stars measure awareness, not fitness for your use case — evaluate based on your platform, workflow, and feature requirements.

Are there other parallel AI agent tools I should consider?

Yes. Look at dmux (worktree-focused CLI), Claude Code Agent Teams (Anthropic's built-in feature), OpenAI Codex (cloud-based), and the broader comparison index.
