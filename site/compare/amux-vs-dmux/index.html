amux vs dmux

Both run parallel AI coding agents. One is a worktree manager. The other is an orchestration platform. Here's exactly when to pick which.

TL;DR

Both amux and dmux solve the same problem — running multiple AI coding agents in parallel without them stepping on each other — but they take very different approaches.

  • dmux is a worktree manager. It automates git worktree creation and gives each agent a clean isolated branch with minimal config. Terminal-first, multi-CLI (Claude, Codex, Gemini, Aider), interactive use case.
  • amux is an orchestration platform. It adds a web dashboard, mobile PWA, self-healing watchdog, kanban task board, token cost tracking, and a REST API. Built for unattended fleets you can monitor from anywhere.

Pick dmux if you want fast, frictionless worktree spin-up at your desk and you mix multiple AI CLIs.
Pick amux if you want to run agents while you're away (overnight, in meetings, on your phone), need spend visibility, or want a REST API for custom automation.

What Each Tool Is

dmux

dmux is an open-source CLI that spawns parallel AI coding sessions, each in its own git worktree. Its core insight is that git worktree + tmux + a CLI agent is enough to parallelize coding work — you just need a tool that automates the boilerplate.

Key features:

  • One command creates a worktree, branch, and tmux pane running your AI CLI of choice
  • Built-in support for Claude Code, Codex CLI, Gemini CLI, and Aider
  • Clean merge tooling that diffs each worktree against main
  • Lightweight: a Node CLI with no server, no web UI, no daemon

If you've ever wanted "tmux new-window but it also makes a git worktree and starts Claude Code in it," dmux is that tool.

amux

amux is an open-source orchestration platform built around amux-server.py — a single Python file that runs a local HTTPS server, manages tmux sessions, and serves a web dashboard. It treats parallel agents as a fleet that needs to be observed, recovered, and coordinated.

Key features:

  • Web dashboard at https://localhost:8822 with live status, token spend, and a send bar
  • Mobile PWA for managing the fleet from your phone
  • Self-healing watchdog: auto-restart crashed agents, auto-compact context overflow, detect stuck sessions
  • SQLite-backed kanban task board with atomic claiming (no two agents work on the same task)
  • Per-session token and cost tracking
  • Full REST API for custom automation, scheduling, and integration
  • Inter-session messaging so agents can delegate work to each other
  • Notes, CRM, and calendar primitives for solo developer workflows

If dmux is "tmux + worktrees with a nice CLI," amux is "a full developer cockpit where parallel agents are first-class citizens."

Architecture Differences

The two tools sit at different points on the same stack. dmux is a thin layer above git and tmux. amux is a full server with persistent state.

dmux architecture

┌──────────────────────────────────────────────┐
│  dmux CLI                                    │
│  (one-shot commands, no daemon)              │
└──────────────────────────────────────────────┘
              │
              ▼
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│ git worktree │ + │     tmux     │ + │  AI CLI of   │
│              │   │   session    │   │   choice     │
└──────────────┘   └──────────────┘   └──────────────┘

dmux is stateless. Each invocation does one thing: create a worktree, list worktrees, merge a worktree, etc. Once a session is started, dmux is out of the loop — you interact with the agent through the tmux pane directly.

amux architecture

┌──────────────────────────────────────────────────────────┐
│           amux-server.py (long-running)                  │
│                                                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌─────────┐ │
│  │ Web UI   │  │ REST API │  │ Watchdog │  │  Board  │ │
│  └──────────┘  └──────────┘  └──────────┘  └─────────┘ │
│                                                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌─────────┐ │
│  │  Notes   │  │   CRM    │  │  Tokens  │  │  Cron   │ │
│  └──────────┘  └──────────┘  └──────────┘  └─────────┘ │
└──────────────────────────────────────────────────────────┘
                          │
                          ▼
        ┌────────┐  ┌────────┐  ┌────────┐
        │ tmux 1 │  │ tmux 2 │  │ tmux N │
        │ Claude │  │ Claude │  │ Claude │
        └────────┘  └────────┘  └────────┘

amux runs as a persistent server. It owns session lifecycle, restarts crashed agents, exposes a REST API, and serves a dashboard. The server is the source of truth for everything — sessions, tasks, costs, notes — and survives across reboots.

What this means in practice

dmux's stateless model is simpler. There's no daemon to restart, no port to manage, no database to back up. If you close your terminal, dmux is gone (the tmux sessions keep running, of course).

amux's server model is more powerful. The dashboard is always available, the watchdog is always running, and you can hit the API from any other process (cron jobs, mobile apps, custom scripts, even other agents). The trade-off: you have one more long-running process to manage.

Workflow Comparison

Same task, two tools. Goal: spin up 5 parallel agents, each working on a different test suite expansion task.

The dmux way

# 5 commands, 5 worktrees, 5 agents
dmux new add-auth-tests
dmux new add-api-tests
dmux new add-payment-tests
dmux new add-notification-tests
dmux new add-search-tests

# Each command:
#   1. Creates a git worktree at ../project-add-auth-tests
#   2. Creates a branch named add-auth-tests
#   3. Opens a tmux window with Claude Code running
#   4. You type your task into Claude Code interactively

# When done, merge:
dmux merge add-auth-tests

Strengths: Fast, low ceremony, zero config. You're talking to Claude Code in seconds.
Limitations: You manually type the task into each agent. No central queue. No way to check progress without attaching to each tmux pane.

The amux way

# Register sessions (one-time)
for module in auth api payment notification search; do
  git worktree add ../project-$module -b agent/$module-tests
  amux register $module --dir ../project-$module --yolo
done

# Queue tasks on the board
amux board add "Add auth tests, target 90% coverage" --session auth
amux board add "Add API tests, target 85% coverage" --session api
amux board add "Add payment tests, target 90% coverage" --session payment
amux board add "Add notification tests, target 80% coverage" --session notification
amux board add "Add search tests, target 85% coverage" --session search

# Launch all sessions
amux start --all

# Open the dashboard
amux serve  # → https://localhost:8822
# Watch all 5 agents simultaneously, see token spend, get phone alerts

Strengths: Tasks are persisted on the board, you can monitor all agents from one dashboard, the watchdog restarts crashes, you can check progress from your phone.
Limitations: More setup ceremony than dmux. Need to think about the board, sessions, and watchdog config.

The takeaway

dmux optimizes for time-to-first-agent. amux optimizes for time-to-fleet-visibility. If your sessions are short and interactive, dmux wins on friction. If your sessions run for hours and you want to walk away, amux wins on observability.

Dashboards & Remote Access

This is the biggest functional gap.

dmux has no dashboard. You manage agents via tmux, the dmux CLI, and your terminal. To check progress, you attach to a tmux session and look. To check progress remotely, you SSH in and attach. To check progress from your phone — you don't.

amux has a full web dashboard. Open https://localhost:8822 in any browser and see:

  • All sessions with live status (running, idle, crashed, stuck)
  • Per-session token spend and cumulative cost
  • Last N lines of output from every agent, scrollable
  • A send bar to type messages directly into any agent
  • The kanban board with drag-and-drop task management

The dashboard works as a PWA on your phone, so you can install it like a native app. With Tailscale, you can access your home or office amux instance from anywhere on the internet without exposing ports. See the Mobile Management PWA guide and Remote Access with Tailscale guide.

If you ever want to start an agent before bed and check on it from your phone in the morning, amux is the only one of the two that can do this.

Unattended Operation & Self-Healing

Running agents overnight is the canonical use case for parallel coding tools. Both can do it — but with very different reliability.

dmux: agents run, you hope they don't crash

dmux starts an agent in a tmux pane and walks away. If the agent crashes (segfault, OOM, network blip), the tmux pane shows the error and the agent is dead until you manually restart it. If the agent hits its context limit and freezes, dmux doesn't notice. If an agent gets stuck in a loop, dmux doesn't notice.

This is fine for a 30-minute interactive sprint. It is not fine for a 6-hour overnight run.

amux: self-healing watchdog

amux's watchdog actively monitors every session and handles common failure modes automatically:

  • Crash: Process dies → auto-restart, replay last user message
  • Context overflow: Detected via output pattern → trigger /compact and continue
  • Stuck: No new output for N minutes → send a nudge or restart
  • Permission prompts: Auto-respond in YOLO mode
  • Rate limits: Back off and retry transparently

The watchdog is what makes overnight runs practical. Without it, one crash at 2am ends the session for the next 6 hours. With it, the agent restarts in 30 seconds and keeps going. See the Self-Healing Configuration guide and Running AI Coding Agents Overnight guide for details.

Multi-CLI Support

This is dmux's biggest strength.

dmux ships with first-class support for Claude Code, Codex CLI, Gemini CLI, and Aider. You pick the agent type when you create a session, and dmux knows how to start each one. This is genuinely useful if you're comparing models on the same task or running a mixed fleet to leverage each model's strengths.

amux is Claude Code-first. The session manager, watchdog, and token tracking are all designed around Claude Code's CLI. You can register a session running Codex CLI or Gemini CLI — each session is just a tmux pane running an arbitrary command — but you'll lose the smart watchdog behaviors (which key off Claude Code's specific output patterns) and token tracking (which parses Claude's spend output).

If you're a multi-model shop, dmux is the better fit. If you're 95%+ on Claude Code, amux's deep integration is more valuable than multi-CLI breadth.

Which Should You Choose?

Choose dmux if...

  • You work interactively at your desk and don't need remote monitoring
  • You use multiple AI CLIs (Claude, Codex, Gemini, Aider) and want first-class support for all of them
  • You want zero-ceremony worktree spin-up — one command, agent running
  • You don't need a task board, dashboard, or REST API
  • You prefer tools that are "just CLIs" with no daemons, servers, or persistent state
  • Your sessions are short (under an hour) and you'll be there to handle failures yourself

Choose amux if...

  • You want to run agents unattended — overnight, while in meetings, while away from your desk
  • You need a web dashboard you can check from any device, including your phone
  • You want self-healing recovery from crashes, context overflows, and stuck sessions
  • You need token spend visibility per session
  • You want a kanban task board with atomic task claiming so agents don't duplicate work
  • You're building custom automation that needs a REST API (cron jobs, scripts, integrations)
  • You're running 10+ agents and need fleet-level observability
  • You want a single-file Python deployment that's easy to audit and easy to deploy to a cloud VM
  • You're 95%+ on Claude Code and want deep, model-aware integration

The honest summary

If your mental model is "I want to run a bunch of agents in worktrees right now and merge them later," dmux is the right tool. It does that one thing extremely well with very little ceremony.

If your mental model is "I want to run a fleet of agents that I can monitor from anywhere, that recovers itself from failures, and that I can build custom automation around," amux is the right tool. It's a bigger surface area but it's solving a bigger problem.

Neither tool is wrong. They're optimized for different points on the interactive ↔ unattended spectrum.

Migrating Between Them

From dmux to amux

If you already have dmux worktrees, you don't need to recreate them. amux can register an existing directory as a session:

# Your existing dmux worktrees are at ../project-add-auth-tests, etc.
# Register each as an amux session
for wt in ../project-add-*; do
  name=$(basename $wt | sed 's/project-//')
  amux register $name --dir $wt --yolo
done

# Start the amux server and dashboard
amux serve

# Now you can monitor your existing worktrees from the dashboard
# without losing any state

Your git history, branches, and uncommitted work are untouched. amux just adds a monitoring and recovery layer on top.

From amux to dmux

The reverse is also straightforward. amux's worktrees are standard git worktrees, so dmux can find and manage them. You'll lose the dashboard, watchdog, board, and token tracking — but the underlying git state is unchanged.

Frequently Asked Questions

What is the main difference between amux and dmux?

dmux is a worktree manager — it automates git worktree creation and lets you run multiple AI CLIs side by side. amux is a full orchestration platform — it adds a web dashboard, mobile PWA, self-healing watchdog, kanban task board, token cost tracking, and a REST API on top of the same parallel-agent pattern.

Should I use amux or dmux?

Use dmux for fast interactive worktree management at your desk, especially if you mix multiple AI CLIs. Use amux when you need unattended operation, remote monitoring, self-healing, cost tracking, or a REST API.

Can I use amux and dmux together?

Technically yes — they don't conflict at the OS level — but most people pick one. amux can register existing dmux worktrees as sessions if you want to migrate.

Does dmux have a web dashboard?

No. dmux is terminal-only. amux provides a full web dashboard at https://localhost:8822 with mobile PWA support.

Does amux support Codex CLI and Gemini CLI like dmux does?

amux is primarily designed for Claude Code. You can register sessions running other CLIs, but the self-healing watchdog and token tracking are Claude-specific. dmux has first-class multi-CLI support out of the box.

Is dmux open source?

Yes, both are MIT-licensed. amux is a single Python file. dmux is a Node/TypeScript CLI.

Which is more reliable for overnight runs?

amux, by a wide margin. The self-healing watchdog handles crashes, context overflows, and stuck sessions automatically. dmux has no equivalent — if an agent dies overnight, it stays dead until you restart it manually.

Which has better cost tracking?

amux. It tracks token spend per session and shows cumulative cost on the dashboard. dmux has no cost tracking layer; you'd need to check Claude's billing dashboard separately.

Are there other parallel AI agent tools I should consider?

Yes — the category is growing. Look at Claude Code Agent Teams (Anthropic's built-in feature), cmux (native macOS), NTM (lightweight TUI), and the broader comparison index. The agentmaxxing guide includes a feature matrix across all of them.
