amux vs OpenAI Symphony

One turns your issue tracker into an autonomous coding pipeline. The other gives you a cockpit for managing a fleet of AI agents. Both orchestrate parallel coding agents — but for very different workflows.

TL;DR

OpenAI Symphony and amux both orchestrate parallel coding agents, but they represent fundamentally different philosophies about how developers should interact with AI agent fleets.

  • Symphony is a ticket-driven autonomous pipeline. It polls your Linear board, claims open issues, spawns a Codex agent for each, and produces pull requests — with no human interaction during execution. Built with Elixir/OTP for process-level fault tolerance. OpenAI describes it as a reference implementation, not a maintained product.
  • amux is a developer-driven orchestration platform. You manage Claude Code sessions from a web dashboard, send messages to agents in real time, track tasks on a built-in kanban board, and monitor everything from your phone. Built as a single Python file for zero-dependency deployment.

Pick Symphony if you want fully autonomous issue-to-PR automation, you're on the Codex/GPT ecosystem, and you use Linear for project management.
Pick amux if you use Claude Code, want a web dashboard with mobile access, need interactive control over agents, or want a built-in task board without depending on an external tracker.

What Each Tool Is

OpenAI Symphony

Symphony is an open-source orchestration layer from OpenAI that turns a Linear project board into a control plane for Codex coding agents. It grew out of a bottleneck OpenAI encountered internally: engineers running multiple Codex sessions could only manage 3–5 before context-switching became painful. Symphony eliminates that by making the issue tracker the interface — every open ticket gets an agent, agents run continuously, and humans only review the resulting pull requests.

Key characteristics:

  • Polls Linear every 30 seconds for open issues, claims them atomically, spawns a Codex agent per issue
  • Built with Elixir on the BEAM/OTP runtime — each agent runs as a supervised process with automatic crash recovery
  • Configured via WORKFLOW.md — a single file (YAML front matter + Markdown prompt template) that defines agent behavior, concurrency limits, and workspace setup
  • Communicates with Codex via JSON-RPC 2.0 over stdio (the Codex App Server protocol)
  • Manages multi-turn execution (up to 20 turns by default), CI monitoring, rebasing, and conflict resolution
  • OpenAI explicitly states they do not plan to maintain Symphony as a standalone product — it is a reference implementation

Some teams at OpenAI reported a 500% increase in landed pull requests during the first three weeks of using Symphony, though no baseline PR rate was published.

amux

amux is an open-source agent orchestration platform built around amux-server.py — a single Python file that runs a local HTTPS server, manages tmux sessions, and serves a web dashboard. It treats parallel agents as a fleet that needs to be monitored, recovered, coordinated, and controlled from anywhere.

Key characteristics:

  • Web dashboard at https://localhost:8822 with live session status, token spend, a send bar, and a kanban board
  • Mobile PWA for managing the fleet from your phone, plus a native iOS app
  • Self-healing watchdog: auto-restart crashed agents, auto-compact context overflow, detect stuck sessions, handle rate limits
  • Built-in SQLite-backed kanban board with atomic task claiming — no external issue tracker required
  • Full REST API for sessions, board, notes, CRM, email, browser automation, and cron
  • Inter-session messaging so agents can discover peers, delegate work, and coordinate via the shared board
  • Actively developed and maintained with regular releases

Architecture Differences

Symphony and amux solve orchestration at different layers. Symphony is an autonomous pipeline — tickets go in, PRs come out. amux is a control plane — you see everything, control everything, and agents coordinate through you and through each other.

Symphony architecture

┌─────────────────────────────────────────────────────────────┐
│  Symphony  (Elixir/OTP long-running daemon)                 │
│                                                             │
│  ┌────────────┐  ┌──────────────┐  ┌──────────────────────┐│
│  │ Linear     │  │ Orchestrator │  │ Workspace Manager    ││
│  │ Poller     │─▶│ (GenServer)  │─▶│ (clone + setup)      ││
│  └────────────┘  └──────────────┘  └──────────────────────┘│
│                         │                                   │
│              ┌──────────┴──────────┐                        │
│              ▼                     ▼                        │
│  ┌──────────────────┐  ┌──────────────────┐                │
│  │ Agent Runner 1   │  │ Agent Runner N   │                │
│  │ (Codex via RPC)  │  │ (Codex via RPC)  │                │
│  └──────────────────┘  └──────────────────┘                │
│              │                     │                        │
│              ▼                     ▼                        │
│        ┌──────────┐         ┌──────────┐                   │
│        │ PR + CI  │         │ PR + CI  │                   │
│        └──────────┘         └──────────┘                   │
└─────────────────────────────────────────────────────────────┘
        ▲                                          │
        │  poll issues                    push PRs  │
        │                                          ▼
   ┌─────────┐                              ┌──────────┐
   │ Linear  │                              │  GitHub  │
   └─────────┘                              └──────────┘

Symphony's data flow is unidirectional: issues enter from Linear, agents produce PRs, humans review on GitHub. The developer's role is reduced to writing tickets and reviewing output. There is no interactive session management — once an issue is claimed, the agent runs autonomously until it succeeds, fails, or times out.

amux architecture

┌──────────────────────────────────────────────────────────┐
│           amux-server.py (long-running)                   │
│                                                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌─────────┐ │
│  │ Web UI   │  │ REST API │  │ Watchdog │  │  Board  │ │
│  └──────────┘  └──────────┘  └──────────┘  └─────────┘ │
│                                                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌─────────┐ │
│  │  Notes   │  │   CRM    │  │  Tokens  │  │  Cron   │ │
│  └──────────┘  └──────────┘  └──────────┘  └─────────┘ │
└──────────────────────────────────────────────────────────┘
              │                          ▲
    manage    │                          │  REST API
    sessions  ▼                          │  calls
        ┌────────┐  ┌────────┐  ┌────────┐
        │ tmux 1 │  │ tmux 2 │  │ tmux N │
        │ Claude │◀─│ Claude │──│ Claude │  ← agents talk
        │ Code   │──│ Code   │─▶│ Code   │    to each other
        └────────┘  └────────┘  └────────┘

amux's data flow is bidirectional. You send messages to agents, agents send messages to each other, the dashboard shows everything in real time, and the REST API is open for any external tool to participate. The developer stays in the loop — not blocking agents, but always able to observe and intervene.

What this means in practice

Symphony is optimized for hands-off throughput. Write well-scoped tickets, let agents grind, review the PRs. This works when your codebase has excellent test coverage, your tickets are unambiguous, and you trust the agent to make good architectural decisions without guidance.

amux is optimized for hands-on control. Start agents, watch them work, redirect when they go off track, send context they're missing, and let them coordinate with each other. This works when tasks are exploratory, when you want to steer agents in real time, or when you're running agents overnight and need self-healing recovery.

Workflow Comparison

Same goal: implement 5 features across your codebase in parallel.

The Symphony way

# 1. Write a WORKFLOW.md that defines your agent behavior
#    (concurrency limits, workspace setup, prompt template)

# 2. Create 5 well-scoped tickets in Linear
#    Each ticket has clear acceptance criteria and context

# 3. Start the Symphony daemon
mise exec -- ./bin/symphony ./WORKFLOW.md

# Symphony now:
#   - Polls Linear every 30 seconds
#   - Claims each ticket atomically
#   - Clones the repo into an isolated workspace per ticket
#   - Spawns a Codex agent with the WORKFLOW.md prompt
#   - Runs up to 20 turns per agent
#   - Commits, pushes, opens PRs
#   - Monitors CI, rebases on conflicts
#   - Handles review feedback (destructive: resets and re-implements)

# 4. You review the 5 PRs on GitHub when they're ready

Strengths: Fully autonomous. You write tickets and review PRs — the middle is automated. Great for well-defined tasks in codebases with strong test suites.
Limitations: You can't steer agents mid-execution. If an agent goes down the wrong path, it burns through all 20 turns before you see the result. You need Linear. You need Codex. Review feedback triggers a destructive reset instead of an incremental fix.

The amux way

# 1. Register sessions and queue tasks
for feature in auth payments search analytics billing; do
  amux register $feature --dir ~/Dev/project --yolo
done

amux board add "Implement OAuth2 flow" --session auth
amux board add "Add Stripe webhook handler" --session payments
amux board add "Build full-text search endpoint" --session search
amux board add "Add PostHog event tracking" --session analytics
amux board add "Create invoice PDF generation" --session billing

# 2. Start all sessions
amux start --all

# 3. Open the dashboard — monitor from any device
amux serve  # → https://localhost:8822

# Now you can:
#   - Watch all 5 agents working simultaneously
#   - Send a message to redirect an agent going off track
#   - See token spend per session in real time
#   - Check progress from your phone via the PWA
#   - Trust the watchdog to restart any crash at 3am
#   - Let agents delegate sub-tasks to each other via the board

Strengths: Interactive control. You can steer agents, monitor spend, check from your phone, and trust the self-healing watchdog for overnight runs. No external dependencies — the task board is built in.
Limitations: More hands-on than Symphony's fire-and-forget model. You're managing a fleet, not just reviewing output. No automated PR generation pipeline (agents open PRs as part of their work, but it's not orchestrated at the system level).

Agent Support & Model Ecosystem

This is the most decisive factor for most teams.

Symphony: Codex-native

Symphony's reference implementation communicates with agents via the Codex App Server protocol (JSON-RPC 2.0 over stdio). This is a Codex-specific protocol. Running Claude Code, Aider, or any other agent requires either implementing the App Server protocol for that agent or rewriting the runner — which is what the DEV Community porting effort did (a full TypeScript rewrite).

If you're on GPT/Codex, Symphony speaks your language natively. If you're on Claude Code, it does not.

amux: Claude Code-native

amux is built around Claude Code. The self-healing watchdog, token tracking, context overflow detection, auto-compact triggers, and permission prompt handling are all specific to Claude Code's CLI output patterns. Other agents (Codex CLI, Aider, Gemini CLI) can run as amux sessions — each session is just a tmux pane — but they won't get the Claude-specific recovery features.

If you're on Claude Code, amux gives you deep, model-aware integration. If you're on Codex, the watchdog features are limited.

The takeaway

The orchestrator you choose is largely determined by the model ecosystem you're in. Claude Code users should use amux. Codex users should use Symphony (or fork the spec for their agent of choice).

Issue Tracking & Task Management

This is where the two tools' philosophies diverge most sharply.

Symphony: your issue tracker is the interface

Symphony has no built-in task management. It uses Linear as both the source of work and the control surface. Want to start a task? Create a Linear ticket. Want to pause it? Move it to a terminal state. Want to change priority? Reorder in Linear.

This is elegant if your team already lives in Linear. It's a hard dependency if you don't. As of April 2026, Linear is the only supported tracker. No GitHub Issues, no Jira, no Asana. The SPEC is designed to be pluggable, and a community GitHub Issues adapter is in development, but it's not ready.

amux: built-in kanban with no external dependencies

amux includes a SQLite-backed kanban board as a first-class feature. Create tasks, assign them to sessions, track status transitions, set due dates, export to iCal. The board supports atomic task claiming so two agents can't pick up the same work.

You don't need Linear, Jira, or any external tool. The board lives in amux's dashboard, is accessible via the REST API, and works from your phone. Agents can create and claim board items programmatically, enabling agent-to-agent delegation.

If you want to integrate with an external tracker, you can — amux's REST API makes it easy to build a sync. But it's not required.

Fault Tolerance & Self-Healing

Both tools handle agent failures, but they address different failure modes.

Symphony: OTP supervision trees

Symphony leverages Elixir/OTP's supervision model. Each agent runs as a supervised process. If it crashes, the supervisor restarts it with exponential backoff (delay = min(10000 × 2^(attempt-1), max_backoff)). The Orchestrator GenServer reconciles state every tick, killing workers for terminal-state issues and detecting stalls.

This is architecturally elegant and well-proven — OTP has decades of production reliability behind it. The trade-off: it handles process-level crashes but not AI-agent-level failures like context overflow, permission prompts, or model-specific stuck states.

amux: AI-agent-aware watchdog

amux's watchdog is purpose-built for the failure modes specific to AI coding agents:

  • Crash: Process dies → auto-restart, replay last user message
  • Context overflow: Detected via Claude Code's output pattern → trigger /compact and continue
  • Stuck: No new output for N minutes → send a nudge or restart
  • Permission prompts: Auto-respond in YOLO mode so agents don't stall on y/n confirmations
  • Rate limits: Back off and retry transparently

Symphony's OTP supervision handles "the process crashed." amux's watchdog handles "the AI agent is stuck, out of context, rate-limited, or waiting for a prompt it'll never get." For overnight coding agent runs, the latter set of failure modes is more common than the former.

Dashboards & Remote Access

Symphony has a minimal status surface. An optional Phoenix LiveView dashboard shows agent states and a JSON API exposes /api/v1/state and per-issue status. This is observability, not control — you can see what agents are doing but you can't interact with them from the dashboard. No mobile support.

amux has a full interactive dashboard. Open https://localhost:8822 in any browser and see all sessions with live status, token spend, output scrollback, and a send bar for typing messages directly into any agent. The kanban board is drag-and-drop. The dashboard works as a PWA on your phone with native-app-like safe-area insets and touch targets.

With Tailscale, amux is accessible from anywhere on the internet without exposing ports. Start agents at your desk, check them from your phone at dinner, review results from your laptop in bed.

Code Review & PR Handling

An important and under-discussed difference.

Symphony: destructive rework

When a reviewer leaves feedback on a Symphony-generated PR, the default workflow resets the entire branch and re-implements from scratch. A reviewer says "rename this variable" and the agent throws away 47 changed files and starts over. This was flagged by the developer who ported Symphony to Claude Code as a significant limitation — it wastes compute and can produce different (potentially worse) results on the second pass.

amux: incremental feedback

amux sessions are interactive. If you see something wrong in an agent's output, you type a message into the send bar and the agent addresses it incrementally. The context is preserved, the work so far is preserved, and the agent makes a targeted fix. This is how human pair programming works — you don't throw away the file and start over when someone suggests a rename.

Setup & Maintenance

Symphony: Elixir ecosystem

# Install Elixir, mise, and dependencies
brew install mise
mise install

# Configure Linear API key and Codex access
export LINEAR_API_KEY=lin_...
export OPENAI_API_KEY=sk-...

# Write your WORKFLOW.md (YAML frontmatter + Markdown prompt)
# Start the daemon
mise exec -- ./bin/symphony ./WORKFLOW.md

Symphony requires the Elixir/OTP runtime, which is not in most developers' toolboxes. The configuration via WORKFLOW.md is elegant once you understand it, but the learning curve includes Elixir tooling, the SPEC's state machine, and Linear API setup. The HN discussion noted that the documentation was characterized by some as difficult to follow.

amux: single-file Python

# Install (Python 3 and tmux are likely already on your system)
git clone https://github.com/mixpeek/amux && cd amux && ./install.sh

# Start using it immediately
amux register myproject --dir ~/Dev/myproject --yolo
amux start myproject
amux serve  # → https://localhost:8822

amux is a single Python file with no dependencies beyond Python 3 and tmux, both of which are pre-installed on most developer machines. No Elixir, no mise, no external API keys for the orchestrator itself. Deploy to a remote server with scp amux-server.py your-server:.

Maintenance commitment

Symphony has 12 commits on GitHub and no versioned releases. OpenAI has explicitly stated it is a reference implementation, not a maintained product. The SPEC.md is the lasting contribution — a well-thought-out specification that others can implement.

amux is actively developed with regular commits, a native iOS app, cloud deployment via Terraform, and ongoing feature development. Bugs get fixed. Features get shipped.

Which Should You Choose?

Choose Symphony if...

  • You want fully autonomous issue-to-PR automation with no human interaction during execution
  • You're on the Codex / GPT ecosystem and want native agent integration
  • Your team already uses Linear for project management
  • Your codebase has excellent test coverage and well-scoped tickets — agents can work without guidance
  • You're comfortable with Elixir/OTP or willing to learn
  • You want to study the SPEC.md as a reference for building your own orchestrator
  • You're okay with a reference implementation that may not be actively maintained by OpenAI

Choose amux if...

  • You use Claude Code and want native, model-aware integration
  • You want a web dashboard you can access from any device, including your phone
  • You want interactive control — send messages to agents, steer them in real time
  • You need self-healing for context overflows, stuck sessions, and permission prompts, not just process crashes
  • You want a built-in task board without depending on Linear or any external tracker
  • You want agents that can coordinate with each other via the shared board and REST API
  • You need token spend visibility per session
  • You want a single-file deployment with zero dependencies beyond Python and tmux
  • You want an actively maintained project, not a reference implementation

The honest summary

Symphony and amux are not competing for the same workflow. They represent two different answers to the question "how should humans interact with a fleet of coding agents?"

Symphony says: "Humans shouldn't interact with agents at all. Write tickets, review PRs. The middle is automated." This is the factory model — issues are raw material, PRs are finished goods, agents are the machines.

amux says: "Humans should have a cockpit. See everything, steer when needed, let agents coordinate, and trust self-healing for the overnight shift." This is the fleet management model — agents are collaborators, not assembly lines.

Neither is wrong. Symphony's model works when tasks are well-defined and your test suite catches mistakes. amux's model works when tasks are exploratory, when context matters, or when you want the peace of mind that comes from being able to check your phone and see that all 15 agents are still running at 3am.

The most telling difference: Symphony optimizes for PR throughput. amux optimizes for developer confidence.

Using Them Together

In theory, these tools can complement each other. Symphony handles your well-defined, ticket-shaped work on the Codex side. amux handles your exploratory, interactive Claude Code sessions with full monitoring.

# Symphony handles the ticket grind
# (runs as a daemon, polls Linear, produces PRs)
mise exec -- ./bin/symphony ./WORKFLOW.md

# amux handles the creative work
# (interactive sessions you steer and monitor)
amux register refactor --dir ~/Dev/project --yolo
amux register prototype --dir ~/Dev/project --yolo
amux start --all
amux serve

In practice, most teams standardize on one model ecosystem (Claude or GPT) and one orchestrator. But if you're a multi-model shop, the combination is viable.

Frequently Asked Questions

What is the main difference between amux and Symphony?

Symphony is a ticket-driven pipeline: it polls Linear, claims issues, spawns Codex agents, and produces PRs — fully autonomous. amux is a developer-driven platform: you manage Claude Code sessions from a web dashboard, send messages to agents in real time, and track tasks on a built-in kanban board. Symphony automates the issue-to-PR pipeline. amux gives you a cockpit for managing a fleet.

Does Symphony work with Claude Code?

Not out of the box. The reference implementation uses the Codex App Server protocol. A community developer has ported the design to Claude Code via a full TypeScript rewrite, but this is not official. amux is built natively for Claude Code.

Does Symphony require Linear?

Yes. Linear is the only supported issue tracker as of April 2026. A community GitHub Issues adapter is in development. amux has a built-in kanban board — no external tracker required.

Is Symphony actively maintained?

OpenAI explicitly states Symphony is a reference implementation, not a standalone product they plan to maintain. The repo has 12 commits and no versioned releases. The SPEC.md is the durable artifact. amux is actively developed with regular releases.

Which has better fault tolerance?

Symphony uses OTP supervision trees — architecturally elegant for process crashes. amux uses a Python watchdog tuned for AI-agent-specific failures: context overflow, stuck sessions, permission prompts, rate limits. For coding agent operations, amux covers more failure modes.

Which is easier to set up?

amux. Single Python file, no dependencies beyond Python 3 and tmux. Symphony requires Elixir/OTP runtime, mise, a Linear API key, and Codex access.

What about the 500% PR increase claim?

OpenAI reported that "some teams" saw a 500% increase in landed PRs during the "first three weeks." No baseline was published. Going from 2 PRs/week to 12 is technically 500%. More PRs does not necessarily mean more value if review burden scales proportionally.

Can I use amux and Symphony together?

Yes. Use Symphony for autonomous Codex ticket-grinding and amux for interactive Claude Code sessions. They don't share state but can run side by side. In practice, most teams pick one model ecosystem.

What does Symphony's destructive review handling mean?

When a reviewer gives feedback on a Symphony PR, the default workflow resets the entire branch and re-implements from scratch — even for minor comments like "rename this variable." This wastes compute and risks producing different results. amux agents address feedback incrementally with full context preserved.

Are there other orchestrators I should consider?

Yes. See cmux (native macOS terminal), dmux (worktree-focused CLI), Claude Code Agent Teams (Anthropic's built-in feature), and the full comparison index.
