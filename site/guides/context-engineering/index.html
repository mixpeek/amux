What Is Context Engineering?

Context engineering is the practice of curating everything an AI coding agent sees — system prompts, project rules, reference docs, tool outputs, conversation history, memory — so that it produces reliable, correct output.

This is not prompt engineering. Prompt engineering optimizes a single input message. Context engineering designs the entire information architecture around an agent: what it sees at startup, what it can load on demand, what it physically cannot do, who it can delegate to, and what persists across sessions.

Prompt engineering:     "Write a function that does X"
                                    ↓
Context engineering:    CLAUDE.md + hooks + skills + subagents + memory + spec
                        ↓         ↓        ↓         ↓          ↓        ↓
                     base rules  guardrails  reference  delegation  history  task scope
                                    ↓
                        Agent sees the right things at the right time
                                    ↓
                        Reliable output, not lucky output

The difference matters because a single-prompt approach breaks the moment you need agents to work on real codebases over hours or days. Context engineering is what makes agents consistently good, not occasionally impressive.

Why it matters now

Three shifts in 2025–2026 made context engineering the critical developer skill:

  • Agents got long-running. Sessions that run for hours need structured context that survives compaction, context overflow, and session restarts — not just a good opening prompt.
  • Fleets became normal. Running 5–50 agents in parallel means your context engineering is multiplied. A bad CLAUDE.md wastes tokens once per session — across 20 agents, it wastes them 20 times.
  • The tooling matured. Claude Code now has six distinct context surfaces (CLAUDE.md, hooks, skills, subagents, memory, specs). Knowing which surface to use when is the skill gap.

The Six Context Surfaces

Every piece of context you give an AI coding agent enters through one of six surfaces. Each has different characteristics:

Surface When loaded Enforcement Scope Persists across sessions
CLAUDE.md Every session start Soft (~70%) Project or directory Yes (in repo)
Hooks On specific events Hard (100%) Global or project Yes (in settings)
Skills On demand (agent decides) Soft Project or user Yes (in repo or ~/.claude)
Subagents On delegation Soft (own context) Per invocation Definition persists, state doesn't
Memory Session start (auto-memory) Soft User or project Yes
Specs Passed per task Soft Per task Yes (checked into repo)

The core skill of context engineering is putting information on the right surface. Put too much in CLAUDE.md and you bloat every session. Put safety rules in CLAUDE.md instead of hooks and they get violated 30% of the time. Put reference docs inline instead of in skills and you waste context window on every turn. The rest of this guide covers each surface in depth.

CLAUDE.md: The Base Layer

CLAUDE.md is the always-loaded project configuration. The agent reads it at session start and it stays in context for the entire conversation. This makes it the most expensive surface — every token here is paid on every turn.

What belongs in CLAUDE.md

  • Build and test commands — npm run test, make lint, python -m pytest tests/
  • Style rules the linter doesn't catch — naming conventions, import ordering preferences, commit message format
  • Forbidden patterns — "never use any in TypeScript", "never push to main directly"
  • Pointers to deeper docs — "for API reference, see docs/api.md"; "for deploy instructions, load the deploy skill"
  • Project structure overview — what lives where, but only if it's non-obvious from the directory layout

What does NOT belong in CLAUDE.md

  • Full API documentation — goes in a skill, loaded on demand
  • Safety-critical rules — goes in hooks, enforced at 100%
  • Large code examples — the agent can read the actual code
  • Deployment runbooks — goes in a skill, loaded only during deploy
  • Things the agent can infer from the code — file structure, framework choice, language version

The litmus test

For every line in CLAUDE.md, ask: "If I remove this, will the agent make a mistake it cannot recover from by reading the code?" If the answer is no, cut it. A lean CLAUDE.md is a fast CLAUDE.md.

Hierarchical CLAUDE.md

You can place CLAUDE.md files in subdirectories. The agent loads the root CLAUDE.md plus any CLAUDE.md in the directory it's currently working in. Use this for context that only matters in one part of the codebase:

myproject/
  CLAUDE.md                  ← global rules (always loaded)
  frontend/
    CLAUDE.md                ← React-specific rules (loaded when working in frontend/)
  backend/
    CLAUDE.md                ← Python/FastAPI rules (loaded when working in backend/)
  infrastructure/
    CLAUDE.md                ← Terraform conventions (loaded when working in infra/)

Real example: a lean, effective CLAUDE.md

# MyProject

Single-page Python app. FastAPI backend, React frontend, PostgreSQL.

## Commands
- `make test` — run all tests (pytest + vitest)
- `make lint` — ruff + eslint
- `make dev` — start dev server on :8000

## Rules
- Never use `any` in TypeScript. Use `unknown` and narrow.
- Always run `make lint` before committing.
- Commit messages: imperative mood, under 72 chars, no period.
- For deploy instructions, load the `deploy` skill.
- For API reference, see `docs/openapi.yaml`.

That's 14 lines. It tells the agent everything it can't infer. Everything else is either in the code (which it can read), in a skill (loaded on demand), or in a hook (enforced mechanically).

Hooks: Hard Enforcement

CLAUDE.md instructions are followed about 70% of the time. For style preferences, that's fine. For rules like "don't push to main" or "don't delete the production database", 70% is a disaster waiting to happen.

Hooks are shell commands that run automatically at specific points in the agent's lifecycle. They can block actions (exit code 2), modify behavior, or inject context. They enforce at 100% because they execute outside the model — the agent cannot choose to ignore them.

The four hook types

HookWhen it runsUse case
PreToolUseBefore a tool call executesBlock dangerous commands, enforce formatting
PostToolUseAfter a tool call completesAuto-format after file edits, run tests
NotificationWhen the agent sends a notificationPush alerts to Slack, phone, or dashboard
SessionStartWhen a session beginsInject dynamic context, check prerequisites

Example: block pushes to main

// .claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "bash -c '[[ \"$CLAUDE_TOOL_INPUT\" =~ \"git push.*main\" ]] && echo \"BLOCKED: never push to main directly\" && exit 2 || exit 0'"
      }
    ]
  }
}

Exit code 2 blocks the tool call and shows the message to the agent. The agent cannot override this — it's enforced at the harness level, not the model level.

Example: auto-format after every file edit

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "bash -c 'make lint-fix 2>/dev/null || true'"
      }
    ]
  }
}

The decision rule

CLAUDE.md = "please do this" (soft, ~70% compliance). Hook = "you physically cannot do otherwise" (hard, 100%). Move any rule you've seen violated from CLAUDE.md to a hook.

Skills: On-Demand Knowledge

Skills are the progressive-disclosure layer. Instead of loading a 5,000-line API reference into CLAUDE.md (wasting tokens on every turn), you put it in a skill. The agent sees only the skill's name and description (~200 tokens) at all times, and loads the full content only when it judges the skill is relevant.

How skills work

.claude/skills/
  deploy.md          ← deployment runbook
  api-reference.md   ← full API docs
  database-schema.md ← schema reference
  security-review.md ← security checklist

Each skill file has YAML frontmatter with a name and description. The description is the routing signal — the agent reads it and decides whether to load the full content.

---
name: deploy
description: Use when the user asks to deploy, release, or ship to production. Contains the full deployment runbook with environment-specific steps.
---

# Deployment Runbook

## Staging
1. Run `make build`
2. Run `make test`
3. Deploy with `make deploy-staging`
4. Verify at https://staging.example.com

## Production
1. Ensure staging is green
2. Create a release tag: `git tag v$(date +%Y%m%d)`
3. Deploy with `make deploy-prod`
4. Monitor https://grafana.example.com/d/prod for 15 minutes
...

That entire runbook is loaded only when the user says "deploy." Every other turn, the agent pays only ~30 tokens for the frontmatter.

Skills vs CLAUDE.md: the decision

  • Needed every turn? → CLAUDE.md
  • Needed sometimes? → Skill
  • Needed never (agent can read the code)? → Neither

Subagents: Delegation With Focused Context

Subagents are the context isolation layer. When a task would generate a lot of intermediate output (20 file reads, 15 grep results), a subagent can do the work in its own fresh context and return a single summarized message to the parent. The parent's context grows by 200 tokens instead of 8,000.

See the complete subagent guide for deep coverage. The context engineering perspective:

Why subagents are a context strategy

Without subagent:
  Parent context: [...8KB of grep output, file reads, intermediate reasoning...]
  → context fills up fast, earlier instructions get pushed out

With subagent:
  Subagent context: [...8KB of work...] → discarded after returning
  Parent context: [...200-token summary...]
  → parent stays focused, instructions stay visible

The context math

A 200K context window sounds enormous until you fill it with tool outputs. A single grep across a large codebase can return 5KB. Read 10 files and you've spent 30KB. After 30–40 tool calls, you're past 100KB and the model starts losing track of your original instructions. Subagents let the parent session stay under 50KB even during complex multi-file tasks, because all the exploratory noise stays in disposable child contexts.

Memory: Cross-Session Persistence

Every new Claude Code session starts with a blank context window (plus CLAUDE.md). Memory is the mechanism that carries knowledge across sessions — what the user prefers, what decisions were made, what mistakes to avoid.

Two memory systems

  • Auto-memory (~/.claude/projects/<project>/memory/) — Claude Code's built-in memory that automatically saves facts, preferences, and feedback across conversations.
  • External memory (amux board, notes API, databases) — structured memory accessible via REST API that multiple agents can read and write.

Memory for solo sessions

Auto-memory works well for a single developer working with one agent. The agent learns your style preferences, remembers past decisions, and avoids repeating mistakes. No extra setup needed — just correct the agent when it's wrong and it saves the feedback.

Memory for parallel agent fleets

Auto-memory breaks down with parallel agents because each session has its own memory file. Agent 1's learnings aren't visible to agent 5. For fleet-wide context, externalize memory to a shared system:

# Shared board — all agents can see task state
curl -sk $AMUX_URL/api/board

# Shared notes — reference docs accessible to any session
curl -sk $AMUX_URL/api/notes/architecture-decisions

# Per-session state — only this agent's context
curl -sk $AMUX_URL/api/sessions/$AMUX_SESSION/peek

The principle: shared knowledge goes in shared stores; session-specific knowledge goes in session-specific stores. Never use the local filesystem as shared state when running parallel agents in separate worktrees.

Specs: Task-Level Instructions

Specs are the newest context surface and the one gaining the most traction in 2026. A spec is a structured document that describes exactly what a task should produce — requirements, constraints, acceptance criteria, and sometimes an implementation plan — before the agent writes any code.

Why specs matter for context engineering

Without a spec, the agent infers intent from a one-line prompt. It fills the gaps with guesses — sometimes brilliant, sometimes wrong. A spec eliminates guessing by pre-loading the agent with everything it needs to make correct decisions.

# spec: add-pagination-to-users-api.md

## Goal
Add cursor-based pagination to GET /api/users.

## Requirements
- Use cursor-based pagination (not offset), keyed on `created_at`
- Default page size: 50, max: 200
- Response includes `next_cursor` and `has_more`
- Backward-compatible: existing clients that don't send cursor get page 1

## Constraints
- Do NOT change the User model or database schema
- Do NOT add new dependencies
- Must pass existing test suite + new pagination tests

## Acceptance criteria
- `make test` passes
- `curl /api/users?limit=10` returns 10 users with `next_cursor`
- `curl /api/users?cursor=X&limit=10` returns the next 10

That spec costs ~300 tokens of context but eliminates 90% of the back-and-forth and wrong turns. For parallel agent fleets, specs are essential — each agent gets a spec in its prompt, works autonomously, and produces output that matches expectations without human supervision.

Spec-driven development workflow

  1. Write specs for each task (5–10 minutes per spec)
  2. Push specs to the board as task descriptions or linked files
  3. Launch agents — each claims a task and loads its spec
  4. Review output against acceptance criteria from the spec

The upfront investment in specs pays for itself because agents run faster (fewer wrong turns), produce more predictable output (clear acceptance criteria), and require less review (you check the spec, not the intent).

Context Engineering for Parallel Agent Fleets

Running one agent well is context engineering 101. Running 10 agents well — each on a different task, in separate git worktrees, for hours at a time — is where the discipline earns its name.

The three-layer model

Layer 1: Fleet-wide context (shared)
┌──────────────────────────────────────────────────────────└
│  CLAUDE.md (checked into repo, all agents see it)      │
│  Hooks (settings.json, enforced on all sessions)       │
│  Skills (.claude/skills/, loadable by any agent)       │
│  Subagents (.claude/agents/, available to all)         │
┐──────────────────────────────────────────────────────────┌

Layer 2: Task-specific context (per agent)
┌──────────────────────────────────────────────────────────└
│  Task spec (requirements, constraints, criteria)      │
│  Prompt (what this specific agent should do)           │
│  Working directory (isolated git worktree)             │
┐──────────────────────────────────────────────────────────┌

Layer 3: Runtime context (dynamic)
┌──────────────────────────────────────────────────────────└
│  Tool outputs (file reads, grep results, test runs)   │
│  Subagent summaries (delegated work returns)           │
│  Board state (shared task coordination)                │
│  External API responses                                │
┐──────────────────────────────────────────────────────────┌

Principle 1: identical agents, differentiated tasks

Every agent in your fleet should see the same CLAUDE.md, the same hooks, the same skills, and the same subagents. The only thing that differs is the task spec and working directory. This makes agents interchangeable — if one crashes, another can pick up its task without special configuration.

Principle 2: externalize shared state

Agents in separate worktrees cannot share files. Use a REST API (amux board, notes, or any HTTP endpoint) for any state that multiple agents need to read or write: task assignments, architectural decisions, discovered bugs, coordination signals.

# Agent claims a task atomically (no two agents grab the same task)
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"session":"'"$AMUX_SESSION"'"}' \
  $AMUX_URL/api/board/TASK-ID/claim

# Agent publishes a finding for others to see
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"title":"Found: auth module uses deprecated bcrypt API","session":"'"$AMUX_SESSION"'","status":"todo"}' \
  $AMUX_URL/api/board

Principle 3: keep per-agent context small

A fleet of 20 agents each consuming 150KB of context is burning through your rate limits fast. Target under 80KB average per session by:

  • Using subagents for exploratory work (grep, multi-file reads)
  • Keeping CLAUDE.md under 500 lines
  • Putting reference docs in skills, not inline
  • Writing tight specs that preempt wrong turns

The Context Budget: Measuring What You Spend

Context has a cost — both in tokens and in model attention. Think of your 200K context window as a budget:

Context surfaceTypical costWhen paid
System prompt (Claude Code built-in)~15K tokensEvery turn
CLAUDE.md (root)200–2,000 tokensEvery turn
Memory index500–2,000 tokensEvery turn
Skill frontmatter (all skills)50–200 tokens per skillEvery turn
Skill full content (loaded)500–5,000 tokensWhen loaded
Subagent prompt200–500 tokensOnce per delegation
Subagent response200–1,000 tokensOnce per delegation
File read (tool output)500–5,000 tokensEach read
Conversation historyGrows per turnCumulative

A session with a lean CLAUDE.md, 5 skill frontmatters, and minimal memory starts at ~20K tokens. That leaves 180K for actual work. A session with a bloated 3,000-line CLAUDE.md and 20 skills starts at ~40K, leaving only 160K — and the model's attention on your instructions is diluted by all the noise.

Diagnosing Failures by Context Surface

When an agent produces bad output, the question is not "why is the model dumb?" but "which context surface failed?" This reframe is the most important mental shift in context engineering:

SymptomLikely causeFix
Agent uses wrong build commandMissing from CLAUDE.mdAdd build command to CLAUDE.md
Agent pushes to main despite instruction not toSoft rule in CLAUDE.md (70% compliance)Move to PreToolUse hook (100%)
Agent doesn't know about deploy processDeploy docs not availableCreate a deploy skill
Agent's context fills up during codebase explorationNo delegation layerCreate an explorer subagent
Agent repeats a mistake it was corrected on last sessionNo cross-session persistenceSave feedback to memory
Agent builds wrong thing despite clear promptAmbiguous requirementsWrite a spec with acceptance criteria
Agent ignores CLAUDE.md instructions late in sessionInstructions pushed out by context growthTrim CLAUDE.md, use subagents to reduce tool output

The fix is almost never "write a better prompt." It's almost always "put the right information on the right context surface."

FAQ

What is context engineering?

Context engineering is the practice of curating everything an AI agent sees — system prompts, project rules, reference docs, tool outputs, memory — so it produces reliable, correct output. Unlike prompt engineering (crafting a single input), context engineering designs the entire information environment across multiple surfaces.

What is the difference between context engineering and prompt engineering?

Prompt engineering optimizes a single input message. Context engineering designs the entire information architecture: what the agent sees at startup (CLAUDE.md), what it can load on demand (skills), what it cannot do (hooks), who it can delegate to (subagents), and what persists across sessions (memory). Prompt engineering is one tool inside context engineering.

How long should CLAUDE.md be?

Under 500 lines for the root file. For each line, ask: would removing this cause the agent to make a mistake it cannot recover from by reading the code? If not, cut it. Use subdirectory CLAUDE.md files for directory-specific context. Use skills for large reference docs.

When should I use a hook instead of a CLAUDE.md instruction?

When the rule must be enforced 100% of the time. CLAUDE.md compliance is ~70%. For style preferences, that's fine. For "never push to main" or "never delete production data", 70% means a guaranteed incident eventually. Hooks execute outside the model and cannot be overridden.

What are Claude Code skills?

Markdown files in .claude/skills/ that contain reference documentation or workflow instructions. The agent sees only the skill's name and description (~200 tokens) at all times, and loads the full content on demand when relevant. Progressive disclosure for your knowledge base.

How does context engineering change when running parallel agents?

Three things change: (1) shared context must be externalized to a REST API or database, because agents run in separate worktrees; (2) per-agent context must be scoped to that agent's task via specs; (3) fleet-wide rules go in the root CLAUDE.md while task-specific instructions go in the prompt. The goal: identical agents, differentiated task context.

Is CLAUDE.md the same as context engineering?

No. CLAUDE.md is one of six context surfaces. Treating it as the entire strategy is the most common mistake teams make — they cram everything into CLAUDE.md, bloat the context window, and wonder why agent quality degrades. The fix is distributing information across all six surfaces.
