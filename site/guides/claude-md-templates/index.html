CLAUDE.md Templates

Copy-paste configs for every project type. Each template is battle-tested across thousands of amux agent sessions and explained line by line.

A good CLAUDE.md is the single highest-leverage thing you can do for AI coding agent quality. It's loaded at the start of every Claude Code conversation, persists across sessions, and gets version-controlled alongside your code. The difference between a project with a well-tuned CLAUDE.md and one without is the difference between an agent that understands your codebase and one that guesses.

These templates are starting points. Copy the one closest to your stack, delete lines Claude already gets right, and add rules for things it gets wrong. A good CLAUDE.md is 50–200 lines of high-signal corrections, not a comprehensive wiki.

How CLAUDE.md works

Claude Code loads instructions from up to three locations, in order of precedence:

LocationScopeCommit to git?
~/.claude/CLAUDE.mdPersonal — applies to all projectsNo
PROJECT_ROOT/CLAUDE.mdProject — shared with teammatesYes
PROJECT_ROOT/subdir/CLAUDE.mdDirectory — scoped overridesYes

Every line in CLAUDE.md costs context tokens on every conversation, so keep it tight. The templates below are designed to stay under the 500-line budget while covering the rules that actually change agent behavior.

The templates

Python FastAPI Service

Python
# my-api

FastAPI service. Python 3.12, uv for dependency management.

## Commands
- `uv run pytest` — run all tests
- `uv run pytest tests/test_foo.py::test_bar` — single test
- `uv run ruff check . --fix && uv run ruff format .` — lint + format
- `uv run mypy src/` — type check

## Structure
- `src/app/` — FastAPI app, routers, dependencies
- `src/app/models/` — SQLAlchemy models
- `src/app/schemas/` — Pydantic request/response schemas
- `tests/` — pytest tests (use `TestClient`, not `httpx` directly)

## Rules
- Always add Pydantic schemas for request AND response bodies.
- Use `Depends()` for shared logic, never import from route handlers.
- Database queries go in `src/app/crud/`, not in route handlers.
- Alembic for migrations. Never edit the models without creating a migration.
- Return 404 with `detail` message, not empty 200.
- Tests must not hit external services. Use `respx` for HTTP mocks.
- Type hints on every function signature. No `Any` unless unavoidable.
Why: Separates CRUD from routes, enforces migration discipline, and prevents the #1 FastAPI mistake (business logic in route handlers).

Next.js App Router

TypeScript
# my-app

Next.js 15 with App Router. TypeScript strict mode. Tailwind CSS v4.

## Commands
- `pnpm dev` — local dev server
- `pnpm test` — vitest
- `pnpm lint` — eslint + prettier
- `pnpm build` — production build (must pass before committing)

## Structure
- `app/` — App Router pages and layouts
- `components/` — shared React components
- `lib/` — utilities, API clients, constants
- `hooks/` — custom React hooks

## Rules
- Use Server Components by default. Only add "use client" when you need
  browser APIs, state, or event handlers.
- Data fetching happens in Server Components or Route Handlers, never
  in client components (no useEffect for data loading).
- Colocate styles with components using Tailwind classes. No CSS modules,
  no styled-components, no inline style objects.
- Every component file exports one component. Name the file after the
  component: `UserCard.tsx` exports `UserCard`.
- Use `next/image` for all images. Never use raw `<img>` tags.
- Forms use Server Actions. No client-side form submission to API routes.
- Error boundaries: every route segment gets a `error.tsx`.
- Loading states: use `loading.tsx` per route, not manual Suspense.
Why: Enforces the Server Components mental model and prevents the most common App Router mistakes (useEffect data fetching, missing error boundaries).

Django + DRF API

Python
# my-project

Django 5.1 with Django REST Framework. PostgreSQL. Celery for async tasks.

## Commands
- `python manage.py test` — run all tests
- `python manage.py test app.tests.test_views::ViewTestCase::test_create` — single
- `ruff check . --fix && ruff format .` — lint
- `python manage.py makemigrations && python manage.py migrate` — migrations

## Structure
- Each Django app is a self-contained domain. Don't import models across apps
  — use signals or a shared `core/` app for cross-cutting concerns.
- Serializers validate input. Views orchestrate. Models hold business logic.
- `tasks/` in each app for Celery tasks.

## Rules
- Never use `Model.objects.raw()` or string-format SQL. Always ORM.
- Wrap multi-step writes in `transaction.atomic()`.
- All list endpoints must be paginated. Use `PageNumberPagination`.
- Celery tasks must be idempotent. Assume they will be retried.
- Test with `APIClient`, not raw Django test client.
- Settings split: `settings/base.py`, `settings/dev.py`, `settings/prod.py`.
  Never `if DEBUG` in application code.
- Permissions go on the view via `permission_classes`, not in serializers.
Why: Prevents cross-app coupling, enforces the Django layering model, and catches the two most common DRF bugs (missing pagination, non-idempotent tasks).

Go Microservice

Go
# my-service

Go 1.23 microservice. net/http stdlib router. PostgreSQL via pgx.

## Commands
- `go test ./...` — all tests
- `go test ./internal/handler/ -run TestCreateUser` — single test
- `golangci-lint run` — lint
- `go build -o bin/server ./cmd/server` — build

## Structure
- `cmd/server/` — main.go, wiring, config loading
- `internal/handler/` — HTTP handlers (thin: parse, validate, call service, respond)
- `internal/service/` — business logic (no HTTP types)
- `internal/repo/` — database layer (pgx queries, no business logic)
- `internal/model/` — domain types (plain structs, no dependencies)

## Rules
- Handlers accept interfaces, not concrete types. Wire in main.go.
- Errors bubble up. Only handlers decide HTTP status codes.
- Context flows down: `func (s *Service) Create(ctx context.Context, ...)`.
- No global state. No init() functions. No package-level vars except `var ErrX`.
- Tests use testcontainers for Postgres. No mocking the DB layer.
- Every exported function has a one-line GoDoc comment.
- Use `slog` for structured logging. No fmt.Println, no log.Printf.
Why: Enforces Go's "accept interfaces, return structs" idiom and the clean handler/service/repo layering that prevents spaghetti in growing services.

Node.js Express API

TypeScript
# my-api

Express.js REST API. TypeScript 5.5 strict mode. Prisma ORM. pnpm.

## Commands
- `pnpm test` — vitest
- `pnpm lint` — eslint
- `pnpm db:migrate` — prisma migrate dev
- `pnpm build` — tsc

## Structure
- `src/routes/` — Express routers (one file per resource)
- `src/services/` — business logic (no Express types)
- `src/middleware/` — auth, validation, error handling
- `prisma/schema.prisma` — database schema (source of truth)

## Rules
- Route handlers call services. Services call Prisma. Never skip layers.
- Validate all request bodies with zod schemas in `src/schemas/`.
- Every route has explicit error handling. Use the global error middleware,
  don't try/catch in individual routes.
- Prisma queries use `select` or `include` explicitly. No implicit full-model fetches.
- Environment variables accessed only through `src/config.ts`. Never
  inline `process.env.X` in business logic.
- Integration tests use a test database. `beforeAll` runs migrations,
  `afterAll` drops the schema.
- No default exports. Always named exports.
Why: Prevents the Express.js antipattern of business logic in route handlers and enforces consistent error handling across all endpoints.

Turborepo Monorepo

Full-stack
# my-monorepo

Turborepo monorepo. pnpm workspaces. TypeScript throughout.

## Commands
- `pnpm test` — run all workspace tests via turbo
- `pnpm --filter @app/web test` — test one workspace
- `pnpm lint` — eslint across all packages
- `pnpm build` — turbo build (respects dependency graph)

## Workspaces
- `apps/web` — Next.js frontend
- `apps/api` — Express backend
- `packages/ui` — shared React component library
- `packages/shared` — types, utils, constants shared across apps
- `packages/config-*` — shared ESLint, TypeScript, Tailwind configs

## Rules
- NEVER import directly between apps (web ↛ api, api ↛ web). Shared
  code goes in a package under `packages/`.
- Shared types live in `packages/shared/src/types/`. Both apps import from there.
- Adding a dependency? Add to the specific workspace, not the root.
  `pnpm --filter @app/web add lodash`, never `pnpm add lodash` at root.
- Each package has its own `tsconfig.json` extending the shared base.
- CI runs `turbo build --filter=[HEAD^1]` for affected-only builds.
  Don't break the turbo dependency graph.
- `packages/ui` components are pure. No API calls, no data fetching,
  no business logic inside UI components.
Why: The #1 monorepo mistake is cross-app imports. This template makes the boundary rules explicit and prevents turborepo cache invalidation problems.

Python Data Pipeline

Data / ML
# my-pipeline

ETL pipeline. Python 3.12, Airflow for orchestration, Polars for transforms.

## Commands
- `uv run pytest tests/` — all tests
- `uv run ruff check . --fix && uv run ruff format .` — lint
- `uv run mypy src/` — type check
- `airflow dags test my_dag 2026-01-01` — backfill test

## Structure
- `dags/` — Airflow DAG definitions (thin: just task wiring)
- `src/extractors/` — data source connectors
- `src/transforms/` — Polars transform functions (pure, testable)
- `src/loaders/` — destination writers (S3, Postgres, BigQuery)
- `tests/fixtures/` — sample input CSVs/JSONs for deterministic testing

## Rules
- Transform functions are PURE: take a DataFrame, return a DataFrame.
  No I/O inside transforms.
- DAG files only wire tasks. Business logic lives in `src/`, not in DAGs.
- Every transform has a test with a fixture file. Test the shape AND values.
- Use Polars, not Pandas. If a library requires Pandas, convert at the
  boundary: `df.to_pandas()` / `pl.from_pandas()`.
- Credential loading via `src/config.py` only. No hardcoded keys, no inline
  `os.environ` calls.
- Partitioned writes: always write with date partitioning. No full-table
  overwrites unless the DAG name starts with `rebuild_`.
Why: Enforces the pure-transform pattern that makes data pipelines testable, and prevents the DAG-as-monolith antipattern.

ML Training Project

Data / ML
# my-model

ML training codebase. PyTorch 2.x, Hydra for config, W&B for experiment tracking.

## Commands
- `uv run pytest tests/` — unit tests (fast, no GPU)
- `uv run python train.py +experiment=baseline` — train with Hydra config
- `uv run ruff check . --fix && uv run ruff format .` — lint

## Structure
- `configs/` — Hydra YAML configs (model, data, training, experiment)
- `src/models/` — model definitions (nn.Module subclasses)
- `src/data/` — datasets, dataloaders, augmentations
- `src/training/` — training loop, callbacks, metrics
- `notebooks/` — exploration only, never imported by src/

## Rules
- Hyperparameters come from Hydra configs. NEVER hardcode learning rates,
  batch sizes, or hidden dims in Python files.
- Every training run logs to W&B. No silent runs.
- Model forward() takes tensors, returns tensors. No data loading or
  preprocessing inside model classes.
- Reproducibility: set seeds in config, log the full Hydra config to W&B.
- Notebooks are throwaway. Anything worth keeping gets extracted to src/.
- GPU memory: always use `torch.cuda.amp` for mixed precision. Profile
  before optimizing — no premature gradient checkpointing.
- Data loading: `num_workers > 0` and `pin_memory=True` by default.
Why: Prevents "it worked on my laptop" by enforcing config-driven training and full experiment logging from day one.

React Native (Expo)

Mobile
# my-mobile-app

React Native with Expo SDK 52. TypeScript strict. Expo Router for navigation.

## Commands
- `npx expo start` — dev server
- `pnpm test` — jest
- `pnpm lint` — eslint
- `eas build --platform ios --profile preview` — build

## Structure
- `app/` — Expo Router file-based routes
- `components/` — shared UI components
- `hooks/` — custom React hooks
- `lib/` — API client, storage, constants
- `assets/` — images, fonts

## Rules
- Use Expo SDK APIs over raw React Native when both exist
  (e.g., expo-camera, not react-native-camera).
- Navigation: Expo Router file-based routes. No manual stack configuration.
- Styles: StyleSheet.create() only. No inline style objects in JSX
  (performance: inline objects create new refs every render).
- All API calls go through `lib/api.ts`. Components never call fetch directly.
- Test components with @testing-library/react-native. No enzyme, no shallow.
- Platform branching: use `.ios.tsx` / `.android.tsx` file suffixes
  for platform-specific code, not `Platform.select()` inline.
- Images must have explicit width/height. No unsized images.
Why: Enforces Expo best practices and the most impactful performance rule (no inline style objects), which causes 90% of "why is my list janky" bugs.

Terraform Infrastructure

Infra
# my-infra

Terraform infrastructure. AWS. Modules pattern with environment workspaces.

## Commands
- `terraform fmt -recursive` — format
- `terraform validate` — syntax check
- `terraform plan -out=plan.tfplan` — always save plan before apply
- `tflint --recursive` — lint
- `checkov -d .` — security scan

## Structure
- `modules/` — reusable Terraform modules (VPC, ECS, RDS, etc.)
- `environments/dev/` — dev environment root
- `environments/staging/` — staging root
- `environments/prod/` — prod root (NEVER modify without a plan review)

## Rules
- NEVER run `terraform apply` without saving a plan first.
- All resources must be tagged: `Environment`, `Team`, `ManagedBy=terraform`.
- No hardcoded AMI IDs, ARNs, or account numbers. Use data sources or variables.
- State is in S3 with DynamoDB locking. Never use local state.
- Secrets go in AWS Secrets Manager, referenced via data source. NEVER
  put secrets in .tf files or terraform.tfvars.
- Module outputs must include the resource ID and ARN at minimum.
- Use `for_each` over `count` for resources that might be removed
  individually (count reindexing destroys/recreates).
- Environments share modules but NEVER share state files.
Why: The for_each vs count rule alone prevents more accidental resource destruction than any other single Terraform practice.

Python CLI Tool

Python
# my-cli

Python CLI tool distributed via pip. Click for argument parsing. Rich for output.

## Commands
- `uv run pytest` — tests
- `uv run ruff check . --fix && uv run ruff format .` — lint
- `uv run mypy src/` — type check
- `uv run python -m my_cli --help` — test locally

## Structure
- `src/my_cli/` — package root
- `src/my_cli/cli.py` — Click command definitions (thin)
- `src/my_cli/core.py` — business logic (no Click imports)
- `src/my_cli/output.py` — Rich formatting and display logic

## Rules
- Click commands are thin wrappers: parse args, call core, format output.
  No business logic in cli.py.
- Core functions return data (dicts, dataclasses). Display logic lives in
  output.py. This lets us test core functions without capturing stdout.
- Exit codes: 0 = success, 1 = user error (bad input), 2 = runtime error.
  Always sys.exit() with a code, never bare exit().
- Stderr for errors and progress. Stdout for actual output (pipeable).
- Config file: `~/.config/my-cli/config.toml`. Use tomli to parse.
  CLI flags override config file values.
- Never prompt for input in non-interactive mode. Check `sys.stdin.isatty()`.
Why: The stdout/stderr separation rule is the most commonly broken CLI convention — enforcing it here means your tool works correctly in shell pipelines from day one.

Rust Application

Rust
# my-app

Rust application. Cargo workspace. tokio async runtime.

## Commands
- `cargo test` — all tests
- `cargo test --package my-core -- test_name` — single test
- `cargo clippy --all-targets -- -D warnings` — lint (must pass, zero warnings)
- `cargo build --release` — optimized build
- `cargo fmt --check` — format check

## Structure
- `crates/my-core/` — domain logic (no I/O, no async)
- `crates/my-server/` — HTTP server (axum), wires core to endpoints
- `crates/my-cli/` — CLI binary (clap), wires core to commands

## Rules
- Core crate has ZERO I/O dependencies. It operates on types, not connections.
  Server and CLI crates handle I/O and pass data into core.
- Use thiserror for library error types, anyhow for binary error types.
  Never `.unwrap()` outside of tests.
- Async at the boundary only. Core logic is sync where possible.
- `#[must_use]` on all functions that return Result or Option.
- Clippy at deny-warnings level. Fix clippy warnings, don't #[allow] them.
- Dependencies: check if there's a std equivalent before adding a crate.
  No leftpad incidents.
Why: The zero-I/O core crate pattern is the key to testable Rust — it means your business logic compiles and tests in milliseconds without standing up a server.

Single-File Project (amux-style)

Meta
# my-project

Single-file project: everything lives in `server.py` (Python server + inline HTML/CSS/JS).

## Workflow
- Commit after every completed task. Don't batch changes.
- The server auto-restarts on file save, so changes are live immediately.
- Always verify syntax after edits: `python3 -c "import ast; ast.parse(open('server.py').read())"`

## Rules
- One file. Don't extract modules, don't create packages, don't split HTML
  into templates. Complexity lives in one place where grep works.
- HTML/CSS/JS is inline in Python string literals. Use f-strings for
  dynamic content, raw strings for static blocks.
- Routes are elif chains in the request handler. Don't add a framework.
- SQLite for persistence. Database file in ~/.my-project/data.db.
- No external Python dependencies beyond stdlib. If you need something,
  vendor it or rewrite the 20 lines you actually need.
Why: For projects that should stay simple. The key rule — "don't extract modules" — fights the instinct to prematurely organize and keeps the whole system greppable.

Personal Global (~/.claude/CLAUDE.md)

Meta
# Global Instructions

## Behavior
- Don't explain what you're about to do. Just do it.
- Don't add comments unless the why is non-obvious.
- Don't refactor adjacent code unless I ask.
- Commit after finishing each task, not in batches.
- Use the existing code style, even if you'd do it differently.

## Coding preferences
- Prefer simple solutions over clever ones.
- Prefer flat code over nested code (early returns, guard clauses).
- Prefer explicit imports over wildcards.
- Prefer composition over inheritance.
- Three similar lines is better than a premature abstraction.

## Don't
- Don't add error handling for scenarios that can't happen.
- Don't add backwards-compatibility shims. Just change the code.
- Don't write multi-line comments or docstrings.
- Don't create documentation files unless I ask.
- Don't add features beyond what I asked for.
Why: Global config controls agent personality across all projects. These rules eliminate the most common annoyances: over-commenting, scope creep, and unnecessary abstraction.

Multi-Agent (amux sessions)

Meta
# my-project

Multi-agent project managed by amux. Multiple Claude Code sessions
work on this codebase simultaneously.

## Coordination
- Check the board before starting: `amux board list`
- Claim your task: `amux board doing TASK-ID`
- Mark done when finished: `amux board done TASK-ID`
- If you need another session's help, post a board task with their session name.

## Git rules (critical for parallel agents)
- ALWAYS create a feature branch: `git checkout -b $AMUX_SESSION/feature-name`
- NEVER push to main directly.
- NEVER rebase or force-push. Merge main into your branch if needed.
- Commit early and often. Small commits are easier to merge.
- If you encounter a merge conflict, stop and describe it on the board.

## Boundaries
- Only modify files related to your assigned task.
- If you need to change a shared file (config, schema, migrations),
  coordinate on the board first.
- Don't install new dependencies without a board discussion.

## Communication
- Before starting, peek at active sessions: `amux peek <session>`
- If stuck for more than 5 minutes, post a help request on the board.
Why: Multi-agent work fails when agents step on each other. This template enforces the branch-per-agent pattern and coordination-before-shared-file-edits rule.

iOS Swift (SwiftUI)

Mobile
# my-ios-app

iOS app. Swift 6, SwiftUI, Swift Package Manager. Minimum iOS 17.

## Commands
- `xcodebuild test -scheme MyApp -destination 'platform=iOS Simulator,name=iPhone 16'` — tests
- `swift build` — build check (SPM packages)
- `swiftlint` — lint

## Structure
- `Sources/Features/` — one folder per feature (screen + view model + tests)
- `Sources/Core/` — shared utilities, networking, persistence
- `Sources/Design/` — design system components, colors, typography

## Rules
- MVVM pattern. Views are dumb. ViewModels are @Observable classes.
- Views never call URLSession, CoreData, or any I/O directly.
- Navigation via NavigationStack with typed destinations. No NavigationLink
  with destination views inline.
- Use Swift concurrency (async/await). No Combine for new code.
  Existing Combine code can stay, but new reactive code uses AsyncSequence.
- Structured concurrency: use TaskGroup over detached Tasks.
- Preview every view. If a view can't be previewed in isolation, its
  dependencies are too tangled.
- Accessibility: every interactive element has an accessibilityLabel.
Why: Enforces @Observable over ObservableObject (Swift 6 standard), prevents the NavigationLink-with-inline-destination antipattern, and mandates previews as a design health check.

Chrome Extension (Manifest V3)

TypeScript
# my-extension

Chrome Extension. Manifest V3. TypeScript. Vite for bundling.

## Commands
- `pnpm dev` — build + watch (load unpacked from dist/)
- `pnpm test` — vitest
- `pnpm build` — production build
- `pnpm lint` — eslint

## Structure
- `src/background/` — service worker (event-driven, no persistent state)
- `src/content/` — content scripts (runs in page context)
- `src/popup/` — popup UI (React)
- `src/options/` — options page
- `src/shared/` — types and utilities shared across contexts

## Rules
- Service worker is EVENT-DRIVEN. No persistent state in memory. Use
  chrome.storage.session for ephemeral state, chrome.storage.local for persistent.
- Message passing between contexts uses typed messages. Define all message
  types in `src/shared/messages.ts`.
- Content scripts must be minimal. Heavy logic runs in the service worker
  and communicates via chrome.runtime.sendMessage.
- Never use `chrome.tabs.executeScript` with string code. Always use
  registered content scripts or `func` parameter.
- Permissions: request only what you need. Use optional_permissions for
  features users might not enable.
- CSP: popup and options pages have strict CSP. No inline scripts, no eval.
Why: MV3's service worker lifecycle is the biggest foot-gun. This template prevents the #1 extension crash: assuming the service worker stays alive between events.

Kubernetes + Helm Charts

Infra
# my-k8s

Kubernetes manifests and Helm charts. ArgoCD for GitOps deployment.

## Commands
- `helm template my-app charts/my-app/ -f values-dev.yaml` — render templates
- `helm lint charts/my-app/` — lint chart
- `kubeval <(helm template ...)` — validate against K8s schema
- `kubectl diff -f manifests/` — diff against live cluster

## Structure
- `charts/my-app/` — Helm chart for the main application
- `charts/shared/` — shared library chart (templates only)
- `manifests/` — raw manifests for infra components (cert-manager, ingress)
- `argocd/` — ArgoCD Application definitions

## Rules
- NEVER `kubectl apply` to production. All prod changes go through ArgoCD
  (commit to main → ArgoCD syncs).
- Resource limits on every container. No unbounded pods.
- Liveness and readiness probes on every deployment. They must be different
  endpoints (readiness = can serve traffic, liveness = not deadlocked).
- Secrets via External Secrets Operator pulling from AWS Secrets Manager.
  Never put base64 secrets in YAML files.
- Pod disruption budgets on every production deployment.
- Anti-affinity rules for multi-replica deployments (spread across nodes).
- Values files per environment: values-dev.yaml, values-staging.yaml,
  values-prod.yaml. Never use `--set` in CI.
Why: The liveness-vs-readiness distinction is the most misunderstood K8s concept. Getting it wrong causes cascading restarts during normal deploys.

SvelteKit Full-stack

Full-stack
# my-app

SvelteKit 2. TypeScript. Tailwind CSS. Drizzle ORM with PostgreSQL.

## Commands
- `pnpm dev` — dev server
- `pnpm test` — playwright e2e + vitest unit
- `pnpm lint` — eslint + svelte-check
- `pnpm build` — production build

## Structure
- `src/routes/` — SvelteKit file-based routes
- `src/lib/server/` — server-only code (DB, auth, email)
- `src/lib/components/` — shared Svelte components
- `src/lib/stores/` — Svelte stores for client state

## Rules
- Data loading: always use +page.server.ts load functions. No fetch() in

- Every dynamic route has a +error.svelte page.

Subdirectory Override (tests/)

Meta
# Tests directory rules

## When editing files in tests/
- Test file names: `test_<module>.py` matching the source file they cover.
- Use pytest fixtures from conftest.py. Don't create test data inline.
- Integration tests that need a database use the `db` fixture (auto-rollback).
- Mocking: only mock I/O boundaries (HTTP, filesystem, clock). Never mock
  internal functions — if the test is hard to write without mocking internals,
  the production code has a coupling problem.
- No sleep() in tests. Use `freezegun` for time-dependent tests.
- Assertion messages on every assert: `assert result == 42, f"Expected 42, got {result}"`.
- Test names describe the behavior: `test_returns_404_when_user_not_found`,
  not `test_get_user_3`.
Why: Place this in `tests/CLAUDE.md` to override project-level rules with test-specific conventions. The "never mock internals" rule prevents brittle tests that break on every refactor.

Writing your own CLAUDE.md

Start from the template closest to your stack, then iterate using this process:

  1. Run Claude Code for a day without CLAUDE.md. Take notes on every mistake it makes or convention it misses.
  2. Write one rule per mistake. Be specific: "use uv, not pip" is better than "follow project conventions."
  3. Delete rules Claude follows naturally. If removing a line doesn't change behavior, it's wasting context tokens.
  4. Test with multiple sessions. Run 3–5 amux sessions on the same task and check if all agents follow the rules consistently.
  5. Version control it. Your CLAUDE.md should evolve with your codebase. Review it in PRs like any other config file.

Documenting the entire architectureBurns context tokens on every conversation; Claude can read the codeOnly document things that aren't obvious from reading the code

Using templates with amux multi-agent sessions

The biggest payoff from a good CLAUDE.md is in multi-agent workflows. When you run 10 parallel Claude Code sessions via amux, all of them load the same CLAUDE.md. This means:

  • Consistent style across every agent's output without per-session prompting
  • No conflicting approaches — agents agree on test frameworks, naming, file structure
  • The Multi-Agent template above adds git coordination rules that prevent merge conflicts
  • Subdirectory overrides let you scope different rules to different parts of the codebase

If you're running multiple agents, start with the template for your stack plus the Multi-Agent template as a project-level addition. The combination ensures both code quality and coordination.

Run multiple agents with consistent CLAUDE.md

amux manages parallel Claude Code sessions that all share your project's CLAUDE.md. Coordinate work via kanban board, monitor output in a web dashboard.

git clone https://github.com/mixpeek/amux && cd amux && ./install.sh
amux register myproject --dir ~/Dev/myproject --yolo
amux start myproject
amux serve  # → https://localhost:8822
