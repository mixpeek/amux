Scaling to 50+ AI Coding Agents

amux runs a fleet of 50+ Claude Code, Codex, or Gemini CLI agents from a single dashboard or your phone — with atomic task claiming, self-healing watchdog, and live monitoring. Here is the architecture you need to do it reliably. Updated July 2026.

What Changes Above 10 Agents

Running 3-5 AI coding agents is mostly about keeping them pointed at different tasks. Scaling to 50 introduces four new failure modes that don't exist at small scale:

  • Task collisions. Two agents claim the same work item from a shared list, do overlapping work, and produce conflicting commits.
  • Rate limit cascades. Agents start simultaneously and spike API requests — all hit 429 errors, back off at the same time, then spike again in lockstep.
  • Monitoring blindness. You can watch 5 terminal panes. You cannot watch 50. Without a unified dashboard, you won't know which agents are stuck, which finished, or which are burning money on repeated failures.
  • Crash accumulation. At 10 agents, 1 crash is a 10% loss. At 50, expecting 2-5 crashes per overnight run is normal — you need automated recovery, not manual restarts.

amux is designed to solve all four. The rest of this guide explains how.

Hardware Sizing

Each agent runs in its own tmux session with a Claude Code (or Codex/Gemini) process. Memory consumption per agent:

tmux session ~20MB Scrollback buffer + pane state
Claude Code process ~150-250MB Node.js runtime + context cache
Working repo clone ~50-200MB Depends on repo size + git objects
Total per agent ~300-500MB Use 400MB as your planning number

CPU is rarely the bottleneck — agents spend most of their time waiting for API responses. 4-8 cores is sufficient for 50 agents. Disk I/O matters for large codebases with many repo clones; use SSD and consider a shared git remote with per-agent worktrees (git worktree isolation guide).

Atomic Task Queue (No Duplicate Work)

The most important architectural decision for a large fleet is how agents claim work. Without atomic claiming, two agents will pick the same task, both spend an hour on it, and you get a merge conflict.

amux solves this with SQLite compare-and-swap. The claim endpoint is a single SQL statement:

UPDATE board_items
SET status='doing', session=?, claimed_at=?
WHERE id=? AND status='todo'

If the row was already claimed (status != 'todo'), the UPDATE affects 0 rows and the agent gets a 409 response — it skips to the next task. This requires no distributed lock manager, no Redis, no external coordination service. It works with SQLite's default WAL mode even with 50 concurrent writers.

To use atomic claiming in your agent loop:

# In CLAUDE.md or as a startup prompt for each agent:
# 1. Find unclaimed tasks for this session:
curl -sk $AMUX_URL/api/board | python3 -c "
import json,sys,os
items = json.load(sys.stdin)
todo = [i for i in items if i['status']=='todo']
if todo: print(todo[0]['id'])
"

# 2. Claim atomically (409 = already taken, try next):
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"session":"'"$AMUX_SESSION"'"}' \
  $AMUX_URL/api/board/TASK-ID/claim

# 3. Work on it, then mark done:
curl -sk -X PATCH -H 'Content-Type: application/json' \
  -d '{"status":"done","desc":"Result: ..."}' \
  $AMUX_URL/api/board/TASK-ID

API Rate Limit Strategies

Rate limits are the most common obstacle when scaling beyond 10 agents. The strategies differ by provider:

Provider Limit type Strategy for 50 agents
Anthropic Max ($200/mo) Token budget (generous) Stagger starts 5-10s each; best for overnight runs
Anthropic API Tier 4 ~60 RPM, 2M TPM Sonnet Cap SYNC_POOL_CAPACITY at 50; use Sonnet not Opus
OpenAI Codex Tier-based; generous at high tiers Mix with Claude for parallel capacity
Antigravity CLI (Google) 1500 RPD free; paid tiers higher Use for longer-running agents with lower frequency

Staggered starts are the single most impactful change. Instead of launching 50 agents simultaneously:

# Stagger launches — 5 seconds between each
for i in $(seq 1 50); do
  amux start "agent-$i" &
  sleep 5
done

This spreads the initial burst of API calls across 4 minutes, avoiding the synchronized spike that triggers rate limiting. After the first few turns, agents naturally desynchronize as tasks have different lengths.

YOLO Mode + Self-Healing at Scale

At 50 agents, you cannot manually approve every tool-use prompt. YOLO mode is not optional — it is required. Enable it when registering sessions:

amux register myproject --dir ~/Dev/myproject --yolo
amux start myproject

As of July 2026, YOLO mode is the default for new session registrations. The watchdog detects the unique "Esc to cancel" marker that only appears on tool-approval prompts and auto-answers, keeping agents unblocked.

The self-healing watchdog handles the most common failure patterns at scale:

  • Process crash: Detects dead tmux pane → restarts process → replays last message
  • Context overflow: Detects context approaching limit → auto-compacts → continues
  • Stuck agent: No output for configurable timeout → restart with fresh context
  • API rate limit 429: Detects error output → exponential backoff → retry

Expected crash rate at 50-agent scale: 2-8 per 8-hour overnight run, all auto-recovered. The watchdog logs every restart with timestamp and reason, visible in the amux board's session activity feed.

Monitoring a 50-Agent Fleet

The amux dashboard provides several monitoring views for large fleets:

Board view (primary): The kanban shows every task with its claiming agent and status. At a glance you can see how many tasks are in todo vs. doing vs. done, and which agents are actively working. The board auto-updates via SSE — no refresh required.

Workspace tiled view: Arranges multiple session peek panels side-by-side in the browser. Useful for spot-checking 6-12 agents at once. On a 4K display you can tile 16 panes.

iOS app / mobile PWA: The iOS app lets you check fleet status from anywhere. Useful for checking in on an overnight run from bed — swipe through sessions, read output, send messages. The board tab gives the same kanban view as the desktop.

Schedules tab in peek: Each session's peek panel now includes a Schedules tab. For agents running recurring tasks, you can see schedule status without leaving the context of that session.

For alerting, set up a simple watchdog notification:

# Schedule an hourly health check that posts to your board
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{
    "title": "Fleet health check",
    "session": "monitor",
    "command": "Check board for stuck tasks (doing > 2h), restart any stuck agents, post summary",
    "schedule_expr": "every 1h"
  }' \
  $AMUX_URL/api/schedules

Mixing Providers (Claude + Codex + Gemini)

One of amux's differentiators at scale is native support for multiple AI runtimes simultaneously. Rather than hitting a single provider's rate limit with 50 agents, you can distribute across providers:

  • Claude Code (Anthropic Max): Best for complex reasoning, architectural decisions, open-ended refactors. Run 20-30 sessions.
  • Codex CLI (OpenAI): Strong at targeted code generation, test writing. Run 10-15 sessions for a different rate limit bucket.
  • Antigravity CLI (Google/Gemini): Large context window useful for whole-codebase analysis. Run 5-10 sessions for review tasks.

All three appear in the same amux board and dashboard. The kanban does not care which runtime claimed a task — agents from different providers compete equally for tasks from the same queue. See the multi-runtime setup guide for configuration details.

FAQ

How much RAM do I need to run 50 AI coding agents?

Plan for 400MB per agent. For 50 agents: 20GB RAM consumed, so a 32GB machine gives comfortable headroom. A 64GB machine can run 100+ agents with room for the OS, dashboard, and burst activity.

How do I avoid API rate limit errors with 50 parallel agents?

Stagger session starts (5-10 seconds between each), set SYNC_POOL_CAPACITY to match your RPM tier, and mix providers. The Claude Max plan ($200/month) has the most permissive rate limits for high-concurrency use.

How do I prevent 50 agents from working on the same task?

Use amux's atomic task claiming: POST /api/board/TASK-ID/claim. This is a SQLite compare-and-swap — only one agent can claim a task in any given instant. Others receive 409 and skip to the next task.

What happens when agents crash at scale?

amux's watchdog detects crashes automatically and restarts the process, replaying the last message. Context overflow triggers auto-compaction before it becomes a crash. At 50-agent scale, expect 2-8 auto-recoveries per overnight run.

Can I mix Claude, Codex, and Gemini agents in the same fleet?

Yes. amux supports Claude Code, Codex CLI, and Antigravity CLI (Google's Gemini successor) in the same dashboard and board. Agents from different providers compete for the same task queue — useful for spreading load across rate limit buckets.
