How to Review AI-Generated Code

Your agents shipped 10 branches overnight. Now you have 90 minutes before standup. This is the playbook for reviewing AI-generated code at scale — what to check, what to skip, and where the bugs hide.

The Shift: Writer to Reviewer

The DORA 2026 report confirmed what developers already felt: the bottleneck has shifted from writing code to verifying it. With 92% of developers using AI coding tools daily and AI generating over 40% of code at adopting companies, the developer's primary job is no longer authoring — it's reviewing.

Fortune called it the "supervisor class". JetBrains found that senior developers now spend more time reviewing agent output than writing code themselves. The problem: most developers were never trained to review 10x more code than they can write.

This guide is the practical side. Not the think piece about role transformation. The actual workflow for a developer who wakes up to 10 completed agent branches and needs to ship by lunch.

Time Budgets: How Long Each Review Should Take

The most common mistake is spending equal time on every branch. Not all agent output deserves the same scrutiny. Budget your review time by task type:

Test generation

3-5 min

Run the tests. Check coverage numbers. Skim for assertions that test the mock instead of real behavior. If tests pass and coverage improved, merge.

Documentation

2-3 min

Spot-check factual accuracy against current code. Look for hallucinated function signatures or deprecated API references. Lowest risk.

Simple endpoints / CRUD

5-8 min

Verify input validation, auth checks, error handling at boundaries. Run the tests. Check the database query patterns for N+1.

Business logic

10-15 min

This is where human judgment matters most. Trace the logic against the spec. Check edge cases explicitly. Verify that the agent's interpretation of requirements matches yours.

Auth / security

12-18 min

The highest-risk category. Check every auth boundary, every input sanitization point, every secret reference. AI agents reproduce insecure patterns from training data. Don't trust, verify.

Refactoring

5-10 min

Verify the refactored code is functionally equivalent. Run the existing tests (they should all still pass). Check that no public API signatures changed unintentionally.

With these budgets, 10 branches from an overnight run take 60-90 minutes to review. If you're regularly exceeding that, your task specs need to be more granular.

Triage: What to Review First

Don't review in completion order. Triage by risk:

  1. Shared files first. If two agents touched the same file (types, schemas, config), review those branches first. Incompatible changes here cascade into every other branch.
  2. Core business logic second. The feature's critical path. If this is wrong, nothing else matters.
  3. Security-sensitive code third. Auth, payment, PII handling. These have the highest blast radius if bugs ship.
  4. Endpoints and CRUD fourth. Standard patterns, lower risk, usually straightforward.
  5. Tests and documentation last. Lowest risk, fastest to review, and if earlier branches need rework, these may need to change anyway.

The AI Code Review Checklist

Print this. Use it until the checks become automatic. These are the failure modes specific to AI-generated code — not general code review advice.

Before reading any code

  • Run the agent's tests. If they fail, stop reviewing and send the agent back with the failure. Don't debug for the AI.
  • Check the diff size vs spec scope. A spec that says "add one endpoint" shouldn't produce a 500-line diff. Large diffs from small specs mean the agent over-engineered.
  • Read the agent's commit messages. AI agents usually summarize what they did. If the summary doesn't match the spec, that's a red flag.

While reading the diff

  • Verify every import exists. AI agents hallucinate package names, deprecated APIs, and functions that don't exist in the version you use. grep for any import you don't recognize.
  • Check error handling at system boundaries. AI writes happy-path code. Look at every external call (database, API, file system) and verify there's error handling. Missing try/catch on I/O is the #1 bug category.
  • Look for hardcoded values. API keys, URLs, credentials, magic numbers. AI agents copy patterns from training data, including hardcoded examples that should be environment variables.
  • Count the abstraction layers. AI loves creating unnecessary interfaces, abstract base classes, factory patterns, and helper modules. If you can inline a helper and the code reads better, the abstraction shouldn't exist.
  • Verify test assertions are meaningful. AI-generated tests often assert that a function returns a value (obvious) but miss asserting the correct value, error cases, or side effects. Check that tests would fail if the code broke.
  • Check database queries. Look for N+1 patterns: a query inside a loop, or an ORM call that triggers lazy loading. AI generates code that works with 10 records and dies with 10,000.

After approving the code

  • Run the full test suite, not just the agent's tests. Agent code that passes its own tests can break existing tests it doesn't know about.
  • Check for unintended file changes. Agents sometimes reformat files, update lockfiles, or modify configs as side effects. git diff --stat catches these.
  • Verify the branch merges cleanly. If it doesn't, check whether the conflict is with another agent's branch before resolving.

Red Flags: When to Reject Immediately

Stop reviewing and restart the agent if you see any of these

  • The agent modified files outside the spec scope. If you asked for an API endpoint and it refactored the auth middleware, that's scope creep. It may have introduced bugs in code that was working.
  • The diff contains commented-out code with "TODO" or "FIXME". The agent couldn't solve the problem and left a marker. This is half-finished work disguised as a deliverable.
  • Tests pass but don't test the actual requirement. AI agents sometimes write tests that verify the code runs without errors but never check the output. A test for "create user" that doesn't verify the user was actually created is worthless.
  • The agent created a new abstraction layer that didn't exist before. Unless the spec explicitly asked for it, a new abstract base class, service layer, or helper module is the agent solving a problem you didn't have.
  • Security-sensitive code with no input validation. An endpoint that accepts user input and passes it directly to a database query, shell command, or template renderer. Reject. No exceptions.
  • The agent's approach is fundamentally wrong. If you'd need to rewrite more than 30% of the diff to fix it, restart from scratch with a better spec. Iterating on a bad approach costs more than starting over.

The 6 AI Code Bug Patterns

Research from CSA, CodeScene, and the DORA team shows that AI-generated code bugs cluster into six patterns. Learn to spot them and your review speed doubles.

1. Hallucinated imports

The agent imports a package, function, or API that doesn't exist in your dependency tree. Looks plausible. Compiles in the agent's imagination. Fails on npm install or first import.

Catch it: Run npm ls / pip freeze and diff against new imports.

2. Happy-path-only logic

Code that handles the normal case beautifully and crashes on the first edge case. No null checks, no empty array handling, no timeout logic. The agent tested with ideal inputs and declared victory.

Catch it: Ask "what happens if this is null/empty/huge?"

3. Security anti-patterns

SQL injection via string concatenation. XSS via innerHTML. Secrets in source. Missing auth middleware on new routes. The agent reproduces patterns from training data, including decades of insecure code.

Catch it: Search for innerHTML, string interpolation in queries, eval().

4. Over-abstraction

A simple task produces a factory, an interface, three helper classes, and a configuration system. AI agents default to "enterprise patterns" when simple code would work. More abstraction = more surface area for bugs.

Catch it: Count new files. A 1-endpoint spec shouldn't create 5 files.

5. N+1 and performance traps

ORM calls inside loops. Unbounded queries without pagination. Synchronous operations that should be async. The code works in dev with 10 rows and melts production with 10,000.

Catch it: Look for database/API calls inside any for/map/forEach.

6. Shallow test assertions

expect(result).toBeTruthy() instead of expect(result.id).toBe(42). Tests that verify the function didn't throw instead of verifying it did the right thing. Green CI with zero confidence.

Catch it: Search test files for toBeTruthy, toBeDefined, bare not.toThrow.

Integration Review: The Parallel Agent Problem

This is the review problem unique to multi-agent workflows. When 5 agents work in isolated git worktrees, they can't see each other's changes. Each branch works in isolation. The bugs appear when you merge them together.

What to check:

  • Type definitions. Did two agents define the same type/interface differently? Compare all shared type files across branches before merging anything.
  • API contracts. If Agent A created an endpoint and Agent B consumes it, do the request/response shapes match? Check query parameters, headers, and response body schemas.
  • Database schema. Did two agents add migrations that conflict? Same table, same column name, incompatible changes?
  • Import paths. One agent created utils/auth.ts while another expects lib/auth.ts. Both work in their worktree. Neither works when merged.
  • Shared state. Two agents writing to the same config file, environment variable, or global store. The last merge wins, silently breaking the first.

In amux, the dashboard shows which files each session committed. A quick scan for overlapping file paths across sessions catches 80% of integration issues before you open a single diff.

The Morning Review Workflow

The concrete sequence for reviewing 10 agent branches before standup.

1

Dashboard scan (5 min)

Open your orchestrator's dashboard. Check: how many agents completed? Any that crashed or stalled? Any that are still running? Note which sessions modified shared files.

2

Run all test suites (parallel, 5-10 min)

Before reviewing any code, run tests in every agent's worktree. This immediately tells you which branches have working code and which need rework. Start reviews only on branches where tests pass.

3

Integration check (10 min)

Compare type definitions, API contracts, and shared files across all branches. Resolve conflicts before reviewing individual branches. This prevents wasted time reviewing code that will change during merge.

4

Review high-risk branches (30-45 min)

Business logic, auth, core features. Use the checklist. Focus on the diff against the spec. Don't read code that hasn't changed.

5

Review low-risk branches (15-20 min)

Endpoints, CRUD, tests, docs. These should go fast. If tests pass and the diff matches the spec, merge.

6

Sequential merge and validate (15-20 min)

Merge one branch at a time into main. Run the full suite after each merge. If something breaks, you know exactly which branch caused it. Never batch-merge.

How to Reject and Retry

When agent output doesn't pass review, your response should be proportional to the problem:

Severity Action Time cost
Cosmetic (naming, formatting) Fix it yourself. Faster than re-prompting. 1-2 min
Localized bug (logic error in one function) Send the agent the failing test or exact error. Let it fix. 5-10 min
Wrong approach (viable but suboptimal) Send a follow-up prompt with the right approach. Include constraints. 10-20 min
Fundamentally wrong (>30% needs rewriting) Kill the session. Rewrite the spec. Start a fresh agent. 20-45 min

The two-strike rule: if an agent fails to fix an issue after two follow-up prompts, restart from scratch. Iterating on a confused agent costs more tokens and time than a clean start with a better spec.

Tooling: Dashboard, Diff, and Automation

The right tools cut review time in half. Here's what to use at each step:

  • Orchestrator dashboard (amux, Claude Squad, Conductor) — triage at a glance. See which agents completed, which crashed, what files each touched. amux's peek mode lets you read agent output without leaving the dashboard.
  • git diff --stat — first thing you run on every branch. Shows how many files changed and how many lines. If the numbers don't match the spec's scope, investigate before reading code.
  • Claude Code /ultrareview — launches a cloud-based fleet of reviewer agents to verify the agent's work before you touch it. Best used as a pre-filter: run /ultrareview, read its report, then focus your human review on what it flagged.
  • Linters and type checkers (ESLint, mypy, TypeScript) — run these automatically on every branch via Claude Code hooks or CI. They catch hallucinated imports, type mismatches, and unused code for free.
  • Security scanners (Semgrep, Bandit, SonarQube) — automated first pass on security. Configure as a hook so every agent branch gets scanned before you see it.

Human Review vs AI Review: When You Need Each

Cross-model review (one AI reviews another AI's code) catches 60-70% of defects that self-review misses. But it has blind spots. Here's how to divide the work:

Let AI review Human must review
Code style, formatting, naming conventions Business logic correctness against requirements
Common security patterns (SQL injection, XSS) Authorization logic and access control design
Test coverage gaps Whether the right things are being tested
Unused imports, dead code, type errors Architecture decisions and integration fit
Documentation accuracy (API signatures) Whether the approach matches team conventions

The pattern that works: Agent writes → Different AI model reviews → Human reviews the review. The human pass is faster because the AI reviewer already flagged likely issues. You're reviewing a report, not doing archaeology.

FAQ

How long should it take to review AI-generated code?

For a well-scoped task (15-45 minutes of agent work), review should take 5-15 minutes. If you're spending longer than 15 minutes on a single agent's output, the task was scoped too broadly. The goal is 3-5 minutes for routine tasks and 10-15 minutes for complex ones. With 10 agents completing work overnight, morning review should take 60-90 minutes.

Should I review every line of AI-generated code?

No. Review every diff, but not every line. Focus on: business logic, security boundaries, integration points, and anything the agent flagged as uncertain. Skip reviewing boilerplate, formatting, and imports — those are what linters and type checkers handle. Your attention is the bottleneck, so spend it where human judgment matters most.

What are the most common bugs in AI-generated code?

Six patterns cover the majority: (1) hallucinated imports, (2) happy-path-only logic with no edge case handling, (3) security anti-patterns (SQL injection, XSS, hardcoded secrets), (4) over-abstraction creating unnecessary complexity, (5) N+1 query patterns, and (6) shallow test assertions that verify existence instead of correctness.

How do I review code from agents that worked on related tasks?

Review shared files (types, schemas, API contracts) across all branches before reviewing implementation code. If two agents defined the same type differently, resolve that first. Tools like amux show committed file diffs per session, making it easy to spot overlapping changes before you open a single diff.

Is it safe to let AI agents review each other's code?

Partially. Cross-model review catches 60-70% of defects that self-review misses. Same-model review suffers from sycophancy bias. Use automated agent review as a first pass, but a human must still review business logic, security-critical code, and integration points.

What should I do when an agent's output is wrong?

Proportional response: cosmetic issues — fix yourself (1-2 min). Localized bugs — send the agent the failing test (5-10 min). Wrong approach — new prompt with constraints (10-20 min). Fundamentally wrong — kill the session, rewrite the spec, start fresh (20-45 min). If an agent fails after two follow-up prompts, always restart.

How does reviewing AI code differ from reviewing human code?

Three key differences: (1) Volume — AI produces 5-10x more code per hour, so you must triage ruthlessly. (2) Failure patterns — humans make typos and logic errors; AI makes hallucination, over-engineering, and security pattern errors. Your checklist must cover different things. (3) No ego — you can reject AI code with a one-line reason. This should make you more decisive, not less rigorous.
