AI Agent Sandboxing in 2026: How to Run Coding Agents Without Risking Your System

AI coding agents execute arbitrary code on your machine — shell commands, package installs, file modifications, network requests. Without isolation, a prompt injection or hallucinated rm -rf can compromise your entire system. This guide compares 8 sandboxing technologies and the built-in safety models of Claude Code, Codex CLI, and Cursor — so you can run agents autonomously without staying up all night watching them. Why AI Agents Need Sandboxing

An AI coding agent is not an autocomplete widget. It is an autonomous process that reads files, writes code, runs shell commands, installs packages, and makes HTTP requests. When you run Claude Code in YOLO mode or Cursor’s agent mode, you are giving an LLM access to your terminal with your user-level permissions.

Most of the time, this works. But AI agents fail in ways that human developers don’t:

  • Hallucinated commands — LLMs occasionally generate destructive commands they “think” are correct
  • Prompt injection — malicious content in files, READMEs, or dependency metadata can hijack an agent’s behavior
  • Supply chain attacks — agents install packages from npm, PyPI, and other registries without verifying integrity
  • Data exfiltration — an agent with network access can send code, secrets, or credentials to external servers

CVE-2026-25592 & CVE-2026-26030: On May 7, 2026, Microsoft disclosed remote code execution vulnerabilities in Semantic Kernel’s AI agent framework. Prompt injection escalated to host-level RCE — the agent’s tool-calling mechanism became a shell. This is the threat model that sandboxing addresses: even if the agent is compromised, the damage stays inside the sandbox.

The Threat Model

What can go wrong when an AI agent runs unsandboxed? Four categories of risk, from most to least common:

Filesystem damage

Agent deletes, overwrites, or corrupts files outside the project. Most common failure mode. Mitigated by mounting project directories read-only or restricting filesystem access to a dedicated working directory.

Network exfiltration

Agent sends source code, .env files, or secrets to an external server via curl, package install scripts, or reverse shells. Mitigated by disabling outbound networking or restricting to an allowlist.

Privilege escalation

Agent exploits host kernel vulnerabilities or misconfigured permissions to escape isolation. Container escapes (CVE-2024-21626) have been demonstrated. Mitigated by microVMs or userspace kernels like gVisor.

Resource exhaustion

Agent spawns infinite loops, fills disk, or consumes all memory. Mitigated by cgroup resource limits, timeout enforcement, and amux’s watchdog which detects and kills stuck sessions.

Container-Based Isolation

Containers use Linux namespaces and cgroups to isolate processes. They share the host kernel — which makes them fast but means a kernel exploit can escape. For most AI agent workloads, a hardened container is sufficient. For untrusted code or red-team scenarios, pair containers with gVisor or switch to Firecracker.

Container

Docker

By Docker Inc. Apache 2.0 Linux, macOS, Windows

The default containerization platform. Docker runs everywhere, integrates with every CI/CD system, and most developers already have it installed. For AI agent sandboxing, the key flags are --network=none (disable networking), --cap-drop=ALL (drop all Linux capabilities), and --read-only (immutable root filesystem).

  • Hardened agent command: docker run --rm --network=none --cap-drop=ALL --security-opt=no-new-privileges -v ./project:/work:ro agent-image
  • GPU support — --gpus flag for NVIDIA GPU pass-through
  • Composable — use Docker Compose to define multi-container agent environments
  • Ecosystem — integrates with gVisor as runtime for stronger isolation (--runtime=runsc)
Best for: teams that want proven, widely-understood isolation without changing their workflow. The default choice for 90% of agent sandboxing needs.
Container

Podman

By Red Hat Apache 2.0 Linux, macOS, Windows

Drop-in Docker replacement with one critical advantage: rootless by default. Podman runs containers without a daemon and without root privileges, reducing the attack surface. A container escape in rootless Podman lands in an unprivileged user namespace, not root.

  • Rootless by default — no daemon, no root, smaller attack surface than Docker
  • Docker-compatible CLI — alias docker=podman works for most commands
  • Pod support — group multiple containers (agent + sidecar tools) into a pod
  • Systemd integration — generate systemd unit files for long-running agent containers
Best for: security-conscious teams who want rootless containers without changing their container workflow.

MicroVM-Based Isolation

MicroVMs run a separate kernel inside hardware-virtualized environments. Unlike containers, an exploit inside a microVM cannot reach the host kernel — it only compromises the guest kernel. This is the strongest isolation model available, and it’s what AWS Lambda and Fargate use under the hood.

MicroVM

E2B AI-native

By E2B Core: Apache 2.0 Cloud-managed

Purpose-built for AI agent code execution. E2B provides cloud sandboxes powered by Firecracker microVMs with an API designed specifically for AI agents: create a sandbox, upload files, execute commands, read output, and tear down — all in milliseconds. Each sandbox is fully isolated from the host and other sandboxes.

  • Sub-100ms cold start — faster than most container startups
  • Agent-first API — Python and TypeScript SDKs designed for agent integration (sandbox.run_code())
  • Custom templates — define base environments with pre-installed languages, frameworks, and tools
  • File system persistence — upload project files, execute code, download results
  • Integrations — works with Claude Code, Codex, LangChain, CrewAI, and any agent framework
Best for: AI platforms and agents that need to execute untrusted code in the cloud with maximum isolation and minimum setup.
MicroVM

Firecracker

By AWS Apache 2.0 Linux (KVM required)

The microVM monitor that powers AWS Lambda. Firecracker launches a minimal virtual machine with its own kernel in under 150ms with less than 5MB of memory overhead. It uses KVM for hardware virtualization, providing the same isolation guarantees as a full VM at near-container performance. E2B is built on top of Firecracker.

  • <150ms boot — comparable to container startup, far faster than traditional VMs
  • <5MB overhead — minimal memory footprint per VM, enabling thousands of concurrent instances
  • Hardware isolation — KVM-backed, separate kernel per VM, no shared kernel attack surface
  • Production-proven — runs millions of Lambda functions per second at AWS scale
  • Minimalist — no BIOS, no USB, no PCI — reduced attack surface by design
Best for: self-hosted infrastructure teams who need VM-level isolation with container-speed performance. Requires Linux with KVM.

Kernel-Level Sandboxing

gVisor takes a different approach: instead of running a separate kernel (like Firecracker) or sharing the host kernel (like Docker), it reimplements ~200 Linux syscalls in a user-space application kernel. This intercepts and validates every system call without hardware virtualization overhead.

Kernel Sandbox

gVisor

By Google Apache 2.0 Linux

gVisor’s application kernel (runsc) runs as a Docker or containerd runtime. Install it, set --runtime=runsc, and every container gets syscall-level isolation. This is what Google Cloud Run, Modal, and other cloud platforms use for multi-tenant isolation.

  • Syscall interception — reimplements ~200 Linux syscalls in userspace Go code
  • Docker-compatible — docker run --runtime=runsc — no code changes to your containers
  • Kernel exploit protection — host kernel never sees untrusted application syscalls directly
  • Netstack — userspace TCP/IP stack for network isolation without iptables complexity
  • 5–15% overhead — mostly on syscall-heavy workloads; near-zero for typical coding tasks
Best for: teams already using Docker who want stronger isolation without switching to microVMs. The best middle ground between containers and VMs.

Cloud Sandbox Platforms

Managed platforms that handle the infrastructure for you — provision an isolated environment via API, execute agent code, and tear it down. Best for teams that don’t want to manage their own container/VM infrastructure.

Cloud Platform

Modal

By Modal Proprietary (managed) Cloud

Serverless cloud platform for AI/ML workloads. Modal’s Sandbox API provides gVisor-isolated containers that spin up in under a second with programmatic command execution. Unlike E2B, Modal also supports GPUs — critical for agents that need to run ML inference or GPU-accelerated builds.

  • gVisor isolation — syscall-level sandboxing for every container
  • GPU support — A100, H100, L40S for ML workloads inside sandboxes
  • Python-native API — modal.Sandbox.create() with file mounting, command execution, and output streaming
  • Sub-second cold start — cached images start near-instantly
Best for: AI platforms and agents that need GPU-accelerated sandboxes in the cloud with zero infrastructure management.
Cloud Platform

Daytona

By Daytona GitHub Apache 2.0

Open-source development environment manager purpose-built for AI agent workflows. Daytona provides full Linux workspaces with IDE support, pre-configured toolchains, and network isolation. Unlike E2B (which is optimized for short-lived code execution), Daytona environments persist — making them better for long-running agents that need to build, test, and iterate.

  • Persistent environments — full dev workspace that survives agent restarts
  • Self-hostable — run on your own infrastructure or use the managed cloud
  • IDE integration — connect VS Code, JetBrains, or any SSH-compatible editor
  • GPU support — GPU pass-through for ML workloads
  • Network policies — fine-grained network access control per workspace
Best for: teams running long-lived AI agents that need persistent, full-featured development environments with network isolation.
Cloud Platform

Fly.io Machines

By Fly.io Proprietary (managed) Cloud

Fast-launching Firecracker microVMs with a REST API. Fly.io Machines boot in under 300ms, run in 35+ regions worldwide, and provide full VM isolation. The Machines API lets you programmatically create, start, stop, and destroy VMs — ideal for agent sandboxes that need to be provisioned on demand.

  • Firecracker-based — hardware-virtualized microVMs, same isolation as Lambda
  • <300ms boot — fast enough for on-demand sandbox provisioning
  • REST API — full API for programmatic VM lifecycle management
  • Global regions — deploy sandboxes close to your users/developers
  • GPU support — L40S GPUs available for ML/AI workloads
Best for: teams that want Firecracker-level isolation with a managed API and global deployment — without running their own KVM infrastructure.

Built-In Tool Sandboxes

The major AI coding agents ship with their own permission and isolation models. These aren’t full sandboxes — they’re guardrails that control what the agent can do within your existing environment.

Built-In

Codex CLI Strongest

By OpenAI Sandbox docs

The only mainstream coding agent with OS-level sandboxing built in. Codex CLI runs all code execution in an isolated container with the filesystem restricted to the working directory and network disabled by default. Three modes: suggest (read-only), auto-edit (file writes, no commands), and full-auto (all permissions).

  • Network disabled by default — no outbound connections unless explicitly enabled
  • Filesystem restricted — can only access the project directory
  • OS-level isolation — runs in a container sandbox, not just permission checks
  • Three permission modes — progressive trust from read-only to full access
Built-In

Claude Code

By Anthropic Sandboxing deep-dive

Claude Code uses a permission-based model rather than OS-level sandboxing. In default mode, it asks for approval before running shell commands, editing files outside the project, or making network requests. In --dangerously-skip-permissions (YOLO) mode, all guardrails are removed. Claude Code’s hooks system lets you add custom validation — block specific commands, require confirmation for destructive operations, or log all actions.

  • Permission prompts — asks before shell commands, file edits outside project, network access
  • Hooks — pre/post tool-use hooks for custom guardrails (e.g., block rm -rf, require git diff review)
  • Allowlists — configure trusted commands in .claude/settings.json
  • No OS-level sandbox — runs with your user permissions; for stronger isolation, run inside Docker
Built-In

Cursor

By Anysphere

Cursor’s agent mode runs terminal commands with user approval by default. Background agents execute in cloud sandboxes (using Cursor’s infrastructure) with isolated filesystems and restricted network access. The YOLO mode bypass lets the agent run commands without confirmation.

  • Agent mode approval — asks before running terminal commands
  • Background agent sandboxes — cloud-based isolated environments for background tasks
  • YOLO mode — bypass approvals for trusted development workflows
  • Rules system — .cursorrules for project-specific behavior constraints

How to Choose Your Sandboxing Approach

Match the isolation level to your risk tolerance and use case:

Trusted code, local dev

Built-in tool permissions (Claude Code prompts, Codex suggest mode) are sufficient. The agent only touches your project, and you’re reviewing changes.

Unattended / overnight agents

Docker with --network=none and project mounted read-write. Add gVisor runtime for kernel exploit protection. Pair with amux’s watchdog for crash recovery.

Untrusted code execution

E2B or Firecracker microVMs. Hardware-level isolation with separate kernels. No shared kernel attack surface.

Cloud / multi-tenant

Modal for GPU workloads. Fly.io for global distribution. Daytona for persistent dev environments.

Enterprise compliance

Firecracker or gVisor on-premises. Both are open-source and self-hosted. Add seccomp, AppArmor, and audit logging for compliance requirements.

Budget-conscious

Docker or Podman (free, runs locally). gVisor (free, adds isolation to existing containers). All Apache 2.0 licensed.

The recommended stack for most teams

  • Layer 1 — Git isolation: Each agent gets its own branch or git worktree to prevent merge conflicts
  • Layer 2 — Container: Docker with --network=none --cap-drop=ALL for filesystem and network isolation
  • Layer 3 — Runtime hardening: gVisor as the container runtime for kernel exploit protection
  • Layer 4 — Orchestration: amux watchdog for crash recovery, resource monitoring, and session management

Sandboxing Agent Fleets

Running 10–50 agents in parallel multiplies both productivity and risk. Each agent is an independent process that can fail independently. Sandboxing at fleet scale requires combining isolation with orchestration.

Frequently Asked Questions

Why do AI coding agents need sandboxing?

AI agents execute arbitrary code — shell commands, package installs, file modifications, network requests. Without sandboxing, prompt injection, hallucinated commands, or malicious dependencies can escalate to full system compromise. Microsoft disclosed CVEs in May 2026 showing how prompt injection in Semantic Kernel achieved host-level RCE. Sandboxing confines the blast radius.

Container vs. microVM: what’s the difference?

Containers (Docker, Podman) share the host kernel — fast startup (<1s) but vulnerable to kernel exploits. MicroVMs (Firecracker, E2B) run a separate kernel — slightly slower (<150ms) but immune to kernel-level escapes. gVisor offers a middle ground: container-speed startup with userspace syscall interception.

Which coding agent has the best built-in sandbox?

Codex CLI has the strongest built-in sandboxing — OS-level isolation with network disabled by default. Claude Code uses permission prompts and hooks (not OS-level isolation). Cursor’s background agents run in cloud sandboxes. For maximum security with any tool, add Docker or E2B on top.

Can I sandbox agents without losing performance?

Yes. Docker adds <2% CPU overhead. gVisor adds 5–15% on syscall-heavy workloads, near-zero on typical coding tasks. E2B sandboxes boot in under 100ms. Firecracker microVMs boot in under 150ms with <5MB memory overhead. The main bottleneck is startup latency, not runtime performance.

How do I sandbox agents running in parallel with amux?

Options: (1) Git worktree isolation — each agent gets its own branch. (2) Docker containers per session with mounted project directories. (3) E2B sandboxes connected via the amux REST API. (4) Combined: git worktrees + Docker for both code and system isolation. amux’s watchdog works with all configurations.

Is Docker secure enough?

Docker with defaults is reasonable but not bulletproof. Container escapes via kernel exploits have been demonstrated (CVE-2024-21626). Hardening checklist: --network=none, --cap-drop=ALL, --security-opt=no-new-privileges, non-root user, seccomp/AppArmor profiles. For higher security, add gVisor as the runtime or switch to Firecracker.

What is E2B and why is it popular?

E2B provides cloud sandboxes built on Firecracker microVMs, purpose-built for AI agent code execution. Each sandbox boots in under 100ms with full Linux and complete isolation. The API is designed for agents: create, upload, execute, download, tear down. Used by AI coding tools, chatbots, and data analysis agents for safe untrusted code execution.

Best free / open-source sandboxing options?

Docker (Apache 2.0) — the most widely available. Podman (Apache 2.0) — rootless by default. Firecracker (Apache 2.0) — microVMs, requires KVM. gVisor (Apache 2.0) — userspace kernel for containers. nsjail (Apache 2.0) — lightweight Linux process isolation. All free and self-hostable.

Does amux run agents in a sandbox?

By default, amux runs agents in tmux panes on the host machine — there is no sandbox layer built in. amux is an orchestration and control plane, not an isolation layer. Sandboxing is layered on top: (1) run each tmux session inside a Docker container with --network=none --cap-drop=ALL; (2) use amux’s built-in git worktree isolation so agents can’t step on each other’s code; (3) for untrusted code execution, connect E2B sandboxes via the amux REST API. amux’s self-healing (auto-compact, crash-restart, last-message replay) works with all these configurations.
