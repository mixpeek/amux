AI Agent Security Hardening

Your AI coding agent has shell access, can read your credentials, write to any file, and make network requests. Here's how to run it safely — whether you're a solo developer or operating a fleet of 20+ agents overnight.

Real Incidents (2026)

These aren't theoretical risks. Every attack vector described in this guide has been exploited in the wild.

Config Poisoning — RCE via CLAUDE.md (March 2026)

Security researchers demonstrated remote code execution through crafted CLAUDE.md files in public repositories. When a developer cloned the repo and ran Claude Code, the agent executed arbitrary commands embedded in the config. The attack bypassed permission prompts by encoding payloads in hook configurations. Every developer who git clones untrusted repos and runs an AI agent is vulnerable.

MCP Supply Chain — 1,184 Malicious Skills (February 2026)

Researchers audited the MCP ecosystem and found over 1,184 malicious packages across community skill registries. Payloads included credential harvesters, reverse shells, and crypto miners. The Model Context Protocol ecosystem grew to 5,800+ servers and 97 million monthly SDK downloads — but with minimal vetting infrastructure.

Credential Exfiltration — .env Leaks via Agent Output (2026)

Multiple reports of AI agents reading .env files during "helpful" debugging and accidentally including API keys in commit messages, PR descriptions, or MCP tool call parameters. In one documented case, an agent running in YOLO mode pushed a commit containing production database credentials to a public GitHub repository.

Prompt Injection via Issues — Agent Hijacking (April 2026)

Attackers created GitHub issues with hidden prompt injection payloads. When developers pointed their coding agents at "fix issue #42," the agent read the issue body, followed the injected instructions, and installed a backdoor dependency. The attack exploited the trust boundary between "user instruction" and "external data the agent reads."

These incidents share a pattern: AI agents operate with the developer's full permissions but without the developer's judgment. Security hardening is about constraining what agents can do without neutering their usefulness.

The Threat Model

AI coding agents face a unique threat landscape. They combine the attack surface of a CI/CD system (arbitrary code execution) with the access of a developer workstation (credentials, SSH keys, private repos).

1. Supply Chain Poisoning Critical

Malicious code in repos you clone: CLAUDE.md with injected hooks, .cursorrules with prompt injections, poisoned MCP server configs, or compromised dependencies that trigger agent actions on install.

Attack vector: You clone a repo → agent reads config → agent executes attacker's payload with your permissions.

2. Credential Theft Critical

Agents can read ~/.aws/credentials, .env, SSH keys, browser cookies, and any secret on the filesystem. Exfiltration routes: git commits, HTTP requests via tools, MCP server calls, or writing to shared storage.

Attack vector: Agent reads secret → includes in tool call / commit / output → secret reaches an external system.

3. Runaway Execution High

Agents in autonomous/YOLO mode making destructive changes: dropping databases, force-pushing to main, deleting files, or running expensive operations that burn through API quotas.

Attack vector: Poorly scoped prompt or hallucinated reasoning → agent takes destructive action → no human in the loop to stop it.

4. Prompt Injection High

Crafted content in code comments, issue descriptions, PR bodies, or dependency READMEs that alter agent behavior. The agent reads "data" but treats it as "instruction."

Attack vector: External text with embedded instructions → agent interprets as commands → performs unintended actions.

5. Lateral Movement Medium

A compromised agent using its access to reach other systems: SSH to production servers, push to other repos, access internal APIs via the developer's network position, or compromise other agents in the fleet.

Attack vector: Compromised agent → uses developer's SSH keys / API tokens / network access → reaches production systems.

6. Data Exfiltration via MCP Medium

MCP servers are network services that the agent calls as tools. A malicious or compromised MCP server can receive any data the agent sends to it — source code, credentials, conversation context.

Attack vector: Agent calls MCP tool with sensitive context → MCP server forwards data to attacker.

No single control stops all attacks. Layer defenses so that when one fails, others catch the threat.

The rest of this guide walks through each layer with concrete implementations. You don't need all six — pick the layers that match your risk tolerance. A solo developer on a personal project needs Layer 1-3. A team running 20 agents overnight on production code needs all six.

Config File Security (CLAUDE.md, .cursorrules, MCP configs)

Config files are the #1 attack vector for AI coding agents. They run before any human reviews them, and they can instruct the agent to do almost anything.

Audit configs before running agents on cloned repos

Before running any AI agent in a newly-cloned repository, inspect these files:

  • CLAUDE.md and .claude/ directory — instructions for Claude Code
  • .cursorrules and .cursor/ — instructions for Cursor
  • .github/copilot-instructions.md — instructions for GitHub Copilot
  • mcp.json, .mcp/, claude_desktop_config.json — MCP server configs
  • Any files referenced in hook configurations (pre/post tool use scripts)

Treat these files with the same suspicion you'd give a Makefile or postinstall script in an untrusted package.

Git hook for config validation

Add a post-checkout hook that flags suspicious patterns in agent config files:

#!/bin/bash
# .git/hooks/post-checkout — warn on suspicious agent configs

SUSPICIOUS_PATTERNS=(
  "curl "
  "wget "
  "nc "
  "bash -c"
  "/dev/tcp"
  "eval("
  "base64"
  "$(("
  "rm -rf"
  "chmod 777"
  "ssh "
)

for file in CLAUDE.md .claude/**/* .cursorrules mcp.json; do
  [ -f "$file" ] || continue
  for pattern in "${SUSPICIOUS_PATTERNS[@]}"; do
    if grep -qi "$pattern" "$file" 2>/dev/null; then
      echo "⚠️  SECURITY: '$file' contains suspicious pattern: $pattern"
      echo "    Review before running AI agents in this repo."
    fi
  done
done

Lock your own CLAUDE.md with trusted-only hooks

In repos you maintain, commit a .claude/settings.json that only allows hooks from paths you control:

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "*",
        "command": "/usr/local/bin/my-verified-hook.sh"
      }
    ]
  }
}

Never allow hook commands that reference relative paths (an attacker can plant files at those paths) or shell expansion patterns.

MCP Server Hardening

The Model Context Protocol ecosystem has 5,800+ servers with 97 million monthly SDK downloads. Many are community-maintained with no security review. Treat MCP servers as third-party code that runs with your agent's permissions.

Pin exact versions

Never use @latest or unpinned versions for MCP servers. Pin to exact commit SHAs or version tags:

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem@0.6.2"],
      "env": {}
    }
  }
}

If the server is installed from a git repo, pin to a specific commit hash, not a branch:

# Good: pinned commit
"command": "node",
"args": ["/opt/mcp-servers/github-server/index.js"]

# Bad: pulls latest from npm on every run
"command": "npx",
"args": ["-y", "@someorg/mcp-server"]

Audit before installing

Before adding any MCP server to your config:

  1. Read the source code — MCP servers are typically small (~500 lines). If you can't read it in 10 minutes, it's doing too much.
  2. Check the dependency tree: npm ls or pip show -f. Excessive dependencies increase supply chain risk.
  3. Verify the publisher's identity. Who maintains this? Is there a verified org? GitHub stars don't equal safety.
  4. Check for network calls. A filesystem MCP server that makes outbound HTTP requests is suspicious.

Restrict MCP filesystem access

Most MCP servers that need filesystem access accept path restrictions. Always scope them to the minimum:

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y", "@modelcontextprotocol/server-filesystem@0.6.2",
        "/home/user/project"
      ]
    }
  }
}

Never give an MCP server access to ~ (home directory), / (root), or paths containing credentials (~/.ssh, ~/.aws).

Run MCP servers in network-isolated containers

For high-security environments, run each MCP server in its own container with no network access:

# Dockerfile.mcp-filesystem
FROM node:22-slim
RUN npm install -g @modelcontextprotocol/server-filesystem@0.6.2
ENTRYPOINT ["mcp-server-filesystem"]

# Run with no network and minimal filesystem
docker run --rm --network=none \
  -v /home/user/project:/workspace:ro \
  mcp-filesystem /workspace

This prevents a compromised MCP server from exfiltrating data over the network or reading outside its designated path.

Credential Protection

AI agents can read any file your user account can read. This includes .env, ~/.aws/credentials, ~/.ssh/id_rsa, browser session tokens, and application secrets. Protect them.

Block access to credential paths

Use Claude Code hooks to block reads of sensitive directories:

#!/bin/bash
# hooks/block-credentials.sh — PreToolUse hook
# Blocks agent from reading credential files

BLOCKED_PATHS=(
  ".env"
  ".env.local"
  ".env.production"
  "credentials"
  "id_rsa"
  "id_ed25519"
  ".aws/"
  ".ssh/"
  ".netrc"
  ".npmrc"
  "secrets"
  "token"
  ".kube/config"
)

INPUT=$(cat)
for path in "${BLOCKED_PATHS[@]}"; do
  if echo "$INPUT" | grep -qi "$path"; then
    echo '{"decision":"block","reason":"Blocked: credential path access"}'
    exit 0
  fi
done
echo '{"decision":"allow"}'

Use secret managers instead of .env files

If your agent never needs direct access to secrets (and it usually doesn't), use a secret manager that injects values at runtime:

  • 1Password CLI: op run -- your-command injects secrets as env vars without writing files
  • AWS Secrets Manager / Parameter Store: application fetches secrets at runtime
  • direnv with encrypted vault: secrets decrypted only in the correct directory

The principle: if the secret isn't on the filesystem, the agent can't read it.

Scan agent output for leaked secrets

Add a PostToolUse hook that scans agent output for accidentally leaked credentials:

#!/bin/bash
# hooks/scan-secrets.sh — PostToolUse hook
# Alerts if agent output contains credential patterns

INPUT=$(cat)
OUTPUT=$(echo "$INPUT" | jq -r '.output // empty' 2>/dev/null)

# Check for common secret patterns
if echo "$OUTPUT" | grep -qEi \
  '(AKIA[0-9A-Z]{16}|sk-[a-zA-Z0-9]{48}|ghp_[a-zA-Z0-9]{36}|-----BEGIN.*PRIVATE KEY)'; then
  echo '{"decision":"block","reason":"ALERT: Potential credential detected in output"}'
  exit 0
fi
echo '{"decision":"allow"}'

For more comprehensive scanning, integrate TruffleHog or detect-secrets into your hook pipeline.

Pre-commit secret scanning

Even with hook-based protection, add a pre-commit safety net. Use git-secrets to prevent credentials from ever being committed:

# Install git-secrets
brew install git-secrets  # macOS
# or: pip install detect-secrets

# Initialize in your repo
git secrets --install
git secrets --register-aws

# Add custom patterns
git secrets --add 'sk-[a-zA-Z0-9]{48}'      # OpenAI keys
git secrets --add 'anthropic_api_key'         # Anthropic keys
git secrets --add 'ghp_[a-zA-Z0-9]{36}'      # GitHub PATs

This catches credentials that slip past hook-level protections when the agent creates commits.

Running YOLO Mode Safely

YOLO mode (auto-approve all tool calls) is how most developers run agents unattended. It's powerful but removes the human checkpoint. Here's how to use it without giving agents unconstrained access.

Scope YOLO to specific commands

Instead of blanket YOLO, use Claude Code's permission configuration to allow specific patterns:

{
  "permissions": {
    "allow": [
      "Bash(git *)",
      "Bash(npm test)",
      "Bash(npm run *)",
      "Bash(python -m pytest *)",
      "Bash(cargo test *)",
      "Read(*)",
      "Write(src/**)",
      "Write(tests/**)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(curl *)",
      "Bash(wget *)",
      "Bash(ssh *)",
      "Bash(scp *)",
      "Bash(git push --force *)",
      "Bash(docker *)",
      "Write(.env*)",
      "Write(*.pem)",
      "Write(credentials*)"
    ]
  }
}

The deny list takes precedence. Even in YOLO mode, these commands will be blocked.

Directory scoping

Restrict the agent to its project directory. It should not be able to read or write outside the repo:

# When registering an amux session, scope to the project dir
amux register myproject --dir ~/Dev/myproject --yolo

# The session runs in ~/Dev/myproject and CLAUDE.md should declare:
# "Do not access files outside this project directory."
# "Do not read ~/.aws, ~/.ssh, ~/.env, or any parent directory."

Combine with filesystem-level enforcement (symlink protections, mount namespaces) for defense in depth.

Session timeouts and cost caps

Prevent runaway execution with hard limits:

  • Time limit: Kill sessions that run longer than expected (e.g., 2 hours for a task that should take 30 minutes)
  • Token cap: Set a per-session token budget. If the agent burns through its allocation, something went wrong.
  • File write limit: Alert if an agent writes more than N files in a single session — could indicate a hallucination loop

In amux, the watchdog's stuck detection catches runaway sessions automatically. Configure it to restart (not continue) when a session exceeds its expected duration.

Network Isolation

If an agent can't make outbound network requests, it can't exfiltrate data. Most coding tasks don't require internet access once dependencies are installed.

Block outbound connections for coding sessions

On macOS, use the application firewall or a tool like Little Snitch. On Linux, use iptables rules per-user or network namespaces:

# Linux: block all outbound for a specific user (used by agent sessions)
sudo iptables -A OUTPUT -m owner --uid-owner agent-user -j DROP

# Allow only the AI API endpoint
sudo iptables -I OUTPUT -m owner --uid-owner agent-user \
  -d api.anthropic.com -p tcp --dport 443 -j ACCEPT
sudo iptables -I OUTPUT -m owner --uid-owner agent-user \
  -d api.openai.com -p tcp --dport 443 -j ACCEPT

This allows the agent to call the AI API (required for operation) but blocks all other outbound traffic. A compromised agent can't phone home.

DNS-level blocking

For simpler setups, use /etc/hosts or a DNS sinkhole to block known exfiltration destinations while allowing the AI API:

# /etc/hosts additions for agent sessions
# Block common exfiltration targets
127.0.0.1 pastebin.com
127.0.0.1 requestbin.net
127.0.0.1 webhook.site
127.0.0.1 ngrok.io
127.0.0.1 burpcollaborator.net

Container Sandboxing

For maximum isolation, run agents inside containers. This provides filesystem isolation, network control, resource limits, and a clean-slate environment. Docker launched Sandboxes in January 2026 specifically for AI coding agent workloads.

Docker-based agent sandbox

# Dockerfile.agent — minimal Claude Code environment
FROM ubuntu:24.04
RUN apt-get update && apt-get install -y \
    git nodejs npm python3 tmux curl ca-certificates \
    && npm install -g @anthropic-ai/claude-code \
    && rm -rf /var/lib/apt/lists/*

# Run as non-root
RUN useradd -m agent
USER agent
WORKDIR /workspace

ENTRYPOINT ["claude"]
# Run with restricted network and resource limits
docker run --rm -it \
  --network=agent-net \          # custom network with egress rules
  --memory=4g \                  # cap memory (prevent fork bombs)
  --cpus=2 \                     # cap CPU
  --read-only \                  # read-only root filesystem
  --tmpfs /tmp:size=1g \         # writable tmp with size limit
  -v ./project:/workspace \      # mount only the project
  -e ANTHROPIC_API_KEY \         # pass API key via env, not file
  agent-sandbox \
  "Implement the auth module per the spec in TASKS.md"

The agent-net Docker network should only allow egress to api.anthropic.com:443 (or api.openai.com:443). All other outbound traffic is dropped.

gVisor for stronger isolation

gVisor intercepts system calls in userspace, providing a security boundary stronger than standard Docker containers. Use it when running agents on untrusted code:

# Run Docker container with gVisor runtime
docker run --runtime=runsc --rm \
  --network=none \
  -v ./project:/workspace \
  agent-sandbox "Fix the failing tests"

gVisor prevents container escapes and limits the kernel attack surface. The performance overhead is ~5-15% for typical coding workloads — acceptable for security-sensitive environments.

Lightweight sandboxing without Docker

If Docker is too heavy, use process-level sandboxing:

  • Linux: Bubblewrap (bwrap) creates minimal namespaces. Firejail provides predefined security profiles.
  • macOS: sandbox-exec (deprecated but still functional) provides profile-based restrictions.
# Linux: run Claude Code in a Firejail sandbox
firejail --noprofile --net=none --private-tmp \
  --whitelist=~/Dev/myproject \
  --read-only=~ \
  claude "Implement feature X"

Fleet Security with Multiplexers

When running multiple agents in parallel, security becomes a fleet concern. One compromised agent can potentially affect others. Multiplexers like amux add a security control layer that applies policies uniformly across all sessions.

Session isolation

Each agent session should be isolated from others:

  • Separate working directories: Each agent gets its own git worktree or repo clone. One agent can't corrupt another's work.
  • Separate tmux sessions: amux runs each agent in its own tmux session. Sessions can't access each other's terminal buffers.
  • API key isolation: In high-security setups, each agent uses a different API key. If one is compromised, revoke it without affecting the fleet.
  • Controlled inter-session communication: Agents communicate only through the amux REST API (board, notes, messages) — never through shared filesystem or environment variables.

Centralized hook enforcement

Define security hooks once, apply them to every session in the fleet. In amux, the global CLAUDE.md and .claude/settings.json are inherited by all sessions:

# ~/.claude/settings.json — applies to ALL Claude Code sessions
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "/usr/local/bin/security-gate.sh"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "*",
        "command": "/usr/local/bin/secret-scanner.sh"
      }
    ]
  },
  "permissions": {
    "deny": [
      "Bash(curl * | bash)",
      "Bash(wget -O - * | sh)",
      "Bash(pip install *)",
      "Bash(npm install *)"
    ]
  }
}

With this configuration, no agent in the fleet can pipe downloads to shell or install packages without explicit approval — regardless of what their per-project CLAUDE.md says.

Atomic task claiming prevents race conditions

When multiple agents claim work from a shared board, use atomic operations to prevent duplicated or conflicting work:

# amux's /api/board/TASK-ID/claim endpoint is atomic
# If two agents try to claim the same task, only one succeeds
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"session":"agent-1"}' \
  https://localhost:8822/api/board/TASK-123/claim

This prevents a compromised agent from claiming all tasks to disrupt the fleet, and ensures no two agents accidentally work on the same code simultaneously.

Monitoring & Anomaly Detection

You can't stop every attack at the gate. Monitoring detects compromises that bypass preventive controls.

What to monitor

Signal Normal Suspicious
Files read Source code, tests, configs .env, ~/.aws, ~/.ssh, credential files
Files written Source files, test files Binary files, files outside project, dotfiles
Commands run git, npm test, python curl to unknown hosts, nc, base64 -d
Network connections AI API, package registries Unknown IPs, non-standard ports, DNS queries to odd domains
Token usage Steady consumption Sudden spikes (looping), zero usage (crashed/stuck)
Session duration 30min–2hrs per task >4 hours with no output (stuck or looping)

Session output scanning

Periodically scan agent output for indicators of compromise:

#!/bin/bash
# monitor-fleet.sh — run via cron every 5 minutes
# Scans all amux session output for suspicious patterns

ALERT_PATTERNS=(
  "BEGIN.*PRIVATE KEY"
  "AKIA[0-9A-Z]{16}"
  "sk-[a-zA-Z0-9]{20,}"
  "password.*="
  "curl.*pastebin"
  "wget.*-O.*|.*sh"
  "/dev/tcp/"
  "reverse.*shell"
)

SESSIONS=$(curl -sk https://localhost:8822/api/sessions | \
  python3 -c "import json,sys; [print(s['name']) for s in json.load(sys.stdin)]")

for session in $SESSIONS; do
  OUTPUT=$(curl -sk "https://localhost:8822/api/sessions/$session/peek?lines=50" | \
    python3 -c "import json,sys; print(json.load(sys.stdin).get('output',''))")
  for pattern in "${ALERT_PATTERNS[@]}"; do
    if echo "$OUTPUT" | grep -qEi "$pattern"; then
      echo "🚨 ALERT: Session '$session' matched pattern: $pattern"
      # Send notification, kill session, or log for review
    fi
  done
done

Audit logging

Enable audit logging so you have a forensic record of every action every agent took:

  • Git history: Every agent commit is timestamped and attributed. Use git log --author= to trace agent actions.
  • amux session logs: The server logs every API call, session start/stop, and watchdog intervention.
  • Claude Code conversation logs: Stored locally in ~/.claude/ — contains the full conversation including every tool call and result.
  • System-level auditing: On Linux, use auditd to log file access and process execution by agent user accounts.

The 15-Point Hardening Checklist

Print this. Run through it when setting up a new project or onboarding a new team member to your agent fleet.

  • 1. Audit CLAUDE.md and MCP configs in every repo you clone before running an agent
  • 2. Pin MCP server versions to exact version numbers or commit hashes — never @latest
  • 3. Block credential path access via PreToolUse hooks (.env, ~/.aws, ~/.ssh)
  • 4. Install pre-commit secret scanning with git-secrets or detect-secrets
  • 5. Use scoped permissions instead of blanket YOLO — allow specific commands, deny dangerous ones
  • 6. Restrict agent working directory — agents should not read/write outside the project
  • 7. Set session timeouts — kill sessions that exceed expected duration
  • 8. Block outbound network except AI API endpoints for unattended/overnight sessions
  • 9. Use secret managers (1Password CLI, AWS Secrets Manager) instead of .env files on disk
  • 10. Scan agent output for leaked credentials via PostToolUse hooks
  • 11. Run each agent in its own worktree/directory to prevent cross-contamination
  • 12. Use atomic task claiming when multiple agents share a work queue
  • 13. Container-sandbox untrusted code with Docker or gVisor for repos you don't control
  • 14. Monitor for anomalies — unusual file access, unexpected network calls, token spikes
  • 15. Review agent git history weekly — look for files that shouldn't have changed, unusual commit patterns

Minimum viable security (solo developer): Items 1, 3, 4, 5, 6. Takes 15 minutes to set up.
Production fleet security (teams): All 15 items. Takes a day to implement, saves you from the next incident.

Frequently Asked Questions

What are the biggest security risks of AI coding agents?

The top risks are: (1) Supply chain poisoning via malicious CLAUDE.md or MCP server configs in cloned repos, (2) Credential exfiltration — agents reading .env files or AWS credentials and leaking them through tool calls, (3) Runaway execution in YOLO mode making destructive changes without oversight, (4) Prompt injection via crafted code comments or issue descriptions that hijack agent behavior, and (5) MCP server vulnerabilities — the 5,800+ community MCP servers have varying security quality, with over 1,000 found to contain malicious code.

Is YOLO mode safe for AI coding agents?

YOLO mode is safe when combined with proper guardrails: directory-scoped permissions that prevent writes outside the project, hook-based allowlists that block dangerous commands (rm -rf, curl to external hosts, credential access), network isolation that prevents data exfiltration, and session timeouts that cap execution duration. Without these layers, YOLO mode gives agents unconstrained access to your system. See the Running YOLO Mode Safely section above for concrete configurations.

How do I secure MCP servers for AI agents?

Pin MCP server versions to exact commits (never use @latest), audit server source code before installation, run MCP servers in network-isolated containers, use an allowlist of approved servers in your team's shared configuration, restrict file system access to the minimum paths needed, and monitor MCP server network traffic for unexpected outbound connections. Never install MCP servers from untrusted sources. See MCP Server Hardening for implementation details.

Can AI agents steal my credentials?

Yes. An AI coding agent with file read access can read .env files, ~/.aws/credentials, SSH keys, and API tokens. It can then exfiltrate them via tool calls (writing to files, making HTTP requests, or encoding them in git commit messages). Mitigations: use secret managers instead of .env files, block read access to credential directories via hooks, and use credential-scanning hooks that abort if secrets appear in agent output. See Credential Protection.

How do multiplexers improve agent security?

Multiplexers like amux add a security control layer between you and your agents: session isolation prevents one compromised agent from affecting others, hook systems enforce permission policies uniformly across all sessions, centralized monitoring detects anomalous behavior, and audit logging creates a record of every action taken. Running agents through a multiplexer with proper configuration is significantly safer than running them directly in unmanaged terminal sessions. See Fleet Security with Multiplexers.

Should I run AI agents in Docker containers?

For untrusted code or overnight unattended runs, yes. Docker containers provide filesystem isolation, network control, and resource limits. Docker Sandboxes (launched January 2026) are specifically designed for AI coding agent workloads. The trade-off is performance overhead and complexity — containerized agents can't access your local tools, databases, or file system without explicit mounts. For trusted projects in active development, hook-based security (Layers 1-3) is usually sufficient. See Container Sandboxing.

What's the minimum security I should set up?

At minimum: (1) Audit configs in cloned repos before running agents, (2) Block credential path access via a PreToolUse hook, (3) Install pre-commit secret scanning with git-secrets, (4) Scope YOLO permissions to specific commands, (5) Restrict the agent's working directory. This takes 15 minutes and stops the most common attack vectors. Add network isolation and container sandboxing for overnight/unattended runs.

How do I handle security for agents reading GitHub issues?

Treat issue content as untrusted input (because it is). Never point an agent at an issue from an unknown contributor without review. Use hooks that detect common prompt injection patterns in agent input. Consider running a "triage agent" that summarizes issues in a sanitized format before passing them to coding agents. The OWASP prompt injection guidelines apply directly to this scenario.
