Conversation Forking: Branch a Session into Parallel Agents

Get a decision in hours instead of days. Fork a Claude Code session at any architectural crossroads and let two or three independent agents each explore a different path — all starting from the same full conversation context. Review the results, pick the winner, merge. Updated July 2026.

Conversation forking is the technique of cloning a running AI agent session — its entire JSONL conversation history, all accumulated context, and its working git state — into multiple new sessions that each proceed independently. Unlike starting fresh agents, forked sessions inherit everything the original session learned: the problem statement, constraints, prior decisions, code patterns, and any files already discussed. Each fork then takes that shared context in a different direction.

amux is an open-source control plane for AI agent teams — run dozens of parallel Claude Code, Codex, or Gemini CLI sessions from your phone or browser with self-healing, kanban, CRM, and email built in. Conversation forking is one of its core workflow primitives for engineering teams that need to evaluate multiple implementation options without wasting time re-briefing agents from scratch.

What is conversation forking?

When you work with a Claude Code session, the agent's "memory" lives in a JSONL file — a log of every message, tool call, and result in the conversation. This is what lets the agent remember decisions made an hour ago, understand the codebase it has been exploring, and maintain coherent long-running work.

Conversation forking copies that JSONL history into one or more new sessions. Each new session is an independent copy of the original: same context, same accumulated knowledge, but now running in its own tmux session and on its own git branch. From the moment of the fork, they diverge. You send each fork a different implementation directive and they proceed in parallel.

The result: in the time it would take one agent to try approach A, evaluate it, abandon it, and try approach B, you get a complete implementation of A and a complete implementation of B side by side. You review both and merge the winner.

Why fork instead of starting fresh sessions?

The most common alternative to forking is starting new sessions and re-briefing each from scratch. This works at small scale, but it has three compounding costs:

  • Token cost. Re-pasting the problem description, constraints, codebase context, and prior decisions into each new session costs thousands of tokens per fork. With the full JSONL history copied, forks pay none of this overhead — they already know everything.
  • Drift risk. When you manually re-brief multiple agents, subtle differences in how you phrase things can lead to agents solving slightly different problems. Forked sessions have identical starting context, so they are directly comparable.
  • Time cost. A planning session where an agent reads files, understands the architecture, and establishes the problem space might take 10-15 minutes. Forking reuses that work; fresh sessions repeat it for every branch.

There is also a cognitive cost to you: briefing three agents from scratch requires you to maintain a coherent mental model and reproduce it three times. With forking, you brief once, fork, then give each fork one focused directive.

When should you fork a Claude Code session?

Forking is most valuable at decision points — moments in a project where multiple viable paths exist and the cost of exploring the wrong one is high. The right time to fork is after you have enough context to know the paths, but before committing to one.

Common triggers:

  • After a planning or design session. You have spent time with an agent analyzing the codebase and problem space. The agent has proposed two or three plausible architectures. Rather than picking one now, fork and let each explore its path in parallel. See which is cleaner, faster, or more maintainable before committing.
  • At an architectural decision point. "Should this be server-side rendered or client-side?" "Should we use Postgres or SQLite for this workload?" "Should the cache be Redis or in-process?" These are classic fork moments — diverging implementations that will produce genuinely different code.
  • Before a risky refactor. If you are about to change something fundamental — a database schema, a core abstraction, an authentication system — fork first. One fork attempts the planned approach; another tries a more conservative path. You can compare diffs before merging anything.
  • When you want a second opinion on implementation quality. Fork the session and give each fork the same task with different quality constraints: "optimize for readability" vs "optimize for performance." Review both and pick the characteristics you want.
  • A/B testing prompting strategies. Fork the same session and give each fork slightly different instructions. This is useful for calibrating how prompt phrasing affects implementation choices at the team level.

Forking is not the right tool when you want a single agent to do multiple independent tasks in parallel — for that, use a fleet with a shared task queue. See the running 10+ agents guide for that pattern.

How to fork a Claude Code session (step-by-step)

Here is the complete workflow from first session to merged result.

Step 1: Run a planning session to build context

Start a normal Claude Code session on your project. Let it explore the codebase, understand the problem, and surface the key design decisions. The goal is to get the agent to a point of deep context — it understands the constraints, has read the relevant files, and can articulate the tradeoffs of different approaches.

amux register myproject --dir ~/Dev/myproject --yolo
amux start myproject
# Send the session your planning prompt:
amux send myproject "Read the codebase. We need to add a caching layer to the
API. Understand the current request flow, identify the hot paths, and outline
two or three viable implementation approaches with their tradeoffs. Do not
implement anything yet."

Wait for the planning session to finish. Check its output in the dashboard or via:

amux peek myproject --lines 80

Step 2: Fork the session

Once planning is complete and the session has established full context, create forks:

# Create 3 forks, each on its own git branch
amux fork myproject --count 3

# Or name the branches explicitly:
amux fork myproject --branches cache-redis cache-lru cache-memcached

amux does the following for each fork:

  1. Copies the JSONL conversation history to a new session directory
  2. Creates a new git branch from the current HEAD of your project
  3. Registers a new amux session (e.g. myproject-fork-1, myproject-fork-2, myproject-fork-3)
  4. Starts each new session in tmux with the copied history loaded

After this command, three new sessions appear in your amux dashboard, all ready to receive directives. The original session is untouched — it continues running as before.

Step 3: Send each fork a different directive

Each fork is a separate session you can message independently:

amux send myproject-fork-1 "Implement the caching layer using Redis.
Use ioredis. Cache at the route handler level. TTL 60s for GET /api/items,
30s for /api/user/:id. Write tests. Open a PR draft when done."

amux send myproject-fork-2 "Implement the caching layer in-process using
an LRU map (no external dependencies). Same cache keys and TTLs as the
Redis approach. Write tests. Open a PR draft when done."

amux send myproject-fork-3 "Implement the caching layer using the HTTP
Cache-Control headers only — no server-side caching. Evaluate whether the
CDN layer is sufficient. Write a brief analysis and a test suite. PR draft."

Step 4: Monitor from the dashboard

All three forks post updates to the amux board. Open the dashboard to see their progress in the kanban view:

amux serve  # → https://localhost:8822

Each fork's status is visible under its session name. You can peek at any fork's output without switching terminals:

amux peek myproject-fork-1 --lines 50
amux peek myproject-fork-2 --lines 50

Step 5: Review and merge the winning branch

When all forks complete (or when you have seen enough to decide), compare their branches:

# See what each fork changed vs main
git diff main...cache-redis --stat
git diff main...cache-lru --stat
git diff main...cache-memcached --stat

# Review the full diff of the winner
git diff main...cache-lru

# Merge the winning branch
git checkout main
git merge cache-lru

# Clean up the others
git branch -d cache-redis
git branch -d cache-memcached

Directing each fork to a different approach

The quality of your fork directives determines the quality of the comparison. A few patterns that work well:

Approach-specific directives: Give each fork a clear implementation strategy. Be specific about the mechanism (Redis vs LRU map vs HTTP headers) rather than repeating the problem statement — the forks already know the problem from the shared history.

Constraint-specific directives: Give forks the same task with different constraints. "Implement with zero new dependencies" vs "Implement with best-in-class libraries for each concern" vs "Implement the simplest possible version." This produces comparable implementations along a real tradeoff axis you care about.

Quality-axis directives: "Optimize this for readability and maintainability" vs "Optimize this for runtime performance" vs "Optimize this for minimal bundle size." Useful when you know what you are building but want to understand the shape of tradeoffs before deciding.

After sending the initial directive, you can continue a conversation with any fork independently:

# Ask fork 1 a follow-up question without affecting fork 2:
amux send myproject-fork-1 "What's the memory overhead of the Redis client
at idle? Is there a connection pooling concern we should address?"

Monitoring forked sessions

Forked sessions appear as first-class sessions in the amux dashboard. The board view shows all their status updates in the kanban. The workspace tiled view lets you watch multiple fork output panels side by side.

To watch all forks simultaneously from the command line:

# List all sessions from the forked parent:
curl -sk $AMUX_URL/api/sessions | python3 -c "
import json, sys
sessions = json.load(sys.stdin)
forks = [s for s in sessions if 'fork' in s['name']]
for s in forks:
    print(s['name'], '|', s.get('status',''), '|', s.get('desc',''))
"

From the iOS app, you can swipe between fork sessions and read their output from anywhere — useful when you kick off forks before leaving for a meeting and want to check in on results an hour later.

For longer-running forks, set up a simple completion check on the board. Each fork should be posting a board item as it completes:

# Each fork's CLAUDE.md should include:
# "When you finish, mark your board task done with a one-line summary
# of what you built and where to find the PR draft."

# Check completion status:
curl -sk $AMUX_URL/api/board | python3 -c "
import json, sys
items = json.load(sys.stdin)
for i in items:
    if 'fork' in i.get('session',''):
        print(i['status'], '|', i.get('session'), '|', i['title'])
"

Reviewing and merging results from forked sessions

Once forks complete, you have three independent implementations on three git branches. The review process is a standard engineering code review, not an AI-specific one:

Diff-based comparison

# Files changed by each fork:
git diff main...cache-redis --name-only
git diff main...cache-lru --name-only

# Complexity proxy: lines changed
git diff main...cache-redis --stat | tail -1
git diff main...cache-lru --stat | tail -1

# Test coverage (if each fork wrote tests):
cd /tmp/fork-redis && npx jest --coverage --silent 2>&1 | tail -5
cd /tmp/fork-lru   && npx jest --coverage --silent 2>&1 | tail -5

Asking an agent to summarize the comparison

You can use a fresh amux session to act as a reviewer — giving it the diffs and asking for a structured comparison:

amux start reviewer
amux send reviewer "Compare these three git branches: cache-redis, cache-lru,
cache-memcached — all branched from main. For each, produce: (1) a one-paragraph
summary of the approach, (2) lines changed, (3) new dependencies added,
(4) test coverage percentage, (5) any obvious concerns. Format as a comparison
table followed by your recommendation."

Cherry-picking across forks

If you want the caching logic from fork-1 but the test structure from fork-2, you are not forced to pick one wholesale. Standard git cherry-pick works across fork branches since they share a common ancestor:

# Cherry-pick specific commits from fork-2 onto fork-1's branch
git checkout cache-lru
git cherry-pick <commit-hash-from-cache-redis>

Forking vs. fresh sessions vs. subagents

Three patterns exist for parallel agent work. They solve different problems:

Dimension Conversation Forking Fresh Sessions Subagents
Starting context Full copy of parent history Zero — must re-brief Depends on parent prompt
Token overhead to start None (history copied) High (full re-brief) Medium (summary injected)
Independence Fully independent Fully independent Coordinated by parent
Git isolation Separate branch per fork As configured manually Usually same branch
Best use case A/B exploring N approaches Truly unrelated tasks Coordinated parallel work
Result combining Pick one (or cherry-pick) Manual Parent synthesizes
Context drift risk None — same starting point High — re-briefing varies Low — same parent

The decision rule is simple:

  • Use forking when you want to compare N complete implementations of the same problem, starting from shared context.
  • Use fresh sessions when the tasks are genuinely unrelated and re-briefing cost is acceptable.
  • Use subagents when a parent agent needs to decompose a task and coordinate workers toward a single result — the parent synthesizes, not you.

Advanced patterns

Forking at multiple depths

You can fork a fork. After the first round of exploration reveals that approach A is promising but has two sub-variants worth exploring, fork the fork-A session into fork-A1 and fork-A2. This creates a tree of explorations, all rooted in the original planning session's context.

# Fork from a fork:
amux fork myproject-fork-1 --count 2 --branches cache-lru-v1 cache-lru-v2

Using forking for code review

Fork an existing session and send each fork to review the same pull request from a different angle. One reviews for correctness, one for performance, one for security. Then merge their findings in a review summary:

amux fork pr-review-session --count 3

amux send pr-review-session-fork-1 "Review this PR for correctness only.
Find logical errors, edge cases, and incorrect assumptions."

amux send pr-review-session-fork-2 "Review this PR for performance only.
Identify any O(n²) operations, unnecessary allocations, or hot paths that
could be optimized."

amux send pr-review-session-fork-3 "Review this PR for security only.
Look for injection vectors, auth bypass, sensitive data exposure, or
dependency vulnerabilities."

See the plan monitoring guide for how to track review sessions on the board.

Forking before a risky deployment

Before a high-stakes change (database migration, infrastructure teardown, public API change), fork your planning session into a "go" fork and a "rollback" fork. The go fork proceeds with the plan; the rollback fork prepares a complete undo script in parallel. You have both ready before you start the real deployment.

Scheduled forks for nightly exploration

Use amux's scheduler to fork a known session each night and send forks in different directions — one exploring performance improvements, one exploring new feature prototypes. Review results each morning. This is how small teams run a continuous exploration process without dedicated research engineering time.

curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{
    "title": "Nightly exploration fork",
    "session": "baseline",
    "command": "Fork this session into two: one exploring performance wins in
the hot path, one prototyping the new search feature. Post results to board.",
    "schedule_expr": "daily at 23:00"
  }' \
  $AMUX_URL/api/schedules

For large-scale parallel exploration across many sessions, see the scaling to 50+ agents guide. For setting up your team to work with shared sessions, see the team workflow setup guide.

Frequently asked questions

Does forking preserve the agent's memory and conversation context?

Yes. amux copies the entire JSONL conversation history into each forked session. Every fork starts with the same complete context — the problem description, constraints, previous decisions, and any code or files discussed. It is identical to the moment of the fork; the forks then diverge from that point forward.

Can I merge results from forked sessions?

Yes — through standard git. Each fork runs on its own git branch. Once the forks complete, you review the branches, pick the best implementation, and merge it to main. If you want to combine parts of two forks, cherry-pick commits between branches just as you would in any git workflow. There is no amux-specific merge command — forks are regular git branches under the hood.

How many forks can I run at once?

There is no hard limit in amux — it creates as many sessions as you specify. In practice, 2-5 forks covers most architectural decision points. Your practical ceiling is API rate limits (each fork makes independent API calls) and available RAM (each session needs roughly 300-500MB). For high-count parallel work, see the 10+ agents guide instead of forking, since that use case calls for a task queue rather than N copies of the same context.

What is the difference between forking and just starting fresh sessions?

Fresh sessions start with zero context. Each fresh agent must be re-briefed from scratch — the problem, constraints, prior decisions, existing code patterns — which costs tokens, time, and risks the agent missing subtle requirements you worked out in the original session. Forked sessions start with the complete conversation history already loaded, so they can proceed directly to implementation without re-establishing context. The comparison is also cleaner: forked sessions start from an identical baseline, so differences in their output are attributable to the different directive, not to drift in how you re-briefed them.

When should I fork versus using subagents?

Fork when you want two or more agents to each produce a complete independent implementation of the same thing, and you will choose one. Use subagents (spawning child agents from within a session) when you want one parent agent to decompose a task and coordinate multiple workers toward a single combined result. Forking is for A/B exploration where you review and pick a winner. Subagents are for parallel execution under a coordinator that synthesizes results.

Do forked sessions share git history?

Each fork starts from the same git commit — the HEAD of your project at the time of the fork. amux creates a new git branch for each fork from that commit, so they share the same ancestor but diverge independently. This means you can diff any fork against main to see exactly what it changed, and merging is a clean branch merge with no unusual history to resolve.

Can I fork from any point in a session's history, not just the current end?

By default, amux fork copies the complete current history up to the moment you run the command. To fork from an earlier point (before a decision was made, for example), you can manually truncate the JSONL history file to the turn you want and then fork. The JSONL format is one JSON object per line, so this is a simple head command on the history file. This advanced use case lets you re-explore a decision that was made several turns ago without re-running the entire planning phase.

What happens to forked sessions if I stop amux?

Forked sessions are registered amux sessions with their own entries in the session registry. When amux restarts, they appear in the dashboard and can be resumed. Their JSONL history is preserved on disk. Any work they committed to their git branches is persisted in git. Work that was in progress but not yet committed when amux stopped may need to be replayed — same behavior as any other session.
