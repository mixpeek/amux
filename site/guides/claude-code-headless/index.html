Claude Code Headless Mode

How to run Claude Code without the interactive terminal — for CI/CD, overnight batches, API-driven workflows, and parallel agent fleets. The complete self-hosting guide.

Why Headless Mode Matters

Claude Code ships as a terminal UI. You type, it responds, you approve tool calls, it executes them. That loop is great for interactive coding sessions and terrible for everything else.

The moment you want to do any of the following, you need headless mode:

  • Run Claude Code from a script. A bash wrapper, a Python subprocess, a Node.js worker — anything that needs to invoke the agent programmatically and capture its output.
  • Use it in CI/CD. A GitHub Action that reviews PRs, a pre-commit hook that fixes lint errors, a nightly job that runs migrations.
  • Run multiple agents in parallel. 10 sessions all working on different branches, with a single operator overseeing the fleet from a dashboard.
  • Operate unattended overnight. Queue tasks before bed, have results waiting in the morning. Not possible with a loop that requires a human to approve tool calls.
  • Integrate with your own UI. Slack bots, web apps, custom IDEs — anything that isn’t the built-in TUI.

Headless mode is the foundation of every advanced Claude Code workflow. It’s also where most of the friction lives: authentication has sharp edges, permissions have dangerous defaults, and supervising a fleet of headless agents is its own problem domain.

This guide walks through all of it.

The Basics: --print and --output-format

The simplest headless invocation is the --print flag (short form -p). It runs a single turn and exits:

# Interactive mode (default)
$ claude
# → enters TUI, waits for input

# Headless mode
$ claude --print "explain the auth flow in src/auth.py"
The auth flow in src/auth.py follows a three-step process...

Without any other flags, --print will still prompt interactively for tool approvals the moment Claude wants to read a file or run a command. That’s rarely useful in automation. For fully non-interactive runs you need to combine --print with permission flags (covered in the next section).

The other flag you’ll use on every headless invocation is --output-format:

FormatOutputWhen to Use
text (default)Human-readable responseQuick one-off scripts, piping to a human
jsonSingle JSON object with response + token usage + costProgrammatic callers, cost tracking, audit logs
stream-jsonNewline-delimited JSON, one event per messageReal-time streaming to a UI, live progress bars

JSON output is the one you want for anything production-grade. It gives you:

{
  "type": "result",
  "subtype": "success",
  "result": "The auth flow in src/auth.py follows...",
  "session_id": "abc123",
  "total_cost_usd": 0.0247,
  "usage": {
    "input_tokens": 1842,
    "output_tokens": 318,
    "cache_read_input_tokens": 12500
  }
}

Now your wrapper script knows how much each run cost, how many tokens it used, and the session ID to resume from.

Authentication: Max Subscription vs API Key

Claude Code supports two authentication paths, and picking the right one for headless workloads has direct cost implications.

MethodHow It WorksBillingBest For
Max subscription claude login stores OAuth tokens in ~/.claude/ Flat monthly fee with weekly rate limit Predictable workloads under the cap; solo developers
API key Set ANTHROPIC_API_KEY environment variable Pay-per-token, no rate cap Bursty workloads, shared team usage, heavy overnight runs
Enterprise Bedrock / Vertex Claude via AWS Bedrock or GCP Vertex AI credentials Billed through your cloud vendor Large orgs with existing cloud contracts and data residency requirements

The practical question most people face: should I run headless agents against my Max subscription, or should I use an API key?

The answer depends on your concurrency and volume:

  • 1–3 agents, steady use: Max subscription is almost always cheaper. You’re paying a flat fee you’ve already committed to.
  • 5+ agents overnight: You’ll hit rate limits within hours. API key is usually the right call for burst capacity.
  • Mixed: Use the Max subscription for your primary interactive agent, and point overnight fleets at ANTHROPIC_API_KEY. You get the flat-rate benefit for everyday work and only pay per-token when bursting.

Switching between the two at runtime is a matter of setting the env var:

# Use Max subscription (default after `claude login`)
$ claude --print "do the thing"

# Force API-key billing for this invocation
$ ANTHROPIC_API_KEY=sk-ant-... claude --print "do the thing"

# Parallel fleet on API key
$ export ANTHROPIC_API_KEY=sk-ant-...
$ for i in {1..10}; do
    claude --print --dangerously-skip-permissions "task $i" &
  done
$ wait

One note on the Max subscription with headless runs: third-party wrappers and harnesses have been the subject of policy changes historically, and Anthropic has increasingly scoped Max subscription usage to first-party Claude Code invocations. If you’re wrapping Claude Code in a larger product, run against an API key — it’s the unambiguous, supported path.

Permission Modes for Unattended Runs

By default, Claude Code asks for human approval before running any tool that modifies state — writing files, running shell commands, making network requests. That’s essential in an interactive session and a blocker in an automated one.

There are three ways to get past the approval prompt:

ApproachFlag / ConfigRisk LevelUse Case
Skip all approvals --dangerously-skip-permissions High — any tool, any action Ephemeral VMs, containers, git worktrees
Allowlist specific tools Permission policy JSON Medium — you define the surface area CI/CD, team environments
Allowlist specific bash prefixes Permission rule Bash(npm test:*) Low — surgical control Scoped workflows (tests only, lint only)

The nuclear option is the aptly-named --dangerously-skip-permissions. It auto-approves every tool call. Use it only when the blast radius is contained:

  • Inside a Docker container with no host volume mounts.
  • Inside a git worktree on a dedicated branch.
  • On an ephemeral cloud VM that gets destroyed after the run.

For anything touching your main checkout, your production credentials, or your laptop’s filesystem, don’t. Use a permission policy instead:

# ~/.claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(npm test:*)",
      "Bash(npm run lint:*)",
      "Bash(git diff:*)",
      "Bash(git status)",
      "Read(src/**)",
      "Read(tests/**)"
    ],
    "deny": [
      "Bash(rm:*)",
      "Bash(git push:*)",
      "Write(.env*)"
    ]
  }
}

This is the pattern most teams converge on: a per-project .claude/settings.json that scopes exactly what headless runs can do, checked into git so the rules evolve with the codebase.

Six Common Headless Patterns

The --print flag is the primitive. Most real workflows compose it into one of these six patterns:

1. One-shot task

Run a single discrete task and capture its output. The simplest pattern, and the starting point for everything else.

$ claude --print --output-format json \
    --dangerously-skip-permissions \
    "Add JSDoc comments to all exported functions in src/" \
    | jq '.result'

2. Resumable session

Run a long task across multiple invocations, maintaining conversation state between them. Useful when each step needs human review before the next continues.

# First turn
$ claude --print --output-format json "Plan the refactor of auth.py" \
    | tee turn1.json
$ SESSION=$(jq -r '.session_id' turn1.json)

# Continue the same session later
$ claude --print --resume $SESSION "Now implement step 1"

3. CI/CD gate

A GitHub Action that runs Claude Code against a PR diff and posts its review as a comment. Headless is the only way to do this.

# .github/workflows/ai-review.yml
- name: AI review
  env:
    ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
  run: |
    diff=$(git diff origin/main...HEAD)
    claude --print --output-format json \
      --dangerously-skip-permissions \
      "Review this diff and flag any bugs or risky changes: $diff" \
      | jq -r '.result' > review.md
    gh pr comment --body-file review.md

4. Batch fan-out

Run the same prompt against many inputs in parallel. Common for test generation, docstring updates, or refactors that apply file-by-file.

$ for file in src/**/*.py; do
    (claude --print --dangerously-skip-permissions \
      "Add type hints to $file" > /dev/null) &
  done
$ wait

5. Pipeline stages

Chain Claude Code into a Unix pipeline, passing output between stages with other tools in between.

$ git log --oneline -20 \
    | claude --print "Group these commits by feature area" \
    | tee changelog-draft.md \
    | claude --print "Rewrite as a user-facing changelog"

6. Scheduled agent

A cron job or scheduler that wakes Claude Code up on a schedule to do maintenance work: update dependencies, regenerate docs, triage new issues.

# /etc/cron.daily/docs-refresh
#!/bin/bash
cd /app
export ANTHROPIC_API_KEY=$(cat /etc/claude-key)
claude --print --dangerously-skip-permissions \
  "Update README.md to reflect any changes in src/ since the last commit"
git add README.md && git commit -m "docs: daily refresh" || true

Supervision: tmux, systemd, and Control Planes

Claude Code headless is not a daemon. Each invocation runs a single turn and exits. For anything that needs to stay alive across turns, across crashes, or across reboots, you need a supervisor.

There are three tiers of supervision, each appropriate for a different scale:

SupervisorWhat It Gives YouWhat It Doesn’t
tmux / screen Persistent session, detach/reattach, scrollback No crash recovery, no cost tracking, no fleet view
systemd / supervisord Auto-restart on crash, log rotation, boot-time startup No conversation state, no inter-agent coordination
Agent control plane (what is this?) All of the above + observability, cost tracking, orchestration, self-healing —

Most teams start with tmux, migrate to systemd when they need crash recovery, and eventually move to a control plane when they’re running enough agents that the operational overhead exceeds the setup cost.

A minimal systemd unit for a single long-lived Claude Code agent looks like this:

# /etc/systemd/system/claude-doc-agent.service
[Unit]
Description=Claude Code — documentation refresher
After=network-online.target

[Service]
Type=simple
User=claude
WorkingDirectory=/app
Environment="ANTHROPIC_API_KEY=sk-ant-..."
ExecStart=/usr/local/bin/claude --print \
    --dangerously-skip-permissions \
    --output-format json \
    "Update documentation based on git log since last run"
Restart=on-failure
RestartSec=60

[Install]
WantedBy=multi-user.target

This works for one or two agents. It does not scale to ten. The moment you have a fleet, you need fleet-level tooling.

Tracking Cost and Token Usage

Headless mode’s biggest operational risk is invisible cost. An agent stuck in a retry loop at 3am can burn through $50–$100 in tokens before anyone notices. You need cost visibility from day one.

Claude Code gives you three levels of cost data, each more detailed than the last:

LevelWhereUse For
Per-invocation total_cost_usd field in JSON output Script-level cost logging, budget gates
Per-session JSONL transcript files in ~/.claude/projects/ Aggregating cost across many turns of a single agent
Per-fleet Control plane dashboard (requires supervision) Budget caps, alerts, team accountability

The JSONL transcript format is the durable record. Every message Claude Code generates gets appended to a file like ~/.claude/projects/<path>/<session-id>.jsonl, with token counts in every entry:

{"type":"assistant","message":{"usage":{"input_tokens":542,"output_tokens":89,"cache_read_input_tokens":14200}},...}
{"type":"assistant","message":{"usage":{"input_tokens":48,"output_tokens":156,"cache_read_input_tokens":15100}},...}

Aggregating these is a few lines of Python:

import json, pathlib

total_input, total_output, total_cached = 0, 0, 0
for line in pathlib.Path("~/.claude/projects/myproj/abc123.jsonl").expanduser().read_text().splitlines():
    msg = json.loads(line).get("message", {})
    u = msg.get("usage", {})
    total_input += u.get("input_tokens", 0)
    total_output += u.get("output_tokens", 0)
    total_cached += u.get("cache_read_input_tokens", 0)

# Claude 4 Opus pricing (example)
cost = (total_input * 15 + total_output * 75 + total_cached * 1.5) / 1_000_000
print(f"Session cost: ${cost:.2f}")

A control plane does this automatically across every session and aggregates by day, by session, and by project. For low-volume use, a cron-driven script that emails you the daily total is enough.

Running Headless Agents in Parallel

The most common use of headless mode is running multiple agents at once. Four things complicate parallel execution that you don’t think about in the single-agent case:

  1. Git conflicts. Two agents modifying the same repo at the same time will step on each other. The standard solution is git worktrees: each agent gets its own branch in its own working directory, merged back only when done.
  2. Rate limiting. On the Max subscription, 10 parallel agents will exhaust your weekly quota in hours. Switch to API key billing for anything over 3–5 concurrent agents.
  3. Output multiplexing. Ten tail’ing terminals is unmanageable. Structured JSON output piped to a central dashboard is the answer.
  4. Task distribution. If you launch 10 agents manually with 10 different prompts, you’re doing orchestration yourself. A shared work queue with atomic task claiming removes that coordination overhead.

A bare-bones parallel launcher looks like this:

# parallel-agents.sh — 5 agents, one task each, separate worktrees
tasks=(
  "Add retry logic to the HTTP client"
  "Improve error messages in auth.py"
  "Write tests for the rate limiter"
  "Update the README install section"
  "Refactor config loading to use pydantic"
)

for i in "${!tasks[@]}"; do
  branch="agent-$i"
  git worktree add "../work-$i" -b "$branch"
  (
    cd "../work-$i"
    claude --print --dangerously-skip-permissions \
      --output-format json "${tasks[$i]}" \
      > "../agent-$i-result.json"
  ) &
done

wait
echo "All 5 agents complete. Check agent-*-result.json"

This works. It also breaks the moment one agent crashes, runs long, or needs the result from another. Which is why the next section exists.

Headless Claude Code in Docker

Running Claude Code in a container is the cleanest way to use --dangerously-skip-permissions safely: the blast radius is the container.

A minimal Dockerfile:

FROM node:22-alpine

RUN apk add --no-cache git bash
RUN npm install -g @anthropic-ai/claude-code

WORKDIR /workspace
ENTRYPOINT ["claude", "--print", "--dangerously-skip-permissions", "--output-format", "json"]

Usage:

$ docker build -t claude-headless .
$ docker run --rm \
    -e ANTHROPIC_API_KEY=sk-ant-... \
    -v "$PWD:/workspace" \
    claude-headless "Refactor the main function to use async/await"

A few things to watch for:

  • Don’t mount your real home directory. Claude stores auth in ~/.claude/ and you don’t want a container writing there.
  • Pin the npm version (@anthropic-ai/claude-code@X.Y.Z) so your container builds are reproducible — headless behavior can drift between releases.
  • Pass the API key via env, not build arg. Build args land in image layers; environment variables at runtime do not.
  • Mount a dedicated project volume rather than bind-mounting the whole repo when you want to isolate changes.

Common Pitfalls

Pitfalls that show up consistently once people start running headless at scale:

  • Forgetting --dangerously-skip-permissions. The script hangs silently waiting for approval. Always pair --print with an explicit permission mode for automation.
  • Letting agents write to ~/.claude/. Container-based runs that accidentally share this path will corrupt each other’s session state. Always use a fresh home directory or an ephemeral volume.
  • No timeouts. A stuck agent will run until you kill it. Always wrap headless invocations in timeout 30m or equivalent.
  • Ignoring exit codes. Claude Code returns non-zero on errors, tool failures, and rate-limit exhaustion. Check the exit code, don’t just tee the output.
  • Unbounded parallelism. Launching 50 agents in parallel against the API will burn through your rate limit in minutes. Cap concurrency with xargs -P, a semaphore, or a queue.
  • No cost limits. A retry loop on a bad prompt can spend $100 in an hour. Always set per-invocation or per-day budget guards.
  • Not saving session IDs. If you don’t capture session_id from the JSON output, you can’t resume. You can’t post-mortem. You can’t debug.

Headless Claude Code with amux

amux is an open-source agent control plane specifically designed for running headless Claude Code at scale. It handles supervision, observability, cost tracking, and orchestration in a single Python file, without requiring any of the scripting above.

The difference is most visible in a side-by-side:

ConcernRaw Headlessamux
Start 10 agents Bash loop + git worktree add + claude --print for each amux start worker-1 worker-2 ... worker-10
See what each is doing 10 log files to tail Web dashboard with per-session live output
Crash recovery systemd unit per agent + Restart=on-failure Built-in watchdog auto-restarts and replays last message
Context overflow Agent fails; you rerun manually tomorrow Auto-compacts and continues; no work lost
Cost tracking Parse JSONL per session, aggregate yourself Per-session and daily dashboards, built-in
Task distribution You pre-split tasks before launching SQLite-backed kanban board; agents claim tasks atomically
Mobile access SSH from your phone Installable PWA + native iOS app

The install is one command:

$ git clone https://github.com/mixpeek/amux && cd amux && ./install.sh
$ amux register myproject --dir ~/Dev/myproject --yolo
$ amux start worker-1 worker-2 worker-3
$ amux serve  # → https://localhost:8822

Under the hood, amux start launches Claude Code in headless mode inside a tmux session per agent, with --dangerously-skip-permissions enabled by the --yolo project flag and cost-tracking auto-wired from the JSONL transcripts. The dashboard at localhost:8822 shows live output, per-session cost, and a kanban board for task claiming — everything the supervision and cost sections of this guide described, already built.

If you’re running one or two headless agents, the raw claude --print approach from earlier in this guide is perfectly fine. If you’re running five or more, or anything overnight, use a control plane — either amux or something equivalent. The scripting to replicate its features yourself is a weekend project, but maintaining that scripting as Claude Code evolves is not.

FAQ

What is Claude Code headless mode?

Headless mode runs Claude Code without the interactive terminal UI. You pass a prompt with --print, Claude executes the task, writes output to stdout, and exits. This is the foundation for CI/CD integration, parallel agents, overnight batches, and any automated workflow.

How do I run Claude Code non-interactively?

Use claude --print "your task". For fully unattended runs, add --dangerously-skip-permissions or a permission policy file. For machine-readable output with token counts and cost, add --output-format json.

Can I use my Max subscription for headless runs?

Yes — headless mode uses the same auth as interactive mode. Your Max subscription covers it under the same weekly rate limit. For sustained parallel usage above your cap, switch to an API key by setting ANTHROPIC_API_KEY.

Should I use the Max subscription or an API key for headless runs?

Max subscription for 1–3 steady agents. API key for 5+ parallel agents or overnight bursts. A common setup combines both: Max for the interactive primary, API key for the overnight fleet.

Is it safe to use --dangerously-skip-permissions?

Only in isolated environments: containers, git worktrees on dedicated branches, or ephemeral VMs. Never on your main laptop with production credentials. For safer unattended operation, use a permission policy that allowlists specific tools.

How do I track cost for headless runs?

Use --output-format json to get total_cost_usd and token counts per invocation. For session-level aggregation, parse the JSONL transcript in ~/.claude/projects/. For fleet-level tracking, use a control plane like amux.

Can I run Claude Code in Docker?

Yes. Install Node.js and the Claude Code CLI in your image, mount your project as a volume, pass ANTHROPIC_API_KEY as an env var, and use claude --print as the entrypoint. Containers are also the safest place to use --dangerously-skip-permissions.

How do I run multiple headless agents in parallel?

Launch each in its own git worktree to avoid conflicts, cap concurrency to stay under rate limits, capture structured JSON output, and use atomic task claiming to prevent duplicate work. A control plane automates all four.

What’s the difference between --print and the interactive REPL?

The REPL loops until you exit; --print runs a single turn and returns. The REPL prompts for tool approvals interactively; --print combined with permission flags is fully non-interactive. Both share authentication and session state.

Can I resume a headless session later?

Yes. Capture the session_id from JSON output and pass --resume <session-id> on the next invocation. The conversation picks up exactly where it left off.
