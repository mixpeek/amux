Understanding the Rate Limits

The Max plan has two types of limits, and confusing them costs people hours of dead time:

Limit type Window What happens Recovery
Short-term (burst)Per minuteSession pauses, retries automatically1–5 minutes
Daily / weekly capRolling 24h / 7dSession blocked until window rollsHours to days

Short-term limits are annoying but manageable — the agent just slows down. Daily and weekly caps are the killer. When you burn through your weekly Opus quota by Tuesday, you're paying $200/month for 2 days of real work.

The limits are per-model. Opus, Sonnet, and Haiku each have independent quotas. This is the single most important thing to understand — and the foundation of every strategy below.

Where Your Tokens Actually Go

Before optimizing, you need to see where the waste is. Most people assume tokens go to code generation. In reality:

~70%
Input tokens — reading files, command output, error messages, context from previous turns
~30%
Output tokens — actual code, tool calls, thinking. The part you care about.

The agent reads your entire codebase context on every turn. A 5,000-line file gets re-read every time the agent makes a change. A failing test suite dumps 500 lines of output that gets processed as input. Most of your quota goes to the agent understanding the world, not changing it.

The top token sinks

  1. Large file reads. The agent reads entire files even when it only needs 10 lines. A 2,000-line file = ~8K tokens per read.
  2. Test output. Running a full test suite that prints 1,000 lines of output = ~4K tokens of context the agent has to process.
  3. Conversation history. Every previous turn stays in context. A 30-turn conversation carries the full history forward.
  4. Exploration. The agent reads files to understand the codebase before making changes. Necessary, but expensive when repeated across sessions.
  5. Retries. When the agent goes off-track and you correct it, you pay for the bad attempt and the correction.

Strategy 1: Model Routing (Biggest Win)

This is the highest-leverage optimization. Route tasks to the cheapest model that can handle them.

Task type Model Why
Complex architecture, multi-file refactorsOpusNeeds deep reasoning across files
Feature implementation, bug fixesSonnetGood enough for 90% of coding tasks
Test generation, documentationSonnetStructured output, doesn't need Opus reasoning
Simple refactors, formatting, renamesHaikuMechanical changes, minimal reasoning needed
Code review, linting suggestionsHaikuPattern matching, not creative work

The math: If you route 60% of your work to Sonnet and 20% to Haiku, you free up your Opus quota for the 20% of tasks that actually need it. That means your Opus quota lasts 5x longer — from Tuesday to the following Monday.

# With amux, set the model per session
$ amux register test-gen --dir ~/Dev/api --yolo --model sonnet
$ amux register arch-review --dir ~/Dev/api --yolo --model opus
$ amux register doc-update --dir ~/Dev/api --yolo --model haiku

Strategy 2: Context Discipline

Every token in the conversation history is re-processed on every turn. A 50-turn conversation with a growing context window becomes exponentially expensive. Cut it ruthlessly.

CLAUDE.md is your best friend

A good CLAUDE.md file gives the agent context once instead of discovering it through expensive exploration. Every minute the agent spends reading random files to understand your project is a minute of wasted Opus tokens.

# Bad: agent explores for 5 minutes, reads 20 files
# Cost: ~40K input tokens just to understand the project

# Good: CLAUDE.md tells it everything upfront
# Cost: ~2K tokens, agent starts coding immediately

Short sessions beat long ones

A 50-turn session carries all 50 turns in context. By turn 50, the agent is re-reading 200K tokens of history on every single response. That's your quota evaporating.

The fix: Short, focused sessions. One task per session. 5–15 turns. When the task is done, close the session. The next task starts fresh with a clean context window.

1 session, 50 turns
~800K total tokens. By turn 40, context is so large that compaction kicks in and the agent forgets early decisions.
5 sessions, 10 turns each
~250K total tokens. Each session is focused, context stays small, agent stays sharp. Same work, 3x fewer tokens.

Targeted file reads

In your CLAUDE.md, tell the agent which files matter for what. This prevents the agent from reading your entire src/ directory to find the auth middleware:

# In CLAUDE.md:
Auth middleware: src/middleware/auth.ts
Database models: src/models/ (one file per table)
API routes: src/routes/ (grouped by resource)
Tests: tests/ (mirrors src/ structure)

Strategy 3: Parallel Agents

This is counterintuitive: running more agents doesn't use more tokens. 5 agents doing 5 tasks in parallel use the same total tokens as 1 agent doing 5 tasks sequentially. But they finish 5x faster.

Why this matters for rate limits: if your daily cap resets at midnight, you want to burn through your quota as efficiently as possible during the hours you have. 1 agent running for 8 hours wastes time between tasks (thinking, exploring, re-reading). 5 agents running for 2 hours do the same work with less per-session overhead.

Parallel agents actually save tokens

Each parallel agent starts with a fresh context window. That means:

  • No accumulated conversation history from previous tasks
  • No context pollution from unrelated file reads
  • No compaction artifacts from hitting context limits
  • Each agent reads only the files relevant to its specific task

In practice, 5 parallel focused sessions use 20–40% fewer total tokens than 1 session doing all 5 tasks sequentially, because the sequential session carries growing context baggage.

# Sequential: 1 agent, 5 tasks, growing context
# Task 1: 50K tokens (clean context)
# Task 2: 65K tokens (carrying Task 1 history)
# Task 3: 80K tokens (carrying Tasks 1-2 history)
# Task 4: 95K tokens (carrying Tasks 1-3 history)
# Task 5: 110K tokens (carrying Tasks 1-4 history)
# Total: 400K tokens

# Parallel: 5 agents, 1 task each, fresh context
# Agent 1: 50K tokens
# Agent 2: 50K tokens
# Agent 3: 50K tokens
# Agent 4: 50K tokens
# Agent 5: 50K tokens
# Total: 250K tokens (37% savings)

Strategy 4: Task Pipelining

Don't let agents sit idle. When one finishes, immediately start the next task in its session slot. Think of it like a factory assembly line, not a single worker doing everything.

The pipeline

Hour 1:  [Agent 1: API endpoints]  [Agent 2: tests]    [Agent 3: docs]
Hour 2:  [Agent 1: auth refactor]  [Agent 2: UI tests]  [Agent 3: README]
Hour 3:  [Agent 1: migration]      [Agent 2: e2e tests] [Agent 3: changelog]

Each agent finishes a task, closes the session (clearing context), and immediately starts the next one fresh. No accumulated context, no wasted quota on history.

Strategy 5: Specs Over Prompts

A vague prompt causes the agent to explore, ask questions, try approaches, backtrack, and retry. Every one of those interactions burns tokens. A clear spec eliminates them.

Vague prompt
"Add pagination to the users API"

Agent explores 8 files. Picks offset-based. You correct to cursor-based. Agent rewrites. You add "don't change the schema." Agent rewrites again.

Result: 3 rounds, ~120K tokens, 45 minutes.
Spec
"Cursor-based pagination on GET /users. Key on created_at. Default 50, max 200. Don't change the schema. Acceptance: curl returns {data, next_cursor}."

Agent reads 2 files. Implements once. Done.

Result: 1 round, ~40K tokens, 12 minutes.

Specs are a 3x token multiplier. Every minute spent writing a spec saves 3 minutes of agent exploration and 3x the tokens. See our spec-driven development guide for templates.

Strategy 7: Schedule Around Resets

Rate limits have rolling windows. If your weekly cap resets on Monday at midnight UTC, schedule your heaviest workloads for Monday and Tuesday. Save Wednesday–Friday for lighter, interactive work.

The weekly rhythm

Monday:    Heavy batch — overnight fleet of 10 agents (test gen, refactors, docs)
Tuesday:   Heavy batch — feature implementation, 5-8 agents
Wednesday: Interactive — architecture decisions, code review (Opus, light usage)
Thursday:  Interactive — bug fixes, small features (Sonnet)
Friday:    Pipeline — ship: merge branches, deploy, cleanup

This matches natural development cycles too: decompose over the weekend, execute Monday–Tuesday, refine Wednesday–Friday.

FAQ

Q Why does Claude Code hit rate limits so fast?
Claude Code is token-hungry by design. Every tool call — reading files, running commands, processing output — counts as input tokens. A single complex session can burn 500K–1M tokens per hour. The key insight: it's not about using fewer tokens per task, it's about getting more useful output per token spent.
Q Can I run multiple sessions on one Max plan?
Yes. The limit is on total tokens per time window, not on session count. 5 agents each doing focused tasks use the same total tokens as 1 agent doing them sequentially — but finish 5x faster. Parallel agents don't increase token spend; they compress calendar time.
Q Is the Max plan worth it for parallel agents?
Yes, if you run more than 3 concurrent agents regularly. The Pro plan ($20/month) hits limits within the first hour of parallel work. Max pays for itself by giving you the headroom to actually run agents at scale. For overnight fleets of 10+ agents, consider supplementing with API billing.
Q Does using Sonnet/Haiku help with rate limits?
Significantly. Rate limits are per-model. Routing 60% of work to Sonnet and 20% to Haiku means those tasks don't count against Opus quota at all. This is the single highest-leverage optimization.
Q Can I use the API instead to avoid caps?
Yes. The Anthropic API has separate, higher rate limits and charges per token with no weekly ceiling. For 10+ parallel agents or overnight runs, API billing is often more cost-effective. Claude Code supports both — you can use the Max plan for daytime interactive work and API keys for overnight batch runs.
Q What happens when I hit the rate limit?
Short-term limits: the session pauses and retries automatically in 1–5 minutes. Daily/weekly caps: the session is blocked until the window resets. With amux, the watchdog detects rate-limited sessions and can switch to a different model tier or queue the work for later.
Q How many parallel agents on the Max plan?
No limit on concurrent sessions — only on total tokens. Practically: 3–5 Sonnet agents are sustainable on the $100 plan. For 10+ agents, use the $200 plan or API billing. Short, focused tasks (15–30 min) use fewer tokens than long exploratory sessions.
Q How do I track which sessions use the most tokens?
amux shows per-session token spend on every session card in the dashboard. Sort by token count to find the heaviest consumers. The Anthropic Console also shows aggregate API usage. Between the two, you can see both per-session and account-level token consumption.
