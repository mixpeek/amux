Vibe Debugging: How to Fix AI-Generated Code That Breaks

Your app worked in the demo. The AI generated it in 20 minutes. Now it’s in production and users are hitting errors you’ve never seen before. Welcome to vibe debugging — the discipline of systematically fixing code you didn’t write, may not fully understand, and need working by tomorrow morning. This guide covers the 7 ways vibe-coded apps break, the tools and workflows that fix them, and why multi-agent investigation is replacing single-threaded debugging.

The short version

What is vibe debugging? Systematically diagnosing and fixing bugs in AI-generated code — code produced by Claude Code, Cursor, Codex, v0, Bolt, or Lovable with minimal human review.

Why it’s different: You’re debugging code you didn’t write and don’t fully understand. That changes the workflow.

Key insight: Use multiple AI agents to debug what a single agent generated. Parallel investigation catches what serial debugging misses.

Best tools: Claude Code or Codex CLI for investigation, TypeScript / Ruff for static analysis, Sentry for production monitoring, amux for running parallel debugging sessions.

What Is Vibe Debugging?

Vibe coding — the practice of describing what you want and letting an AI build it — went mainstream in 2025 after Andrej Karpathy coined the term. Tools like Cursor, Claude Code, v0, Bolt, Lovable, and Replit made it possible for anyone to generate working applications in minutes.

But “working” in a demo and “working” in production are different things. By mid-2026, a pattern has emerged: vibe-coded apps that handle 10 users collapse at 100. Features that passed a quick manual test break under real-world input. Error handling that looked correct turns out to be a thin wrapper around try/catch blocks that swallow every exception.

Vibe debugging is the discipline that emerged from this gap. It’s the systematic process of diagnosing and fixing code that:

  • You didn’t write and may not fully understand
  • Was generated in a single pass without architectural planning
  • Works for the happy path but fails on edge cases
  • Has few or no tests
  • Uses libraries and patterns you didn’t choose

The term has been spreading across developer communities on Reddit and Hacker News since early 2026, as developers who enthusiastically adopted vibe coding hit the wall of maintaining what they generated.

The 7 Ways Vibe-Coded Apps Break

After analyzing thousands of AI-generated codebases, these are the failure patterns that appear again and again. Recognizing which pattern you’re dealing with is the first step of any vibe debugging session.

1. Shallow Logic

The code handles the happy path perfectly but has no concept of edge cases. A form works with valid input but crashes on empty strings. An API endpoint returns correct data for the first page but breaks on pagination. The AI optimized for the demo, not for reality.

2. Hallucinated APIs

The AI referenced functions, methods, or API endpoints that don’t exist in the library version you’re using. response.json.data when the actual field is response.body.results. A React hook that was renamed three versions ago. These are silent until the code path executes.

3. Error Handling Theater

The AI added try/catch blocks everywhere — but the catch blocks either swallow errors silently, log a generic message, or re-throw without context. The app looks like it handles errors. It doesn’t. When something fails, you get no useful information about what went wrong.

4. Dependency Roulette

The AI pulled in 12 packages where 3 would suffice. Some are unmaintained. Some have overlapping functionality. Some have known vulnerabilities that npm audit or pip-audit will flag. The dependency tree is a liability you didn’t choose.

5. Implicit State Assumptions

The code assumes things about state that are true in a single-user demo but false under concurrency. Two users editing the same resource. A WebSocket that reconnects but doesn’t re-sync state. A cache that’s never invalidated. Race conditions that only appear under load.

6. The Test Desert

No tests. Or worse — tests that the AI generated alongside the code that test implementation details instead of behavior, mock everything so they can’t catch real bugs, and pass even when the code is broken. A green test suite that provides zero confidence.

7. Copy-Paste Propagation

The AI duplicated a pattern across files instead of abstracting it. When one instance has a bug, every instance has the same bug. Fix it in one place and you’ve fixed 1 of 7. Grep for the pattern, and you’ll find it everywhere — each copy slightly different, each copy slightly broken.

The compounding problem: These failures rarely occur in isolation. A vibe-coded app typically has shallow logic (1) with hallucinated APIs (2) and no tests (6). Fixing one bug reveals three more. This is why serial debugging fails — you need parallel investigation to map the full scope of the problem before you start fixing.

Vibe Debugging vs Traditional Debugging

Vibe debugging isn’t just “debugging but for AI code.” The workflow is fundamentally different because your starting point is different.

Dimension Traditional Debugging Vibe Debugging
Code familiarity You wrote it (or your team did) AI wrote it — you may not understand the architecture
First step Reproduce the bug Understand the code, then reproduce the bug
Root cause analysis Read the code, trace the logic Ask an AI agent to explain the code, then trace the logic
Test coverage Usually some tests exist Often zero meaningful tests
Bug distribution Localized to the change that introduced it Spread across files via copy-paste propagation
Dependency knowledge Team chose the libraries deliberately AI chose libraries — may be unfamiliar or inappropriate
Fix confidence High (you understand the system) Low without tests — fixing one thing often breaks another
Best approach Single developer, focused session Multiple AI agents investigating in parallel

The Vibe Debugging Toolkit

Effective vibe debugging requires five categories of tools. You don’t need all of them for every bug, but you should have each category available.

Investigation

Claude Code

Terminal-based AI agent. Ask it to explain unfamiliar code, trace data flow, or generate hypotheses about why something broke. Best for deep investigation of complex bugs. Supports context engineering via CLAUDE.md.

Investigation

Codex CLI

OpenAI’s terminal agent. Strong at step-by-step debugging with its built-in sandboxing. Good for investigating bugs in isolation without risking side effects. Filesystem restrictions prevent accidental damage during debugging.

Investigation

Aider

Open-source AI pair programming tool with strong git integration. Useful for vibe debugging because it shows exactly what it’s changing in diff format, making it easy to review fixes before committing. Works with multiple LLM backends.

Static Analysis

TypeScript / Mypy

Type checkers catch hallucinated APIs immediately. If the AI called response.data.items but the type says response.body.results, the type checker flags it before runtime. The single highest-ROI tool for vibe debugging.

Static Analysis

ESLint / Ruff

Linters catch patterns that type checkers miss — unused variables, unreachable code, inconsistent returns, unsafe regex. Ruff checks an entire Python project in under a second. Run these before you start debugging to find the low-hanging bugs.

Monitoring

Sentry / Datadog

Error monitoring gives you the actual stack traces from production. Without it, vibe debugging is guesswork. Sentry’s free tier is enough for most projects. Feed the stack traces to your AI debugging agent for faster root cause analysis.

Testing

pytest / Jest / Playwright

You need tests to verify your fixes actually work and don’t break something else. pytest for Python, Jest for JavaScript, Playwright for end-to-end. Write the test before you fix the bug — confirm it fails, then make it pass.

Orchestration

amux

Agent multiplexer for running parallel debugging sessions. Spin up 3–5 Claude Code agents investigating different subsystems simultaneously. Shared board for tracking findings. Self-healing watchdog restarts agents that crash mid-investigation.

6 Debugging Workflows That Work

Each workflow targets a specific failure pattern. Start with the one that matches your symptoms.

1. The Comprehension-First Debug

Best for: unfamiliar AI-generated codebases Time: 30–60 min

When you’re staring at code you didn’t write and don’t understand, investigation must precede debugging. Don’t guess — map the system first.

1

Map the architecture. Ask an AI agent: “Read every file in this project and give me a one-paragraph summary of the architecture — what frameworks, what data flow, what the entry points are.”

2

Trace the bug path. Starting from the user-facing symptom, ask the agent to trace the code path from the UI action to the backend response. Identify every function in the chain.

3

Check assumptions. For each function in the chain, verify: does this function handle null input? Does it handle the error case? Does the return type match what the caller expects?

4

Write a failing test. Before fixing anything, write a test that reproduces the bug. Now you have a concrete target.

When to use: This is your default workflow for any vibe-coded project you’re seeing for the first time. Skip step 1 only if you already understand the architecture.

2. The Static Analysis Sweep

Best for: hallucinated APIs, type errors, dead code Time: 10–20 min

Before debugging individual symptoms, let machines find the obvious bugs. This workflow catches failure patterns 2 (hallucinated APIs) and 7 (copy-paste propagation) in minutes.

1

Run the type checker. tsc --noEmit for TypeScript, mypy . for Python with Mypy, pyright for Pyright. Every error is a real bug or a real gap in type coverage.

2

Run the linter. ruff check . for Python, eslint . for JavaScript/TypeScript. Focus on errors, not style warnings.

3

Audit dependencies. npm audit or pip-audit. Check for known vulnerabilities and unused packages.

4

Feed results to an AI agent. Copy the type checker and linter output into Claude Code or Codex and ask it to fix all errors. Review the fixes before committing.

When to use: Run this workflow first on any vibe-coded project, even before you know what the specific bug is. It finds 30–40% of issues automatically.

3. The Git Bisect + AI Hybrid

Best for: regressions in projects with commit history Time: 15–30 min

If the app used to work and now it doesn’t, git bisect finds the breaking commit. Then an AI agent analyzes what changed.

1

Start bisecting. git bisect start, mark a known-good commit and the current broken state. Let git find the culprit.

2

Analyze the breaking commit. Once bisect identifies the commit, ask an AI agent: “This commit introduced a regression. Read the diff and explain what it changed and why it might cause [symptom].”

3

Generate a targeted fix. With the breaking change identified, the fix is usually narrow. Ask the agent to fix the specific issue while preserving the intent of the original commit.

When to use: Any time you can say “this used to work.” Especially effective for vibe-coded apps where the AI made multiple commits and one of them introduced a subtle break.

4. The Test Harness Retrofit

Best for: test deserts (pattern 6), building confidence for refactoring Time: 1–2 hours

You can’t safely fix code that has no tests. This workflow creates a test safety net before you start changing things.

1

Identify the critical paths. What are the 5–10 most important user flows? Login, create, read, update, delete, payment, etc.

2

Generate integration tests. Ask an AI agent to write integration tests (not unit tests) for each critical path. Use pytest with a real database, or Playwright for end-to-end browser tests.

3

Run them and fix the tests, not the code. The tests should pass against the current code. If they don’t, fix the tests — you’re characterizing existing behavior, not changing it.

4

Now debug with confidence. With a test harness in place, every fix you make can be validated. Run tests after each change. Regressions are caught immediately.

When to use: Before any significant refactoring or bug-fixing in a test-free codebase. The time investment pays back immediately. Use multiple AI agents in parallel — each writing tests for a different subsystem.

5. The Dependency Audit

Best for: dependency roulette (pattern 4), unexplained runtime errors Time: 20–40 min

When errors point to library code or the app breaks after npm install / pip install, the problem is in the dependency tree.

1

Map what’s installed. Run npm ls --depth=0 or pip list. Ask an AI agent: “Which of these packages are redundant? Which have overlapping functionality?”

2

Check for version conflicts. npm ls 2>&1 | grep ERESOLVE or pip check. AI-generated code often pins incompatible versions.

3

Audit for vulnerabilities. npm audit or pip-audit. AI agents pull in packages without checking the security advisory database.

4

Remove and replace. Drop packages that are unmaintained, insecure, or redundant. Ask the AI agent to replace them with well-maintained alternatives. Run tests after each removal.

When to use: When your node_modules is 800MB, when you see deprecation warnings on startup, or when errors originate from inside a dependency.

6. The Production Trace

Best for: bugs that only appear in production Time: varies

Some vibe-coded bugs only manifest under real traffic — race conditions, memory leaks, state corruption under concurrency. You need production data to debug them.

1

Add monitoring if you haven’t. Sentry takes 5 minutes to set up. You need actual stack traces, not user reports that say “it’s broken.”

2

Collect error patterns. Group errors by endpoint, user action, or time of day. Look for correlation — does the error only happen during high traffic? After a specific action? With specific input?

3

Feed traces to an AI agent. Give the agent the stack trace, the request that triggered it, and any relevant log context. Ask it to identify the root cause and suggest a fix.

4

Write a load test. Use k6, Locust, or Playwright to simulate the traffic pattern that triggers the bug. Verify your fix survives load before deploying.

When to use: When local testing passes but production is broken. Especially common with vibe-coded apps that have implicit state assumptions (pattern 5).

Multi-Agent Debugging: Divide and Conquer

The most effective vibe debugging approach in 2026 isn’t a single developer staring at code. It’s multiple AI agents investigating different angles of the same problem simultaneously.

Why? Because vibe-coded bugs compound. Fixing one reveals three more. A single debugging agent gets stuck in a serial loop — fix, test, discover new bug, context-switch, fix, test. With parallel agents, you map the full problem space before committing to fixes.

The 4-Agent Investigation Pattern

This pattern works for any vibe-coded app with multiple interacting issues. Each agent runs in its own session with amux, reporting findings to a shared board.

Agent 1: Architecture Mapper

Reads every file, maps the module structure, data flow, and API boundaries. Reports: “Here’s how the system actually works.” Identifies dead code, orphaned routes, and circular dependencies.

Agent 2: Type & Lint Auditor

Runs TypeScript / Mypy, ESLint / Ruff, and dependency audits. Reports every type error, unused import, and vulnerable package. Fixes what it can automatically.

Agent 3: Test Writer

Generates integration tests for every critical path using pytest, Jest, or Playwright. Runs them against the current code. Reports which paths pass and which are broken.

Agent 4: Bug Hunter

Takes the specific reported symptoms and traces each one from the user action to the root cause. Uses findings from agents 1–3 to accelerate investigation. Proposes targeted fixes.

Why this works: Agent 1 gives you a map. Agent 2 finds the surface-level bugs. Agent 3 creates a safety net. Agent 4 fixes the actual problem. They run in parallel, so the whole investigation takes 15–20 minutes instead of 2 hours. Use amux to manage the sessions and aggregate findings on the shared board. Each agent works in its own git worktree to avoid conflicts.

Approach Time to Root Cause Bug Coverage Scales to Complex Bugs Requires Codebase Knowledge
Manual debugging Hours Low (serial) Yes (if experienced) Yes
Single AI agent 30–60 min Medium Sometimes No
Multi-agent parallel 15–20 min High (parallel) Yes No

When to Debug vs. When to Rewrite

Not every vibe-coded project is worth debugging. Sometimes the fastest path forward is to regenerate the code with better constraints. Here’s the decision framework.

The one-sentence test

If you can explain the bug to a colleague in one sentence (“the pagination endpoint returns duplicates when page size changes”), debug it. If you need a whiteboard (“the state management is broken in ways that affect authentication, caching, and real-time updates simultaneously”), consider rewriting.

Debug when…

  • The issue is localized — 1–2 files, one subsystem
  • The architecture is sound — structure is good, implementation has bugs
  • Some tests exist — you can verify your fixes
  • The code is deployed — rewriting means downtime
  • You understand the fix — you can describe what needs to change

Rewrite when…

  • No tests, no types — you can’t verify anything works
  • 3+ subsystems are broken — bugs are intertwined
  • The architecture is wrong — not bad implementation but bad design
  • Debugging takes longer than generation — the ROI has flipped
  • You’re pre-launch — no users, no migration concerns

If you decide to rewrite, don’t just re-prompt the same way. Use spec-driven development — write a detailed spec with acceptance criteria, break it into small tasks, and generate each piece separately with tests. An amux session per task keeps agents focused and prevents the sprawling, interconnected code that caused the bugs in the first place.

Prevention: Stop Bugs Before They Ship

The best vibe debugging is the debugging you never have to do. These practices catch AI-generated bugs before they reach production.

Type everything

Use TypeScript instead of JavaScript. Use Mypy or Pyright with strict mode for Python. Type checkers catch hallucinated APIs at compile time instead of in production at 2am.

Lint aggressively

Ruff checks a Python project in milliseconds. ESLint with typescript-eslint catches unsafe patterns. Use Claude Code hooks to run linting after every AI edit.

Require tests

Configure your AI agent to write tests alongside code. In CLAUDE.md: “Every new function must have a corresponding test. Every bug fix must include a regression test.” Enforce with pre-commit hooks.

Review before merging

AI-generated PRs need review just like human PRs. Use a cross-model review pattern — if Cursor wrote the code, have Claude Code review it. Different models catch different mistakes. See the review guide.

Write specs first

Don’t vibe code from a one-line prompt. Write a spec with acceptance criteria, data models, and error cases. The AI produces dramatically better code from a detailed spec. See spec-driven development.

Constrain the AI

Use CLAUDE.md, .cursorrules, or AGENTS.md to restrict which libraries the AI uses, enforce coding standards, and require error handling patterns. See config files compared.

Use quality gates

Automate the prevention pipeline: type check → lint → test → security scan → cross-model review. Any failure blocks the merge. See quality gates guide for full pipeline configurations.

Break large tasks

One prompt generating an entire app produces tangled, untestable code. Decompose into small, independent tasks with clear interfaces. Use amux’s board to track them and assign each task to a separate agent session.

FAQ

What is vibe debugging?

Vibe debugging is the process of systematically diagnosing and fixing bugs in AI-generated code — code that was produced by tools like Claude Code, Cursor, Codex, v0, Bolt, or Lovable with minimal human review. The term emerged in 2026 as a counterpart to “vibe coding” — where vibe coding is about generation, vibe debugging is about the cleanup that follows when generated code breaks.

Why does AI-generated code break in production?

Seven common reasons: (1) shallow logic that handles the happy path but not edge cases, (2) hallucinated APIs or functions that don’t exist in the library version you’re using, (3) error handling theater — try/catch blocks that swallow errors silently, (4) dependency bloat from packages the AI chose without evaluating alternatives, (5) implicit state assumptions that fail under concurrency, (6) no meaningful tests, and (7) copy-paste propagation that duplicates bugs across files.

How is vibe debugging different from normal debugging?

Traditional debugging starts with understanding the code you wrote. Vibe debugging starts with understanding code you didn’t write and may not fully comprehend. The developer didn’t make the architectural decisions, didn’t choose the libraries, and may not understand the internal data flow. This means vibe debugging requires an extra “comprehension” phase before root-cause analysis. Multi-agent approaches help by parallelizing this comprehension — separate agents can map the architecture, trace data flow, and audit dependencies simultaneously.

Can I use AI to debug AI-generated code?

Yes — and it’s often the most effective approach. A single AI agent writing code can produce bugs because of context window limits, hallucinations, and missing constraints. But multiple AI agents reviewing that code from different angles (security, performance, correctness, edge cases) catch problems the generating agent missed. The key is using different prompts and sometimes different models for investigation versus generation. Claude Code, Codex CLI, and Aider are all effective debugging agents. Running them in parallel with amux lets you investigate multiple hypotheses simultaneously.

What tools are best for vibe debugging?

The most effective stack combines: (1) a terminal AI agent like Claude Code or Codex CLI for investigation, (2) static analysis with TypeScript, ESLint, Ruff, or Mypy to catch type errors and code smells, (3) error monitoring with Sentry or Datadog, (4) test frameworks like pytest, Jest, or Playwright, and (5) an agent orchestrator like amux for parallel debugging sessions.

When should I debug vibe-coded code versus rewrite it?

Debug when: the issue is localized (1–2 files), the architecture is sound but the implementation has bugs, some test coverage exists, or the code is deployed and a rewrite would cause downtime. Rewrite when: there are no tests, no types, the bugs span 3+ subsystems, the architecture itself is wrong (not just the implementation), or you’ve spent more time debugging than the original generation took. A useful heuristic: if you can explain the bug in one sentence, debug it. If you need a whiteboard, rewrite it.

How do I prevent vibe coding bugs in the first place?

Write specs before generating code. Use type checking (TypeScript, Mypy) and linting (ESLint, Ruff). Require tests alongside generated code. Review AI output before committing. Use CLAUDE.md or .cursorrules to constrain the AI’s choices. Break large features into small, testable units. See the quality gates guide for full pipeline configurations.
