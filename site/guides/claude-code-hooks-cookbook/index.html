Claude Code Hooks Cookbook

20 production-ready hook recipes — copy the JSON into your settings, drop the script into .claude/hooks/, and you’re done. Battle-tested across thousands of parallel agent sessions.

Hooks are the enforcement layer of context engineering. Your CLAUDE.md tells the agent what to do; hooks guarantee it. A CLAUDE.md instruction is followed ~70% of the time. A hook enforces at 100% — the tool call physically cannot proceed if the hook blocks it.

Every recipe below is self-contained. Copy the settings.json snippet, save the optional script, and it works. No build step, no dependencies beyond bash and python3.

How Hooks Work

Hooks are shell commands configured in .claude/settings.json that fire on specific events. The agent doesn’t decide whether to run them — they execute unconditionally.

PreToolUse

Runs before a tool call. Exit 2 to block the action. Use for safety gates, policy enforcement, and input validation.

PostToolUse

Runs after a tool call succeeds. Use for side effects: testing, linting, notifications, dashboard updates.

Notification

Fires on agent notifications. Use for alerts, logging, or forwarding to Slack/Discord/email.

Stop

Runs when the agent stops. Use for cleanup, final reports, or triggering the next step in a pipeline.

The matcher field filters which tool triggers the hook. Use Bash, Edit, Write, Read, or any tool name. Combine with | for multiple tools. Use * to match everything.

Anatomy of a hook config 
{
  "hooks": {
    "PreToolUse": [           // event type
      {
        "matcher": "Bash",    // which tool(s) to intercept
        "hooks": [
          {
            "type": "command",
            "command": ".claude/hooks/my-hook.sh"
          }
        ]
      }
    ]
  }
}
What the hook script receives on stdin 
{
  "tool_name": "Bash",
  "tool_input": {
    "command": "rm -rf /tmp/data"
  }
}

// PostToolUse also includes:
{
  "tool_name": "Edit",
  "tool_input": { "file_path": "src/main.py", "old_string": "...", "new_string": "..." },
  "tool_output": "File edited successfully"
}

0Allow the tool callSuccess — stdout fed back to model
2Block the tool call — stderr sent as errorN/A (tool already ran)
OtherWarning logged, tool proceedsWarning logged

Security & Safety

PreToolUse hooks that prevent dangerous actions before they happen. These are non-negotiable for unattended or autonomous agent runs.

1. Block Secret Leaks in File Writes

Security

Scans every file write for API keys, tokens, passwords, and private keys. Blocks the write and tells the agent what it found.

settings.json 
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{
          "type": "command",
          "command": ".claude/hooks/block-secrets.sh"
        }]
      }
    ]
  }
}
.claude/hooks/block-secrets.sh 
#!/bin/bash
set -euo pipefail
INPUT=$(cat)
NEW=$(echo "$INPUT" | python3 -c "
import sys, json
d = json.load(sys.stdin)
# For Edit, check new_string; for Write, check content
print(d.get('tool_input',{}).get('new_string',''))
print(d.get('tool_input',{}).get('content',''))
")

PATTERNS=(
  'AKIA[0-9A-Z]{16}'           # AWS access key
  'sk-[a-zA-Z0-9]{20,}'        # OpenAI / Anthropic API key
  'ghp_[a-zA-Z0-9]{36}'        # GitHub personal access token
  'xoxb-[0-9]+-[a-zA-Z0-9]+'   # Slack bot token
  '-----BEGIN .* PRIVATE KEY'   # PEM private key
  'password\s*=\s*["\x27][^"\x27]{8,}' # Hardcoded password
)

for pat in "${PATTERNS[@]}"; do
  if echo "$NEW" | grep -qE "$pat"; then
    echo "BLOCKED: detected secret pattern: $pat" >&2
    echo "Use an environment variable or .env file instead." >&2
    exit 2
  fi
done
exit 0
Works with both Edit and Write tools. Add your own regex patterns for internal tokens.

2. Block Destructive Bash Commands

Security

Blocks rm -rf /, DROP TABLE, git push --force, and other commands that cause irreversible damage.

settings.json 
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": ".claude/hooks/block-destructive.sh"
        }]
      }
    ]
  }
}
.claude/hooks/block-destructive.sh 
#!/bin/bash
set -euo pipefail
CMD=$(cat | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('command',''))")

BLOCKED=(
  "rm -rf /"
  "rm -rf ~"
  "rm -rf \$HOME"
  "rm -rf ."
  "git push --force"
  "git push -f"
  "git reset --hard"
  "DROP TABLE"
  "DROP DATABASE"
  "TRUNCATE TABLE"
  ":(){:|:&};:"
  "mkfs."
  "dd if="
  "> /dev/sda"
)

for pattern in "${BLOCKED[@]}"; do
  if echo "$CMD" | grep -qiF "$pattern"; then
    echo "BLOCKED: destructive command detected: $pattern" >&2
    echo "This command is on the blocklist. Ask the user for confirmation." >&2
    exit 2
  fi
done
exit 0
Essential for YOLO mode and autonomous runs. Catches both exact matches and substrings.

3. Enforce Branch Policy

Security

Blocks direct commits to main and master. Forces agents to work on feature branches.

settings.json 
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": ".claude/hooks/enforce-branch.sh"
        }]
      }
    ]
  }
}
.claude/hooks/enforce-branch.sh 
#!/bin/bash
set -euo pipefail
CMD=$(cat | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('command',''))")

# Only check git commit commands
echo "$CMD" | grep -q "git commit" || exit 0

BRANCH=$(git rev-parse --abbrev-ref HEAD 2>/dev/null || echo "unknown")
if [[ "$BRANCH" == "main" || "$BRANCH" == "master" ]]; then
  echo "BLOCKED: cannot commit directly to $BRANCH" >&2
  echo "Create a feature branch first: git checkout -b feat/your-feature" >&2
  exit 2
fi
exit 0
Combine with recipe #2 to also block git push --force to protected branches.

4. Sandbox File Writes to Project Directory

Security

Prevents the agent from writing files outside the project root. Catches path traversal and accidental writes to system directories.

settings.json 
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{
          "type": "command",
          "command": ".claude/hooks/sandbox-writes.sh"
        }]
      }
    ]
  }
}
.claude/hooks/sandbox-writes.sh 
#!/bin/bash
set -euo pipefail
PROJECT_ROOT=$(git rev-parse --show-toplevel 2>/dev/null || pwd)
FILE_PATH=$(cat | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('file_path',''))")

# Resolve to absolute path
RESOLVED=$(cd "$PROJECT_ROOT" && python3 -c "import os; print(os.path.realpath('$FILE_PATH'))")

if [[ ! "$RESOLVED" == "$PROJECT_ROOT"* ]]; then
  echo "BLOCKED: write target is outside project directory" >&2
  echo "Target: $RESOLVED" >&2
  echo "Allowed: $PROJECT_ROOT/*" >&2
  exit 2
fi
exit 0
Critical for multi-tenant setups or when agents run in shared environments.

5. Protect .env and Credentials Files

Security

Blocks edits to .env, credentials.json, *.pem, and other sensitive config files.

settings.json 
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{
          "type": "command",
          "command": ".claude/hooks/protect-config.sh"
        }]
      }
    ]
  }
}
.claude/hooks/protect-config.sh 
#!/bin/bash
set -euo pipefail
FILE=$(cat | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('file_path',''))")
BASE=$(basename "$FILE")

PROTECTED=(
  ".env" ".env.local" ".env.production"
  "credentials.json" "service-account.json"
  ".npmrc" ".pypirc"
)

for p in "${PROTECTED[@]}"; do
  if [[ "$BASE" == "$p" ]]; then
    echo "BLOCKED: $BASE is a protected config file" >&2
    echo "Editing credentials files is not allowed in automated runs." >&2
    exit 2
  fi
done

# Also block by extension
case "$BASE" in
  *.pem|*.key|*.p12|*.pfx|*.jks)
    echo "BLOCKED: cannot modify certificate/key file: $BASE" >&2
    exit 2 ;;
esac
exit 0
Add your own filenames to the PROTECTED array for project-specific secrets.

Code Quality

PostToolUse hooks that validate output after every change. These catch mistakes while the agent is still in context and can fix them immediately.

6. Validate Python Syntax After Every Edit

Quality

Parses edited Python files with ast.parse() after every write. Catches syntax errors before they snowball into broken imports across the project.

settings.json 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{
          "type": "command",
          "command": ".claude/hooks/check-python.sh"
        }]
      }
    ]
  }
}
.claude/hooks/check-python.sh 
#!/bin/bash
set -euo pipefail
FILE=$(cat | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('file_path',''))")

# Only check Python files
[[ "$FILE" == *.py ]] || exit 0

if ! python3 -c "import ast; ast.parse(open('$FILE').read())" 2>/tmp/hook-syntax-err.txt; then
  echo "SYNTAX ERROR in $FILE:" >&2
  cat /tmp/hook-syntax-err.txt >&2
  exit 2
fi
echo "Syntax OK: $FILE"
This is the hook amux uses in production for its single-file Python server.

7. Auto-Format Code on Save

Quality

Runs the appropriate formatter after every file write — Black for Python, Prettier for JS/TS/CSS/HTML, gofmt for Go.

settings.json 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{
          "type": "command",
          "command": ".claude/hooks/auto-format.sh"
        }]
      }
    ]
  }
}
.claude/hooks/auto-format.sh 
#!/bin/bash
set -euo pipefail
FILE=$(cat | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('file_path',''))")

case "$FILE" in
  *.py)
    command -v black >/dev/null && black --quiet "$FILE" 2>/dev/null
    ;;
  *.js|*.ts|*.tsx|*.jsx|*.css|*.html|*.json)
    command -v prettier >/dev/null && prettier --write "$FILE" 2>/dev/null
    ;;
  *.go)
    command -v gofmt >/dev/null && gofmt -w "$FILE" 2>/dev/null
    ;;
  *.rs)
    command -v rustfmt >/dev/null && rustfmt "$FILE" 2>/dev/null
    ;;
esac
exit 0
Uses command -v to skip silently if the formatter isn’t installed. No crashes, no noise.

Quality

Runs ESLint, Ruff, or your language’s linter after each edit. Feeds warnings back to the model so it fixes them in the same turn.

settings.json 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{
          "type": "command",
          "command": ".claude/hooks/lint-check.sh"
        }]
      }
    ]
  }
}
.claude/hooks/lint-check.sh 
#!/bin/bash
set -euo pipefail
FILE=$(cat | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('file_path',''))")

case "$FILE" in
  *.py)
    command -v ruff >/dev/null && ruff check "$FILE" --output-format text 2>&1 || true
    ;;
  *.js|*.ts|*.tsx|*.jsx)

    ;;
esac
exit 0
Stdout goes back to the model as context. The agent sees lint warnings and fixes them without being asked.

9. TypeScript Type Check on Save

Quality

Runs tsc --noEmit after TypeScript file changes. Catches type errors early, before they cascade.

settings.json 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{
          "type": "command",
          "command": ".claude/hooks/typecheck.sh"
        }]
      }
    ]
  }
}
.claude/hooks/typecheck.sh 
#!/bin/bash
set -euo pipefail
FILE=$(cat | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('file_path',''))")

[[ "$FILE" == *.ts || "$FILE" == *.tsx ]] || exit 0
command -v npx >/dev/null || exit 0

# Run tsc on just the changed file's project
ERRORS=$(npx tsc --noEmit 2>&1 | grep -c "error TS" || true)
if [ "$ERRORS" -gt 0 ]; then
  echo "TypeScript: $ERRORS type error(s) after editing $FILE"
  npx tsc --noEmit 2>&1 | head -20
fi
exit 0
Runs the full project type check, not just the single file, to catch cross-module breakage.

10. Enforce Maximum File Size

Quality

Warns when a file exceeds a line count threshold. Catches the “one-more-function” drift that turns modules into 5,000-line monsters.

settings.json 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{
          "type": "command",
          "command": ".claude/hooks/file-size-check.sh"
        }]
      }
    ]
  }
}
.claude/hooks/file-size-check.sh 
#!/bin/bash
set -euo pipefail
FILE=$(cat | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('file_path',''))")

MAX_LINES=500  # adjust per project
[ -f "$FILE" ] || exit 0

LINES=$(wc -l < "$FILE")
if [ "$LINES" -gt "$MAX_LINES" ]; then
  echo "WARNING: $FILE is $LINES lines (limit: $MAX_LINES)"
  echo "Consider splitting into smaller modules."
fi
exit 0
Outputs a warning, not a block — the agent sees it and can decide to refactor. Change exit 0 to exit 2 in a PreToolUse hook to make it a hard limit.

Automation

Hooks that trigger side effects automatically — tests, notifications, log entries — so the agent doesn’t have to remember to run them.

11. Auto-Run Tests After Code Changes

Automation

Runs the relevant test file after every source file edit. The agent gets test results in context and fixes failures immediately.

settings.json 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{
          "type": "command",
          "command": ".claude/hooks/auto-test.sh"
        }]
      }
    ]
  }
}
.claude/hooks/auto-test.sh 
#!/bin/bash
set -euo pipefail
FILE=$(cat | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('file_path',''))")

# Skip test files themselves (avoid infinite loops)
echo "$FILE" | grep -qE "(test_|_test\.|\.test\.|\.spec\.)" && exit 0

# Python: look for matching test file
if [[ "$FILE" == *.py ]]; then
  DIR=$(dirname "$FILE")
  BASE=$(basename "$FILE" .py)
  for TEST_FILE in "${DIR}/test_${BASE}.py" "${DIR}/tests/test_${BASE}.py" "tests/test_${BASE}.py"; do
    if [ -f "$TEST_FILE" ]; then
      echo "Running: pytest $TEST_FILE -x -q"
      pytest "$TEST_FILE" -x -q 2>&1 | tail -10
      exit 0
    fi
  done
fi

# JavaScript/TypeScript: look for .test. or .spec. files
if [[ "$FILE" =~ \.(js|ts|tsx|jsx)$ ]]; then
  DIR=$(dirname "$FILE")
  BASE=$(basename "$FILE" | sed 's/\.[^.]*$//')
  for TEST_FILE in "${DIR}/${BASE}.test."* "${DIR}/__tests__/${BASE}."*; do
    if [ -f "$TEST_FILE" ]; then
      echo "Running: npx jest $TEST_FILE --no-coverage"
      npx jest "$TEST_FILE" --no-coverage 2>&1 | tail -15
      exit 0
    fi
  done
fi
exit 0
The agent sees test output on stdout and will fix failures in its next action. No “remember to run the tests” instruction needed.

12. Slack Notification on Commit

Automation

Sends a Slack message every time the agent creates a commit. Great for overnight runs — you see progress in real time.

settings.json 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": ".claude/hooks/slack-on-commit.sh"
        }]
      }
    ]
  }
}
.claude/hooks/slack-on-commit.sh 
#!/bin/bash
set -euo pipefail
CMD=$(cat | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('command',''))")

# Only trigger on git commit commands
echo "$CMD" | grep -q "git commit" || exit 0

WEBHOOK_URL="${SLACK_WEBHOOK_URL:-}"
[ -n "$WEBHOOK_URL" ] || exit 0

MSG=$(git log -1 --pretty=format:"%h %s" 2>/dev/null || echo "commit")
BRANCH=$(git rev-parse --abbrev-ref HEAD 2>/dev/null || echo "?")
REPO=$(basename "$(git rev-parse --show-toplevel 2>/dev/null)" || echo "repo")

curl -s -X POST "$WEBHOOK_URL" \
  -H 'Content-Type: application/json' \
  -d "{\"text\":\"Agent committed to *${REPO}* (${BRANCH}): \`${MSG}\`\"}" \
  >/dev/null 2>&1 &
exit 0
Set SLACK_WEBHOOK_URL in your environment. The curl runs in background (&) so it doesn’t slow down the agent.

13. Audit Log of All Tool Calls

Automation

Logs every tool call to a JSONL file with timestamp, tool name, and input. Full audit trail for compliance or debugging.

settings.json 
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "*",
        "hooks": [{
          "type": "command",
          "command": ".claude/hooks/audit-log.sh"
        }]
      }
    ]
  }
}
.claude/hooks/audit-log.sh 
#!/bin/bash
INPUT=$(cat)
LOG_DIR="${HOME}/.claude/audit-logs"
mkdir -p "$LOG_DIR"

TIMESTAMP=$(date -u +%Y-%m-%dT%H:%M:%SZ)
SESSION="${AMUX_SESSION:-local}"

echo "$INPUT" | python3 -c "
import sys, json
d = json.load(sys.stdin)
entry = {
    'ts': '$TIMESTAMP',
    'session': '$SESSION',
    'tool': d.get('tool_name', ''),
    'input': d.get('tool_input', {})
}
print(json.dumps(entry))
" >> "$LOG_DIR/$(date +%Y-%m-%d).jsonl"
exit 0
One file per day, JSONL format. Easy to grep, parse, or ship to your logging pipeline. The * matcher catches every tool.

Agent Coordination

Hooks that integrate with amux or other orchestration systems to coordinate work across parallel agent sessions.

15. Report Status to amux Dashboard

Coordination

Posts a status update to the amux board every time the agent finishes a tool call. Gives you real-time visibility into what each agent is doing.

settings.json 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": ".claude/hooks/amux-status.sh"
        }]
      }
    ]
  }
}
.claude/hooks/amux-status.sh 
#!/bin/bash
AMUX_URL="${AMUX_URL:-https://localhost:8822}"
SESSION="${AMUX_SESSION:-}"
[ -n "$SESSION" ] || exit 0

CMD=$(cat | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('command','')[:100])" 2>/dev/null || echo "?")

# Only report git commits and test runs
echo "$CMD" | grep -qE "(git commit|pytest|npm test|jest)" || exit 0

curl -sk -X POST -H 'Content-Type: application/json' \
  -d "{\"title\":\"[$SESSION] $CMD\",\"status\":\"doing\",\"session\":\"$SESSION\"}" \
  "$AMUX_URL/api/board" >/dev/null 2>&1 &
exit 0
Only reports significant actions (commits, test runs) to avoid flooding the board. Runs in background so it never blocks.

16. Notify Other Sessions on File Change

Coordination

When an agent modifies a shared file (like a schema, API contract, or config), notify all other sessions via the amux messaging API so they can adapt.

settings.json 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{
          "type": "command",
          "command": ".claude/hooks/notify-file-change.sh"
        }]
      }
    ]
  }
}
.claude/hooks/notify-file-change.sh 
#!/bin/bash
set -euo pipefail
AMUX_URL="${AMUX_URL:-https://localhost:8822}"
SESSION="${AMUX_SESSION:-}"
[ -n "$SESSION" ] || exit 0

FILE=$(cat | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('file_path',''))")

# Only notify for shared files — customize this list
SHARED_PATTERNS=("schema" "migration" "api" "proto" "openapi" "package.json" "pyproject.toml")
MATCH=false
for pat in "${SHARED_PATTERNS[@]}"; do
  echo "$FILE" | grep -qi "$pat" && MATCH=true
done
$MATCH || exit 0

# Get all other sessions and notify them
SESSIONS=$(curl -sk "$AMUX_URL/api/sessions" | python3 -c "
import json, sys, os
me = os.environ.get('AMUX_SESSION','')
[print(s['name']) for s in json.load(sys.stdin) if s['name'] != me and s.get('status') == 'running']
" 2>/dev/null)

for s in $SESSIONS; do
  curl -sk -X POST -H 'Content-Type: application/json' \
    -d "{\"text\":\"@$SESSION modified shared file: $FILE — you may need to pull or adapt.\"}" \
    "$AMUX_URL/api/sessions/$s/send" >/dev/null 2>&1 &
done
exit 0
Customize SHARED_PATTERNS for your project. Common shared files: database schemas, API specs, shared configs, lock files.

17. Auto-Claim Board Task on Session Start

Coordination

When a session starts, automatically claim the next unclaimed task from the amux board. Prevents two agents from working on the same thing.

settings.json 
{
  "hooks": {
    "Stop": [
      {
        "matcher": "*",
        "hooks": [{
          "type": "command",
          "command": ".claude/hooks/claim-next-task.sh"
        }]
      }
    ]
  }
}
.claude/hooks/claim-next-task.sh 
#!/bin/bash
set -euo pipefail
AMUX_URL="${AMUX_URL:-https://localhost:8822}"
SESSION="${AMUX_SESSION:-}"
[ -n "$SESSION" ] || exit 0

# Find the first unclaimed todo item
TASK_ID=$(curl -sk "$AMUX_URL/api/board" | python3 -c "
import json, sys
items = json.load(sys.stdin)
for item in items:
    if item['status'] == 'todo' and not item.get('session'):
        print(item['id'])
        break
" 2>/dev/null)

[ -n "$TASK_ID" ] || exit 0

# Atomic claim via CAS
curl -sk -X POST -H 'Content-Type: application/json' \
  -d "{\"session\":\"$SESSION\"}" \
  "$AMUX_URL/api/board/$TASK_ID/claim" >/dev/null 2>&1

echo "Claimed task $TASK_ID for session $SESSION"
Uses the Stop hook event so each time an agent finishes its current work, it pulls the next task. The atomic /claim endpoint prevents race conditions.

Workflow

Hooks that enforce development workflow policies — review requirements, deployment freezes, naming conventions.

18. Enforce Conventional Commit Messages

Workflow

Blocks commits that don’t follow the type(scope): description format. Keeps your git log machine-parseable for changelogs.

settings.json 
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": ".claude/hooks/conventional-commit.sh"
        }]
      }
    ]
  }
}
.claude/hooks/conventional-commit.sh 
#!/bin/bash
set -euo pipefail
CMD=$(cat | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('command',''))")

# Only check git commit commands
echo "$CMD" | grep -q 'git commit' || exit 0

# Extract the commit message from -m flag
MSG=$(echo "$CMD" | python3 -c "
import sys, re
cmd = sys.stdin.read()
m = re.search(r'-m\s+[\"'\''](.*?)[\"'\'']', cmd)
if m: print(m.group(1).split('\n')[0])
else: print('')
")

[ -n "$MSG" ] || exit 0

PATTERN='^(feat|fix|docs|style|refactor|perf|test|chore|ci|build|revert)(\(.+\))?: .{3,}'
if ! echo "$MSG" | grep -qE "$PATTERN"; then
  echo "BLOCKED: commit message doesn't follow conventional format" >&2
  echo "Expected: type(scope): description" >&2
  echo "Types: feat, fix, docs, style, refactor, perf, test, chore, ci, build, revert" >&2
  echo "Got: $MSG" >&2
  exit 2
fi
exit 0
The agent will retry with a conforming message when blocked. Remove (scope) from the pattern if you don’t use scopes.

19. Block Pushes During Deploy Freeze

Workflow

Prevents git push during a defined freeze window. Drop a .deploy-freeze file to activate, remove it to deactivate.

settings.json 
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": ".claude/hooks/deploy-freeze.sh"
        }]
      }
    ]
  }
}
.claude/hooks/deploy-freeze.sh 
#!/bin/bash
set -euo pipefail
CMD=$(cat | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('command',''))")

echo "$CMD" | grep -q "git push" || exit 0

PROJECT_ROOT=$(git rev-parse --show-toplevel 2>/dev/null || pwd)
FREEZE_FILE="$PROJECT_ROOT/.deploy-freeze"

if [ -f "$FREEZE_FILE" ]; then
  REASON=$(cat "$FREEZE_FILE" 2>/dev/null || echo "No reason given")
  echo "BLOCKED: deploy freeze is active" >&2
  echo "Reason: $REASON" >&2
  echo "Remove .deploy-freeze to lift the freeze." >&2
  exit 2
fi
exit 0
Create .deploy-freeze with a reason: echo "Mobile release cut — freeze until May 2" > .deploy-freeze. Add it to .gitignore.

20. Require Test File for New Source Files

Workflow

When the agent creates a new source file, check that a corresponding test file also exists. Warns if test coverage is missing.

settings.json 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "hooks": [{
          "type": "command",
          "command": ".claude/hooks/require-tests.sh"
        }]
      }
    ]
  }
}
.claude/hooks/require-tests.sh 
#!/bin/bash
set -euo pipefail
FILE=$(cat | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('file_path',''))")

# Skip test files, configs, and non-code files
echo "$FILE" | grep -qE "(test_|_test\.|\.test\.|\.spec\.|__)" && exit 0
echo "$FILE" | grep -qE "\.(py|js|ts|tsx|jsx)$" || exit 0
echo "$FILE" | grep -qiE "(config|setup|__init__|index)" && exit 0

DIR=$(dirname "$FILE")
BASE=$(basename "$FILE" | sed 's/\.[^.]*$//')

# Check for common test file patterns
FOUND=false
for PATTERN in "test_${BASE}" "${BASE}_test" "${BASE}.test" "${BASE}.spec"; do
  find "$DIR" "$(dirname "$DIR")/tests" "$(dirname "$DIR")/__tests__" \
    -name "${PATTERN}.*" -type f 2>/dev/null | grep -q . && FOUND=true
done

if ! $FOUND; then
  echo "WARNING: No test file found for $FILE"
  echo "Consider creating a test file: test_${BASE}.py or ${BASE}.test.ts"
fi
exit 0
Outputs a warning, not a block. The agent receives the hint and often creates the test file in its next action.

Combining Multiple Hooks

You can stack multiple hooks on the same event. They run in order — if any PreToolUse hook exits 2, the action is blocked regardless of what other hooks return.

Example: security + quality stack 
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": ".claude/hooks/block-secrets.sh" },
          { "type": "command", "command": ".claude/hooks/protect-config.sh" },
          { "type": "command", "command": ".claude/hooks/sandbox-writes.sh" }
        ]
      },
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": ".claude/hooks/block-destructive.sh" },
          { "type": "command", "command": ".claude/hooks/enforce-branch.sh" },
          { "type": "command", "command": ".claude/hooks/deploy-freeze.sh" }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": ".claude/hooks/check-python.sh" },
          { "type": "command", "command": ".claude/hooks/auto-format.sh" },
          { "type": "command", "command": ".claude/hooks/lint-check.sh" },
          { "type": "command", "command": ".claude/hooks/auto-test.sh" }
        ]
      }
    ]
  }
}

FAQ

What are Claude Code hooks?
Hooks are shell commands that run automatically before or after Claude Code tool calls. They’re configured in .claude/settings.json and can block actions (exit code 2), log activity, enforce rules, or trigger side effects. Hook events include PreToolUse, PostToolUse, Notification, and Stop. Hooks receive JSON on stdin with tool details and can return feedback on stdout.
Where do I put hooks?
For project-level hooks shared with your team, use .claude/settings.json (committed to git). For personal hooks, use .claude/settings.local.json (gitignored). For global hooks across all projects, use ~/.claude/settings.json.
How do I block an action with a hook?
In a PreToolUse hook, have your script exit with code 2. This blocks the tool call and sends your stderr output back to the model as an error message. Exit code 0 allows the action. Any other exit code logs a warning but doesn’t block.
What data does a hook receive on stdin?
JSON with tool_name (e.g. “Bash”, “Edit”, “Write”) and tool_input (the parameters). For Bash, tool_input.command is the shell command. For Edit, you get file_path, old_string, new_string. PostToolUse hooks also receive tool_output with the result.
Can I use hooks with parallel agents?
Yes. Hooks in .claude/settings.json are inherited by every agent running in that project. This is how you enforce fleet-wide policies — define once, every session follows. For inter-agent coordination, hooks can call the amux REST API to post status updates, claim tasks, or message other sessions.
How do I debug a hook?
Test it directly: echo '{"tool_name":"Bash","tool_input":{"command":"ls"}}' | bash .claude/hooks/your-hook.sh. Check the exit code with echo $?. Make sure the script is executable (chmod +x) and the matcher matches the tool name (case-sensitive, | for alternatives).
What’s the difference between PreToolUse and PostToolUse?
PreToolUse runs before the tool and can block it (exit 2). Use for safety: secrets, destructive commands, branch policy. PostToolUse runs after success. Use for automation: testing, formatting, notifications. PreToolUse is your guard; PostToolUse is your automation layer.

Hooks + amux = production-grade agents

Hooks enforce local policy. amux adds fleet-wide observability, task coordination, and self-healing. Together, they turn raw Claude Code sessions into infrastructure you can run overnight.
