Every AI Coding Agent Config File, Compared

CLAUDE.md vs .cursorrules vs AGENTS.md vs copilot-instructions.md vs .windsurfrules — what each does, how they differ, and how to set up one repo for all of them.

The average developer uses 2.3 AI coding tools. Each tool has its own config file format, its own syntax, and its own quirks. You end up with instructions scattered across CLAUDE.md, .cursorrules, .github/copilot-instructions.md, and whatever Windsurf calls theirs this month.

This guide covers every format — what it does, where it lives, what it supports, and what it doesn’t. Then it shows you how to keep a single repo working with multiple tools without duplicating rules by hand.

CLAUDE.md Claude Code

CLAUDE.md

Most features

The richest config system of any AI coding tool. CLAUDE.md is a markdown file loaded at the start of every Claude Code conversation. It supports three scoping levels, glob-based rule files, hooks that run shell commands on tool events, permission allow/deny lists, on-demand skills, and subagent definitions.

Location
PROJECT_ROOT/CLAUDE.md
Global
~/.claude/CLAUDE.md
Per-directory
subdir/CLAUDE.md
Format
Markdown (any valid markdown)
Max recommended
~500 lines (loaded every turn)
Commit to git?
Yes (project), No (global)
Project CLAUDE.md example 
# Project Name

## Build & test
- Build: `npm run build`
- Test: `npm test`
- Lint: `npm run lint`
- Single test: `npm test -- --grep "test name"`

## Code style
- TypeScript strict mode, no `any`
- Prefer named exports over default
- Use Zod for runtime validation at API boundaries
- Error messages must include the operation that failed

## Forbidden patterns
- Never use `console.log` in production code — use the logger
- Never import from `src/internal/` outside that directory
- Never commit `.env` files or hardcoded secrets

## Architecture
- `src/api/` — HTTP handlers (Express)
- `src/services/` — Business logic (no HTTP imports)
- `src/db/` — Database queries (Drizzle ORM)
- Services never import from api/; api/ imports from services/
File tree for a full CLAUDE.md setup
CLAUDE.md ← project-wide rules (committed) ~/.claude/CLAUDE.md ← personal global preferences .claude/settings.json ← hooks, permissions, model config .claude/settings.local.json ← personal overrides (gitignored) .claude/rules/api-routes.md ← glob-scoped rules (globs: ["src/api/**"]) .claude/rules/database.md ← glob-scoped rules (globs: ["src/db/**"]) .claude/skills/deploy.md ← on-demand skill (loaded when relevant) .claude/agents/code-reviewer.md ← subagent definition src/api/CLAUDE.md ← per-directory override

What makes it unique: Hooks. CLAUDE.md is the only config system that can run shell commands before/after tool calls. A PreToolUse hook can block an action (exit code 2), enforce branch policy, or scan for secrets. A PostToolUse hook can auto-run tests, format code, or notify a dashboard. No other tool has this. See the Hooks Cookbook for 20 production recipes.

Limitations: Only works with Claude Code. Every line costs context tokens on every conversation, so bloated files waste money. No conditional logic — you can’t say “only apply this rule on Mondays.”

.cursorrules & .cursor/rules/ Cursor

.cursorrules → .cursor/rules/

Glob scoping

Cursor has two config systems: the legacy .cursorrules file (single file, plain text) and the newer .cursor/rules/ directory (multiple files, MDC format with frontmatter). The directory format is the recommended path forward.

Legacy location
.cursorrules (project root)
New location
.cursor/rules/*.mdc
Format
MDC (markdown + YAML frontmatter)
Glob scoping
Yes, via globs in frontmatter
Global
No (user settings only)
Commit to git?
Yes
Legacy .cursorrules 
You are an expert TypeScript developer.

- Use functional components with hooks, never class components
- Prefer named exports
- Use Tailwind CSS for styling, no inline styles
- Always add loading and error states to data-fetching components
- Write tests for all business logic in src/services/
.cursor/rules/api-style.mdc (new format) 
---
description: API route conventions for Express handlers
globs: ["src/api/**/*.ts"]
alwaysApply: false
---

# API Route Style

- Every handler must validate input with Zod schemas
- Return { data, error, status } response envelope
- Log errors with structuredLogger, never console.log
- Rate limit all public endpoints
- Include OpenAPI JSDoc annotations
File tree
.cursorrules ← legacy (still works, lower priority) .cursor/rules/general.mdc ← alwaysApply: true (loaded every time) .cursor/rules/api-style.mdc ← scoped to src/api/**/*.ts .cursor/rules/database.mdc ← scoped to src/db/**/*.ts .cursor/rules/testing.mdc ← scoped to **/*.test.ts

What makes it unique: The alwaysApply flag. Rules with alwaysApply: false are loaded only when the agent works on files matching the globs pattern. This is similar to CLAUDE.md’s .claude/rules/ but with explicit opt-in/opt-out control. The @docs feature lets you reference external documentation URLs that get pulled into context.

Limitations: No hooks, no permission system, no subagent definitions. The legacy .cursorrules format has no scoping — everything applies everywhere. The MDC format requires learning a new file extension. No per-directory nesting like CLAUDE.md.

.github/copilot-instructions.md GitHub Copilot

copilot-instructions.md

Simplest

The simplest config file of the bunch. A single markdown file that gives Copilot project-level context. GitHub also supports organization-level instructions and per-language files, but the repo-level file is what most teams use.

Location
.github/copilot-instructions.md
Per-language
.github/copilot-instructions-{lang}.md
Org-level
Via GitHub org settings
Format
Markdown (plain)
Glob scoping
No
Commit to git?
Yes
.github/copilot-instructions.md 
# Project Instructions for Copilot

This is a Next.js 15 app with TypeScript, Drizzle ORM, and Tailwind CSS.

## Conventions
- Use server components by default, client components only when needed
- All database queries go through src/db/queries/ — never raw SQL in components
- Use next-safe-action for server actions with Zod validation
- Error boundaries in every route segment

## Do not
- Suggest class components
- Use CSS modules or styled-components
- Import from node_modules directly when a wrapper exists in src/lib/

What makes it unique: Organization-level instructions. Enterprise teams can set instructions that apply across all repos in the org, with per-repo files adding specifics. This is the only config system with org-level inheritance.

Limitations: No per-directory scoping, no glob matching, no hooks, no permissions, no skill loading. It’s a single flat file — effective for simple rules, but it can’t express complex per-module policies. GitHub’s own documentation acknowledges that “instructions are not guaranteed to be followed” — there’s no enforcement mechanism.

AGENTS.md Emerging Standard

AGENTS.md

Vendor-neutral

AGENTS.md is an emerging open proposal for a vendor-neutral agent config file. The idea: instead of maintaining CLAUDE.md + .cursorrules + copilot-instructions.md separately, you write one AGENTS.md that all tools can read. Adoption is early but growing, driven by developers frustrated with maintaining 3+ config files for the same rules.

Location
AGENTS.md (project root)
Format
Markdown + optional YAML frontmatter
Standard
Community proposal (not owned by any vendor)
Tool support
Growing (some tools read it, most don’t yet)
Nested
Proposed but not standardized
Commit to git?
Yes
AGENTS.md example 
---
version: "1.0"
tools: ["claude-code", "cursor", "copilot"]
scope: project
---

# Agent Instructions

## Project Overview
Next.js 15 application with TypeScript, Drizzle ORM, Tailwind CSS.

## Build Commands
- Dev server: `npm run dev`
- Build: `npm run build`
- Test: `npm test`

## Code Standards
- TypeScript strict mode, no implicit any
- Functional components with hooks
- Named exports only
- Zod for all API input validation

## Architecture Rules
- src/api/ handles HTTP only — no business logic
- src/services/ handles business logic — no HTTP imports
- src/db/ handles data access — Drizzle ORM queries only
- Never import across layer boundaries in the wrong direction

What makes it unique: Vendor neutrality. If adoption grows, it could become the .editorconfig of AI tools — one file, every tool respects it. The YAML frontmatter lets you declare which tools should read it and version the format.

Limitations: Adoption is early. Most tools don’t read it natively yet. No hooks, no permissions, no scoping, no skill loading. It only covers the subset of features that all tools have in common (plain text instructions), so you lose tool-specific features like Claude Code hooks or Cursor’s @docs references. In practice, teams that adopt AGENTS.md still maintain tool-specific files for advanced features.

.windsurfrules Windsurf

.windsurfrules

Windsurf / Codeium

Windsurf (formerly Codeium) uses a .windsurfrules file at your project root. The format is nearly identical to the legacy .cursorrules — plain text or markdown with project instructions. Windsurf 2.0 added Spaces for team-shared rules, but the file-based config remains the standard.

Location
.windsurfrules (project root)
Format
Plain text / markdown
Glob scoping
No
Global config
No (use Spaces for team sharing)
Commit to git?
Yes
.windsurfrules example 
You are working on a Python FastAPI application.

Project structure:
- app/routers/ — API endpoints
- app/models/ — Pydantic models and SQLAlchemy ORM
- app/services/ — Business logic
- tests/ — Pytest test files

Rules:
- Use async/await for all endpoint handlers
- All routes must have response_model defined
- Use dependency injection for database sessions
- Write docstrings for all public functions
- Run `pytest -x` after making changes

Limitations: No per-directory scoping, no hooks, no permissions, no glob matching. The simplest format alongside Copilot’s. If you’re migrating from Cursor, you can often copy your .cursorrules content directly into .windsurfrules.

.clinerules Cline / Roo Code

.clinerules

Cline / Roo Code

Cline (formerly Roo Code) uses .clinerules at the project root, plus optional .clinerules-{mode} files for its custom modes feature. Custom modes let you define specialized agent personas (e.g., architect, code reviewer, security auditor) with their own rules and auto-approve settings.

Location
.clinerules (project root)
Mode-specific
.clinerules-architect, .clinerules-reviewer
Format
Markdown
Global config
Via VS Code settings
Commit to git?
Yes
.clinerules example 
# Project Rules

## Stack
- Python 3.12, FastAPI, SQLAlchemy 2.0, Alembic
- Frontend: React 19, TypeScript, Vite

## Conventions
- Use type hints everywhere
- Alembic for all schema changes — never raw ALTER TABLE
- API responses use standard envelope: { data, error, meta }
- Tests required for all service functions

What makes it unique: Custom modes. You can define a .clinerules-security that turns Cline into a security auditor with different auto-approve rules and a security-focused system prompt. This is conceptually similar to Claude Code’s .claude/agents/ subagent definitions.

Limitations: No per-directory nesting, no glob scoping, no hooks. The mode-based system is powerful but Cline-specific — rules don’t transfer to other tools.

.aider.conf.yml Aider

.aider.conf.yml

Aider

Aider uses a YAML config file that controls both behavior settings and custom instructions. It’s the only config that uses YAML instead of markdown, which makes it more structured but less readable for prose instructions.

Location
.aider.conf.yml (project root)
Global
~/.aider.conf.yml
Format
YAML
Conventions file
CONVENTIONS.md (loaded via read: flag)
Commit to git?
Yes
.aider.conf.yml example 
model: claude-sonnet-4-6
auto-commits: true
auto-lint: true
lint-cmd: "ruff check --fix"
test-cmd: "pytest -x"
read:
  - CONVENTIONS.md
  - docs/ARCHITECTURE.md

What makes it unique: Built-in lint and test integration. The auto-lint and lint-cmd settings run your linter automatically after every edit — similar to what Claude Code achieves with PostToolUse hooks, but declarative instead of imperative. The read: directive loads additional context files without them being editable.

Limitations: Instructions go in a separate CONVENTIONS.md, not in the YAML file itself. No per-directory scoping, no glob matching, no hooks beyond lint/test. YAML is awkward for long prose instructions. No MCP or tool integration.

Unified Setup: One Repo, Multiple Tools

Most teams use 2–3 AI tools. Here’s how to set up a single repo that works with all of them without duplicating rules across files.

Approach 1: Maintain each file separately

The simplest approach. Each tool gets its own config file. Rules may overlap, but you review them together in PRs.

CLAUDE.md ← Claude Code rules .cursorrules ← Cursor rules (or .cursor/rules/) .github/copilot-instructions.md ← Copilot rules .windsurfrules ← Windsurf rules (if used)

Pros: Each file uses the tool’s full feature set (hooks in CLAUDE.md, glob scoping in Cursor, org-level in Copilot). No build step. Simple to understand.
Cons: Rules drift apart. When you update a convention in one file, you might forget the others.

Approach 2: Shared source file + symlinks

Keep a canonical CODING_RULES.md and include it in each tool’s config. Tool-specific features go in their own files.

CODING_RULES.md ← shared rules (single source of truth) CLAUDE.md ← includes CODING_RULES.md content + hooks .cursor/rules/shared.mdc ← includes CODING_RULES.md content + globs .github/copilot-instructions.md ← includes CODING_RULES.md content

In CLAUDE.md, reference the shared rules:

# Project Rules

See CODING_RULES.md for shared conventions.

## Claude Code-specific

### Hooks
- PostToolUse runs `./scripts/check-python-syntax.sh` on every edit
- PreToolUse blocks git push --force

### Permissions
- Allow: Bash(npm test*), Bash(git commit*), Edit(src/**)

Approach 3: Generate from a single source

A build script reads one source file and generates tool-specific configs. Best for teams with 5+ developers where config drift is a real problem.

#!/bin/bash
# scripts/sync-agent-config.sh
# Generates tool-specific config files from a single source

SOURCE="CODING_RULES.md"

# Claude Code: copy as-is (markdown), prepend tool-specific header
{
  cat .claude/claude-header.md
  echo ""
  cat "$SOURCE"
} > CLAUDE.md

# Cursor: wrap in MDC frontmatter
{
  echo "---"
  echo "description: Shared project coding rules"
  echo "alwaysApply: true"
  echo "---"
  echo ""
  cat "$SOURCE"
} > .cursor/rules/shared.mdc

# Copilot: copy as-is
cp "$SOURCE" .github/copilot-instructions.md

# Windsurf: copy as-is
cp "$SOURCE" .windsurfrules

echo "Synced agent config files from $SOURCE"

Add this to your CI or pre-commit hook to catch drift.

Which Format Should You Use?

You use one tool only

Use that tool’s native format. CLAUDE.md for Claude Code, .cursor/rules/ for Cursor, etc. You get full feature access and zero complexity.

You use 2–3 tools, small team

Maintain separate files (Approach 1). The duplication is manageable. Review them together in PRs. Use a shared CODING_RULES.md reference for the rules both tools should follow.

You use 2–3 tools, large team

Generate from a single source (Approach 3). Add the sync script to CI. Tool-specific features (hooks, glob scoping) go in separate config files that aren’t generated.

You want maximum capability

CLAUDE.md with Claude Code. It’s the only config system with hooks, permissions, skills, subagents, and per-directory nesting. If you need enforcement (not just instructions), this is the only option.

FAQ

Can I use CLAUDE.md and .cursorrules in the same repo?
Yes. Each tool reads only its own config file. They coexist without conflict. Many teams maintain both. The key is keeping shared rules in sync — update one, update the other.
Does AGENTS.md replace CLAUDE.md?
No, not today. AGENTS.md is a vendor-neutral proposal for shared instructions, but it can’t express tool-specific features like hooks, permissions, glob scoping, or skill loading. Think of AGENTS.md as the lowest common denominator — good for basic rules that all tools should follow, but you still need CLAUDE.md (or .cursor/rules/) for advanced features. If AGENTS.md adoption grows, tools may add native support for it as a base layer.
Which config file has the most features?
CLAUDE.md, by a wide margin. It’s the only format with hooks (shell commands that run on tool events), permission allow/deny lists, on-demand skills, subagent definitions, and three-level scoping (global → project → directory). Cursor’s .cursor/rules/ is second with glob-based scoping and MDC frontmatter. Everything else is a flat markdown file with no enforcement mechanism.
How long should my config file be?
As short as possible while covering the rules the AI gets wrong without guidance. For CLAUDE.md: under 500 lines (loaded every conversation, costs tokens). For .cursorrules / copilot-instructions: under 300 lines. If your config is getting long, split into scoped files (Cursor globs, CLAUDE.md per-directory) or use on-demand loading (CLAUDE.md skills).
Should I commit these files to git?
Yes, commit all project-level config files. They’re part of your development infrastructure, like .eslintrc or pyproject.toml. The only exception is personal preferences — ~/.claude/CLAUDE.md (global Claude Code) or .claude/settings.local.json (personal hooks/permissions). Those stay out of the repo.
Do these config files work with amux parallel agents?
Yes. Every amux session runs Claude Code in your project directory, so all sessions inherit the same CLAUDE.md and hooks. This is one of the key advantages of file-based config for multi-agent workflows: define rules once, every parallel agent follows them. For agent-specific overrides, use per-directory CLAUDE.md files in each agent’s worktree.
