How to Run an AI Engineering Team

An AI engineering team uses coding agents as the primary unit of work, coordinated by one shared dashboard. This guide shows how to set it up — from a solo developer running 10 agents to a 5-person team running 50.

Answer in 3 sentences

Running an AI engineering team means launching dozens of parallel coding agents — Claude Code, Codex, or Gemini CLI — and coordinating their work through a shared board, dashboard, and self-healing watchdog instead of sprint planning and Slack threads. The human role shifts from writing code to writing specs, reviewing output, and making architectural decisions. amux is the open-source control plane that makes this practical: one Python file gives you parallel session management, a kanban board, a mobile-first dashboard, and crash recovery — all visible from your phone.

What Is an AI Engineering Team?

A traditional engineering team has five developers, each writing code. An AI engineering team has one to five developers, each orchestrating five to twenty coding agents. The output is the same — shipped features, fixed bugs, merged PRs — but the workflow is fundamentally different.

The key insight: coding agents are fast but need direction. Without coordination infrastructure, five developers each running their own agents in isolation is five times the chaos, not five times the output. The agents step on each other's work, the board lives in five different heads, and nobody knows what shipped until someone checks.

An AI engineering team solves this with a shared control plane: one place where every agent's status is visible, every task is tracked, and work is claimed atomically so two agents never build the same feature at the same time. That's what amux provides.

This approach works at every scale — a solo developer running 10 agents overnight, a two-person startup shipping 5x faster than their competition, or a 10-person team where every developer has their own agent fleet visible on a shared dashboard. The agentic engineering methodology applies regardless of headcount.

How amux Works as the Team Control Plane

amux is a single Python file — no containers, no SaaS subscription, no external dependencies beyond Python 3 and tmux. It runs on any Mac, Linux machine, or cloud VM and serves as the operational backbone for your agent fleet.

Session management
Launch, monitor, and kill agent sessions without touching a terminal. Each session is a tmux window — visible in the dashboard, addressable via API.
Kanban board
SQLite-backed task board with atomic claiming — no two agents pick up the same task. Visible to the whole team, syncs to iCal for due dates.
Self-healing watchdog
Monitors every session. When an agent crashes, overflows context, or gets stuck, the watchdog auto-restarts and replays — no human intervention required.
Mobile dashboard
Mobile-first PWA and native iOS app. Check agent status, move board tasks, and send messages to running agents from anywhere.
Multi-runtime support
Manages any CLI-based agent: Claude Code, Codex CLI, Gemini CLI, Aider. Mix runtimes on one instance — the right tool for each task.
Built-in CRM & email
Contact management and email automation included. Agents can send status emails, log customer interactions, or trigger follow-ups automatically.

The dashboard auto-refreshes via SSE — every developer sees real-time agent status without polling or refresh. When you're on your phone and an agent finishes a feature branch at 2am, the board updates instantly.

How to Set Up amux for a Team

The full setup takes about 20 minutes for a shared server configuration. Per-developer installs take 5 minutes each.

Step 1: Install amux on a shared server or each developer's machine

# On a Mac or Linux machine (shared server or individual)
$ git clone https://github.com/mixpeek/amux && cd amux && ./install.sh

# Verify the install
$ amux --version
$ which amux   # should be in /usr/local/bin or ~/.local/bin

Prerequisites: Python 3.9+, tmux, and git. The install script handles everything else. For a shared server, run this once on the server — all team members connect to the same instance via browser.

Step 2: Register projects with amux register

# Register each project/repo you want agents to work on
$ amux register api --dir ~/Dev/myapp-api --yolo
$ amux register frontend --dir ~/Dev/myapp-frontend --yolo
$ amux register infra --dir ~/Dev/myapp-infra --yolo

# --yolo enables unattended operation (auto-approves safe operations)
# --worktree gives each session its own git branch automatically
$ amux register api --dir ~/Dev/myapp-api --yolo --worktree

Each registered project is a project template — when you start a session, you pick the project and amux sets the working directory, CLAUDE.md context, and git configuration automatically.

Step 3: Establish session naming conventions

Before anyone launches their first agent, agree on a naming scheme. Good names make the shared dashboard readable at a glance. See Session Naming Conventions below for patterns.

Step 4: Configure the shared kanban board

# Create the initial sprint tasks before launching agents
$ amux board add "Auth: JWT refresh endpoint" --session eh-auth
$ amux board add "Payments: Stripe webhook handler" --session jd-payments
$ amux board add "Tests: billing module coverage" --session team-tests

# Or use the dashboard UI — drag tasks between columns
# All team members see the same board at https://amux-host:8822

The board is the single source of truth. Anything not on the board isn't being tracked. Agents claim tasks atomically — no race conditions, no duplicated work. Read the kanban board guide for the full coordination patterns.

Step 5: Set up mobile access (PWA or iOS app)

Mobile access is how you maintain oversight without sitting at a terminal all day.

  • iOS app: Install the amux app from the App Store. Point it at your amux host URL. Get push notifications when sessions need attention.
  • Android PWA: Open https://your-amux-host:8822 in Chrome, tap the install prompt, and add to home screen.
  • Safari PWA: Open in Safari on iOS, tap the Share button, and choose Add to Home Screen for a full-screen experience.

Once set up, you can start, stop, and monitor agents from your phone. No terminal required during the day — just check the board during meetings and intervene when the watchdog flags something that needs human input.

Step 6: Enable the self-healing watchdog

# Enable watchdog globally for all sessions (recommended for teams)
$ amux config set watchdog.enabled true
$ amux config set watchdog.restart_on_crash true
$ amux config set watchdog.compact_on_overflow true

# Or per-session at launch time
$ amux start auth-feature --watchdog "Implement JWT refresh endpoint per spec"

The watchdog handles the three most common failure modes automatically: agent crashes (restart + replay), context window overflow (compact + resume), and stuck permission prompts (YOLO mode auto-approves). With watchdog enabled, overnight runs complete reliably without babysitting. See the self-healing configuration guide for advanced options.

Step 7: Set up remote access via Tailscale

# On the amux server
$ curl -fsSL https://tailscale.com/install.sh | sh
$ sudo tailscale up
$ tailscale ip -4   # note the Tailscale IP, e.g. 100.x.y.z

# Team members install Tailscale on their device, join the same tailnet
# Then access amux at https://100.x.y.z:8822 — no VPN config, no port forwarding

Tailscale provides zero-config private networking. The amux dashboard is accessible from any team member's phone or laptop without exposing port 8822 to the internet. The remote access with Tailscale guide covers the full setup including iOS, Android, and certificate trust.

Team Configurations: Solo vs Shared

There are three common deployment patterns. Choose based on team size, compute availability, and how much centralization you want.

Dimension Each developer on their own Shared amux server Hybrid (per-dev + shared board)
Setup time 5 min per developer 20 min one-time 5 min per developer + 10 min board setup
Compute Each developer's own machine Centralized (cloud VM or server) Each developer's machine, shared board
Visibility No shared view — siloed One dashboard, entire team visible Shared board, separate session views
Coordination Manual (Slack, standups) Board + dashboard, fully automated Board-based, manual for session details
Agent conflicts High — no shared state None — atomic claiming Low — board prevents duplication
Overnight runs Only while developer's machine is on Always-on, any time Depends on individual machines
Remote access SSH tunnel or Tailscale per machine One Tailscale node for whole team Tailscale per machine
Best for Solo developers Teams of 3+, startup with cloud VM Distributed teams, contractors

For most teams shipping real product, the shared server configuration is the right default. One URL, one board, one source of truth. Developers access it from anywhere. Agents run 24/7 even when everyone's laptop is closed.

Session Naming Conventions

A shared dashboard with 20 active sessions only works if names are meaningful. Pick a convention before your first session and enforce it in the team's CLAUDE.md or onboarding doc.

Format 1: Owner + domain + task type

# [initials]-[domain]-[task-type]
eh-auth-feature      # Ethan, auth service, feature work
jd-payments-tests    # Jordan, payments, test generation
sa-infra-deploy      # Sara, infrastructure, deployment script
team-db-migration    # Shared task, database, migration

Format 2: Sprint ticket reference

# [ticket-id]-[short-description]
FEAT-042-notif-api     # Feature ticket 42: notification API
BUG-078-auth-timeout   # Bug 78: auth timeout fix
REFAC-015-billing-svc  # Refactor 15: billing service
TEST-009-payments      # Test task 9: payments coverage

Format 3: Runtime-prefixed (for multi-runtime teams)

# [runtime]-[domain]-[task]  — useful when mixing Claude Code, Codex, Gemini CLI
cc-api-feature      # Claude Code session, API work
gm-refactor-auth    # Gemini CLI session, auth refactor
cx-tests-frontend   # Codex CLI session, frontend tests

The dashboard truncates long session names on mobile. Keep names under 24 characters. Avoid spaces and special characters — amux uses session names as tmux window identifiers and board tags.

Board Workflow for Teams

The board is where team coordination actually happens. Here are the patterns that work in practice — borrowed from teams running 20+ agents daily.

Sprint kickoff: seed the board, then launch

# 1. One team lead seeds the board with the sprint's tasks
$ amux board add "Auth: JWT refresh" --desc "specs/auth-refresh.md" --session eh-auth
$ amux board add "Payments: webhook" --desc "specs/stripe-webhook.md" --session jd-payments
$ amux board add "Tests: auth module" --desc "specs/auth-tests.md"

# 2. Each developer launches agents that claim their assigned tasks
$ amux start eh-auth "Check the board, claim your task, implement it"

# 3. Unassigned tasks are picked up by any available agent
# Agents claim via: POST /api/board/TASK-ID/claim

Status discipline on a shared board

  • todo: Task is defined, not yet claimed. Any agent can pick it up.
  • doing: An agent has claimed it and is actively working. Touch it only if the agent is stuck.
  • done: Implementation complete, not yet verified. The diff exists — someone needs to review it.
  • verified: Confirmed working end-to-end. Tests passed, merged, deployed, confirmed in prod.

Review column for agent output

For teams that require human review before merging agent output, add an intermediate status. The agent moves its task to "review" when it creates a PR or diff, and a human moves it to "done" after approval. This keeps the board honest — nothing gets marked done that hasn't been reviewed.

# Agent marks its own task ready for review
$ curl -sk -X PATCH -H 'Content-Type: application/json' \
  -d '{"status":"review","desc":"PR #142 created — auth refresh endpoint"}' \
  $AMUX_URL/api/board/FEAT-042

# Human reviews, approves, merges — then marks done
$ amux board done FEAT-042

Daily board review as the standup

For remote teams, a shared dashboard link replaces the traditional standup. Every team member opens https://amux-host:8822 asynchronously and sees: which sessions are WORKING vs IDLE, which tasks are in progress vs done, and which tasks need attention (watchdog flags, stuck sessions). The dashboard is the standup. The standup is the dashboard.

Monitoring Your Agent Fleet from Your Phone

The single biggest quality-of-life improvement in running an AI engineering team is getting the dashboard on your phone. When your agents run overnight or during meetings, you need ambient awareness — not another screen open on your desk.

What you can do from the phone

  • Check session status — see which agents are WORKING (green), IDLE (yellow), or CRASHED (red) at a glance
  • Read live output — tap any session card to see the last N lines of terminal output, no SSH required
  • Move board tasks — drag todo to doing to done; agents see the board update immediately
  • Send messages to agents — if an agent is stuck, send it a clarifying message from the board card
  • Kill or restart sessions — stop a runaway agent or manually trigger a restart without touching a terminal
  • Add tasks — ideas during a meeting go straight to the board, and idle agents pick them up automatically

Overnight monitoring workflow

# Before bed: queue the overnight tasks, verify watchdog is active
$ amux board add "Refactor: billing service" --desc "specs/billing-refactor.md"
$ amux board add "Tests: payments module" --desc "specs/payments-tests.md"
$ amux board add "Docs: API reference update" --desc "specs/api-docs.md"

$ amux start billing-refactor --watchdog "Claim and implement your board task"
$ amux start payments-tests  --watchdog "Claim and implement your board task"
$ amux start api-docs         --watchdog "Claim and implement your board task"

# Morning: check from phone while coffee brews
# Board shows done items, session cards show final output
# Review diffs on the go, merge from laptop when ready

5 Questions Teams Ask Before Getting Started

Q Do all team members need to know how to use the terminal?
No. The amux dashboard and iOS app give non-technical team members full visibility into agent status and board tasks without touching a terminal. Engineers set up and launch agents; anyone can monitor the board, review status, and move tasks. The CLI is only needed for initial setup and launching new agent sessions.
Q How do we handle code review when agents generate most of the code?
The review process shifts from line-by-line authoring review to spec-compliance review. Each task has acceptance criteria; the reviewer checks that criteria are met, not that they'd have written it identically. Review time per PR drops significantly — you're reading diffs of well-scoped, single-purpose tasks, not sprawling feature branches. See the reviewing AI agent code guide for the full process.
Q What happens when two agents write code that conflicts?
Three things prevent this: (1) task decomposition — scoped tasks touch different files by design; (2) git worktrees — each agent works in its own branch and directory, so conflicts only appear at merge time when they're cheap to resolve; (3) atomic board claiming — agents can't take the same task, so they can't produce conflicting implementations of the same feature. The git conflict avoidance guide covers the full strategy.
Q How much does running a team of agents cost per month?
amux itself is MIT-licensed and free. The cost is your AI API usage — Claude Code, Codex, or Gemini CLI tokens. A well-run team of 5 developers, each running 5–10 agents per day, typically spends $200–$800/month in API costs. That's equivalent to a fraction of one developer salary, for output that rivals a team twice the size. The AI coding agent cost guide breaks down the math and optimization strategies.
Q Can we use amux if we're not all using Claude Code? We have some people using Codex and some using Gemini CLI.
Yes — amux manages tmux sessions, and any CLI-based agent runs inside one. You can have Claude Code sessions, Codex CLI sessions, and Gemini CLI sessions all running simultaneously, all visible on the same dashboard, all pulling from the same board. The runtime is a property of the session, not of amux. See how amux compares to single-runtime tools and the docs for per-runtime configuration.
