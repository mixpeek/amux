How to Build a Multi-Agent Coding Orchestrator

The missing infrastructure layer between "run more agents" and "ship reliable code overnight."

Every guide about multi-agent coding starts with the same advice: use worktrees, give each agent a clear task, review the PRs in the morning. None of them explain what happens when agent 7 crashes at 2am, agent 3 picks up the same task as agent 5, and agent 9 burns $40 on a circular refactor nobody asked for.

This guide covers the coordination architecture — the layer between your agents and your codebase that prevents multi-agent coding from becoming multi-agent chaos. We built amux to solve these problems; the patterns here apply whether you use amux, build your own, or bolt orchestration onto Claude Code Agent Teams.

The five layers of a multi-agent system

Most people think about multi-agent coding as "run N agents in parallel." That's layer 1 of 5.

LayerWhat it doesWhat happens without it
1. RuntimeStart/stop agent processes, isolate workspacesCan't run more than one agent
2. Task distributionAssign work, prevent duplicates, track progressAgents duplicate each other's work
3. CommunicationInter-agent messaging, shared state, notificationsAgents work in silos, miss dependencies
4. Fault toleranceCrash recovery, context compaction, self-healingOvernight runs die at 2am, you wake up to nothing
5. GovernanceCost tracking, kill switches, observabilityRunaway costs, no visibility into what agents are doing

Claude Code gives you layer 1. Agent Teams adds worktree isolation to layer 1. Everything above that — you need to build or use an orchestrator.

Layer 1: Runtime — process isolation and lifecycle

The runtime layer manages agent processes. Each agent needs:

  • An isolated workspace. Git worktrees are the standard approach — each agent gets its own branch and working directory. No merge conflicts during development; conflicts resolve at PR time.
  • A stable process container. tmux sessions work well for this. Each agent runs in its own tmux pane with a persistent conversation ID. The session survives terminal disconnects.
  • Environment injection. Every agent needs to know its own identity ($AMUX_SESSION) and how to reach the orchestrator ($AMUX_URL). Inject these as environment variables at session startup.
# Register a project with 8 parallel agent sessions
amux register myproject --dir ~/Dev/myproject --yolo
amux start myproject

# Each session gets its own tmux pane, conversation ID, and env vars
# $AMUX_SESSION=worker-1, $AMUX_URL=https://localhost:8822

The key design decision: agents should be long-running, not ephemeral. Short-lived agents lose their conversation context on every task. Long-running agents accumulate project understanding across tasks — they learn the codebase, remember past failures, and make better decisions over time.

Layer 2: Task distribution — the shared board

This is where most DIY multi-agent setups fail. Without a shared task board, you either:

  • Manually assign tasks to each agent (doesn't scale, blocks on you)
  • Let agents self-assign from a list (race conditions, duplicated work)
  • Use a parent agent to delegate (single point of failure, context overhead)

The solution is a shared board with atomic task claiming.

Atomic claiming via Compare-and-Swap

When an agent wants to pick up a task, it doesn't just update the status — it performs a CAS (Compare-and-Swap) operation. The claim only succeeds if the task is still unclaimed at the moment of the write.

# Agent attempts to claim a task
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"session":"worker-3"}' \
  $AMUX_URL/api/board/TASK-42/claim

# Returns 200 if claimed, 409 if another agent got there first

With SQLite, this is a single UPDATE with a WHERE clause:

UPDATE board SET session = ?, status = 'doing'
WHERE id = ? AND status = 'todo' AND session IS NULL

If rowcount == 0, someone else claimed it. No locks, no Redis, no distributed consensus — just SQLite's built-in transaction isolation. This works reliably for 50+ agents on a single machine.

Task lifecycle

Every task flows through four states:

  1. todo — created, waiting for an agent to claim it
  2. doing — claimed by an agent, work in progress
  3. done — completed, results written to the task description
  4. blocked — waiting on a dependency or human decision

Agents write their results back to the task when they finish. Other agents can read those results to inform their own work. This creates an asynchronous information flow that doesn't require direct messaging.

# Mark task done with results
curl -sk -X PATCH -H 'Content-Type: application/json' \
  -d '{"status":"done","desc":"Refactored auth module. PR #412 opened. Breaking change: session token format changed from JWT to opaque."}' \
  $AMUX_URL/api/board/TASK-42

Layer 3: Communication — inter-agent messaging

Some coordination requires real-time communication. An agent discovers a breaking change that affects a peer's work. A triage agent needs to reassign a stuck task. A test agent wants to notify the feature agent that tests are failing.

Direct messaging

The simplest pattern: one agent sends a message directly to another agent's terminal input.

# Agent worker-1 notifies worker-3 about a breaking change
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"text":"@worker-3 heads up: I changed the session token format from JWT to opaque. You may need to update the auth middleware tests."}' \
  $AMUX_URL/api/sessions/worker-3/send

The receiving agent sees this as a new message in its conversation. It can decide whether to act on it immediately or finish its current task first.

Peek — non-intrusive observation

Sometimes an agent needs to check what another agent is doing without interrupting it. Peek reads the last N lines of a session's terminal output:

# Check what worker-5 is currently doing
curl -sk "$AMUX_URL/api/sessions/worker-5/peek?lines=50" \
  | python3 -c "import json,sys; print(json.load(sys.stdin).get('output',''))"

This is critical for triage agents — agents whose sole job is to monitor other agents, reassign stuck tasks, and escalate blockers.

Shared notes

For information that multiple agents need to reference — API contracts, migration plans, shared decisions — use a notes system rather than broadcasting messages:

# Agent writes a shared decision to notes
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"content":"# Auth Migration Plan\n\nAll endpoints now use opaque tokens.\nOld JWT validation removed in PR #412.\nTest fixtures updated in fixtures/auth.json."}' \
  $AMUX_URL/api/notes/auth-migration

# Any other agent can read it
curl -sk $AMUX_URL/api/notes/auth-migration

Layer 4: Fault tolerance — self-healing

This is the layer that separates "I ran 10 agents for a demo" from "I run 10 agents every night and ship the PRs in the morning."

Failure mode 1: Context overflow

Claude Code has a context window. Long-running agents hit it. When they do, the agent stalls with a compaction prompt. An unattended agent that hits context overflow at 2am just stops.

Solution: A watchdog process monitors terminal output. When it detects context overflow patterns (e.g., "context is getting large"), it automatically triggers the /compact command. The agent resumes with a compressed context.

# Watchdog detects context overflow and triggers compaction
# The agent sees: "Context compacted. Resuming from turn 42."
# No human intervention needed.

Failure mode 2: Process crashes

Claude Code crashes. The tmux pane dies. An unattended setup loses the session permanently.

Solution: The watchdog detects dead sessions and restarts them. It replays the last message to restore the agent's state. The agent picks up where it left off.

Failure mode 3: Stuck permission prompts

Claude Code asks "Allow this tool?" The agent can't proceed because nobody is there to approve. In YOLO mode, the orchestrator auto-answers these prompts.

Failure mode 4: Circular behavior

An agent gets stuck in a loop — refactoring the same file, running the same failing test, or writing and deleting the same code. Without monitoring, this burns tokens forever.

Solution: Track per-session token spend. Set kill thresholds. If an agent burns more than X tokens without producing a commit or moving a task to "done," flag it for human review.

Layer 5: Governance — cost control and observability

Per-session cost tracking

Every agent should report its token consumption. Parse JSONL transcripts to extract input/output token counts, deduplicate across restarts, and compute per-session cost:

# Check per-session token spend
curl -sk $AMUX_URL/api/sessions | python3 -c "
import json, sys
for s in json.load(sys.stdin):
    cost = s.get('total_cost', 0)
    if cost > 0:
        print(f\"{s['name']:20s} \${cost:.2f}\")
"

The dashboard

Running 10+ agents without a dashboard is like running 10 servers without monitoring. You need to see:

  • Which agents are active, idle, or stuck
  • What task each agent is working on
  • Terminal output (peek) without SSH-ing into each session
  • Token spend per session and total
  • Board status: how many tasks are done, in progress, blocked

Mobile access matters more than you'd expect. If you kick off 10 agents at 11pm, you want to check progress from your phone at midnight — not walk back to your desk.

Kill switches

You need the ability to stop any agent instantly, stop all agents, or pause the entire system. API-driven, accessible from the dashboard and from your phone.

Agent Teams and a full orchestrator are complementary. Use Agent Teams for short parallel tasks within a conversation. Use an orchestrator for long-running agent fleets that operate overnight.

Real-world orchestration patterns

Pattern 1: The decomposition pattern

One human (or one "lead" agent) breaks work into tasks on the board. Worker agents claim tasks independently.

# Human or lead agent creates tasks
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"title":"Refactor auth module to use opaque tokens","status":"todo"}' \
  $AMUX_URL/api/board

curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"title":"Update API tests for new token format","status":"todo"}' \
  $AMUX_URL/api/board

curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"title":"Update OpenAPI spec with new auth schema","status":"todo"}' \
  $AMUX_URL/api/board

# Worker agents claim tasks atomically — no coordination needed

Pattern 2: The triage pattern

A dedicated triage agent monitors the board and other agents. It reassigns stuck tasks, splits tasks that are too large, and escalates blockers.

# Triage agent's CLAUDE.md instructions:
# "You are a triage agent. Every 5 minutes:
# 1. Check the board for tasks stuck in 'doing' for >30 minutes
# 2. Peek at the agent working on each stuck task
# 3. If the agent is stuck, send it a helpful nudge
# 4. If the agent has crashed, move the task back to 'todo'
# 5. Report status to the #triage note"

Pattern 3: The pipeline pattern

Tasks have dependencies. Agent A's output feeds into Agent B's input. The board tracks dependencies; agents only claim tasks whose prerequisites are "done."

# Task with a dependency
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"title":"Write integration tests for new auth flow","status":"todo","desc":"Depends on: TASK-42 (auth refactor). Do not start until TASK-42 is done."}' \
  $AMUX_URL/api/board

Implementation checklist

Building your own orchestrator? Here's what you need, in priority order:

  1. Process manager — start/stop agents in tmux sessions with env var injection
  2. Task board with atomic claiming — SQLite is perfect for single-machine orchestration
  3. Watchdog — monitor terminal output, detect crashes and context overflow, auto-recover
  4. REST API — so agents can interact with the board and each other
  5. Token tracking — parse JSONL transcripts for cost visibility
  6. Dashboard — web UI for monitoring all sessions at a glance
  7. Inter-agent messaging — send text to another session's terminal input
  8. Notes / shared state — persistent documents any agent can read/write
  9. Scheduling — cron-style recurring tasks for overnight runs
  10. Mobile access — PWA for monitoring from your phone

Or skip the build and use amux, which implements all 10 in a single Python file with zero dependencies.
