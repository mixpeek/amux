Kanban Board for AI Agents

Post tasks. Agents claim them atomically. The board is the single source of truth for your AI engineering team. Updated July 2026.

When you run 10 or 20 parallel AI coding agents, the hardest coordination problem isn't launching them — it's ensuring no two agents pick up the same task. amux solves this with a SQLite-backed kanban board and a single atomic endpoint: POST /api/board/:id/claim. You post tasks; agents claim them atomically; the board shows who has what in real time, on your phone or browser. This is the coordination layer that turns a fleet of independent agents into a functioning AI engineering team.

Core mechanism: The claim endpoint is an atomic SQLite compare-and-swap. Even with 20 agents racing for the same queue, each task is assigned exactly once. If a task is already claimed, the call returns 409 and the agent moves to the next available task. No Redis, no external coordination service.

Why a kanban board is essential for multi-agent coordination

The naive alternative — hardcoding which agent gets which task — breaks at 5+ agents. You can't predict which agent will finish first, which will get stuck, or which task will expand in scope. A shared queue with atomic claiming lets agents self-organize around available work.

Without atomic claiming, you get the classic duplicate-work problem: two agents both see a task in the queue, both start it simultaneously, both make changes, and you end up with a merge conflict and 2× the token spend. With atomic claiming, the first agent to claim a task wins — every subsequent agent gets a 409 and moves on.

The amux board adds one more layer: the verified status. Most team tools conflate "done" with "working in production." amux makes the distinction explicit, so you always know the real state of your fleet's work.

The task status lifecycle

Status Meaning Who sets it
todo Queued, available to claim You (or an orchestrator agent)
doing Actively owned by a session Agent (via /claim or PATCH)
done Implementation complete — NOT confirmed in prod Agent on task completion
verified Confirmed working in production with evidence Human reviewer or verification agent

The done vs verified distinction is critical. An agent that opens a PR marks the task done. The task should only reach verified after CI passes, the change is deployed, and production behavior is confirmed. Don't let agents mark tasks verified — that's a human judgment or a dedicated verification agent that checks actual prod behavior.

Atomic task claiming — how it works

The claim operation is a single SQLite transaction:

# Claim a task atomically
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"session":"my-agent-session"}' \
  $AMUX_URL/api/board/TASK-ID/claim

# Response 200 → task is yours
# Response 409 → already claimed, find another task

Inside, this runs a SQL transaction that checks WHERE id=? AND status='todo' and updates to status='doing', session=? atomically. If the row was already updated by another agent between your check and your write, the transaction returns nothing matched → 409. No lost updates, no double claims, even under concurrent load.

Wiring the board into your agents' CLAUDE.md

Add this workflow to the CLAUDE.md for each agent (or a shared project CLAUDE.md):

## Task workflow (MANDATORY — do not skip)

### Before starting any task
1. Check the board for available work:
   curl -sk $AMUX_URL/api/board | python3 -c "
   import json,sys
   tasks = [t for t in json.load(sys.stdin) if t['status']=='todo']
   [print(t['id'], t['title']) for t in tasks[:5]]
   "

2. Claim the first relevant task:
   curl -sk -X POST -H 'Content-Type: application/json' \
     -d '{"session":"$AMUX_SESSION"}' \
     $AMUX_URL/api/board/TASK-ID/claim
   # If 200: proceed. If 409: try the next task.

### While working
   curl -sk -X PATCH -H 'Content-Type: application/json' \
     -d '{"status":"doing","desc":"Working on X..."}' \
     $AMUX_URL/api/board/TASK-ID

### When done
   curl -sk -X PATCH -H 'Content-Type: application/json' \
     -d '{"status":"done","desc":"Result: opened PR #123 — implemented OAuth refresh flow"}' \
     $AMUX_URL/api/board/TASK-ID

Adding tasks to the board

Via CLI (quickest)

amux board add "Implement OAuth refresh flow"
amux board add "Add Stripe webhook handler"
amux board add "Write integration tests for payments module"
# Each is auto-tagged to the current $AMUX_SESSION and set to todo

Via REST API (for scripts and orchestrators)

# Add a single task
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"title":"Implement OAuth refresh flow","status":"todo","desc":"See spec in docs/auth.md"}' \
  $AMUX_URL/api/board

# Bulk-add from a list
for task in "Auth refresh" "Stripe webhooks" "Payment tests" "API docs"; do
  curl -sk -X POST -H 'Content-Type: application/json' \
    -d "{\"title\":\"$task\",\"status\":\"todo\"}" \
    $AMUX_URL/api/board
done

From the dashboard or iOS app

Multi-agent coordination patterns

Pattern 1: Single shared queue (simplest)

All agents pull from one todo queue. Each agent is a generalist — it claims whatever is available. Works well when tasks are similar in scope and any agent can handle any task.

Pattern 2: Role-filtered queues (pipeline)

Add a role field to each task (or prefix the title). Implementer agents look for [implement] tasks; reviewer agents look for [review] tasks. This creates a natural pipeline: implementers finish, retitle their task [review] ..., and reviewer agents pick it up.

# Implementer agent's task query
tasks=$(curl -sk $AMUX_URL/api/board | python3 -c "
import json,sys
[print(t['id'], t['title']) for t in json.load(sys.stdin)
 if t['status']=='todo' and '[implement]' in t['title'].lower()]
")

# After completing, hand off to reviewer:
curl -sk -X PATCH -H 'Content-Type: application/json' \
  -d '{"title":"[review] Auth refresh — PR #123","status":"todo","desc":"PR opened"}' \
  $AMUX_URL/api/board/TASK-ID

Pattern 3: Session-tagged tasks (dedicated assignments)

Pre-assign tasks to specific sessions by setting the session field at creation time. Agents check for tasks tagged to their own $AMUX_SESSION. Use this when different agents have different specializations or access rights.

# Create a task for a specific session
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"title":"Audit payments module for PCI issues","session":"security-agent","status":"todo"}' \
  $AMUX_URL/api/board

# The security-agent checks for its own tasks
curl -sk $AMUX_URL/api/board | python3 -c "
import json,sys,os
s = os.getenv('AMUX_SESSION','')
[print(t['id'], t['title']) for t in json.load(sys.stdin)
 if t.get('session')==s and t['status']=='todo']
"

Monitoring the board from your phone

The amux board view is designed for one-handed phone use. From the iOS app (App Store: amux) or the PWA on Android:

  • Swipe between columns (todo / doing / done / verified)
  • Tap any task to see the full description, session owner, and timestamps
  • Add new tasks with a single tap — the title field opens directly
  • Move tasks between statuses by tapping the status pill
  • The board updates live via Server-Sent Events — you see changes from all agents within ~2 seconds

For teams using the board across multiple developers: everyone accessing the same amux server (via Tailscale or a shared cloud VM) sees the same live board state. One server, one board, many participants.

iCal integration for due dates

Board items support a due field (ISO date). Items with due dates are exported as an iCal feed:

# Add a task with a due date
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"title":"Ship v2 API","status":"todo","due":"2026-07-15"}' \
  $AMUX_URL/api/board

# Subscribe to the iCal feed
# Local: https://localhost:8822/api/calendar.ics
# S3 (public, stable): configure AMUX_S3_BUCKET in ~/.amux/server.env

Subscribe the iCal URL from Google Calendar, Apple Calendar, or any calendar app that supports iCal feeds. The feed auto-updates on every board write — when an agent marks a task done, the calendar event disappears within the next calendar sync.

Complete board REST API reference

Method Endpoint Description
GET /api/board List all tasks (all statuses)
POST /api/board Create a task — body: {title, status?, desc?, session?, due?}
PATCH /api/board/:id Update any field — status, title, desc, session, due
DELETE /api/board/:id Delete a task
POST /api/board/:id/claim Atomic claim — 200 on success, 409 if already claimed
POST /api/board/clear-done Archive all tasks in done or verified status
GET /api/calendar.ics iCal feed for tasks with due dates

CLI shorthands

amux board add "Task title"      # create as todo, tag current session
amux board doing TASK-ID         # mark in progress
amux board done TASK-ID          # mark complete
amux board todo TASK-ID          # move back to todo

The board as the team's shared source of truth

When you run multiple agents across multiple developers on a shared amux server, the board is the one place everyone — human and agent — reads to understand what's being worked on, what's done, and what's next. Sessions declare their work before starting (doing), update when complete (done), and the board prevents duplication automatically.

This matters for multi-session orchestration: an orchestrator agent can scan the board, see what's claimed, and spawn new agents only for unclaimed work. A reviewer can look at the board in the morning and understand overnight output without reading any logs. The board tells the story of your AI engineering team's work.
