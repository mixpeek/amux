A2A + MCP: How the Two Agent Protocols Work Together

MCP connects agents to tools. A2A connects agents to each other. Here's how they fit together in production systems.

Two protocols now dominate the agent infrastructure landscape. Anthropic's Model Context Protocol (MCP) — launched November 2024 — gives agents a standard way to access tools, databases, and APIs. Google's Agent-to-Agent Protocol (A2A) — launched April 2025, now at 150+ participating organizations — gives agents a standard way to discover and delegate to each other.

Most guides cover them separately. This guide covers the architecture where they work together — because in any multi-agent system, you need both.

The two axes of agent connectivity

Think of agent connectivity as two axes:

MCP (vertical axis)A2A (horizontal axis)

MCP is the nervous system — it lets each agent sense and act on its environment. A2A is the coordination layer — it lets agents form teams.

MCP: the tool protocol

MCP defines a standard interface between an agent (the "client") and external capabilities (the "server"). Before MCP, every agent framework invented its own tool-calling format. MCP standardized it.

How MCP works

  1. Server declares capabilities. An MCP server exposes a list of tools (functions the agent can call), resources (data the agent can read), and prompts (templates the agent can use).
  2. Client discovers tools. The agent reads an mcp.json config file that lists available servers. At startup, it connects to each server and fetches the tool list.
  3. Agent calls tools. During a conversation, the LLM decides to call a tool. The agent framework routes the call to the appropriate MCP server, which executes it and returns the result.
// mcp.json — declares which MCP servers this agent can use
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "ghp_..." }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": { "DATABASE_URL": "postgresql://..." }
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/home/user/project"]
    }
  }
}

The key insight: MCP servers are per-agent. Each agent gets its own set of tools, configured in its own mcp.json. An agent with a GitHub MCP server can create PRs. An agent with a Postgres MCP server can query databases. An agent with both can do both.

MCP in multi-agent systems

When you run multiple agents, MCP lets you give each agent a different toolkit:

Agent roleMCP serversCapabilities
Frontend developerfilesystem, browser-toolsRead/write files, inspect rendered UI
Backend developerfilesystem, postgres, redisRead/write files, query databases
Code reviewergithub, filesystemRead PRs, post comments, read code
DevOps agentkubernetes, aws, terraformDeploy, scale, manage infrastructure

This is least-privilege by design. The frontend agent can't accidentally drop a database table because it doesn't have the Postgres MCP server. The DevOps agent can't modify application code because it doesn't have write access to the filesystem.

A2A: the coordination protocol

MCP solves "what can this agent do?" but not "how do agents work together?" That's A2A's job.

How A2A works

A2A defines four primitives:

  1. Agent Cards. Every A2A agent publishes a JSON document at /.well-known/agent.json that describes its capabilities, accepted input formats, and authentication requirements. This is how agents discover each other.
  2. Tasks. The core unit of work. A client agent sends a task to a remote agent. Tasks have an ID, state machine (submitted → working → done/failed/canceled), and can carry structured or unstructured data via "parts" (text, files, structured data).
  3. Messages. Within a task, agents exchange messages — the request, progress updates, and the final result. Messages can contain multiple "parts" (text, file data, JSON).
  4. Streaming. Long-running tasks stream progress via SSE. The client receives real-time updates as the remote agent works.
// Agent Card — published at /.well-known/agent.json
{
  "name": "code-reviewer",
  "description": "Reviews pull requests for bugs, security issues, and style violations",
  "url": "https://reviewer.internal:8080",
  "version": "1.0.0",
  "capabilities": {
    "streaming": true,
    "pushNotifications": false
  },
  "skills": [
    {
      "id": "review-pr",
      "name": "Review Pull Request",
      "description": "Reviews a GitHub PR and posts comments with findings",
      "tags": ["code-review", "security", "quality"]
    }
  ],
  "defaultInputModes": ["text/plain", "application/json"],
  "defaultOutputModes": ["text/plain"]
}

The A2A task lifecycle

Client Agent                          Remote Agent
     |                                      |
     |  POST /tasks/send                    |
     |  {"skill":"review-pr",               |
     |   "message":"Review PR #42"}         |
     |-------------------------------------→|
     |                                      |
     |  {"status":"working",                |
     |   "message":"Analyzing 12 files..."}  |
     |←-------------------------------------|
     |                                      |
     |  {"status":"working",                |
     |   "message":"Found 3 issues..."}     |
     |←-------------------------------------|
     |                                      |
     |  {"status":"done",                   |
     |   "message":"Review complete.        |
     |    3 issues posted as PR comments."} |
     |←-------------------------------------|

The key insight: A2A is model-agnostic. The client agent doesn't know or care what LLM powers the remote agent. A Claude agent can delegate to a GPT agent can delegate to a Gemini agent — the protocol abstracts the implementation behind Agent Cards and standard messages.

Where A2A and MCP intersect

In a production multi-agent system, A2A and MCP occupy distinct but complementary roles:

┌──────────────────────────────────────────────────────────────┐
│                      Orchestrator                            │
│                                                              │
│   ┌─────────────┐    A2A     ┌─────────────┐                │

│   │              │           │              │                │

│   │ │ GitHub  │  │           │ │ Postgres│  │                │

MCP = vertical (agent → tools)     A2A = horizontal (agent → agent)

Agent A uses MCP to read files and create GitHub PRs. Agent B uses MCP to query Postgres and Redis. When Agent A finishes implementing a feature, it uses A2A to delegate a review task to Agent B. Agent B inspects the changes (using its own MCP tools), then responds via A2A with findings.

Three integration patterns

Pattern 1: MCP server wraps A2A agent

Expose a remote A2A agent as a local MCP tool. The calling agent doesn't know it's talking to another agent — it just calls a tool.

// MCP server that delegates to an A2A code-reviewer agent
{
  "mcpServers": {
    "code-review": {
      "command": "node",
      "args": ["a2a-mcp-bridge.js", "--agent=https://reviewer.internal:8080"]
    }
  }
}

// The MCP server internally:
// 1. Receives tool call from local agent
// 2. Creates A2A task on remote agent
// 3. Streams progress back as MCP results
// 4. Returns final output when A2A task completes

When to use: When you want agents to delegate without being aware of A2A. The MCP bridge handles discovery, task creation, and progress tracking. The calling agent just sees a tool called request_code_review.

Pattern 2: A2A for orchestration, MCP for execution

The orchestrator uses A2A to assign work to specialized agents. Each agent uses MCP to execute its assigned work.

# Orchestrator discovers available agents via Agent Cards
GET https://frontend-agent:8080/.well-known/agent.json
GET https://backend-agent:8081/.well-known/agent.json
GET https://test-agent:8082/.well-known/agent.json

# Orchestrator delegates tasks via A2A
POST https://frontend-agent:8080/tasks/send
  {"skill": "implement-component", "message": "Build the user settings page..."}

POST https://backend-agent:8081/tasks/send
  {"skill": "implement-endpoint", "message": "Add PUT /api/users/:id/settings..."}

# Each agent uses its own MCP tools to do the actual work
# Frontend agent: filesystem MCP, browser-tools MCP
# Backend agent: filesystem MCP, postgres MCP
# Test agent: filesystem MCP, playwright MCP

When to use: Classic multi-agent orchestration. The orchestrator is the brain (task decomposition, dependency tracking), agents are the hands (each with specialized tools via MCP).

Pattern 3: Peer-to-peer A2A with shared MCP resources

No central orchestrator. Agents discover peers via Agent Cards and delegate tasks directly. All agents share access to a common MCP resource (like a shared database or board) for coordination.

# Agent A discovers Agent B
GET https://agent-b:8080/.well-known/agent.json
→ skills: ["database-migration", "schema-review"]

# Agent A delegates directly
POST https://agent-b:8080/tasks/send
  {"skill": "schema-review", "message": "Review this migration before I apply it: ..."}

# Both agents share an MCP server for coordination state
{
  "mcpServers": {
    "shared-board": {
      "url": "https://orchestrator:8822/mcp",
      "tools": ["create_task", "claim_task", "complete_task", "list_tasks"]
    }
  }
}

When to use: When agents are autonomous and tasks emerge dynamically. No predefined workflow — agents request help from peers as needed, coordinating through shared state.

How amux implements these patterns today

amux has supported agent-to-agent communication since before A2A existed. Its REST API implements the same primitives — just with a different wire format:

A2A conceptamux equivalentAPI
Agent Card (discovery)Session listGET /api/sessions
Task creationBoard itemPOST /api/board
Task claimingAtomic claim (SQLite CAS)POST /api/board/:id/claim
Message passingSession sendPOST /api/sessions/:name/send
Progress streamingSession peek + SSEGET /api/sessions/:name/peek
Task state machineBoard status (todo → doing → done)PATCH /api/board/:id
Shared resourcesNotesGET/POST /api/notes/:slug
Push notificationsChannels (@mentions)POST /api/channels/:name/send

The difference: amux doesn't require agents to implement a protocol. Agents are Claude Code sessions — they call curl to interact with the REST API. The orchestrator (amux) handles discovery, state management, and coordination. Agents just need HTTP.

# In amux: agent discovers peers
curl -sk $AMUX_URL/api/sessions | python3 -c "
import json, sys
for s in json.load(sys.stdin):
    print(f'{s[\"name\"]:20s} {s.get(\"status\",\"\")} — {s.get(\"desc\",\"\")}')
"

# Agent delegates a task
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"title":"Review the auth migration","session":"reviewer","status":"todo"}' \
  $AMUX_URL/api/board

# Agent sends a direct message to a peer
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"text":"I finished the auth refactor — can you review PR #42?"}' \
  $AMUX_URL/api/sessions/reviewer/send

# Agent checks peer's progress
curl -sk "$AMUX_URL/api/sessions/reviewer/peek?lines=20" \
  | python3 -c "import json,sys; print(json.load(sys.stdin).get('output',''))"

MCP in amux sessions

Each amux session can have its own MCP configuration. The centralized mcp.json applies to all sessions, and per-project .claude/settings.json can add project-specific servers:

# Global MCP config (shared by all sessions)
cat mcp.json
{
  "mcpServers": {
    "github": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-github"] },
    "memory": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-memory"] }
  }
}

# Per-project MCP config (only for sessions registered to this project)
cat ~/Dev/myproject/.claude/settings.json
{
  "mcpServers": {
    "postgres": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-postgres"] }
  }
}

Choosing your architecture

The protocol you need depends on your deployment:

ScenarioProtocol stackWhy
Single agent, multiple toolsMCP onlyNo agent-to-agent communication needed
Multiple agents, same machineREST API (amux)A2A is overkill for local orchestration — HTTP is simpler
Multiple agents, same org, different machinesA2A + MCPA2A handles cross-machine discovery and delegation
Agents from different organizationsA2A + MCP + OAuthA2A's auth model handles cross-org trust
Heterogeneous models (Claude + GPT + Gemini)A2A + MCPA2A's model-agnostic design is essential

When A2A is overkill

If all your agents run on the same machine, managed by the same orchestrator, you don't need A2A's formal protocol. A REST API with session discovery, message passing, and a task board gives you the same coordination without the protocol overhead. That's what amux provides.

A2A becomes valuable when:

  • Agents run on different machines (distributed fleet)
  • Agents use different LLMs (Claude talks to GPT)
  • Agents belong to different organizations (cross-org delegation)
  • You need standardized discovery (Agent Cards at well-known URLs)

When MCP is not enough

MCP tools are stateless function calls — request in, response out. They don't model:

  • Long-running tasks that take minutes or hours
  • Multi-turn conversations between agents
  • Progress tracking with intermediate status updates
  • Task cancellation for runaway work

If your agents need to delegate work that takes more than a few seconds, you need something beyond MCP — either A2A or a task-based API like amux's board.

Implementation checklist

Setting up a multi-protocol agent system:

  1. Start with MCP. Configure MCP servers for each agent's role. Test that each agent can call its tools independently.
  2. Add local coordination. Use amux's REST API (or your own) for session discovery, message passing, and task tracking. Verify agents can delegate to each other.
  3. Add a task board. Atomic task claiming prevents duplicate work. SQLite CAS is sufficient for single-machine orchestration.
  4. Add observability. Dashboard showing all sessions, token spend per session, and terminal peek. You need to see what agents are doing.
  5. Add fault tolerance. Watchdog for crash recovery, context compaction, and message replay. Agents will fail — the system shouldn't.
  6. Add A2A (if needed). Only when you need cross-machine or cross-model agent coordination. Publish Agent Cards, implement the task endpoint, and bridge to your existing coordination layer.

Steps 1–5 are what amux gives you out of the box. Step 6 is where you extend for distributed deployments.

The future: protocol convergence

A2A and MCP are converging. MCP's roadmap includes richer tool descriptions that look like A2A skills. A2A's roadmap includes resource access patterns that look like MCP resources. The end state is likely a unified agent protocol — but we're not there yet.

For now, the practical approach is:

  • Use MCP for tools. It's mature, widely adopted, and built into every major agent framework.
  • Use A2A for agent coordination across boundaries. Machines, models, organizations.
  • Use a control plane for everything else. Process management, observability, cost control, scheduling — the operational layer that neither protocol covers.
