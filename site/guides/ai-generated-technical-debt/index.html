AI-Generated Technical Debt: How to Use Agents to Clean Up After Agents

Your codebase has six months of agent-written code. 40% of it will be rewritten. Here's the systematic cleanup workflow.

The AI coding boom created a new category of technical debt — and it's accumulating faster than anyone expected.

The numbers are stark: 40% of AI-generated code gets rewritten within two weeks of being committed. Code churn has doubled. Refactoring activity dropped 60% as teams prioritize shipping features over cleanup. Developers spend 42% of their week managing technical debt, and now that debt is being generated at machine speed.

The irony is that vibe coding was supposed to eliminate the boring parts of software engineering. Instead, it created a new boring part: cleaning up after agents that optimize for "it compiles and looks right" rather than "it's maintainable for the next two years."

This guide covers the systematic cleanup workflow — using the same parallel agent infrastructure that created the debt to methodically pay it down. Not one-off refactors. An ongoing process that keeps AI-generated code from becoming a time bomb.

The five types of AI-generated debt

AI agents create debt differently than humans. Understanding the categories helps you target cleanup effectively.

Debt typeWhat it looks likeWhy agents create itCleanup difficulty
Structural debtDuplicated logic across files, inconsistent naming, God classes, 500-line functionsEach agent session starts fresh — it doesn't know what other agents already wroteMedium
Dependency debtOutdated packages, hallucinated libraries, pinned versions from 2024, conflicting version rangesModels suggest packages from training data, not the current registryEasy
Test debtMissing coverage, superficial happy-path tests, tests that test the mock instead of the codeAgents skip tests unless explicitly told to write them, and when they do, they test the obvious pathMedium-Hard
Documentation debtStale comments describing old behavior, misleading docstrings, README sections that reference removed featuresAgents add verbose comments on first write, then modify code without updating themEasy
Integration debtInconsistent API patterns, incompatible assumptions between modules, tangled state managementParallel agents build features independently — their code works alone but clashes togetherHard

The most dangerous is integration debt. When five agents each build a feature in isolation, they each make reasonable but incompatible assumptions about shared state, error handling, and API contracts. The code works in each agent's test — and breaks when combined.

Detecting AI-generated debt

Before you clean up, you need to know what to clean. Here's a three-agent audit workflow:

Agent 1: The code health auditor

This agent scans the codebase for structural problems. It doesn't fix anything — it creates an inventory.

# Code health auditor's CLAUDE.md instructions
# "You are a code health auditor. Scan the entire codebase and report:
#
# 1. DUPLICATION: Find functions/methods with >70% similarity across files.
#    Report pairs with file paths and line numbers.
#
# 2. COMPLEXITY: Find functions over 50 lines or with cyclomatic complexity >10.
#    These are the 'God functions' that agents love to create.
#
# 3. DEAD CODE: Find exported functions/classes with zero callers.
#    Agents often write helper functions that nothing uses.
#
# 4. INCONSISTENCY: Find places where the same pattern is implemented
#    differently (e.g., error handling, logging, API response format).
#
# 5. STALE COMMENTS: Find comments that contradict the code they describe.
#
# Post results as a board task with severity ratings (critical/high/medium/low).
# Do NOT fix anything. Audit only."
# Run the auditor
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"title":"Code health audit — full codebase","status":"todo","session":"auditor"}' \
  $AMUX_URL/api/board

Agent 2: The dependency auditor

# Dependency auditor's CLAUDE.md instructions
# "You are a dependency auditor. Check all dependency files and report:
#
# 1. OUTDATED: Packages more than 2 major versions behind
# 2. VULNERABLE: Packages with known CVEs (check npm audit / pip-audit / cargo audit)
# 3. UNUSED: Packages in the manifest that nothing imports
# 4. HALLUCINATED: Package names that don't exist in the registry
#    (agents sometimes invent plausible-sounding package names)
# 5. CONFLICTING: Version ranges that could resolve to incompatible versions
#
# Post results to the board. Include the exact command to fix each issue."

Agent 3: The test coverage auditor

# Test coverage auditor's CLAUDE.md instructions
# "You are a test coverage auditor. Analyze test quality and report:
#
# 1. UNCOVERED: Functions/methods with zero test coverage
# 2. SHALLOW: Tests that only cover the happy path — no error cases,
#    no edge cases, no boundary conditions
# 3. MOCK-HEAVY: Tests where the mocks are more complex than the code
#    they test — these often pass even when the real code is broken
# 4. BRITTLE: Tests that depend on implementation details (exact string
#    matches, specific call counts, private method spying)
# 5. MISSING INTEGRATION: Features that have unit tests but no
#    integration tests — the most common AI test gap
#
# Post results to the board with a prioritized list of what needs tests first."

Run all three auditors in parallel. They examine different aspects of the codebase and don't conflict with each other. With amux, this takes minutes instead of the hours a human audit would take.

# Launch all three audits in parallel
for agent in auditor dep-auditor test-auditor; do
  curl -sk -X POST -H 'Content-Type: application/json' \
    -d "{\"title\":\"Audit: $agent\",\"status\":\"todo\",\"session\":\"$agent\"}" \
    $AMUX_URL/api/board
done

The cleanup pipeline

Once auditors have reported, cleanup agents address each category. The key principle: one agent per cleanup type, one PR per fix. Atomic changes are easier to review and safer to revert.

Phase 1: Quick wins (1-2 hours)

Start with the easy, high-impact items that a single agent can fix without risk:

Cleanup taskAgent approachRisk level
Remove dead codeDelete unused exports identified by auditor. Run test suite after each deletion.Low
Update dependenciesBump outdated packages one at a time. Run tests between each bump.Low
Fix stale commentsDelete comments that contradict code. Remove auto-generated boilerplate comments.Very low
Remove unused importsSimple tree-shaking pass. No behavior change.Very low
 
# Dead code cleanup agent
# "You are a dead code cleanup agent. For each item in the audit report:
# 1. Verify the function/class truly has zero callers (grep the entire codebase)
# 2. Delete it
# 3. Run the full test suite
# 4. If tests pass: commit with message 'cleanup: remove unused FUNCTION_NAME'
# 5. If tests fail: revert and flag as false positive on the board
# IMPORTANT: One commit per deletion. Never batch multiple deletions."

Phase 2: Structural refactoring (2-4 hours)

Address the duplicated code, inconsistent patterns, and oversized functions:

# Deduplication agent
# "You are a deduplication agent. For each pair of duplicated functions:
# 1. Determine which version is better (more complete, better error handling)
# 2. Extract to a shared module if both are in different files
# 3. Replace all callers to use the single canonical version
# 4. Run tests
# 5. Commit: 'refactor: deduplicate FUNCTION_NAME into SHARED_MODULE'
#
# RULE: Write characterization tests BEFORE refactoring if the function
# has no existing tests. Never refactor untested code."
# Pattern consistency agent
# "You are a pattern consistency agent. For each inconsistency found:
# 1. Identify the dominant pattern (most common implementation)
# 2. Refactor outliers to match the dominant pattern
# 3. If no dominant pattern exists, pick the cleanest one and standardize
# 4. Run tests after each change
# 5. Commit: 'refactor: standardize ERROR_HANDLING/LOGGING/API_PATTERN'
#
# Examples of what to standardize:
# - Error handling: try/catch vs .catch() vs result types
# - Logging: console.log vs logger.info vs custom wrapper
# - API responses: {data, error} vs {result, status} vs raw returns
# - Null handling: optional chaining vs explicit checks vs defaults"

Phase 3: Test backfill (2-4 hours)

The highest-ROI cleanup activity. AI-generated code without tests is a liability. AI-generated tests for that code turn it into an asset.

# Test backfill agent
# "You are a test backfill agent. For each uncovered function in the audit:
# 1. Read the function and understand its contract
# 2. Write tests that cover:
#    - Happy path with realistic inputs
#    - Edge cases: empty inputs, null, undefined, zero, max values
#    - Error paths: what happens when dependencies throw?
#    - Boundary conditions: off-by-one, integer overflow, empty arrays
# 3. Run tests — if any fail, you found a BUG. Report it on the board
#    as a separate issue instead of fixing it.
# 4. Target: branch coverage > 80% for each function
# 5. Commit: 'test: add coverage for FUNCTION_NAME'
#
# IMPORTANT: Do NOT fix bugs you discover. Report them. Your job is
# to add tests, not change production code. Mixing test additions with
# bug fixes makes both harder to review."

The test backfill agent frequently discovers bugs. This is the single best argument for running it: existing AI-generated code that "works" often has subtle bugs that only surface when you write edge-case tests. Finding these during cleanup is 10-50x cheaper than finding them in production.

Phase 4: Integration cleanup (4-8 hours)

The hardest category. Integration debt requires understanding how modules interact, which usually means reading code across the entire codebase. This is where parallel agents shine — each agent can focus on one integration boundary.

# API consistency agent
# "You are an API consistency agent. Review all internal API boundaries:
# 1. List all REST endpoints / RPC methods / function interfaces
# 2. Check for inconsistencies:
#    - Different error formats (some return {error: string}, others throw)
#    - Different auth patterns (some check tokens, others assume auth)
#    - Different pagination approaches (cursor vs offset vs none)
#    - Different naming conventions (camelCase in some, snake_case in others)
# 3. Propose a canonical pattern for each inconsistency
# 4. Post proposals to a note at /api/notes/api-cleanup-plan
# 5. Wait for human approval before making changes
#
# IMPORTANT: Do NOT refactor APIs without human approval.
# Breaking API changes affect every consumer."

Integration cleanup always requires human review. The agent proposes changes and writes them to a shared note; a human reviews the plan before the agent executes. This is the one place where fully autonomous cleanup is too risky.

Scheduling recurring cleanup

One-off cleanup is a bandaid. The debt comes back because the same agents keep generating it. The solution is scheduled cleanup cycles that run automatically.

Nightly: audit only

# Schedule nightly audit at 2am
# Auditors run in parallel, post findings to the board
# Cost: ~$5-10 per night with API keys, $0 with Max subscription

# In amux, use the scheduler:
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{
    "name": "nightly-code-audit",
    "cron": "0 2 * * *",
    "action": "send",
    "session": "auditor",
    "message": "Run a full code health audit. Post findings to the board. Focus on changes from the last 24 hours."
  }' \
  $AMUX_URL/api/scheduler

Weekly: audit + cleanup

# Schedule weekly cleanup on Sunday at midnight
# Full pipeline: audit → quick wins → structural refactoring → test backfill
# Cost: ~$20-50 per week with API keys, $0 with Max subscription

curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{
    "name": "weekly-cleanup",
    "cron": "0 0 * * 0",
    "action": "send",
    "session": "cleanup-lead",
    "message": "Run the full cleanup pipeline: 1) Launch audit agents 2) Wait for audit results 3) Assign cleanup tasks to available agents 4) Verify all cleanup PRs pass tests 5) Post summary to /api/notes/weekly-cleanup-report"
  }' \
  $AMUX_URL/api/scheduler

Monthly: deep integration review

# Monthly deep review — includes integration debt, which needs human approval
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{
    "name": "monthly-integration-review",
    "cron": "0 9 1 * *",
    "action": "send",
    "session": "integration-auditor",
    "message": "Run a full integration consistency audit. Post the API cleanup plan to /api/notes/api-cleanup-plan. Flag items that need human approval on the board."
  }' \
  $AMUX_URL/api/scheduler

Measuring debt over time

You can't manage what you don't measure. Track these metrics weekly:

MetricWhat it tells youHow to measureHealthy trend
Code churn rate% of lines changed within 14 days of being writtengit log analysis — compare line agesDecreasing
Duplication ratio% of codebase that's duplicated logicAudit agent report< 5%
Test coverageBranch coverage of production codeCoverage tool outputIncreasing
Dead code volumeExported symbols with zero callersAudit agent reportDecreasing
Dependency freshnessAverage age of pinned dependency versionsDependency auditor< 6 months
Cleanup velocityDebt items resolved per weekBoard task completion rateStable or increasing
Debt-to-feature ratioCleanup commits vs feature commitsgit log categorization1:3 to 1:5

The debt-to-feature ratio is the most important long-term metric. A healthy codebase generates roughly one cleanup commit for every 3-5 feature commits. If the ratio drops below 1:10, debt is accumulating. If it's above 1:2, you're spending more time cleaning than building — scale back AI code generation until the debt is under control.

The feedback loop: preventing new debt

Cleanup without prevention is a treadmill. The most impactful step is feeding audit findings back into your implementation agents' instructions.

Pattern: audit-driven CLAUDE.md updates

# After each weekly cleanup, update the implementer's CLAUDE.md:
# If the auditor finds duplicated error handling patterns 3+ times:
# → Add to CLAUDE.md: "Use the shared errorHandler from src/utils/errors.ts
#   for all error handling. Do not create new try/catch patterns."

# If the auditor finds inconsistent API responses:
# → Add to CLAUDE.md: "All API responses must use the format
#   {data: T, error: string | null, status: number}.
#   See src/types/api.ts for the type definition."

# If the test auditor finds missing integration tests:
# → Add to CLAUDE.md: "Every new API endpoint must include at least one
#   integration test that hits the real database. No mocked DB tests."

Over time, the implementation agents learn from past mistakes. The auditor finds fewer issues. The cleanup pipeline gets faster. This is the self-improving loop — the same principle from quality gates, applied to long-term codebase health.

Pattern: cleanup-agent-generated rules

Take it a step further — have the cleanup agent itself propose CLAUDE.md rules:

# Rule generation agent
# "After completing a cleanup task, consider whether the issue
# could have been prevented. If yes, propose a CLAUDE.md rule:
#
# Format:
# - RULE: [one sentence describing what to do]
# - EVIDENCE: [the cleanup you just did that motivated this rule]
# - LOCATION: [which CLAUDE.md file this should go in]
#
# Post proposed rules to /api/notes/proposed-rules.
# A human will review and merge them into CLAUDE.md."

Real-world cleanup workflow

Here's a concrete example of running the full pipeline on a codebase with six months of agent-generated code:

# Monday morning: launch the full audit (3 agents in parallel)
amux register myproject --dir ~/Dev/myproject --yolo --sessions 8
amux start myproject

# Assign audit tasks
for role in "code-health" "dependencies" "test-coverage"; do
  curl -sk -X POST -H 'Content-Type: application/json' \
    -d "{\"title\":\"Audit: $role\",\"status\":\"todo\",\"session\":\"auditor\"}" \
    $AMUX_URL/api/board
done

# Audits complete in 15-30 minutes
# Results appear as board tasks with severity ratings

# Monday afternoon: launch cleanup agents for quick wins
# Dead code removal, dependency updates, stale comments
# These are safe for fully autonomous operation

# Tuesday: structural refactoring
# Deduplication, pattern consistency
# Agents work on separate git worktrees to avoid conflicts

# Wednesday: test backfill

Anti-patterns

1. Big-bang cleanup

Wrong: Let debt accumulate for months, then do a massive refactoring sprint.

Right: Continuous small cleanups. One PR per fix. Never a 50-file refactor.

Big-bang refactors break things, are impossible to review, and demoralize the team. Continuous cleanup is safer, more reviewable, and maintains momentum.

2. Cleanup without tests

Wrong: Refactor code that has no test coverage.

Right: Always write characterization tests before refactoring.

If you refactor untested code, you have no way to verify the refactoring preserved behavior. The cleanup might introduce more bugs than the debt it removed.

3. Fixing everything at once

Wrong: Assign all audit findings to cleanup agents simultaneously.

Right: Prioritize by severity. Fix critical items first. Leave low-severity items for next cycle.

Flooding the board with 200 cleanup tasks overwhelms agents and creates merge conflicts. Process 20-30 items per weekly cycle.

4. Cleaning without preventing

Wrong: Run cleanup weekly but never update implementation agent instructions.

Right: Every recurring issue becomes a CLAUDE.md rule.

If you clean up the same pattern every week, you're paying for the same fix repeatedly. Update the source — the implementer's instructions — so the debt stops being generated.
