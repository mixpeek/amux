How to Run a Parallel Security Audit with Claude Code Agents

One agent finds surface bugs. Ten agents running in parallel — each focused on a different module — find the vulnerabilities that actually matter. Here’s the complete setup.

Why AI Security Audits Beat Traditional Scanners

In April 2026, Claude Code discovered a 23-year-old exploitable vulnerability in the Linux kernel. The bug had survived decades of manual code review, thousands of contributor eyes, and every static analysis tool in the ecosystem. An AI agent found it in a single session.

This wasn’t luck. It was a fundamental difference in approach:

DimensionTraditional Scanners (Semgrep, CodeQL, Snyk)AI Agent Audits (Claude Code)
Detection method Pattern matching against known vulnerability signatures Semantic reasoning about what the code does
Strengths Known CVEs, dependency vulns, SAST rule violations Logic bugs, auth bypasses, race conditions, business logic flaws
False positive rate High — flags patterns regardless of context Lower — understands surrounding code and intent
Novel vulnerabilities Cannot find bugs without a matching rule Can reason about previously unseen vulnerability classes
Cross-function analysis Limited to call-graph heuristics Follows data flow through complex control paths
Speed Seconds to minutes Minutes to hours (per module)

The key insight: traditional scanners and AI audits find different classes of bugs. Scanners are fast and comprehensive for known patterns. AI agents are slow but find the bugs that scanners structurally cannot — the ones that require understanding what the code is supposed to do, not just what it looks like.

The practical recommendation: run both. Use Semgrep or CodeQL in CI for fast, pattern-based coverage. Use AI agent audits periodically for deep semantic analysis. This guide covers the second half.

The Single-Agent Approach (and Its Limits)

The simplest AI security audit is Claude Code’s built-in /security-review command. One agent, one pass, entire codebase:

$ claude
> /security-review

This works. For small codebases (under 10k lines), it’s often sufficient. The agent reads the relevant files, reasons about common vulnerability patterns, and produces a report.

It breaks down in three ways as the codebase grows:

  • Context window ceiling. A 200k-token context window holds roughly 50k–80k lines of code. A medium-sized application exceeds this. The agent can’t read every file, so it samples — and sampling means missed surface area.
  • Shallow coverage. When one agent reviews everything, each module gets cursory attention. An auth system that deserves 30 minutes of focused analysis gets 2 minutes in a sweep. Subtle vulnerabilities — timing attacks, TOCTOU races, logic flaws in edge cases — require depth, not breadth.
  • No specialization. Different parts of a codebase have different threat models. The database layer needs SQL injection analysis. The API layer needs auth bypass analysis. The file upload handler needs path traversal analysis. A single agent applying a generic security lens misses domain-specific risks.

The solution is the same pattern that works for every other large-scale agent task: decompose, parallelize, aggregate.

Parallel Audit Architecture

A parallel security audit assigns one agent per module or security boundary. Each agent gets a focused scope, a specialized prompt, and full context window depth on its area. Findings are aggregated on a shared board.

The architecture has three layers:

LayerWhatHow
Decomposition Split the codebase into audit zones One agent per module, service, or trust boundary
Execution Run all agents in parallel with focused prompts amux sessions or headless claude --print
Aggregation Collect, deduplicate, and prioritize findings Shared board + optional triage agent

How to decompose

The decomposition should follow security boundaries, not code organization. Good splits:

  • Auth & session management — login, registration, password reset, session tokens, OAuth flows
  • API endpoints — input validation, authorization checks, rate limiting
  • Database layer — query construction, ORM usage, raw SQL, migration safety
  • File handling — uploads, path construction, temporary files, permissions
  • Third-party integrations — API keys in code, webhook validation, external data trust
  • Frontend / client-side — XSS vectors, CSRF, CSP configuration, sensitive data in DOM
  • Infrastructure & config — env vars, secrets management, TLS, CORS, headers
  • Cryptography — key management, hashing, token generation, randomness

For a microservices architecture, you can also split by service: one agent per service, each with the full OWASP scope but limited to that service’s code.

Step-by-Step Setup with amux

Step 1: Register your project

$ amux register myapp --dir ~/Dev/myapp --yolo
$ amux serve  # → https://localhost:8822

Step 2: Create audit tasks on the board

Post one task per audit zone. Each task includes the scope (which directories/files) and the specific threat model to investigate:

# Auth audit
$ curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"title":"Security audit: auth & session management","status":"todo","desc":"Scope: src/auth/, src/middleware/session.*, templates/login*\nFocus: session fixation, credential storage, password reset flow, OAuth state validation, brute-force protection, token expiration"}' \
  https://localhost:8822/api/board

# API endpoints audit
$ curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"title":"Security audit: API endpoints","status":"todo","desc":"Scope: src/api/, src/routes/\nFocus: input validation, authorization checks on every endpoint, IDOR, mass assignment, rate limiting, error message information leakage"}' \
  https://localhost:8822/api/board

# Database layer audit
$ curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"title":"Security audit: database layer","status":"todo","desc":"Scope: src/models/, src/db/, migrations/\nFocus: SQL injection (raw queries, ORM bypasses), NoSQL injection, migration rollback safety, connection string handling, privilege escalation via DB"}' \
  https://localhost:8822/api/board

# Repeat for each audit zone...

Step 3: Start the audit agents

$ amux start sec-auth sec-api sec-db sec-files sec-integrations sec-frontend sec-infra sec-crypto
  8 sessions started

Step 4: Send each agent its assignment

Each agent claims a task from the board and begins its focused audit:

# Send the same instruction to all audit agents
for session in sec-auth sec-api sec-db sec-files sec-integrations sec-frontend sec-infra sec-crypto; do
  curl -sk -X POST -H 'Content-Type: application/json' \
    -d '{"text":"Check the board for your security audit task. Claim one task, read the scope and focus areas, then perform a thorough security audit of those files. For each finding, report: severity (critical/high/medium/low), file and line number, vulnerability type (OWASP category), description, proof of concept or exploit scenario, and recommended fix. Post each finding as a separate board item with severity in the title."}' \
    https://localhost:8822/api/sessions/$session/send
done

Step 5: Monitor from the dashboard

Open https://localhost:8822 in your browser. The dashboard shows all 8 agents working in parallel, with live output streaming and per-session cost tracking. Findings appear on the kanban board as agents discover them.

A typical 8-agent audit of a medium codebase takes 30–60 minutes and costs $15–40 in API tokens.

Security Prompts That Work

The quality of a security audit depends heavily on the prompt. Generic “find vulnerabilities” prompts produce generic findings. Specific, threat-model-aware prompts find real bugs.

CLAUDE.md security section

Add a security context section to your project’s CLAUDE.md so every audit agent understands the application’s trust model:

# CLAUDE.md — Security context

## Security model

- Auth: JWT tokens stored in HttpOnly cookies, refresh via /api/auth/refresh
- All API endpoints require authentication except: /api/auth/login, /api/auth/register, /health
- Database: PostgreSQL via SQLAlchemy ORM. Raw SQL is used in 2 places (see src/db/reports.py)
- File uploads: stored in S3 via presigned URLs, never on local filesystem
- Secrets: loaded from environment variables, never hardcoded
- Trust boundary: all user input crosses a trust boundary at the API layer

## Known accepted risks (do not flag these)

- /health endpoint is unauthenticated (intentional, used by load balancer)
- Admin users can access all tenant data (multi-tenant admin model)
- Rate limiting is per-IP, not per-user (accepted trade-off)

The “known accepted risks” section is critical. Without it, every audit agent will flag your health endpoint as unauthenticated and your admin access as an IDOR — wasting time on intentional design decisions.

Effective per-agent prompts

Each audit agent should receive a prompt that specifies:

  1. Scope — exactly which files to read
  2. Threat model — which vulnerability classes to focus on
  3. Output format — how to report findings
  4. Severity criteria — what counts as critical vs low

Example for the auth module agent:

You are performing a focused security audit of the authentication and session
management system. Read every file in src/auth/ and src/middleware/session.*.

Check for:
- Session fixation (can an attacker set or predict session tokens?)
- Credential storage (are passwords hashed with bcrypt/argon2? salt per user?)
- Password reset flow (is the token single-use? time-limited? tied to user?)
- OAuth state parameter (is it validated on callback? tied to session?)
- Brute-force protection (is there rate limiting on login? account lockout?)
- Token expiration (do JWTs expire? are refresh tokens rotated?)
- Logout (is the session actually invalidated server-side?)

For each finding, report:
- Severity: critical / high / medium / low
- OWASP category (e.g. A01:2021 Broken Access Control)
- File and line number
- Description of the vulnerability
- Proof of concept: how an attacker would exploit this
- Recommended fix with code example

Post each finding as a board item. Title format: "[CRITICAL] SQL injection in login query"
Ignore anything listed under "Known accepted risks" in CLAUDE.md.

Triaging the Findings

Eight agents auditing eight modules will produce findings. Some are real vulnerabilities. Some are false positives. Some are style suggestions masquerading as security issues. You need a triage process.

Option 1: Human triage

Open the amux board. All findings are there as board items with severity in the title. Sort by severity, review each one, move confirmed vulnerabilities to “doing,” dismiss false positives.

Option 2: Triage agent

Start a ninth agent dedicated to reviewing the findings from the other eight. This agent reads each finding on the board, cross-references the actual code, and filters out false positives:

$ amux start sec-triage
$ curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"text":"Review all security findings on the board. For each one: (1) read the referenced file and line number, (2) verify the vulnerability is real and exploitable, (3) check if it is covered by Known Accepted Risks in CLAUDE.md, (4) if it is a false positive, move it to done with a note explaining why. If it is real, add an exploitation difficulty rating (trivial/moderate/difficult) to the description. Prioritize critical and high findings first."}' \
  https://localhost:8822/api/sessions/sec-triage/send

The triage agent typically eliminates 30–50% of findings as false positives, saving significant human review time.

Fix-in-session workflow

For confirmed vulnerabilities, you can have an agent fix them immediately:

# The same agent that found the bug can fix it
$ curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"text":"Fix the SQL injection vulnerability you found in src/db/reports.py:47. Use parameterized queries. Write a test that verifies the fix. Create the fix on a new git branch named fix/sql-injection-reports."}' \
  https://localhost:8822/api/sessions/sec-db/send

Each fix goes to its own branch, keeping the main branch clean until a human reviews and merges.

AI Audits vs Traditional Tools

Neither approach is sufficient alone. Here’s where each excels:

Vulnerability ClassTraditional ScannersAI Agent AuditBest Approach
Known CVEs in dependencies Excellent — database-driven Poor — doesn’t track CVE databases Snyk / Dependabot
SQL injection (obvious) Good — pattern matching Good — semantic understanding Either
SQL injection (ORM bypass) Poor — ORM usage looks safe Good — understands ORM semantics AI agent
XSS (reflected) Good — taint analysis Good — follows data flow Either
Authorization bypass / IDOR Poor — requires business logic understanding Excellent — reasons about access control AI agent
Race conditions / TOCTOU Poor — requires runtime analysis Good — reasons about concurrency AI agent
Business logic flaws Cannot detect Good — understands intent AI agent
Secrets in code Excellent — regex patterns Good — but slower Scanner (TruffleHog, gitleaks)
Misconfigurations (CORS, CSP, headers) Good — config rules Good — contextual analysis Either
Cryptographic weaknesses Moderate — known-bad patterns Good — understands crypto context AI agent + manual review

The recommended stack: Semgrep or CodeQL in CI (catches the fast, pattern-based stuff on every commit) + parallel AI audit weekly or on-demand (catches the deep, semantic stuff that scanners miss) + annual professional pentest (provides adversarial testing and compliance documentation).

Scheduling Recurring Audits

A one-time audit is useful. A recurring audit is infrastructure. With amux’s cron scheduler, you can run a full parallel audit on a regular cadence:

# Full audit every Sunday night at 10pm
$ amux cron add weekly-security-audit "0 22 * * 0" \
    --session sec-coordinator \
    --prompt "Run the full parallel security audit workflow: post audit tasks to the board for auth, API, database, files, integrations, frontend, infra, and crypto modules. Start 8 audit sessions. Wait for all to complete. Then run the triage agent. Post a summary to the security-audit note with date, finding count by severity, and links to board items."

Over time, this builds a security audit history in your notes: weekly snapshots showing which modules have persistent issues, which categories of vulnerabilities recur, and whether your security posture is improving or degrading.

For teams with compliance requirements (SOC 2, ISO 27001), the audit history on the board serves as evidence of continuous security monitoring — a complement to, not a replacement for, formal compliance tooling.

FAQ

Can Claude Code find real security vulnerabilities?

Yes. In April 2026, Claude Code (Opus 4.6) discovered a 23-year-old exploitable vulnerability in the Linux kernel that survived decades of manual review and traditional static analysis. AI agents find bugs that pattern-matching scanners miss because they reason about code semantics.

How is a parallel audit different from /security-review?

/security-review runs a single agent against your entire codebase in one pass. A parallel audit decomposes by module, assigning a dedicated agent to each. Each agent gets full context window depth on its module. Findings are aggregated on a shared board. The result: deeper coverage with fewer false negatives.

How many agents should I run?

One per module or security boundary. Typical web app: 6–8 agents (auth, API, database, files, integrations, frontend, infra, crypto). Microservices: one per service. Diminishing returns beyond 15–20 for most codebases.

What vulnerabilities can AI agents find that scanners miss?

Logic bugs, authorization bypasses, race conditions, TOCTOU, business logic flaws, and subtle injection vectors that require understanding program semantics. Use traditional scanners for known CVEs, dependency vulns, and SAST rules. Use both.

How much does it cost?

An 8-agent audit of a medium codebase (50k–200k lines) costs $15–40 in API tokens and takes 30–60 minutes. On a Max subscription, roughly a day’s quota. Use amux’s per-session cost tracking to monitor in real time.

Does this replace penetration testing?

No. AI audits complement pentesting. They’re excellent for continuous scanning between formal pentests. Professional pentesting provides adversarial creativity, social engineering vectors, and compliance-grade reporting that AI audits do not.

How do I reduce false positives?

Three techniques: (1) Add a CLAUDE.md security section with your trust model and known accepted risks. (2) Use severity-focused prompts that prioritize exploitability. (3) Run a dedicated triage agent to filter findings before they reach your board.

Can I schedule recurring audits?

Yes. Use amux’s cron scheduler for weekly or nightly audits. Each run produces findings on the board, giving you a continuous security dashboard.
