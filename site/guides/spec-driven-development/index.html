What Is Spec-Driven Development?

Spec-driven development (SDD) is a workflow where you write a structured specification before any code is written — by you or by an AI agent. The spec defines the goal, requirements, constraints, and acceptance criteria. The agent implements against the spec. You review against the spec. No vibes, no guessing, no prompt-and-pray.

Traditional AI coding:
  You: "add pagination to the users API"
  Agent: [guesses what kind, what defaults, what constraints]
  You: "no, not offset-based, cursor-based"
  Agent: [rewrites everything]
  You: "and don't change the schema"
  Agent: [rewrites again]
  Total: 3 rounds, 45 minutes, frustration

Spec-driven:
  You: [write 15-line spec in 5 minutes]
  Agent: [reads spec, implements exactly once]
  You: [review against acceptance criteria]
  Total: 1 round, 15 minutes, done

The core insight: the time you spend writing a spec is always less than the time you spend correcting an agent that guessed wrong. And the gap widens with every additional agent — correcting 10 agents individually is 10x the cost of writing one good spec.

Why Specs Beat Prompts

Prompts and specs are both natural language. The difference is structure:

Dimension Prompt Spec
AmbiguityHigh — agent fills gaps with guessesLow — requirements and constraints are explicit
ReviewabilitySubjective — "does this look right?"Objective — "do these acceptance criteria pass?"
ReusabilityOne-shot, conversation-specificChecked into git, reusable across sessions
ParallelizationRequires human in the loop per agentAgent works autonomously against criteria
Debugging"Why did it do that?""The spec was incomplete" or "the agent didn't follow the spec"

The real unlock: autonomous agents

A prompt requires you to be present — the agent asks questions, you answer, it adjusts. A spec makes the agent autonomous. It has everything it needs to work without you. This is the difference between running 1 agent interactively and running 10 agents overnight. Specs are what make the jump possible.

Anatomy of a Good Spec

A spec has four required sections and one optional one:

# spec: add-cursor-pagination

## Goal
Add cursor-based pagination to GET /api/users.

## Requirements
1. Use cursor-based pagination, keyed on `created_at` (not offset)
2. Default page size: 50, max: 200
3. Response includes `next_cursor` (opaque base64 string) and `has_more` (boolean)
4. Backward-compatible: clients that don't send a cursor get page 1
5. Cursor is stable across concurrent writes (no skipped/duplicated rows)

## Constraints
- Do NOT change the User model or database schema
- Do NOT add new dependencies
- Do NOT modify existing test fixtures
- Keep the existing GET /api/users response format for the user objects themselves

## Acceptance criteria
- `make test` passes (all existing + new tests)
- `curl /api/users?limit=10` returns 10 users with `next_cursor` and `has_more`
- `curl /api/users?cursor=X&limit=10` returns the next 10 users
- `curl /api/users?limit=999` returns 200 users (capped at max)
- `curl /api/users` (no params) returns 50 users (default)

## Context (optional)
- The User model is in `src/models/user.py`
- Existing endpoint is in `src/api/users.py:get_users()`
- We use SQLAlchemy with PostgreSQL
- See `src/api/orders.py` for an example of cursor pagination on another endpoint

Why each section matters

  • Goal — One sentence. Tells the agent what success looks like at the highest level. If the agent gets confused mid-implementation, it re-reads this.
  • Requirements — Numbered list of what must be true when done. Each requirement is independently verifiable. No vague adjectives ("fast", "clean", "modern").
  • Constraints — What the agent must NOT do. This is the most underused section and the one with the highest ROI. Agents love to "improve" things you didn't ask for. Constraints prevent scope creep.
  • Acceptance criteria — Specific commands, tests, or checks that prove the implementation is correct. These should be copy-pasteable. If you can't write a concrete acceptance criterion, the requirement is too vague.
  • Context — Pointers to files, docs, or examples. Not required, but saves the agent 5–10 minutes of codebase exploration.

7 Real Spec Examples (Copy-Paste Ready)

1. API endpoint

# spec: create-invoice-endpoint

## Goal
Add POST /api/invoices to create a new invoice.

## Requirements
1. Accepts JSON body: { customer_id, line_items: [{ description, amount }], due_date }
2. Validates: customer_id exists, at least 1 line item, amounts > 0, due_date in future
3. Returns 201 with the created invoice (including generated `id` and `total`)
4. Returns 422 with field-level errors for invalid input

## Constraints
- Do NOT add authentication (handled by middleware)
- Do NOT send emails or side effects — just create the record
- Use the existing Invoice model, do not modify the schema

## Acceptance criteria
- `make test` passes
- Valid POST returns 201 with `id`, `total`, and `status: "draft"`
- Missing customer_id returns 422 with `{"errors": {"customer_id": "required"}}`
- Empty line_items returns 422
- Negative amount returns 422

2. Bug fix

# spec: fix-duplicate-notifications

## Goal
Fix bug where users receive duplicate email notifications for the same event.

## Requirements
1. Add idempotency check: if a notification with the same (user_id, event_id, channel) exists, skip
2. Use the existing notifications table — add a unique constraint if needed
3. Log skipped duplicates at INFO level

## Constraints
- Do NOT change the notification sending logic itself
- Do NOT add new tables
- Migration must be safe for zero-downtime deploy (no table locks > 1s)

## Acceptance criteria
- `make test` passes
- Calling `send_notification(user_id=1, event_id=42, channel="email")` twice results in 1 row
- Existing tests still pass (no regressions in notification delivery)

3. Test coverage expansion

# spec: tests-for-auth-module

## Goal
Bring test coverage for src/auth/ from 34% to 85%+.

## Requirements
1. Cover: login, logout, token refresh, password reset, email verification
2. Cover edge cases: expired tokens, invalid tokens, locked accounts, rate limiting
3. Use pytest fixtures consistent with existing test style (see tests/conftest.py)
4. Mock external services (email sender, Redis) — do not hit real services

## Constraints
- Do NOT modify any source code in src/auth/
- Do NOT change existing tests
- Tests must run in < 30 seconds total

## Acceptance criteria
- `pytest tests/auth/ -v` — all pass
- `pytest --cov=src/auth --cov-report=term` shows 85%+ coverage
- No tests require network access or running services

4. Database migration

# spec: add-teams-table

## Goal
Add a teams table and associate users with teams (many-to-many).

## Requirements
1. New table: teams (id, name, created_at, updated_at)
2. New join table: team_members (team_id, user_id, role, joined_at)
3. role is an enum: "owner", "admin", "member"
4. Alembic migration, reversible (upgrade + downgrade)

## Constraints
- Do NOT modify the existing users table
- Do NOT add API endpoints (separate task)
- Migration must be idempotent (safe to run twice)

## Acceptance criteria
- `alembic upgrade head` succeeds on a fresh database
- `alembic downgrade -1` removes both tables cleanly
- `alembic upgrade head && alembic upgrade head` does not error (idempotent)

5. Frontend component

# spec: user-avatar-component

## Goal
Create a reusable UserAvatar React component.

## Requirements
1. Shows user's profile image if available, otherwise initials on a colored background
2. Sizes: sm (24px), md (32px), lg (48px), xl (64px)
3. Color derived deterministically from user's name (same name = same color always)
4. Accessible: alt text, role="img" for initials variant
5. Matches design tokens in src/styles/tokens.css

## Constraints
- No external image libraries
- No network requests (image URL passed as prop)
- Must work without JavaScript (SSR-safe)

## Acceptance criteria
- Storybook stories for all 4 sizes, with image, with initials
- Same name always produces same background color
- `npm run test -- UserAvatar` passes
- Lighthouse accessibility score 100 on Storybook page

6. Refactor

# spec: extract-email-service

## Goal
Extract email-sending logic from 6 scattered callsites into a single EmailService class.

## Requirements
1. New file: src/services/email_service.py with EmailService class
2. Methods: send_welcome, send_reset, send_invoice, send_notification
3. All 6 existing callsites refactored to use EmailService
4. Centralized retry logic (3 attempts, exponential backoff)
5. Centralized logging (log every send attempt with recipient + template name)

## Constraints
- Do NOT change email templates
- Do NOT change the SMTP configuration
- Preserve exact same behavior for all 6 callsites (no user-visible changes)

## Acceptance criteria
- `make test` passes
- `grep -r "smtp\|send_email\|sendmail" src/ --include="*.py"` returns only email_service.py
- All 6 callsites import from EmailService

7. Documentation

# spec: api-docs-for-orders

## Goal
Write OpenAPI 3.1 spec for all /api/orders/* endpoints.

## Requirements
1. Cover all 5 endpoints: list, get, create, update, cancel
2. Include request body schemas, response schemas, error responses
3. Include realistic examples for each endpoint
4. Document pagination parameters (cursor, limit)

## Constraints
- Do NOT modify any source code
- OpenAPI spec goes in docs/openapi/orders.yaml
- Use $ref for shared schemas (Error, Pagination)

## Acceptance criteria
- `npx @redocly/cli lint docs/openapi/orders.yaml` passes with no errors
- All 5 endpoints documented with at least one example each

Decomposing Specs Into Parallel Tasks

A single large spec can often be split into multiple parallel tasks. The rule: if two tasks touch different files, they can run in parallel. If they touch the same file, they must be sequential.

Example: splitting a feature spec

# Original spec: "Add team management feature"
# Too big for one agent. Decompose:

Task 1: Database migration (add teams + team_members tables)
  → Touches: alembic/versions/, src/models/
  → Agent 1, worktree: agent/teams-migration

Task 2: API endpoints (CRUD for teams)
  → Touches: src/api/teams.py (new file), tests/api/test_teams.py (new)
  → Agent 2, worktree: agent/teams-api
  → Depends on: Task 1 (needs the model)

Task 3: Team invitation flow
  → Touches: src/api/invitations.py (new), src/services/email_service.py
  → Agent 3, worktree: agent/teams-invitations
  → Depends on: Task 1

Task 4: Frontend team settings page
  → Touches: src/components/TeamSettings/ (new directory)
  → Agent 4, worktree: agent/teams-frontend
  → Independent (can mock API)

The dependency graph

Task 1 (migration)  ──→  Task 2 (API)     ──→  merge all
        │                                         ↑
        └──→  Task 3 (invitations)  ──────────────┘
                                                   ↑
Task 4 (frontend)  ────────────────────────────────┘

Tasks 1 and 4 can start immediately in parallel. Tasks 2 and 3 wait for Task 1's migration to land. All four merge at the end. Total wall-clock time: ~the time of the longest chain (1 → 2), not the sum of all four.

Rules for good decomposition

  • No overlapping files. If two tasks edit the same file, merge conflicts are guaranteed. Either sequence them or split the file first.
  • Each task is independently testable. If you can't run make test on a task's output alone, the task is too entangled.
  • 15–45 minutes per task. Under 15 minutes: the spec overhead isn't worth it. Over 45 minutes: the agent's context window fills up and quality drops.
  • Dependencies are explicit. If Task 2 needs Task 1's model, say so in the spec. The agent can read the model file from Task 1's branch (or you merge Task 1 first).

The Full SDD Workflow

Here's spec-driven development end to end, from backlog to merged PRs:

Phase 1: Spec writing (30–60 minutes for 10 tasks)

# You write specs — 5 minutes each, 10 specs
specs/
  add-cursor-pagination.md
  fix-duplicate-notifications.md
  tests-for-auth-module.md
  extract-email-service.md
  add-teams-migration.md
  add-teams-api.md
  add-teams-invitations.md
  teams-frontend.md
  api-docs-for-orders.md
  upgrade-react-18.md

Phase 2: Push to board (2 minutes)

# Push each spec as a board task
for spec in specs/*.md; do
  title=$(head -1 "$spec" | sed 's/# spec: //')
  amux board add "$title" --desc "$(cat $spec)"
done

# Or via curl:
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"title":"add-cursor-pagination","desc":"...spec contents...","status":"todo"}' \
  https://localhost:8822/api/board

Phase 3: Launch agents (5 minutes)

# Register 10 agents, each in its own worktree
for i in $(seq 1 10); do
  amux register worker-$i --dir ~/Dev/myproject --worktree --yolo
  amux start worker-$i
done

# Each agent's prompt (automatically from the board):
# "Claim a task from the board. Read its spec. Implement it.
#  Run the acceptance criteria. Commit when all pass."

Phase 4: Agents work (1–3 hours, unattended)

Each agent claims a task atomically (no duplicates), reads the spec from the task description, implements in its isolated worktree, runs the acceptance criteria, and commits. The self-healing watchdog restarts any agent that crashes.

Phase 5: Review (30–45 minutes)

# Check results on the board
amux board list

# For each completed task, review the branch:
git diff main..agent/add-cursor-pagination

# Review is fast because you check against acceptance criteria:
# ✓ make test passes
# ✓ curl /api/users?limit=10 returns next_cursor
# ✓ curl /api/users?cursor=X returns next page
# Ship it.

The math

Writing 10 specs: ~60 minutes. Agent implementation (parallel): ~2 hours wall clock. Review: ~45 minutes. Total: ~4 hours for 10 tasks. Without specs, those same 10 tasks take 2–3 hours each sequentially with a single agent (back-and-forth, corrections, rework) = 20–30 hours. SDD with parallel agents is a 5–7x throughput multiplier.

Writing Acceptance Criteria That Actually Work

Acceptance criteria are the most important part of a spec. They are what make review objective instead of subjective, and what let agents self-verify.

Rules for good acceptance criteria

  1. Executable. Every criterion should be a command you can run: make test, curl ..., grep ..., lighthouse .... If you can't run it, it's an opinion, not a criterion.
  2. Binary. Pass or fail. No "should be reasonably fast" or "code should be clean." Those are style preferences for CLAUDE.md, not acceptance criteria.
  3. Independent. Each criterion tests one thing. Don't bundle "tests pass AND coverage is 85% AND no network calls" into one criterion — split them so you know exactly what failed.
  4. Ordered by importance. Put the most critical criterion first. If the agent runs out of context, you want it to at least satisfy the top criteria.

Good vs bad acceptance criteria

BadGood
"Code should be clean"make lint passes with 0 warnings
"Should be fast"time curl /api/users completes in < 200ms
"Should handle errors"curl -X POST /api/users -d '{}' returns 422 with field errors
"Tests should pass"pytest tests/auth/ -v — all pass, 0 skipped
"Should be well-documented"npx @redocly/cli lint docs/openapi.yaml — 0 errors

Constraints: The Most Underused Section

Requirements tell agents what to build. Constraints tell agents what NOT to do. Constraints have disproportionate impact because AI agents love to "help" by doing extra work you didn't ask for:

  • Refactoring nearby code that "could be improved"
  • Adding dependencies "to make it easier"
  • Changing the database schema "to be more normalized"
  • Modifying existing tests "to be more comprehensive"
  • Adding error handling you didn't need

Every line in the constraints section prevents one of these scope-creep patterns. Common constraints worth adding to most specs:

## Constraints
- Do NOT add new dependencies
- Do NOT modify existing tests
- Do NOT change the database schema
- Do NOT refactor code outside the specified files
- Do NOT add features not listed in Requirements
- Keep the public API surface unchanged (backward-compatible)

Specs for Parallel Agent Fleets

Specs become essential when running multiple agents in parallel. Without specs, each agent needs a human babysitter. With specs, agents are autonomous — they have everything they need to work independently.

The fleet workflow

You (30 min)          amux board              10 agents (2 hours)
─────────────         ──────────              ───────────────────
Write 10 specs  ──→  Push to board  ──→  Each agent claims a task
                                          Reads the spec
                                          Implements in worktree
                                          Runs acceptance criteria
                                          Commits and marks done
                                                  │
You (45 min)    ←─────────────────────────────────┘
Review 10 PRs against acceptance criteria
Merge the clean ones

What changes at fleet scale

  • Specs must be self-contained. An agent can't tap you on the shoulder to ask a question. Every ambiguity in the spec becomes a wrong guess multiplied across agents.
  • Constraints prevent merge conflicts. If you don't constrain which files each agent touches, two agents will edit the same file and create a conflict. Use constraints to define boundaries.
  • Acceptance criteria enable batch review. Reviewing 10 PRs against vibes takes hours. Reviewing 10 PRs against checklists takes 45 minutes.
  • The board is the spec registry. Push specs as board task descriptions. Agents claim tasks atomically. No duplicate work.

Anti-Patterns: Specs That Waste Tokens

1. Over-specifying implementation

Specs describe what, not how. If you specify "use a for loop on line 42 of file X," you're writing the code yourself with extra steps. Let the agent make tactical decisions.

# Bad: specifying implementation
"Use a dictionary comprehension to build the lookup table,
 then iterate with enumerate()"

# Good: specifying outcome
"Build a lookup table mapping user_id to email.
 Must handle 100K users in < 1 second."

2. Vague requirements

"Make it better" is not a requirement. "Make it handle 10K concurrent users without degradation" is. Every requirement should be independently verifiable by a machine or a human looking at the output for 10 seconds.

3. Missing constraints

An unconstrained agent will "improve" your codebase in ways you didn't want. Always add: what files are off-limits, what dependencies can't be added, what behaviors can't change.

4. Specs for trivial tasks

"Fix the typo on line 42" does not need a spec. The overhead of writing Goal/Requirements/Constraints/Acceptance exceeds the task itself. Use specs for tasks that take more than 15 minutes.

5. Specs without acceptance criteria

A spec without acceptance criteria is just a prompt with extra formatting. The criteria are what make spec-driven development work — they give the agent a self-verification loop and give you an objective review framework.

Tooling: Where Specs Live and How Agents Consume Them

File-based specs (simplest)

# Store specs in a directory, check into git
specs/
  add-cursor-pagination.md
  fix-duplicate-notifications.md
  ...

# Pass to an agent via prompt:
claude -p "Implement the spec in specs/add-cursor-pagination.md"

Board-based specs (best for parallel fleets)

# Push specs to the amux board as task descriptions
amux board add "add-cursor-pagination" --desc "$(cat specs/add-cursor-pagination.md)"

# Agents claim tasks from the board and read the spec from the description
# Atomic claiming prevents two agents from working on the same spec

Spec-driven tools

ToolApproachBest for
amuxSpecs as board tasks, parallel agents in worktrees, atomic claimingParallel fleet execution at scale
cc-sddSlash commands that enforce Requirements → Design → Tasks → ImplementationSingle-agent structured workflow
KiroIDE with built-in spec editor and agent executorTeams wanting an IDE-native experience
Plain markdownSpecs in a directory, pass via -p flagSimple setups, 1–3 agents

The spec format is tool-agnostic. Any AI coding agent can consume a markdown spec. The orchestration layer — how agents discover, claim, and report on specs — is where tools differ. For parallel execution at scale, you need atomic task claiming and isolated worktrees, which is where an orchestrator like amux fits in.

FAQ

What is spec-driven development?

A workflow where you write a structured specification (goal, requirements, constraints, acceptance criteria) before any code is written. The agent implements against the spec. You review against the spec. It replaces prompt-and-pray with a planning-first approach that produces predictable output.

How is it different from prompt engineering?

A prompt is a one-shot instruction. A spec is a structured document with explicit requirements, constraints, and acceptance criteria. Prompts rely on the agent guessing your intent. Specs eliminate guessing.

How long should a spec be?

10–30 lines. Under 10 means you left out constraints. Over 50 means you're over-specifying implementation details. Sweet spot: 5–10 minutes of writing.

Why is it important for parallel agents?

Without specs, each agent needs human guidance during implementation. That doesn't scale to 10 agents. Specs make agents autonomous: each gets a complete description, works independently, and produces output reviewable against objective criteria.

Should I write specs for small tasks?

No. Under 15 minutes of agent work, just use a prompt. Specs pay off for complex tasks (multiple files, many requirements) and especially for unattended parallel agents that can't ask you questions.

What tools support spec-driven development?

Any AI coding agent can consume specs — they're just markdown. Specialized tools include cc-sdd (Claude Code commands), Kiro (IDE-native), and amux (board-based orchestration with parallel agents). The spec format is tool-agnostic; the orchestration layer is where tools differ.
