What Is Agentic Engineering?

Andrej Karpathy drew the line in early 2026: vibe coding is prompting an AI and accepting what comes back. Agentic engineering is the discipline of decomposing software work into tasks that AI agents execute in parallel, under your architectural supervision.

The difference isn't the AI model — it's the developer's role. A vibe coder sits with one agent and iterates interactively. An agentic engineer launches ten agents with clear specs and reviews completed work. One is pair programming. The other is running a team.

This guide is the practical version. No theory about multi-agent frameworks. No LangGraph diagrams. Just the workflow that works today, with tools you can install in 30 seconds.

The core loop
  1. Decompose — break the work into independent, well-scoped tasks
  2. Spec — write a short spec for each task (goal, requirements, constraints)
  3. Isolate — one git worktree per agent, branched from main
  4. Launch — start agents in parallel, each with its spec
  5. Monitor — watch status from a dashboard, intervene only when stuck
  6. Review & merge — review against acceptance criteria, merge sequentially

Vibe Coding vs Agentic Engineering

Both use the same AI models. The difference is operational discipline.

Vibe coding is fine for exploration, prototyping, and tasks where you need real-time feedback. Agentic engineering is for production work at scale — features, tests, refactors, migrations. Most developers use both throughout the day.

Task Decomposition: The Core Skill

The single most important skill in agentic engineering isn't prompting — it's decomposition. A well-decomposed task set runs cleanly in parallel. A poorly decomposed one creates merge conflicts, duplicated work, and wasted tokens.

The independence test

For each pair of tasks, ask: can agent A finish without knowing what agent B did? If the answer is no, the tasks aren't independent. Either restructure them or sequence them.

Good decomposition examples

# Feature: User notifications system
# Total: 8 tasks, 8 agents, ~2 hours wall clock

Task 1: Database schema + migration (notifications table)
Task 2: REST API — POST /notifications, GET /notifications
Task 3: REST API — PATCH /notifications/:id/read, DELETE
Task 4: WebSocket real-time push (subscribe to user channel)
Task 5: Email notification sender (background job)
Task 6: React notification bell component + dropdown
Task 7: React notification preferences page
Task 8: Tests for all API endpoints

Why this works: Tasks 1 is a prerequisite, but 2–8 are independent once the schema exists. Launch Task 1 first, then 2–8 in parallel.

Bad decomposition

# Don't do this:
Task 1: Build the backend
Task 2: Build the frontend
Task 3: Write the tests

# Why: Too broad. "Build the backend" touches dozens of files.
# The agent will make architectural decisions you didn't intend.
# And Task 3 can't start until 1 and 2 are done.

Rules of thumb

  • 15–45 minutes per task. Under 15 minutes, the overhead of a separate agent isn't worth it. Over 45, the task is probably decomposable further.
  • Different files. If two tasks touch the same file, one of them will create a merge conflict. Restructure.
  • Explicit interfaces. When tasks must interact (e.g., API endpoint + frontend that calls it), define the interface (URL, request/response shape) in the spec. Both agents code to the contract.
  • One agent, one branch. Never have two agents commit to the same branch.

Git Isolation: One Worktree Per Agent

Git worktrees are the secret weapon of agentic engineering. Each agent gets its own complete checkout of the repo, on its own branch, in its own directory. They share the same .git database but never interfere with each other's working tree.

# Create a worktree for each agent task
$ git worktree add ../myrepo-notif-schema  -b feat/notif-schema
$ git worktree add ../myrepo-notif-api     -b feat/notif-api
$ git worktree add ../myrepo-notif-ws      -b feat/notif-ws
$ git worktree add ../myrepo-notif-email   -b feat/notif-email
$ git worktree add ../myrepo-notif-ui      -b feat/notif-ui
$ git worktree add ../myrepo-notif-prefs   -b feat/notif-prefs
$ git worktree add ../myrepo-notif-tests   -b feat/notif-tests

Each agent session points at its own worktree directory. No locks, no conflicts, no coordination overhead during implementation. Conflicts only surface at merge time — where they're cheap to resolve because each branch is small and well-scoped.

Full Walkthrough: 8 Agents, One Feature

Let's walk through building a notification system with 8 parallel agents. This is a real workflow, not a toy example.

Step 1: Write the master spec (10 minutes)

# notification-system.md

## Goal
Add a notification system. Users receive in-app and email
notifications for: new comments on their posts, mentions,
and follower activity.

## Architecture Decisions (already made)
- PostgreSQL table: notifications (id, user_id, type, payload jsonb,
  read boolean, created_at)
- REST API under /api/notifications
- WebSocket channel per user for real-time push
- Background job for email (not inline)
- React component in the existing header

## Constraints
- No new npm dependencies (use existing socket.io)
- No changes to the existing auth middleware
- Email sends via the existing Mailgun wrapper in lib/email.ts

Step 2: Create task specs (15 minutes)

Split the master spec into individual task files. Each references the master for context but has its own acceptance criteria.

# task-2-api-read.md

## Goal
Implement GET /api/notifications and POST /api/notifications

## Requirements
- GET returns paginated list (cursor-based, 20 per page)
- Filter by ?unread=true
- POST creates a notification (used by internal services)
- Auth required (existing middleware)

## Acceptance Criteria
- curl -H "Authorization: Bearer $TOKEN" /api/notifications
  returns { data: [...], cursor: "..." }
- POST with { user_id, type, payload } returns 201
- Unauthorized requests return 401

Step 3: Push to the board and launch

# Push task specs to amux board
$ amux board add "Notif: schema + migration" --desc "See task-1-schema.md"
$ amux board add "Notif: GET/POST API" --desc "See task-2-api-read.md"
$ amux board add "Notif: PATCH/DELETE API" --desc "See task-3-api-write.md"
$ amux board add "Notif: WebSocket push" --desc "See task-4-ws.md"
$ amux board add "Notif: email sender" --desc "See task-5-email.md"
$ amux board add "Notif: bell component" --desc "See task-6-ui-bell.md"
$ amux board add "Notif: preferences page" --desc "See task-7-ui-prefs.md"
$ amux board add "Notif: API tests" --desc "See task-8-tests.md"

# Register sessions and launch
$ amux register notif-schema --dir ~/Dev/api --yolo
$ amux start notif-schema "Read task-1-schema.md and implement it"

# Once schema is done, launch the rest in parallel
$ for task in api-read api-write ws email ui-bell ui-prefs tests; do
>   amux register notif-$task --dir ~/Dev/api --yolo
>   amux start notif-$task "Read task spec and implement"
> done

Step 4: Monitor from the dashboard

Open https://localhost:8822. You'll see 8 session cards, each showing live status (WORKING/IDLE), token spend, current task description, and a peek at the terminal output. Most will finish in 15–30 minutes. The watchdog handles any context overflows or crashes.

Step 5: Review and merge

Each agent creates a commit on its branch. Review each diff against the task spec's acceptance criteria. Merge sequentially:

$ git merge feat/notif-schema      # prerequisite — merge first
$ git merge feat/notif-api-read
$ git merge feat/notif-api-write
$ git merge feat/notif-ws
$ git merge feat/notif-email
$ git merge feat/notif-ui-bell      # may need minor conflict resolution
$ git merge feat/notif-ui-prefs
$ git merge feat/notif-tests        # may need import fixes after merges

Wall-clock time: ~2 hours from spec to merged feature. A single developer writing this by hand: 2–3 days. A single agent doing it sequentially: 6–8 hours.

Agent Coordination Patterns

Not all tasks are embarrassingly parallel. Here are the three patterns that cover 90% of real work:

Pattern 1: Fan-out (most common)

  [spec]
    |
    v
[prerequisite task]
    |
    +----+----+----+----+
    v    v    v    v    v
  [T2] [T3] [T4] [T5] [T6]
    |    |    |    |    |
    v    v    v    v    v
       [merge all]

One prerequisite (schema, config, interface definition), then N independent tasks. The prerequisite runs alone; everything else runs in parallel after it completes.

Pattern 2: Pipeline

[T1: API] --> [T2: integration tests] --> [T3: docs]

Sequential dependency chain. Less parallelism, but each stage can still be decomposed. For example, T1 might be 5 parallel API endpoint agents, all of which complete before T2 launches.

Pattern 3: Map-reduce

[T1: refactor module A] \
[T2: refactor module B]  >--> [T-final: integration test suite]
[T3: refactor module C] /

N independent agents produce output, then a final agent validates the combined result. Great for large refactors, test generation across modules, and documentation.

Running Agents Overnight

The real productivity gain isn't running 8 agents for 2 hours while you watch. It's queuing 20 tasks before bed and waking up to completed branches.

What breaks overnight

  • Context overflow. Long-running agents hit the context window limit and crash. Solution: a watchdog that auto-compacts context and restarts the session.
  • Permission prompts. Claude Code asks for confirmation before destructive operations. Solution: YOLO mode auto-approves safe operations.
  • Thinking-block corruption. Rare but real — malformed thinking blocks kill the session. Solution: watchdog detects the error pattern and restarts with the last message.
  • Rate limits. Too many concurrent agents can hit API rate limits. Solution: stagger launches by 30–60 seconds.

The overnight checklist

# 1. Queue tasks on the board
$ amux board add "Refactor auth module" --desc "specs/auth-refactor.md"
$ amux board add "Write tests for billing" --desc "specs/billing-tests.md"
$ amux board add "Update API docs" --desc "specs/api-docs.md"
# ... repeat for all tasks

# 2. Register sessions with YOLO mode
$ amux register auth-refactor --dir ~/Dev/api --yolo
$ amux register billing-tests --dir ~/Dev/api --yolo
$ amux register api-docs --dir ~/Dev/api --yolo

# 3. Launch with staggered starts
$ amux start auth-refactor "Claim and implement your board task"
$ sleep 30 && amux start billing-tests "Claim and implement your board task"
$ sleep 30 && amux start api-docs "Claim and implement your board task"

# 4. The watchdog runs automatically — go to sleep

Reviewing Agent Output at Scale

Reviewing 8 branches is different from reviewing 1 PR. Here's the process that works:

  1. Triage first. Skim each branch's diff size. Branches with unexpectedly large diffs (2× what you'd expect) probably went off-scope. Review those first.
  2. Check acceptance criteria. Each task spec has explicit criteria. Run them. If they pass, the implementation is correct by definition — even if you'd have written it differently.
  3. Don't rewrite working code. Agentic engineering output won't match your personal style. That's fine. If it's correct, tested, and readable, merge it. Style consistency matters less than shipping.
  4. Merge sequentially. Don't merge all 8 branches at once. Go one at a time, run tests after each merge, fix conflicts before proceeding.
  5. Log what went wrong. If a task produced bad output, the spec was incomplete. Fix the spec pattern for next time, not the prompt.

Mistakes Everyone Makes (And How to Avoid Them)

1. Tasks that are too broad

"Build the backend for feature X" is not a task — it's a project. The agent will make architectural decisions you didn't intend, touch files you didn't expect, and produce output that's hard to review. Break it down until each task takes 15–45 minutes.

2. Overlapping file ownership

Two agents editing the same file creates merge conflicts and wasted work. If two tasks must touch the same file (common with routes.ts, index.ts, or config files), either combine them into one task or make one a prerequisite of the other.

3. Missing constraints

Specs without constraints produce technically correct but architecturally wrong output. "No new dependencies", "use the existing auth middleware", "don't modify the database schema" — these constraints save more rework than any amount of detailed requirements.

4. Micromanaging running agents

The temptation to peek at every agent's output and send corrections mid-task is strong. Resist it. If the spec is good, let the agent finish. If you're constantly correcting, the spec wasn't good enough — fix the spec next time, not the agent mid-flight.

5. Skipping the prerequisite step

Launching 8 agents in parallel when Task 1 (schema, interface, config) isn't done yet means 7 agents will guess at the interface. They'll guess differently. You'll throw away all their work. Always run prerequisites to completion before launching dependent tasks.

FAQ

Q What is agentic engineering?
Agentic engineering is the practice of decomposing software work into tasks that AI coding agents execute in parallel with minimal human supervision. Popularized by Andrej Karpathy in early 2026, it distinguishes the engineering discipline of managing agent fleets from casual "vibe coding." The developer's role shifts from writing code to writing specs, decomposing work, and reviewing output.
Q How is agentic engineering different from vibe coding?
Vibe coding is interactive and single-threaded: one developer, one agent, real-time prompting. Agentic engineering is parallel and asynchronous: one developer, many agents, upfront specs, and batch review. The throughput difference is 5–10× for well-decomposed tasks.
Q How many agents can I run in parallel?
The practical limit is 20–30 on a modern Mac or Linux machine with 16GB+ RAM. The bottleneck is API rate limits, not local resources. Most developers start with 3–5 and scale up as they improve at task decomposition. More agents with bad specs just means more rework.
Q What kinds of tasks work best with parallel agents?
Independent, well-scoped tasks: separate API endpoints, tests for different modules, features that touch different files, documentation for different components, refactoring across isolated packages. Tasks that don't parallelize well: schema migrations, global refactors, and tightly coupled features where one task's output is another's input.
Q Is agentic engineering only for large teams?
The opposite — it's most transformative for solo developers and small teams. A single developer running 10 agents can match the throughput of a 5-person team. The 10× leverage is felt most by individuals who were previously bottlenecked on their own time.
Q How do I prevent merge conflicts?
Three techniques: (1) scope tasks so they touch different files, (2) use git worktrees so each agent has its own checkout, and (3) merge branches one at a time, resolving conflicts before proceeding to the next. Good decomposition eliminates 90% of conflicts before they happen.
Q What tools do I need?
At minimum: an AI coding agent (Claude Code) and git worktrees. For scale beyond 3–4 agents, you need orchestration — a dashboard, task board, and crash recovery. amux provides all of this as a single Python file: session management, kanban board, REST API, self-healing watchdog, and mobile access.
Q What does a typical day look like?
Morning: review overnight output, merge completed branches, close board items. Midday: decompose the next batch of work, write specs, launch agents. Evening: queue overnight tasks (tests, docs, refactoring), let the watchdog handle crashes. The developer's time shifts from writing code to writing specs and making architectural decisions.
