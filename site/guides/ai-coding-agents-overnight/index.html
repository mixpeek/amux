The Overnight Agent Promise (and Reality)

The dream is simple: queue 20 tasks before bed, and wake up to 20 reviewed pull requests. No more choosing between sleep and shipping.

A year ago, this was fantasy. AI coding agents would crash from context overflow at 3am, silently loop on the wrong approach for hours, or produce code that looked right but broke in subtle ways. Running agents unattended was a gamble.

That has changed. Modern AI models (Claude Opus 4, Sonnet 4) are dramatically more reliable at long-running tasks. And tooling has caught up -- self-healing watchdogs, atomic task boards, and fleet monitoring mean you can set real guardrails instead of crossing your fingers.

This guide is the missing manual. Not the pitch -- the playbook. You will learn exactly how to decompose work, configure agents, set safety boundaries, and build a morning review workflow that makes overnight coding practical and safe.

Who this is for: Developers who are already using AI coding agents (Claude Code, Codex CLI, Gemini CLI) and want to run them unattended at scale. If you have never used an AI coding agent, start with Getting Started first.

Architecture Overview

A reliable overnight agent setup has four layers:

Task Board          Agent Orchestrator       Isolated Worktrees       Output
 ┌─────────┐        ┌──────────────┐          ┌──────────┐          ┌──────┐
 │ TASK-1   │───────▶│  Agent 1     │─────────▶│ branch-1 │─────────▶│ PR 1 │
 │ TASK-2   │───────▶│  Agent 2     │─────────▶│ branch-2 │─────────▶│ PR 2 │
 │ TASK-3   │───────▶│  Agent 3     │─────────▶│ branch-3 │─────────▶│ PR 3 │
 │  ...     │        │  Watchdog    │          │  ...     │          │ ...  │
 └─────────┘        └──────────────┘          └──────────┘          └──────┘
   SQLite              tmux + amux              git worktrees          GitHub

Task Board -- a shared queue (amux's SQLite-backed board, GitHub Issues, or Linear) where each task is atomic and independently verifiable.

Agent Orchestrator -- manages agent lifecycle, restarts crashed sessions, and prevents duplicate work. This is where amux lives -- it wraps tmux with a watchdog, REST API, and web dashboard.

Isolated Worktrees -- each agent works on its own git branch (ideally a git worktree) so agents never conflict with each other.

Output -- each agent produces a PR, a commit, or a well-defined artifact that you review in the morning.

Why tmux alone is not enough

You can open 10 tmux panes and start 10 Claude Code sessions. But without an orchestration layer:

  • If one crashes at 2am, nobody restarts it
  • Two agents might claim the same task
  • You have no idea which agents are still working vs. stuck
  • There is no spending visibility until you check billing the next day
  • You cannot check status from your phone

An orchestrator like amux solves all of these. The rest of this guide assumes you have one set up.

Task Decomposition: The Most Important Step

The difference between waking up to 20 clean PRs and waking up to 20 confused agents is entirely in how you define the tasks. This section is worth more than all the tooling combined.

The sleeping test

Before queuing a task, ask: "Can I verify this task's output in under 2 minutes in the morning?" If the answer is no, the task is too vague or too large.

Good vs. bad overnight tasks

Good (atomic, verifiable) Bad (vague, unbounded)
"Add unit tests for auth/login.py -- target 90% coverage" "Improve test coverage"
"Migrate UserService from JavaScript to TypeScript" "Convert the project to TypeScript"
"Fix bug #142: login form submits twice on slow connections" "Fix the bugs in the login flow"
"Update axios from v0.21 to v1.6 and fix breaking changes" "Update all dependencies"
"Write API docs for the /api/users endpoints in OpenAPI format" "Document the API"
"Add input validation to the POST /api/orders endpoint" "Make the API more robust"

The task spec template

For overnight tasks, write more context than you would for a live session. The agent cannot ask clarifying questions at 3am.

## Task: Add rate limiting to the public API

### Context
The /api/v2/* endpoints are public and currently have no rate limiting.
We've seen abuse from scrapers hitting /api/v2/search at 100+ req/sec.

### Requirements
- Add rate limiting middleware to all /api/v2/* routes
- Limit: 60 requests per minute per IP
- Return 429 with Retry-After header when exceeded
- Use in-memory store (no Redis dependency)
- Add tests for the rate limiter

### Files to modify
- src/middleware/rateLimit.ts (create)
- src/routes/v2/index.ts (add middleware)
- tests/middleware/rateLimit.test.ts (create)

### Do NOT
- Modify any v1 routes
- Add external dependencies
- Change the response format of existing endpoints

Using the board for task queuing

With amux, queue your overnight tasks on the board before starting agents:

# Queue tasks for the night
amux board add "Add unit tests for auth/login.py -- target 90% coverage"
amux board add "Migrate UserService from JS to TS"
amux board add "Fix bug #142: double-submit on slow connections"
amux board add "Update axios v0.21 -> v1.6, fix breaking changes"
amux board add "Write OpenAPI docs for /api/users endpoints"

# Each agent claims tasks atomically -- no duplicates
# POST /api/board/TASK-1/claim uses SQLite CAS

The atomic claiming API means two agents will never work on the same task, even if 20 agents all start polling the board simultaneously.

Setting Up Your Agent Fleet

Install and configure

# Install amux (one command)
git clone https://github.com/mixpeek/amux && cd amux && ./install.sh

# Register sessions -- one per task area
amux register auth-tests --dir ~/Dev/myapp --yolo
amux register ts-migration --dir ~/Dev/myapp --yolo
amux register bug-142 --dir ~/Dev/myapp --yolo
amux register axios-upgrade --dir ~/Dev/myapp --yolo
amux register api-docs --dir ~/Dev/myapp --yolo

Isolate with git branches

Each agent should work on its own branch. If two agents modify the same files on the same branch, you will wake up to merge conflicts instead of PRs.

# Option 1: amux detects shared branches and warns you
# The dashboard shows a conflict indicator when two sessions share dir + branch

# Option 2: Set up branches before starting
cd ~/Dev/myapp
git worktree add ../myapp-auth-tests -b overnight/auth-tests
git worktree add ../myapp-ts-migration -b overnight/ts-migration
git worktree add ../myapp-bug-142 -b overnight/bug-142

# Then register each session with its own worktree directory
amux register auth-tests --dir ~/Dev/myapp-auth-tests --yolo
amux register ts-migration --dir ~/Dev/myapp-ts-migration --yolo
amux register bug-142 --dir ~/Dev/myapp-bug-142 --yolo

How many agents to run

Environment Comfortable limit Bottleneck
Laptop (16GB RAM) 5-7 agents RAM, API rate limits
Desktop (32GB RAM) 10-15 agents API rate limits
Cloud VM (4 cores, 16GB) 20-30 agents API rate limits
Cloud VM (8+ cores, 32GB) 50+ agents API rate limits, cost

The bottleneck is almost never local compute -- it is API rate limits. Stagger agent starts by 10-30 seconds to avoid hitting rate limits simultaneously:

# Staggered start
for session in auth-tests ts-migration bug-142 axios-upgrade api-docs; do
  amux start "$session"
  sleep 15
done

Model selection per task

Not every task needs the most expensive model. Assign models based on task complexity:

Task type Recommended model Why
Writing tests, docs, boilerplate Sonnet Fast, cheap, reliable for structured output
Bug fixes, small features Sonnet or Opus Depends on codebase complexity
Complex refactors, migrations Opus Needs deep reasoning across many files
Architecture decisions, design Opus Needs judgment, not just execution

Guardrails: Safety While You Sleep

Guardrails are what separate "agents running overnight" from "agents running amok overnight." Set these up before you start your first overnight run.

CLAUDE.md as the constitution

Every agent reads CLAUDE.md in the project root. For overnight runs, make it stricter than usual:

# CLAUDE.md (overnight-hardened)

## Rules
- Run `npm test` before committing. Do NOT commit if tests fail.
- Run `npm run lint` before committing. Fix all lint errors.
- Never modify files in src/auth/, src/payments/, or infrastructure/
- Never add new dependencies without explicit approval
- Never modify .env, docker-compose.yml, or CI/CD configs
- Create a new git branch for your work: overnight/{task-name}
- Commit frequently with descriptive messages
- If you are unsure about an approach, write your reasoning in
  a comment and move on. Do not block on ambiguity.

## Code standards
- TypeScript strict mode. No `any` types.
- All public functions must have JSDoc comments.
- No console.log in production code (use the logger).

Git hooks as safety nets

Pre-commit hooks run automatically, even at 3am:

# .git/hooks/pre-commit
#!/bin/sh
npm run lint --quiet || exit 1
npm test --quiet || exit 1

If the tests fail, the commit is blocked. The agent will see the failure and attempt to fix it, rather than pushing broken code.

File-level restrictions

Use .claude/settings.json or CLAUDE.md to restrict which files agents can modify. For overnight runs, be explicit about the boundary:

# In the task prompt, be explicit:
"Only modify files in src/services/user/ and tests/services/user/.
Do NOT touch any other directories."

Cost caps

Set spending limits so a runaway agent does not burn through your budget:

  • Claude Max plan ($200/month): Unlimited usage -- no cost cap needed, but watch for rate-limited agents spinning on retries
  • API pricing: Set per-session token limits. amux's token tracking dashboard shows per-session spend in real time
  • Anthropic Console: Set workspace-level spending limits as a hard backstop

The blast radius principle

Each agent's potential damage should be contained. The worst case for any single agent should be "I have to delete one branch" -- never "I have to restore the whole repo."

  • Separate git branches (or worktrees) per agent
  • No force-push permissions on main
  • No deployment triggers from agent branches
  • No access to production databases or secrets

Monitoring & Observability

Before you go to sleep

After starting all agents, spend 5-10 minutes verifying they have started correctly:

# Check all agents are running
amux ls

# Peek at each agent's output to confirm they understood the task
amux peek auth-tests
amux peek ts-migration

# Or open the dashboard on your phone
amux serve  # https://localhost:8822 (or your Tailscale address)

The watchdog

amux's self-healing watchdog runs automatically and handles the three most common overnight failures:

  1. Context overflow: Watchdog detects context nearing the limit and triggers /compact automatically
  2. Crash/corruption: Detects thinking-block corruption, restarts the session, and replays the last message
  3. Stuck agent: Detects tool-approval prompts (the "Esc to cancel" marker) and auto-responds, keeping the agent moving

No configuration needed -- the watchdog is always on. It is what makes overnight runs practical instead of theoretical.

Mobile monitoring

Install the amux PWA on your phone. If you wake up at 3am and want to check status without opening your laptop:

  • See which agents are running, idle, or stopped
  • Peek at any agent's latest output
  • Send a quick course correction if needed
  • Check token spend across all sessions

For remote access, use Tailscale -- amux auto-provisions TLS certs.

The Morning Review Workflow

You have 8 hours of agent output to review. Here is how to do it efficiently without spending all morning on it.

Step 1: Triage (2 minutes)

Open the amux dashboard. Glance at status indicators:

  • Green (idle) -- agent finished and is waiting. This is the ideal state.
  • Amber (working) -- still running. Check if it is making progress or stuck.
  • Red (stopped) -- crashed and the watchdog could not recover. Review logs.

Step 2: Quick-accept simple tasks (10 minutes)

Tasks like "add tests" or "write docs" are quick to verify:

# For each completed agent, check the diff
git diff main..overnight/auth-tests -- '*.test.ts'

# Run the tests locally
cd ~/Dev/myapp-auth-tests && npm test

# If tests pass and the diff looks clean, merge
git checkout main && git merge overnight/auth-tests

Step 3: Deep-review complex tasks (20-30 minutes)

Migrations, refactors, and bug fixes need more attention. For these:

  1. Read the agent's full conversation (peek scrollback) to understand its reasoning
  2. Review the diff carefully, focusing on edge cases
  3. Run the full test suite, not just the new tests
  4. If the agent's approach was wrong, note it for next time -- adjust your CLAUDE.md or task specs

Step 4: Clean up (5 minutes)

# Delete merged worktrees
git worktree remove ../myapp-auth-tests
git worktree remove ../myapp-ts-migration

# Clear completed tasks from the board
amux board done TASK-1
amux board done TASK-2

# Stop all overnight agents
amux stop auth-tests ts-migration bug-142 axios-upgrade api-docs

Total morning review time: 30-45 minutes to process an entire night's worth of coding output. Compare that to the 8-12 developer-hours the agents replaced.

Cost Management at Scale

The math

Pricing model 10 agents x 8 hours Notes
Claude Max ($200/mo) $0 marginal Flat rate. Best for heavy overnight users.
API (Sonnet only) $40-160 ~$0.50-2/agent-hour depending on task verbosity
API (mixed Opus + Sonnet) $80-300 Use Opus only for complex tasks

Optimization strategies

  • Route by complexity: Use Sonnet for test writing, docs, and boilerplate. Reserve Opus for refactors and bug fixes.
  • Prompt caching: Agents that reference the same codebase context benefit from Anthropic's prompt caching -- subsequent turns are cheaper.
  • Concise task specs: A 200-word task spec costs less in input tokens than a 2000-word brain dump.
  • CLAUDE.md size: Keep it focused. A 50-line CLAUDE.md is better than a 500-line one -- every agent reads it on every turn.
  • Set hard limits: Use the Anthropic Console spending limits as a hard backstop.

ROI calculation

A developer-hour costs $75-200 (fully loaded). If 10 overnight agents each complete the equivalent of 1-2 hours of developer work, that is 10-20 developer-hours saved per night. At even conservative estimates:

  • Cost: $0-200/night (depending on pricing model)
  • Value: $750-4,000 in developer-hours
  • ROI: 4-20x on API pricing. Infinite on Max plan.

5 Real-World Overnight Workflows

1. Test suite expansion

One of the highest-value overnight workflows. Agents are excellent at writing tests because the expected behavior is already defined by the code.

# Queue one task per module
amux board add "Write unit tests for src/services/auth/ -- target 90% coverage"
amux board add "Write unit tests for src/services/billing/ -- target 90% coverage"
amux board add "Write integration tests for src/api/v2/ endpoints"
amux board add "Write e2e tests for the checkout flow"

Expected output: 4 PRs, each adding a comprehensive test file. Morning review: run the tests, check coverage reports, merge.

2. Dependency upgrades across a monorepo

Upgrading dependencies is tedious, well-scoped, and perfectly suited for agents.

amux board add "Upgrade React from v18 to v19 in packages/web -- fix breaking changes"
amux board add "Upgrade Express from v4 to v5 in packages/api -- fix breaking changes"
amux board add "Upgrade TypeScript from 5.3 to 5.6 in all packages -- fix type errors"
amux board add "Upgrade Jest from v29 to v30 across the monorepo"

Expected output: 4 PRs, each upgrading one dependency with all breaking changes fixed and tests passing.

3. Documentation generation

Agents can write docs faster than most humans, and they actually read the source code.

amux board add "Generate OpenAPI spec for all /api/v2 endpoints"
amux board add "Write README.md for the packages/sdk package"
amux board add "Add JSDoc to all exported functions in src/utils/"
amux board add "Create architecture decision record for the new caching layer"

4. Bug fix queue

Pull bugs from your issue tracker and queue them as overnight tasks. Works best for well-reported bugs with reproduction steps.

amux board add "Fix #142: Login form double-submits on slow connections"
amux board add "Fix #155: CSV export truncates Unicode characters"
amux board add "Fix #161: Dashboard chart shows wrong timezone"
amux board add "Fix #167: Email notification sends HTML as plain text"

Morning review: Check each fix against the bug report. Run repro steps. If it works, merge.

5. Code migration (JavaScript to TypeScript)

Large migrations are the killer app for overnight agents. Break the migration into one file or module per agent.

amux board add "Convert src/services/user.js to TypeScript with strict types"
amux board add "Convert src/services/order.js to TypeScript with strict types"
amux board add "Convert src/services/inventory.js to TypeScript with strict types"
amux board add "Convert src/middleware/*.js to TypeScript with strict types"
amux board add "Convert src/utils/*.js to TypeScript with strict types"

Expected output: 5 PRs, each converting one module. The key is that each agent works on a separate set of files, so there are no merge conflicts.

Failure Modes & Troubleshooting

Agent loops

Symptom: Agent keeps trying the same approach, failing, reverting, and retrying.

Cause: Task is too ambiguous, or the agent does not have enough context to find a different approach.

Fix: More specific task specs. Add a "if X does not work, try Y" fallback in the prompt. Set a max-iteration limit in CLAUDE.md: "If you have tried 3 approaches and none work, commit what you have with a TODO comment and stop."

Context window exhaustion

Symptom: Agent slows down, loses track of earlier work, or crashes with a context error.

Cause: Long-running task filled the context window.

Fix: amux's watchdog auto-compacts before this happens. For very long tasks, break them into smaller sub-tasks. Tell the agent to commit frequently: "Commit after each completed function."

Rate limit cascades

Symptom: All agents stop responding simultaneously.

Cause: Too many agents hit the API at the same moment (common at startup).

Fix: Stagger agent starts by 10-30 seconds. The rate limits recover quickly -- agents will resume automatically after the cooldown.

Git conflicts between agents

Symptom: Agent's branch cannot merge because another agent modified the same file.

Cause: Two agents worked on overlapping files on separate branches.

Fix: Use git worktrees. Assign non-overlapping file scopes. amux's dashboard shows conflict warnings when two sessions share the same directory and branch.

The "confidently wrong" problem

Symptom: Agent produces code that passes tests but introduces a subtle bug.

Cause: Insufficient test coverage for edge cases, or the agent wrote tests that validate its own buggy logic.

Fix: Require agents to run existing tests, not just new ones. Review agent-written tests carefully in your morning review -- check that they are testing the right behavior, not just that the code runs.

Always-on agents

On a cloud VM, agents can run 24/7 -- not just overnight. Use amux's built-in cron scheduler to queue tasks at specific times:

# Schedule a nightly test run at 11pm
amux cron add --schedule "0 23 * * *" \
  --session test-runner \
  --command "Run all unit tests and report failures"

Team access

Share the dashboard with your team via Tailscale. Multiple developers can queue tasks on the same board, and agents claim work atomically -- no coordination overhead.

FAQ

Is it safe to run AI coding agents overnight on production codebases?

Yes, with proper guardrails. Use separate git branches per agent, restrict file access via CLAUDE.md, enforce pre-commit hooks with linting and tests, and set cost caps. Never give overnight agents access to deployment pipelines, auth systems, or payment code.

How many AI coding agents can I run overnight?

On a laptop: 5-7 agents comfortably (constrained by RAM and API rate limits). On a cloud VM with 4+ cores and 16GB RAM: 20-50 agents. The bottleneck is usually API rate limits, not local compute. Stagger agent starts by 10-30 seconds to avoid hitting limits simultaneously.

What tasks should I NOT give to overnight AI agents?

Avoid tasks that touch authentication, payment processing, infrastructure/deployment, database migrations on production, or anything requiring human judgment about business logic. Also avoid open-ended tasks like "refactor the codebase" -- agents need specific, bounded scope to work unattended.

How much does it cost to run AI agents overnight?

With Claude Code on the Max plan ($200/month), you get unlimited usage -- running 10 agents for 8 hours costs the same flat rate. On API pricing, expect $0.50-2 per agent-hour for Sonnet, $5-15 per agent-hour for Opus. A typical 8-hour overnight run with 10 agents on mixed models costs $50-100 on API pricing.

What happens if an agent gets stuck or crashes overnight?

With amux's self-healing watchdog, crashed agents are automatically restarted and their last message is replayed. Context overflow triggers auto-compaction. Stuck agents (no output for a configurable timeout) are detected and restarted. You can also set hard time limits per agent.

Can I use this with agents other than Claude Code?

amux works with any CLI agent that runs in a terminal: Claude Code, Codex CLI, Gemini CLI, Aider, and others. The orchestration layer (task board, watchdog, dashboard) is agent-agnostic.
