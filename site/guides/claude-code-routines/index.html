Claude Code Routines vs Self-Hosted Automation

Anthropic launched Routines on April 14. OpenAI shipped Codex background agents on April 16. Both run coding agents in the cloud on a schedule. Both have hard limits. Here’s when to use them, when to self-host, and how to migrate.

What Are Claude Code Routines?

Routines are Anthropic’s built-in scheduled automation for Claude Code, launched April 14, 2026 as a research preview. The idea: define a task, set a trigger, and Claude Code executes it in the cloud without you being present.

A Routine has three parts:

  • A prompt. The task you want Claude to do — “Review all open PRs and post comments,” “Run the test suite and fix any failures,” “Update the changelog from this week’s commits.”
  • A trigger. Cron schedule (“every weekday at 9am”), GitHub webhook (on PR open, on push to main), or manual.
  • Execution environment. Anthropic’s cloud — a fresh checkout of your repo, Claude Code runs the task, results appear in the Routines dashboard or as GitHub comments/PRs.

Routines are included with Max and Enterprise subscriptions at no extra per-run cost. They’re the first official answer to “how do I run Claude Code on a schedule without babysitting a terminal?”

That’s the pitch. The reality is more constrained.

The short version: Routines and Codex are managed, cloud-only, and capped. Self-hosted is unlimited, local, and yours to operate. They solve the same problem at different points on the convenience-vs-control spectrum.

Where Routines Fall Short

Routines are a research preview, and the constraints reflect that. Five limitations matter in practice:

1. Daily run caps

Max subscribers get 5–25 Routine runs per day, depending on subscription tier. That’s fine for a daily PR review. It’s not enough for a CI pipeline that triggers on every push, a test suite that runs on every commit, or an overnight batch of 50 tasks.

For comparison: a self-hosted setup can run hundreds of agent invocations per day, limited only by your API quota or machine resources.

2. No persistent state

Each Routine invocation starts fresh. There is no conversation history, no memory of what the last run did, no accumulated context. If your Routine reviews PRs daily, it cannot reference what it said about yesterday’s PR. If it tracks a refactoring across multiple days, it starts over each time.

This is the single biggest architectural limitation. Many useful automations — incremental refactors, multi-day migration tracking, agents that learn from previous runs — fundamentally require persistent state.

3. No local access

Routines execute on Anthropic’s infrastructure. They can clone your GitHub repo, but they cannot:

  • Access files not in the repo (local configs, .env, generated assets)
  • Connect to localhost services (databases, APIs, Docker containers)
  • Reach services behind your VPN or firewall
  • Use tools installed on your machine (custom linters, proprietary CLIs)

If your automation needs any of these, Routines can’t help. Self-hosted automation runs on the same machine as your development environment, with full access to everything.

4. No inter-agent coordination

Each Routine runs in isolation. There is no mechanism for one Routine to trigger another, for Routines to share a task queue, or for a Routine to message another running agent. If you need agents that coordinate — one finishes the database schema, another starts the API layer — Routines don’t support it.

5. Research preview instability

Routines are explicitly labeled “research preview.” Feature behavior, run caps, and availability may change without notice. Building critical infrastructure on a research preview is a calculated risk — one that self-hosted solutions avoid entirely.

Self-Hosted Alternative: Unlimited Automation

The self-hosted approach runs Claude Code on your own machine using the same headless mode that powers Routines behind the scenes — but without the caps, without the cloud lock-in, and with full local access.

amux provides four automation primitives that cover every Routine use case and more:

Cron scheduler

Named recurring jobs with cron expressions, managed through a web UI or CLI. The direct equivalent of a Routine’s cron trigger, but with no daily cap.

# Review open PRs every weekday at 9am
$ amux cron add pr-review "0 9 * * 1-5" \
    --session reviewer \
    --prompt "Review all open PRs on main and post comments"

# Run tests every hour during business hours
$ amux cron add test-sweep "0 9-17 * * 1-5" \
    --session tester \
    --prompt "Run the full test suite. If anything fails, fix it and push."

# List active cron jobs
$ amux cron list
pr-review    0 9 * * 1-5   reviewer   next: 2026-04-17 09:00
test-sweep   0 9-17 * * 1-5  tester   next: 2026-04-16 14:00

/loop for adaptive scheduling

For tasks that shouldn’t run on a fixed schedule but should repeat until a condition is met — like monitoring a deploy, or retrying a flaky migration:

# Agent self-paces: checks every few minutes until the deploy succeeds
/loop Check if the staging deploy passed. If it did, run smoke tests and report. If not, wait.

The agent decides how long to wait between checks based on context. No cron expression needed.

REST API triggers

Trigger agent runs from any webhook — GitHub, Slack, your CI pipeline, a custom script. The equivalent of Routines’ GitHub webhook trigger, but for any HTTP source.

# Trigger an agent from a GitHub webhook
$ curl -sk -X POST -H 'Content-Type: application/json' \
    -d '{"text":"A new PR was opened: #142 - Add retry logic"}' \
    https://localhost:8822/api/sessions/reviewer/send

Board-driven orchestration

For complex workflows where multiple agents need to coordinate: post tasks to the kanban board, agents claim and execute them atomically. No duplicate work, no manual assignment.

# Post a batch of tasks
$ for task in "Update auth module" "Refactor config" "Add API tests"; do
    curl -sk -X POST -H 'Content-Type: application/json' \
      -d "{\"title\":\"$task\",\"status\":\"todo\"}" \
      https://localhost:8822/api/board
  done

# Agents claim tasks automatically via atomic /claim endpoint

All four primitives share the same persistent session infrastructure: agents maintain full conversation history across runs, cost is tracked per-session, and crashes auto-recover via the self-healing watchdog.

When to Use What

There’s no universally better option. The right choice depends on what you need:

Use CaseBest OptionWhy
Daily PR review for a small team Routines Simple, low-frequency, cloud repo access is sufficient
Weekly dependency update + PR Routines Fits within daily cap, stateless is fine
CI gate on every push Self-hosted High frequency blows through Routine caps
Multi-day refactoring across modules Self-hosted Requires persistent state between runs
Automation that reads local databases Self-hosted Routines can’t reach localhost
10+ agents working overnight Self-hosted Need orchestration, cost tracking, fleet observability
Quick ad-hoc cloud task Codex No setup, fire-and-forget, GitHub integration
Team with no DevOps capacity Routines + Codex Zero infrastructure to maintain
Full control over execution Self-hosted Your machine, your rules, your data

The pragmatic approach: start with Routines for the simple stuff. When you hit a cap — daily runs, persistent state, local access, or coordination — migrate that specific workflow to self-hosted. The two aren’t mutually exclusive.

Migration: From Routines to Self-Hosted

Moving a Routine to amux takes about five minutes. The prompt stays the same — only the scheduling and execution layer changes.

Step 1: Install amux

$ git clone https://github.com/mixpeek/amux && cd amux && ./install.sh

Step 2: Register your project

$ amux register myproject --dir ~/Dev/myproject --yolo

The --yolo flag enables auto-approval mode for headless operation — the same permission model Routines use implicitly in the cloud.

Step 3: Create the cron job

Take your Routine’s prompt and schedule, and create an amux cron entry:

# Your Routine was: "Review open PRs" at 9am weekdays
$ amux cron add pr-review "0 9 * * 1-5" \
    --session pr-reviewer \
    --prompt "Review all open PRs on the main branch. For each PR, post a review comment covering: correctness, test coverage, and potential issues. Approve PRs that look good."

Step 4: Start the dashboard

$ amux serve
Dashboard: https://localhost:8822

You now have a web dashboard showing the session status, cost tracking, and cron schedule. The same PR review runs on the same schedule, but on your machine, with access to your full environment, and no daily cap.

Step 5: Add persistent state (optional)

The key advantage of self-hosted: your agent maintains conversation history. If yesterday’s PR review identified a pattern (“this team keeps forgetting error handling”), today’s review can reference that context and provide more targeted feedback.

# The session persists across cron runs
$ amux cron add pr-review "0 9 * * 1-5" \
    --session pr-reviewer \
    --resume \
    --prompt "Review new open PRs since your last run. Reference patterns from previous reviews."

Cost Comparison

The cost model differs significantly between approaches. Here’s a realistic breakdown for a team running 10 automated tasks per day:

ApproachMonthly Cost (10 tasks/day)Billing ModelCost Visibility
Claude Code Routines $0 beyond Max subscription ($100–$200/mo) Included in subscription Run count only
OpenAI Codex ~$150–$500/mo (depends on task complexity) Per-token API pricing API dashboard
Self-hosted (Max sub) $0 beyond Max subscription ($100–$200/mo) Included in subscription Per-session + per-day dashboards
Self-hosted (API key) ~$100–$400/mo (depends on task complexity) Per-token Per-session + per-day dashboards

If you’re already paying for a Max subscription and your tasks fit within the daily cap, Routines are effectively free. The economics shift when:

  • You exceed the daily cap and need to add API-key runs to supplement.
  • You need cost visibility per task. Routines show run counts, not token spend. Self-hosted with amux shows per-session cost, per-day totals, and model breakdowns.
  • You’re running a team. Multiple Max subscriptions at $200/mo each adds up. A single API key shared across a self-hosted fleet with budget caps is often cheaper.

FAQ

What are Claude Code Routines?

Anthropic’s built-in scheduled automation for Claude Code, launched April 14, 2026 as a research preview. Define a task, set a cron schedule or GitHub webhook trigger, and Claude runs it in the cloud without you present. Included in Max and Enterprise subscriptions.

What are the limits of Routines?

5–25 daily runs (tier-dependent), no persistent state between runs, cloud-only execution (no local file/DB access), no inter-agent coordination, and research-preview stability. These constraints matter most for high-frequency, stateful, or local-access workflows.

How do Routines compare to OpenAI Codex?

Both run coding agents in the cloud. Codex focuses on on-demand tasks in sandboxed VMs at API pricing. Routines focus on scheduled automation included in the subscription. Both are cloud-only and stateless. Neither supports inter-agent coordination.

Can I run Routines on my own server?

No. Routines execute on Anthropic’s infrastructure. For local execution, use Claude Code in headless mode with a self-hosted scheduler like amux’s cron or systemd timers.

Should I use Routines or self-hosted?

Routines for simple, low-frequency tasks that don’t need local access. Self-hosted when you need more than 25 runs/day, persistent state, local file/DB access, inter-agent coordination, or full environment control. They’re not mutually exclusive.

How do I migrate from Routines to self-hosted?

Install amux, register your project, create a cron job with the same schedule and prompt, start the dashboard. About five minutes. The prompt stays identical — only the execution layer changes.

Can amux replace Routines entirely?

For the automation use case, yes. amux’s cron scheduler, /loop, board-driven orchestration, and REST API triggers cover everything Routines does plus persistent state, unlimited runs, and local execution. The trade-off: you host it yourself.

What does Codex cost?

Codex uses the codex-mini model at OpenAI’s API pricing. Cost depends on task complexity and token usage. There’s no flat-rate subscription option — every run is billed per-token.
