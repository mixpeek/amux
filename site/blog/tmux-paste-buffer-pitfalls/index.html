tmux paste-buffer Silently Eats Your Newlines (And How We Fixed It)

Two bugs that only manifest under concurrency, and a barely-documented flag that fixes everything.

amux orchestrates dozens of Claude Code sessions running inside tmux panes. Users type prompts in a web dashboard, and amux sends them to the right pane programmatically. This sounds simple. It isn't.

The setup: send-keys works until it doesn't

For short text, tmux send-keys -l works perfectly:

tmux send-keys -t mysession -l "write tests for auth.py"
tmux send-keys -t mysession Enter

The -l flag sends the text literally (no key name interpretation), and then we send Enter separately. Clean and simple.

The problem: send-keys has an undocumented buffer limit around 500 characters. Anything longer gets silently truncated. No error, no warning — the tail of your text just vanishes. For AI agent prompts that include context, code snippets, or detailed instructions, 500 characters is nothing.

The obvious fix: load-buffer + paste-buffer

tmux has a paste buffer system designed for exactly this. Write your text to a temp file, load it into tmux's buffer, then paste it into the target pane:

# Write text to temp file
echo "$LONG_PROMPT" > /tmp/send.txt

# Load into tmux buffer, paste into pane
tmux load-buffer /tmp/send.txt
tmux paste-buffer -t mysession

This bypasses the send-keys limit entirely. Ship it, right?

Bug 1: Newlines become Enter keypresses

Here's what we sent:

Refactor the auth module to use JWT tokens.
Make sure to update the tests.
Don't change the public API.

Here's what the terminal received:

$ Refactor the auth module to use JWT tokens.
bash: Refactor: command not found
$ Make sure to update the tests.
bash: Make: command not found
$ Don't change the public API.
bash: Don't: command not found

paste-buffer doesn't paste text literally. It interprets each \n as if the user pressed the Enter key. Every line of your multi-line text executes as a separate shell command.

For an AI agent prompt, this is catastrophic. Fragments of your English-language instructions execute as shell commands. Most fail harmlessly with "command not found" — but imagine a prompt that includes rm -rf build/ as an example of what to clean up.

Bug 2: The global buffer race condition

tmux has a single default paste buffer. When you call load-buffer without specifying a name, it loads into this shared buffer. When you call paste-buffer without a name, it pastes from it.

Now imagine two concurrent sends to different sessions:

Thread A: tmux load-buffer /tmp/prompt-a.txt     # buffer = prompt A
Thread B: tmux load-buffer /tmp/prompt-b.txt     # buffer = prompt B (overwrites A!)
Thread A: tmux paste-buffer -t session-a          # pastes prompt B 😱
Thread B: tmux paste-buffer -t session-b          # pastes prompt B (again)

Session A gets Session B's prompt. Session B gets its own prompt. This is a classic TOCTOU race, and it's completely silent — no errors, no warnings. The wrong text just appears in the wrong terminal.

This bug is particularly insidious because it only triggers under concurrency. Manual testing never hits it. You only discover it when you're running 10+ agents and occasionally notice one executing the wrong task.

Why this is hard to debug

The failure mode is the intersection of three conditions:

  • Text must be long (>500 chars) — short text uses send-keys, which doesn't have either bug
  • Text must contain newlines — single-line long text pastes fine
  • Sends must overlap — sequential sends to the same or different sessions work correctly

During development, you test with short prompts. They work. You test with long single-line prompts. They work. You test sequential sends. They work. You deploy, run 20 agents, and get intermittent garbled prompts that you can't reproduce on demand.

The fix: named buffers + the -p flag

Two tmux features solve both bugs completely:

Named buffers eliminate the race condition

# Each send gets a unique buffer name
BUF="send-$$-$(date +%s%N)"

tmux load-buffer -b "$BUF" /tmp/prompt.txt
tmux paste-buffer -b "$BUF" -t mysession
tmux delete-buffer -b "$BUF"

The -b flag on both load-buffer and paste-buffer specifies a named buffer. Each concurrent send uses its own buffer, so they can't clobber each other. Clean up with delete-buffer to avoid leaking.

The -p flag pastes literally

tmux paste-buffer -p -b "$BUF" -t mysession

The -p flag tells tmux to paste the buffer contents without interpreting newlines as Enter keypresses. The text appears in the terminal exactly as-is — newlines and all — as if you'd pasted from your system clipboard.

What the man page says

Here's the full documentation for -p from man tmux:

paste-buffer [-dpr] [-s separator] [-b buffer-name] [-t target-pane]
    ...
    If -p is specified, paste bracket control codes are inserted around
    the buffer if the application has requested bracketed paste mode.

That's it. One line. And the description is about bracketed paste mode, which sounds like a terminal protocol detail you'd skip over. What it doesn't say clearly: -p prevents newlines from being interpreted as Enter keypresses. This is a side effect of bracketed paste — the terminal treats everything between the bracket codes as literal paste data, not as individual keystrokes.

Bracketed paste mode (\e[200~ ... \e[201~) was designed so terminals can distinguish typed input from pasted input. The practical consequence for automation: pasted text doesn't trigger line-by-line execution.

The complete solution

Here's the pattern we use in production (Python, but the tmux commands are the same in any language):

import subprocess, tempfile, time, os, threading

# One lock per session — serialize sends to the same pane
_send_locks = {}

def send_text(session_name, text):
    lock = _send_locks.setdefault(session_name, threading.Lock())
    with lock:
        target = f"amux-{session_name}"

        if len(text) <= 400:
            # Short text: send-keys -l is fine
            subprocess.run(
                ["tmux", "send-keys", "-t", target, "-l", text],
                check=True, timeout=10
            )
        else:
            # Long text: named buffer + paste-buffer -p
            buf = f"amux-{session_name}-{int(time.time() * 1000)}"
            with tempfile.NamedTemporaryFile(
                mode='w', delete=False, suffix='.txt'
            ) as f:
                f.write(text)
                tmp = f.name
            try:
                subprocess.run(
                    ["tmux", "load-buffer", "-b", buf, tmp],
                    check=True, timeout=10
                )
                subprocess.run(
                    ["tmux", "paste-buffer", "-p", "-b", buf, "-t", target],
                    check=True, timeout=10
                )
                subprocess.run(
                    ["tmux", "delete-buffer", "-b", buf],
                    timeout=5
                )
            finally:
                os.unlink(tmp)

        # Brief pause for readline to process, then send Enter
        time.sleep(0.15)
        subprocess.run(
            ["tmux", "send-keys", "-t", target, "Enter"],
            check=True, timeout=5
        )

Key details:

  • Per-session locks serialize sends to the same pane. Without this, two rapid sends to the same session can interleave their text. Named buffers prevent cross-session races, but same-session serialization needs a mutex.
  • Unique buffer names include session name + millisecond timestamp. Collisions are practically impossible.
  • Temp file cleanup is in a finally block so it runs even if tmux commands fail.
  • 0.15s sleep before Enter gives readline time to process all the pasted characters. Without this, Enter can arrive before the paste is fully processed, especially with long text.
  • Timeout on every subprocess call — tmux commands can hang if the server is in a bad state. 10s is generous; they normally complete in milliseconds.

Lessons for anyone automating tmux

  1. Never use the default paste buffer for programmatic sends. Always use named buffers (-b). The global buffer is fine for interactive use; for automation under concurrency, it's a race condition waiting to happen.
  2. Always use -p with paste-buffer unless you specifically want newlines interpreted as Enter. The flag name is misleading — it's not about "paste mode," it's about "paste literally."
  3. Use send-keys -l for short text, paste-buffer -p for long text. send-keys is simpler and doesn't need buffer management, but hits the ~500 char limit.
  4. Test with concurrency from day one. Both bugs are invisible under sequential testing. Spin up 10 threads sending simultaneously before you ship.
  5. Always clean up named buffers. tmux delete-buffer -b name after every paste. tmux doesn't garbage-collect them, and they accumulate in memory.

tmux is one of the most reliable pieces of software I've used. But its paste buffer system was designed for interactive use by one human, not programmatic use by concurrent threads. Once you know the two flags (-b and -p), the fix is trivial. Finding them is the hard part.
