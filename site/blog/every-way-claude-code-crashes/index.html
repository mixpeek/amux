Every way Claude Code crashes (and what we learned fixing them)

After running 30+ agents continuously for months, we found that Claude Code sessions don’t die randomly. They fail in specific, deterministic patterns — and every one is recoverable if you catch it in time.

We built amux to run dozens of Claude Code agents in parallel. The self-healing system checks every session every 60 seconds, parsing raw tmux output to detect state. Over months of running agents 24/7 across real codebases, we catalogued every failure mode we hit. Here’s the full list, what causes each one, and what actually fixes it.

1. The thinking block corruption

This is the worst one. The session emits redacted_thinking … cannot be modified in the terminal and becomes completely unrecoverable. No amount of retrying or continuing will fix it — the conversation state is corrupted.

When it happens: Extended thinking blocks + context compaction. When Claude Code compacts a conversation that contains extended thinking, the serialization can corrupt the thinking block references. It’s deterministic — once the compacted state is written, every subsequent API call fails with the same error.

The fix: There’s no recovery within the same conversation. You have to start a new conversation entirely. But you don’t want to lose the user’s intent, so we extract the last meaningful message (>20 characters) from the JSONL history and replay it into the fresh session. Short messages like “ok” or “y” are skipped — replaying an acknowledgment would just repeat the failure cycle.

We rate-limit this to one restart per 120 seconds to prevent cascading restart storms.

2. Context fills up silently

Claude Code shows a context percentage in its UI, but it doesn’t act on it. When context hits 0%, the session either crashes or produces degraded output. Most users don’t notice until it’s too late because the percentage is only visible in the active terminal.

When it happens: Long-running tasks that generate lots of tool output — editing large files, running tests with verbose output, reading many files. A single test suite run can consume 20% of context.

The fix: Send /compact proactively at 20% remaining. But there’s a subtlety: we learned to back up the JSONL at 30%, before compaction triggers. Compaction itself takes time, and if context fills up during compaction, you lose the pre-compact state. The backup at 30% guarantees you always have a recovery point.

We debounce compaction to once per 5 minutes. More frequent compaction thrashes disk I/O and produces worse summaries because there’s less new content to compress.

3. The post-compaction stall

This one surprised us. After /compact finishes, the session returns to the idle prompt (❯) and just … sits there. It doesn’t continue working. The agent forgets what it was doing.

Why it happens: Compaction rewrites the conversation history. The compacted summary doesn’t preserve the “I was in the middle of X” working state. Claude sees a clean prompt and waits for instruction.

The fix: After detecting that compaction completed (idle state + compaction sent in the last 30–90 seconds), we send a continuation message: “Continue with whatever you were doing.” This nudge is enough for the compacted summary to trigger recall of the prior task.

The 30-second minimum wait is important — sending the continuation before compaction finishes interrupts the process.

4. Safety prompts block unattended sessions

Claude Code asks for confirmation before running commands, writing files, and other potentially destructive actions. In unattended mode, these prompts halt the agent indefinitely.

The non-obvious part: Not all prompts are safety prompts. Claude also asks genuine questions — “which approach do you prefer?” or “should I use library X or Y?” Auto-answering those with “yes” produces garbage.

What we found: Claude Code’s safety prompts always include the text esc to cancel in the UI chrome. Model-level questions never do. This single substring is the reliable signal that distinguishes “I need permission to run rm -rf” from “I’m asking you a real question.”

We match against specific patterns — command contains command substitution, do you want to proceed, wants to execute — and only auto-respond when the escape-to-cancel chrome is present.

5. The dingbat signal

Detecting whether Claude Code is actively working or finished is harder than it looks. There’s no API for session state. We had to reverse-engineer the terminal output.

What we found: Claude Code uses Unicode dingbat characters (U+2700–U+27BF) as spinners. The presence or absence of an ellipsis (U+2026, “…”) after the spinner determines the state:

  • ✻ Thinking… (1m 24s) — active, task in progress
  • ✻ Completed for 4m 8s — idle, task finished

That single ellipsis character is the difference between “working” and “done.” We scan only the last 12 lines of output (not the full scrollback, which can exceed 5000 lines) to keep detection fast.

This is fragile by nature — it could break with any Claude Code update — but after months of running, the pattern has been stable across multiple CLI versions.

6. Shell exit after crash

Sometimes Claude Code exits entirely and drops to a bare shell ($ or % prompt). This usually happens after an unhandled error, an OOM, or a Node.js crash in the CLI itself.

How we detect it: If we see a $ or % at the end of a line but not the Claude Code prompt character (❯, U+276F), the CLI has exited. This distinction matters — some users have $ in their shell prompt alongside ❯, and false-positive restarts are worse than missed ones.

The fix: Restart Claude Code, but only if it was alive within the last 10 minutes. If a session has been dead for longer than that, it’s probably intentionally stopped. Rate-limited to one restart per 90 seconds.

What we learned

The big takeaway: Claude Code sessions don’t fail randomly. There are about 6 distinct failure modes, each with a specific trigger and a specific recovery. Once you handle all of them, sessions can run unattended for 12+ hours reliably.

The recovery strategies form a hierarchy:

  1. Proactive — back up at 30%, compact at 20%
  2. Nudge — continue after compaction, auto-respond to safety prompts
  3. Restart within conversation — shell exits, stuck states
  4. Nuclear — new conversation + replay intent (thinking block corruption)

Each layer has its own debounce timer to prevent cascading failures. The timers aren’t arbitrary — they came from months of watching agents fail at 3am and tuning the thresholds.

If you’re running Claude Code agents at scale, the most important thing you can do is instrument your sessions. Parse the output, detect the state, and react programmatically. The failure modes are finite and the fixes are mechanical.

amux handles all of this automatically

Open-source self-healing multiplexer for Claude Code. Monitors every session every 60 seconds, handles all six failure modes, runs on your machine. Python 3 + tmux.

git clone https://github.com/mixpeek/amux && cd amux && ./install.sh
amux register myproject --dir ~/Dev/myproject --yolo
amux start myproject
amux serve   # → https://localhost:8822
