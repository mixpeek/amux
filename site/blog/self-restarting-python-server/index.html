os.execv: The One-Syscall Way to Hot-Reload a Python Server

Replace the running process with a new version of itself. Same PID, same port, new code.

amux is a single Python file — about 25,000 lines of server, dashboard HTML/CSS/JS, and agent orchestration logic. During development, this file changes constantly. Sometimes by me, sometimes by the AI agents that amux is orchestrating (they edit the file, the server picks up the changes, and the agents see the new behavior immediately). The server needs to reload without manual intervention, without a process manager, and without losing the port binding for more than a fraction of a second.

The solution is one syscall: os.execv.

Why not importlib.reload?

importlib.reload(module) re-executes a module's source code and updates the module object in-place. This works for simple cases but falls apart for a server:

  • Module-level state isn't reset. Global variables, class definitions, route registrations — reload executes the code again, but existing references to old objects remain. You get a Frankenstein process with some old state and some new state.
  • The HTTP server can't re-bind. The HTTPServer object, its socket, its thread pool — these are all live objects holding resources. Reloading the module doesn't restart them.
  • Threads keep running. Background threads (file watchers, SSE push, session monitors) reference old function objects. You'd need to track and kill every thread, then restart them with new function references.

For a simple library with stateless functions, reload is fine. For a long-running server, it creates more problems than it solves.

Why not subprocess?

The classic approach: start a child process with the new code, shut down the old one.

# Don't do this
subprocess.Popen([sys.executable, __file__])
sys.exit(0)

Problems:

  • Port conflict. The old process holds port 8822. The new process tries to bind it and gets EADDRINUSE. You need careful coordination — shut down the old server, wait for the OS to release the port, then start the new one. Or use SO_REUSEADDR/SO_REUSEPORT, which have their own edge cases.
  • Two processes. For a brief window, both are running. If the parent is managed by systemd/launchd, killing it might trigger a restart of the old code.
  • PID changes. Anything monitoring the process by PID (log aggregators, process managers, health checks) sees a new process, not a restart.

os.execv: replace the process image

os.execv(path, args) is a thin wrapper around the POSIX execv(2) syscall. It replaces the current process image with a new one. The old code is gone — completely, atomically. The new code starts from the top (if __name__ == '__main__'). Same PID, same parent, same environment.

os.execv(sys.executable, [sys.executable] + sys.argv)

That's it. One line. The Python interpreter is replaced by a fresh Python interpreter running the same script with the same arguments. If the script has changed on disk, the new version runs.

The complete pattern

Here's a minimal working example:

import os, sys, time, ast, threading
from pathlib import Path
from http.server import HTTPServer, BaseHTTPRequestHandler

class Handler(BaseHTTPRequestHandler):
    def do_GET(self):
        self.send_response(200)
        self.end_headers()
        self.wfile.write(b"Hello from the server\n")

def watch_and_restart(server):
    """Watch our own source file and restart on change."""
    script = Path(__file__).resolve()
    mtime = script.stat().st_mtime
    DEBOUNCE = 3  # seconds

    while True:
        time.sleep(1)
        new_mtime = script.stat().st_mtime
        if new_mtime == mtime:
            continue

        # File changed — debounce
        print(f"[restart] file changed, debouncing {DEBOUNCE}s...")
        while True:
            time.sleep(DEBOUNCE)
            settled = script.stat().st_mtime
            if settled == new_mtime:
                break
            new_mtime = settled

        # Validate syntax before restarting
        try:
            ast.parse(script.read_text())
        except SyntaxError as e:
            print(f"[restart] ABORTED — syntax error: {e}")
            mtime = new_mtime
            continue

        # Shut down the server
        threading.Thread(target=server.shutdown, daemon=True).start()
        time.sleep(0.5)
        try:
            server.socket.close()
        except Exception:
            pass

        # Replace ourselves
        os.execv(sys.executable, [sys.executable] + sys.argv)

if __name__ == "__main__":
    server = HTTPServer(("", 8822), Handler)
    watcher = threading.Thread(
        target=watch_and_restart, args=(server,), daemon=True
    )
    watcher.start()
    print("Serving on :8822")
    server.serve_forever()

That's 50 lines for a self-reloading HTTP server. Let's walk through the important parts.

The debounce problem

When an AI agent edits a file, the write isn't atomic from the filesystem's perspective. An editor might:

  1. Write to a temp file
  2. Rename temp file over the original (atomic on most filesystems)

But some tools do multiple writes in quick succession — format on save, lint fixes, multiple edits. Without debouncing, the watcher sees the first write, restarts, then sees the second write and restarts again.

The debounce logic: when mtime changes, wait 3 seconds. Check mtime again. If it changed during the wait, reset the timer. Keep waiting until the file is stable for a full 3 seconds. Only then proceed with the restart.

3 seconds is a sweet spot — long enough to absorb editor save-format-save cycles, short enough that you're not waiting around.

Syntax validation: don't restart into a broken file

This is critical when AI agents are editing your server:

try:
    ast.parse(script.read_text())
except SyntaxError as e:
    print(f"[restart] ABORTED — syntax error: {e}")
    mtime = new_mtime  # update mtime, wait for next change
    continue

ast.parse parses the Python source without executing it. If the file has a syntax error, we log it and keep running the old version. The watcher continues monitoring — when the file is fixed (next save), it'll pass validation and restart successfully.

Without this check, a partially-written file kills your server. With AI agents that sometimes produce incomplete Python mid-edit, this happens more often than you'd expect.

The socket cleanup dance

Before calling os.execv, you must release the listening socket. Otherwise the new process can't bind to the same port.

# 1. Shut down the server (stop accepting, finish in-flight)
t = threading.Thread(target=server.shutdown, daemon=True)
t.start()
t.join(timeout=3)  # don't wait forever

# 2. Close the socket explicitly
try:
    server.socket.close()
except Exception:
    pass
try:
    server.server_close()
except Exception:
    pass

# 3. Brief pause for the OS to release the socket
time.sleep(0.5)

# 4. Replace the process
os.execv(sys.executable, [sys.executable] + sys.argv)

The 3-second timeout on server.shutdown() is important. If a request handler is stuck (slow external API call, deadlock), shutdown() blocks forever waiting for it to finish. The timeout ensures we restart even if something is wedged.

The 0.5-second sleep after closing the socket is a pragmatic concession. Even with SO_REUSEADDR, there's a brief window where the OS hasn't fully released the port. Half a second is enough on every OS I've tested.

What survives execv

SurvivesDies
Process ID (PID)All Python interpreter state
Parent processAll variables, objects, classes
Working directoryAll threads
Environment variablesAll open connections
File descriptors (unless CLOEXEC)In-flight HTTP requests
Signal dispositionsTimers, async tasks

The key insight: environment variables survive. This means you can load config from a file at startup using os.environ.setdefault(), and those values persist across restarts. Process-level overrides (export VAR=val) also survive and take precedence, since setdefault doesn't overwrite existing keys.

# Load config file at startup — values persist across execv restarts
for line in Path("~/.myapp/server.env").expanduser().read_text().splitlines():
    if "=" in line and not line.startswith("#"):
        k, v = line.split("=", 1)
        os.environ.setdefault(k.strip(), v.strip())

The consequence: dropped connections

Every in-flight HTTP request dies. Every SSE (Server-Sent Events) connection drops. Every WebSocket disconnects. There is no graceful handoff.

For amux, this is acceptable because:

  • The dashboard uses SSE for real-time updates. The browser's EventSource auto-reconnects within seconds.
  • API requests from agents retry on failure.
  • The tmux sessions (the actual agents) are completely independent of the Python server — they keep running regardless.

If you need zero-downtime reloads with connection handoff, you want SO_REUSEPORT + a load balancer, or the fork-then-exec pattern with file descriptor passing. Those are complex. os.execv is simple, with a brief (sub-second) interruption.

Gotchas

  • File descriptors leak across execv unless you set FD_CLOEXEC. Python 3 sets this by default on most file opens, but inherited sockets from socket.accept() might not have it. If you see "address already in use" after restart, a leaked FD is holding the port.
  • sys.argv must be correct. os.execv takes the full argument list. If you modify sys.argv at runtime, the restarted process gets the modified version. Use a snapshot: _original_argv = sys.argv[:] at startup.
  • Virtual environments. sys.executable points to the venv's Python, which is correct. But if you deactivate/change the venv between restarts, the old sys.executable path might be stale.
  • Don't execv in a signal handler. Signal handlers run asynchronously and can interrupt critical sections. Execv from a signal handler can leave locks held, files half-written, etc. Use a flag that the main loop checks.

When to use this pattern

This works well for:

  • Development servers that need instant reload
  • Single-process services where brief downtime is acceptable
  • Tools that AI agents edit while running (the server picks up agent changes without human intervention)
  • Long-running scripts that need to update their own logic

It doesn't work for:

  • Production services with zero-downtime requirements
  • Multi-process worker pools (use gunicorn's HUP signal instead)
  • Windows (no execv — use subprocess + exit instead)

For amux, it's been running this pattern in production for months. The server restarts in under a second, the dashboard reconnects automatically, and the AI agents never notice. One syscall, no process manager, no orchestration.
