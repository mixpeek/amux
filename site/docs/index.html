Installation

Prerequisites

  • Python 3.8+
  • tmux — brew install tmux (macOS) or apt install tmux (Linux)
  • Claude Code CLI — installed and authenticated (claude command available)

Install

git clone https://github.com/mixpeek/amux && cd amux
./install.sh

This symlinks amux and amux-server.py to /usr/local/bin. Also installs a cc alias for convenience.

Verify the installation:

amux --version

Updating

cd ~/path/to/amux && git pull && ./install.sh

The server auto-restarts when amux-server.py changes, so updates take effect immediately if the dashboard is running.

Quick start

# 1. Register a session (points to your project directory)
amux register myproject --dir ~/Dev/myproject --yolo

# 2. Start the session (launches Claude Code in tmux)
amux start myproject

# 3. Start the web dashboard
amux serve  # → https://localhost:8822

# 4. Send a task to your agent
amux send myproject "add unit tests for the auth module"

# 5. Check what the agent is doing
amux peek myproject

That's it. Open https://localhost:8822 in your browser to see the dashboard with live status, token tracking, and a send bar.

Core concepts

Sessions

A session is a Claude Code instance running inside a tmux pane. Each session has a name, a working directory, and optional flags (model, permissions, system prompt). Sessions persist across restarts — amux auto-resumes them when the server starts.

Board

The board is a kanban-style task tracker backed by SQLite. Agents and humans post tasks, claim them atomically, and move them through statuses (todo → doing → done). Agents use the board to coordinate work without stepping on each other.

Dashboard

The web dashboard is a PWA that shows all sessions, the board, notes, CRM, email, files, and more. It works on desktop and mobile, installs as a home-screen app on iOS/Android, and supports offline queuing via Background Sync.

REST API

Every dashboard feature is accessible via a REST API at https://localhost:8822/api/. Agents use this API to communicate with each other, read/write board tasks, access notes, and more.

Web dashboard

Start the dashboard with amux serve. It runs at https://localhost:8822 by default.

The dashboard provides:

  • Session cards — live status, last output preview, token usage, send bar
  • Kanban board — drag-and-drop task management with custom statuses
  • Notes — rich markdown editor (Quill) for documents and write-ups
  • CRM — contacts with interaction logs and follow-up tracking
  • Email — send and read email via Mail.app integration
  • Files — file explorer with syntax highlighting and markdown editing
  • Logs — filterable request log with category badges (session, board, notes, etc.)
  • Scheduler — cron job management UI

The dashboard is a single HTML page served from amux-server.py. It uses Server-Sent Events (SSE) for live updates — sessions, board changes, and notifications push to all connected clients within ~2 seconds.

Kanban board

Creating items

# CLI
amux board add "Refactor auth module"

# API
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"title":"Refactor auth module","status":"todo"}' \
  https://localhost:8822/api/board

Moving items

# CLI shortcuts
amux board doing ITEM-ID

# API
curl -sk -X PATCH -H 'Content-Type: application/json' \
  -d '{"status":"done"}' \
  https://localhost:8822/api/board/ITEM-ID

Atomic task claiming

When multiple agents need to claim tasks without duplication, use the claim endpoint:

curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"session":"worker-1"}' \
  https://localhost:8822/api/board/ITEM-ID/claim

This uses SQLite CAS (compare-and-swap) to guarantee only one agent claims each task. If another agent already claimed it, the request returns a conflict error.

Custom statuses

Add custom status columns beyond the defaults (todo, doing, done):

curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"name":"review","color":"#f59e0b"}' \
  https://localhost:8822/api/board/statuses

Clearing done items

curl -sk -X POST https://localhost:8822/api/board/clear-done

Importing from GitHub

amux board import gh:owner/repo#123

This fetches the GitHub issue, parses its body for actionable tasks, and creates board items for each one.

Notes

Notes are markdown documents that any session can read and write. Use them for specs, plans, reports, research — anything meant to be read rather than done.

# Create or update a note
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"content":"# Migration Plan\n\nStep 1: ..."}' \
  https://localhost:8822/api/notes/migration-plan

# Read a note
curl -sk https://localhost:8822/api/notes/migration-plan

# List all notes
curl -sk https://localhost:8822/api/notes

# Delete a note (moves to trash)
curl -sk -X DELETE https://localhost:8822/api/notes/migration-plan

The dashboard has a full rich-text editor (Quill) for creating and editing notes visually. Notes can also be pinned for quick access.

CRM

Built-in contact management with interaction logging and follow-up tracking.

# Add a contact (CLI)
amux crm add "Jane Smith" company=Acme email=jane@acme.com role=CTO

# Add via API
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"name":"Jane Smith","company":"Acme","email":"jane@acme.com","role":"CTO"}' \
  https://localhost:8822/api/crm/contacts

# Log an interaction
amux crm log PPL-1 "Discussed partnership. Follow up in 2 weeks."

# Check pending follow-ups
amux crm fu

# Update a contact
amux crm update PPL-1 phone=555-0123 linkedin=linkedin.com/in/janesmith

Contact fields: name, company, role, email, phone, linkedin, twitter, notes. Interaction types: call, email, meeting, dm, event, other.

Email

Send and read email through the macOS Mail.app integration.

# Send an email
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"to":"alice@example.com","subject":"Hello","body":"Hi Alice, ..."}' \
  https://localhost:8822/api/email/send

# Check inbox
curl -sk https://localhost:8822/api/email/inbox

# Search emails
curl -sk "https://localhost:8822/api/email/search?q=invoice"

Email syncs automatically on a recurring interval. You can trigger a manual sync with POST /api/email/sync.

Browser automation

A shared Playwright browser instance accessible to all sessions. Supports saved auth profiles so you don't need to log in every time.

# Start browser with default auth profile
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"profile":"default","url":"https://example.com"}' \
  https://localhost:8822/api/browser/start

# Take a screenshot
curl -sk https://localhost:8822/api/browser/screenshot
# Returns {"path": "~/.amux/browser-screenshots/latest.jpg"}

# Click at coordinates
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"action":"click","x":640,"y":400}' \
  https://localhost:8822/api/browser/action

# Type text
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"action":"type","text":"hello world"}' \
  https://localhost:8822/api/browser/action

# Run JavaScript
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"action":"eval","script":"document.title"}' \
  https://localhost:8822/api/browser/action

# AI agent mode — describe a task in plain English
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"task":"Find the latest invoice and download it","profile":"default"}' \
  https://localhost:8822/api/browser/agent

# Stop browser
curl -sk -X POST https://localhost:8822/api/browser/stop

Auth profiles

Auth profiles save cookies and local storage so the browser starts already logged in. Profiles are stored in ~/.amux/playwright-auth/.

# List available profiles
curl -sk https://localhost:8822/api/browser/profiles

Scheduler

Run recurring tasks on cron schedules. The scheduler is built into the server — no external cron daemon needed.

# Schedule a nightly security audit at midnight
amux cron add "0 0 * * *" "Run OWASP top-10 scan on src/"

# Schedule a weekly dependency check
amux cron add "0 9 * * MON" "Check for outdated dependencies and create update PR"

The dashboard includes a scheduler management UI where you can view, edit, pause, and delete scheduled jobs.

Skills & slash commands

Skills are project-level custom commands that any agent can invoke. They live in the skills/ directory and appear as slash commands in the dashboard.

# List available skills
curl -sk https://localhost:8822/api/skills

# Run a skill
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"args":"--fix"}' \
  https://localhost:8822/api/skills/lint

Skills can contain any logic — shell scripts, Python code, or complex multi-step workflows. They're the primary way to teach agents project-specific operations like “deploy to staging” or “run the test suite.”

Channels

1:1 inter-session messaging with @mentions. Sessions can send messages to each other for real-time coordination.

# Send a message from one session to another
curl -sk -X POST -H 'Content-Type: application/json' \
  -d '{"text":"Auth module refactor is done, you can start on the API tests now"}' \
  https://localhost:8822/api/sessions/worker-2/send

# Peek at another session's recent output
curl -sk "https://localhost:8822/api/sessions/worker-1/peek?lines=50"

Sessions discover each other via GET /api/sessions and coordinate through a combination of messages and board tasks.

File explorer

The dashboard includes a file explorer with:

  • Directory browsing with syntax highlighting for code files
  • Markdown editor with find-in-page
  • File upload via drag-and-drop
  • File creation, renaming, and deletion
# List files in a directory
curl -sk "https://localhost:8822/api/ls?dir=/Users/you/Dev/myproject"

# Read a file
curl -sk "https://localhost:8822/api/file?path=/Users/you/Dev/myproject/README.md"

# Upload a file
curl -sk -X POST -F "file=@local.txt" https://localhost:8822/api/fs/upload

Self-healing

amux monitors sessions and automatically handles common failure modes:

  • Context overflow — detects when Claude Code hits its context limit and triggers /compact to compress the conversation
  • Crash recovery — restarts crashed sessions and replays the last message
  • Stale detection — identifies stuck sessions and prompts them to continue
  • Auto-resume — when the server starts, it re-attaches to any tmux sessions that survived a restart

The watchdog runs every 3 seconds and requires no configuration. It's on by default for all sessions registered with --yolo.

Mobile & PWA

The dashboard is a Progressive Web App. Install it on your phone:

  1. Open https://localhost:8822 (or your Tailscale/LAN IP) in Safari or Chrome
  2. Tap “Add to Home Screen”
  3. The app works offline — queued actions (send message, create task) sync when connectivity returns via Background Sync

There's also a native iOS app on the App Store.

Authentication

By default, the dashboard has no authentication (all endpoints are public). To enable auth:

# Generate a token
python3 -c "import secrets; print(secrets.token_urlsafe(32))" > ~/.amux/auth_token

# The server reads this file on startup
# All API requests now require: Authorization: Bearer <token>
# Or the ?token=<token> query parameter

When auth is enabled, the dashboard prompts for the token on first visit and stores it in local storage.

TLS & HTTPS

amux generates a self-signed TLS certificate on first run. This enables:

  • Service worker & offline mode (requires HTTPS)
  • Secure WebSocket connections
  • iOS PWA installation

If you have Tailscale installed, amux automatically uses your Tailscale certificate for trusted HTTPS — no certificate warnings on any device on your tailnet.

To disable TLS: amux serve --no-tls

Configuration

Session config

Session configs are stored in ~/.amux/sessions/<name>.env. Each file contains the flags passed at registration.

Server env

Persistent server configuration lives in ~/.amux/server.env. Loaded at startup via os.environ.setdefault.

# Example ~/.amux/server.env
AMUX_S3_BUCKET=ethan-personal
AMUX_S3_KEY=amux/calendar.ics
AMUX_S3_REGION=us-east-2

After editing server.env, touch amux-server.py to trigger a reload.

Global defaults

Set default flags that apply to all new sessions:

amux defaults edit
# Opens ~/.amux/defaults.env in your editor
# Example: CC_MODEL=sonnet CC_YOLO=1

MCP config

Configure MCP servers for all sessions in mcp.json at the project root, or per-session with --mcp-config.

Architecture

amux is a single Python file (amux-server.py) with no external dependencies beyond Python 3 and tmux. The same file runs locally and in the cloud.

  • Server — Python HTTP server with threading, serves the dashboard + API
  • Dashboard — inline HTML/CSS/JS, no build step, no bundler
  • Storage — SQLite for board, CRM, logs, scheduler. Flat files for session configs and notes.
  • Sessions — tmux panes running Claude Code CLI. Managed via tmux send-keys and tmux capture-pane.
  • Live updates — Server-Sent Events (SSE) with a 10-second ping heartbeat. Clients reconnect after 18 seconds of silence.
  • Auto-restart — the server watches its own file and calls os.execv on change

Data directory

All persistent data lives in ~/.amux/:

~/.amux/
├── sessions/          # Session .env configs
├── board/             # SQLite database
├── notes/             # Markdown note files
├── crm/               # CRM data
├── logs/              # Request logs
├── tls/               # TLS certificates
├── playwright-auth/   # Browser auth profiles
├── server.env         # Server configuration
├── defaults.env       # Global session defaults
└── auth_token         # Auth token (if enabled)
