Mobile PWA

Install amux on your phone. Manage agents from anywhere — even offline.

amux’s dashboard is a full Progressive Web App. Install it on iOS or Android for native-like access to your agent fleet. Every mutation you make while offline — starting sessions, moving board items, sending commands — queues locally and replays automatically when you reconnect.

How offline sync works

Most PWAs just cache static pages. amux goes further: it queues mutations and replays them in order, with smart conflict resolution, so you can manage agents from your phone on a subway or a plane and have everything sync when you surface.

1. The offline queue

Every API call goes through a single wrapper function: apiCall(). When the network is down, mutations (POST, PATCH, DELETE) are pushed onto an offline queue stored in both localStorage and IndexedDB. Read-only GET requests return cached data instead.

Each queued operation records:

  • The HTTP method and URL
  • The request body
  • A timestamp
  • The operation type (e.g. start-session, board-move, send-command)

Dual storage (localStorage + IndexedDB) means the queue survives both page refreshes and service worker restarts.

2. Queue reconciliation

Before replaying, amux runs reconcileQueue() to remove contradictory operations. If you started a session then stopped it while offline, both operations are dropped — the net effect is zero, and replaying them in sequence would waste API calls and potentially leave the session in the wrong state.

The reconciler walks the queue and removes inverse pairs:

  • start + stop on the same session → skip both
  • Multiple board moves on the same item → keep only the last

This keeps replay fast and idempotent.

3. Background Sync

When you queue an operation, amux registers a sync event with the browser’s Background Sync API. This means the browser will fire a sync event when connectivity returns — even if the PWA tab is closed.

The service worker’s sync handler reads the queue from IndexedDB (service workers can’t access localStorage), replays each operation in order, and clears the queue on success.

4. The sync banner

When the app comes back online (or you open the PWA after being offline), a progress banner appears at the top of the screen showing each queued operation as it replays:

  • Operation name and target (e.g. “Starting session worker-3”)
  • Progress count (e.g. “2 / 5”)
  • Success/failure status for each item

Drafts (sessions you created while offline) sync first, since other operations may depend on them existing.

5. Auto-retry on visibility change

As a fallback for browsers that don’t support Background Sync, amux also listens for visibilitychange events. When you switch back to the amux tab, it checks for queued operations and replays them immediately. This covers Safari on iOS, which has limited Background Sync support.

6. Service worker caching

The service worker uses a two-strategy cache:

  • Cache-first for static assets (CSS, JS, images, fonts) — instant loads, updated in the background
  • Network-first for HTML pages — always gets fresh content, falls back to cache when offline
  • Skip for API calls — the app-level apiCall() wrapper handles offline queueing, so the service worker stays out of the way

Offline drafts

You can create new sessions while offline. These are saved as “drafts” in IndexedDB with a local UUID. When connectivity returns, drafts are synced first (before the rest of the queue), converting them into real registered sessions. Any commands you queued for a draft session are replayed against the newly-created real session.

Install

iOS: Open amux in Safari → tap Share → Add to Home Screen.

Android: Open amux in Chrome → tap the install banner or Menu → Install app.

The PWA uses HTTPS with a service worker, so your browser will prompt for installation automatically. For certificate setup on your phone, see the getting started guide.

Get started with amux

Run dozens of Claude Code agents in parallel. Python 3 + tmux. Open source.

git clone https://github.com/mixpeek/amux && cd amux && ./install.sh
amux register myproject --dir ~/Dev/myproject --yolo
amux start myproject
amux serve  # → https://localhost:8822
